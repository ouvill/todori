# ADR-023: Tenantを暗号・membership・同期境界とし、Listを通常Recordにする

**日付**: 2026-07-24
**状態**: Accepted（2026-07-24 プロダクトオーナー承認）

### 背景

従来はListごとにDEK、key generation、key bundle、upload APIを持ち、Listが配置と暗号scopeを兼ねていた。このため同じ共有範囲内のList間Task移動にもkey選択と再暗号化が必要で、List削除がTask削除と鍵retirementへ結合していた。

別のSpace entityを追加する案も検討したが、既存Tenantがすでにaccount membership、同期route、cursor、RLS、課金認可の境界を持つ。過去にrevertした全面E2EE再設計は復元せず、mainの現行設計からTenant境界だけを切り替える。

### 決定

1. AccountとTenantは`TenantMembership`による多対多とする。Account登録時にPersonal Tenantを1つ作り、登録Accountだけをactive ownerにする。Personal Tenantは複数Listを持てる。
2. Tenantを暗号・membership・同期・local SQLCipher DBの境界とする。List、Task、Tag、Template、Schedule、Timer Session、将来のComment等はTenant内の暗号化Recordであり、List IDは配置情報であってkey scopeではない。
3. Tenant Root DEKを唯一のcontent key generationとする。Record KeyはHKDF-SHA256でTenant Root DEKからtenant ID、generation、collection、record IDへdomain separationして32 bytes導出し、保存・配送しない。List DEK、`list_key_generations`、`list_key_bundles`、local List key cache、pending List key queue、List key APIを廃止する。
4. wire envelopeはv5（`TDE5` / `TDA5`）、Tenant専用key manifestは`TKM2`、sync protocolはv7、local schemaはv21とする。`TKM2`からscope discriminatorとList IDを除き、tenant ID、suite、generation、status、minimum write generation、previous manifest hash、recipient集合だけを認証する。一般配布前のbreaking baselineとして旧reader / writer、dual read / write、List key fallbackを残さない。
5. 同一Tenant内のTaskのList間移動はplacement変更とする。record IDとTenant generationが変わらないためRecord Keyは同じであり、新nonceで更新plaintextを暗号化する。Tenant間移動は現行releaseで公開しない。将来提供する場合は、移動先Tenantで新record IDを作るcopyと移動元deleteとして扱う。
6. List削除はTaskへcascadeしない。同じlocal transactionで既知Taskをdefault Inboxへrehomeしてlive outboxを作り、List tombstoneを作る。remote List tombstoneや削除後に到着したTaskも、復号したclientがdefault Inboxへrehomeしてlive Recordとして再pushする。TaskのReminder、Timer Session、Undo履歴は保持する。
7. domainにはPersonal / Shared / EnterpriseのTenant kindとmembership型を置くが、現行releaseで有効なのはPersonal Tenantの単一active ownerだけとする。Shared / Enterprise Tenantの作成、招待、承諾、離脱、除名、ownership transfer、role別read/write認可、member端末へのTenant Root DEK配送は後続のcritical設計判断までfail closedにする。server同期routeもPersonal owner以外を拒否する。
8. Personal Tenantの課金は既存Account entitlementを維持する。Enterprise Tenantでmembershipを課金seatとして扱うか、Tenant subscriptionを課金主体にするかは、共有membershipと認可の設計時に同時に裁定し、本変更では有効化しない。
9. Tenantごとのlocal DB catalogと複数Tenant切替UIはShared / Enterprise公開時の後続実装とする。現行Personal profileは1つのTenant-bound SQLCipher DBを開き、別Tenantのkey、record、cursorを同じprofileへ混在させない。

### 帰結

- List追加・削除・同一Tenant内移動からkey lifecycleがなくなり、Personal Tenant内の複数Listを同じ暗号・同期scopeで扱える。
- serverは引き続きList linkageを観測せず、Tenant ID、collection、record ID、generation、opaque blobだけを扱う。
- List削除後もTask identityと履歴が維持される。default Inboxは削除不可であり、rehome先がない状態はcorruptionとしてfail closedになる。
- Shared / EnterpriseのUIやAPIが存在するだけで共有を安全とみなさず、membership state machineとwrite authorizationが承認されるまで利用不能にする。

### 却下した代替案

- **Listを暗号・共有境界のまま維持する案**: placementとkey lifecycleの結合、List間移動時の不要なkey切替を残すため不採用。
- **新しいSpace entityをTenantと並立させる案**: membership、RLS、課金、sync routingの境界を二重化するため不採用。
- **Tenant Root DEKで全Recordを直接暗号化する案**: 同一鍵をそのまま全AEADへ使うため不採用。Record identityごとの導出鍵で分離する。
- **List削除時にTaskをtombstone化する案**: 配置entityの削除で独立Recordと履歴を失うため不採用。
- **key possessionだけでShared writeを許可する案**: 離脱、role変更、stale device、server-side authorizationを説明できないため不採用。
