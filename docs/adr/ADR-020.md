# ADR-020: 暗号suiteと鍵世代を明示し、Organization共有は認証済みhybrid PQC端末配送とする

**日付**: 2026-07-15
**状態**: Accepted（2026-07-15 プロダクトオーナー承認）

### 背景

現行実装はXChaCha20-Poly1305、HKDF-SHA256、SQLCipherを用いるが、OPAQUEはRFC 9807以前、Recovery Keyは独自形式、record envelopeとkey bundleは世代を持たず、server保存のdevice公開鍵は対応する秘密鍵の継続保有と真正性を保証しない。これでは、鍵漏洩・端末除名・algorithm移行時に新規writeを旧鍵から切り離し、live dataを安全に移行・retireする契約がない。

一般配布前で既存開発account、wire、DBを再作成できるため、互換reader / writerや無期限fallbackを追加せず最終形式へ直接移行する。Organization共有ではserver侵害と公開鍵差し替えを脅威に含め、長期保存される暗号文のharvest-now-decrypt-laterリスクも扱う。

脅威とするserverは、保存する公開鍵、certificate、manifest、recipient、暗号blob、generationを改変・追加・削除・replayし、任意の応答を拒否できる。clientは改変・追加・replayとdowngradeを暗号学的に拒否するが、削除・応答拒否によるavailability低下とtraffic量・時刻・ID等の既定metadata露出は防げない。侵害済みendpoint、画面上のplaintext、memory読取、除名以前に端末が取得済みの鍵・plaintextは保護対象外であり遠隔消去できない。hybrid PQCはOrganization DEK配送とaccount / device真正性の将来耐性を対象とし、今回対象外のPQC TLSに依存しないためtransport metadataとavailabilityは従来のTLS脅威境界に残る。

### 決定

1. 暗号形式は数値`version`とunsigned 16-bit big-endian `suite_id`で識別する。production `suite_id`は`0x0002`で、RFC 9807 OPAQUEのRistretto255 + 3DH、Argon2id 64 MiB / t=3 / p=4、HKDF-SHA256、XChaCha20-Poly1305を表す。JSONでは整数`2`、Postgresでは`SMALLINT 2`、OPAQUE recordと同じ`users.opaque_suite_id`へ保存する。server setup、registration / login state、request / responseもIDへ束縛し、未知・旧suiteへのdowngradeをserverとclientの双方で拒否する。
2. Recovery KeyはBIP39 English 24語、256-bit entropy、checksum、NFKD正規化とする。表示phrase文字列を直接KDFへ入れず、検証後に復元した32-byte entropyをHKDF-SHA256の入力とし、`taskveil/recovery-key-wrap-key/v1`でMK wrap keyを導出する。
3. Taskveil key wrap v1 AADは63 bytes固定で、ASCII `TWK1`、purpose `u8`、suite ID `u16be`、generation `u64be`、user / tenant / list UUID各16-byte network orderの順とする。該当しないUUIDはall-zero、該当必須UUIDのall-zeroは拒否する。purposeは`1=mk-password`、`2=mk-recovery`、`3=mk-device`、`4=account-secret-mk`、`5=tenant-dek-mk`、`6=list-dek-mk`、`7=local-tenant-dek-mk`、`8=local-list-dek-mk`とする。purpose 4のpayloadはbaselineでは既存account X25519秘密鍵、Organization公開前にはhybrid account root秘密鍵へ置換し、同時に旧payloadを削除する。同一ciphertextを別用途・主体・scope・世代へ移植できず、pre-Taskveil wrap contractは読まない。Taskveil namespace v1は暗号suite、sync protocol、DB schema、capsule plaintext形式のversionとは独立して固定する。
4. serverは`user_key_generations`、`tenant_key_generations`、`list_key_generations`、`key_recipients`を正規化して保持する。`key_recipients`のscopeは`scope_kind u8 + tenant UUID + nullable list UUID`、主キーは`(scope_kind, tenant_id, list_id, generation, device_id)`、同generation内の`recipient_key_fingerprint`は一意とする。generationはscopeごとに単調増加し、suite、status、manifest authenticator、wrapper revision、recipient fingerprintへ束縛する。
5. record envelope v4 bytesはASCII `TDE4`、suite ID `u16be`、key generation `u64be`、24-byte XChaCha nonce、ciphertext + 16-byte tagの順とする。ciphertext込みのblob上限は64 KiBで、短すぎるblob、未知suite、generation 0をparse時に拒否する。AADはASCII `TDA4`、suite ID `u16be`、generation `u64be`、tenant UUID 16 bytes、collection UTF-8の`u16be length + bytes`、record UUID 16 bytesの順とする。serverはscopeのminimum write generation未満のlive pushを拒否し、clientは認証済みmanifestと一致しないsuite / generation / recipientを拒否する。tombstoneはcontent blobを持たないため再暗号化対象外とする。
6. rotation状態は`prepared -> active -> migrating -> retired`だけを許す。`prepared`は新鍵と全残存端末bundleを作るがwrite generationを変えない。`active`は1 transactionでactive generationとminimum write generationを更新する。`migrating`はlive headだけを新鍵で再暗号化する。`retired`は全live head移行、全非expired端末のcontinuity ACK、30日history保持終了後に旧bundleを削除する。
7. password変更とRecovery再発行はMKを変更せずwrapper revisionだけを進める。MK rotationはTenant / List DEKを新MKでrewrapしcontentを再暗号化しない。Tenant / List DEK rotationだけが対象recordを再暗号化する。rotationはmember / device除名、鍵侵害疑い、Recovery漏洩、algorithm移行で発火し、定期的な全data再暗号化は行わない。
8. offline端末が旧generationしか持たない場合、対象scopeの新規writeはfail closedにし、sync後に新generationを取得して再開する。旧鍵での新規暗号化、無期限fallback、generation 0互換を許可しない。
9. local capsule plaintextは`version=2`、DK generation `u64`、32-byte DK、optional DK-wrapped MK length + bytesを一体で保持する。未登録profileは`wrapped_mk=None`、登録済みprofileは`Some`だけを許す。OS secret storeはcapsule全体を保護して`active` / `pending` slotへ保存する。slotとsession tokenはcanonical local profile rootのSHA-256先頭128-bitをhex化した非秘密namespaceで分離し、別profileやintegration testからの上書き・削除・認証状態混線を防ぐ。同一profile配下のtenant DB群は同じnamespaceとDKを共有する。DK rotationはpending保存、SQLCipher `PRAGMA rekey`、新DKでの再open検証、active昇格、旧capsule削除の順とする。起動時にpendingがある場合だけ双方を試してrollbackまたはcommitを完了し、通常互換fallbackには使わない。
10. Android productionはKeystoreのnon-exportable AES-256-GCM鍵でcapsuleをsealし、平文`device.key`はdevelopment / test以外で起動失敗とする。Apple platformはData Protection Keychainを維持し、実機でDK rotation、再起動、DB reopen、ゼロプロンプトをrelease条件にする。
11. Organization共有のaccount rootはEd25519 + ML-DSA-65 hybrid identity key、deviceは署名鍵とX25519 + ML-KEM-768 hybrid KEM keyを持つ。account root秘密鍵はMKでwrapし、device秘密鍵は端末secret store外へ出さない。account rootはdevice certificateへ署名し、登録にはserver challengeへのproof-of-possessionを必須とする。
12. Organization招待は両account root fingerprint（hybrid公開鍵payloadのSHA-384）をlexicographic byte順に並べ、ASCII `taskveil-safety-number-v1`と連結してSHA-384した48 bytesを正本とする。人間表示は先頭30 bytesを10進60桁へ変換して5桁ごとにgroup化し、QRはversion byte `1` + 48-byte digestを持つ。別経路で一致確認するまで`unverified`としTenant / List DEKを配送しない。root変更時は再確認する。server提供のunsigned、revoked、unknown-suite device keyをclientが拒否する。key transparency logは今回導入しない。
13. Tenant / List DEKは残存する各deviceへ個別配送する。hybrid transcriptはASCII `taskveil-hybrid-dek-wrap-v1`に、各fieldを`u32be length + bytes`でsender / recipient certificate fingerprint、sender / recipient X25519公開鍵、sender / recipient ML-KEM公開鍵、ML-KEM ciphertext、scope kind、scope UUID、generation `u64be`、recipient key fingerprintの順に連結する。certificate fingerprintはroot署名対象payloadのSHA-384 48 bytes、recipient key fingerprintはASCII `taskveil-recipient-key-fingerprint-v1`、suite、X25519公開鍵、ML-KEM-768公開鍵をSHA-256した32 bytesであり、同一値として扱わない。`salt=SHA-384(transcript)`、IKM=`X25519 shared secret || ML-KEM shared secret`、HKDF-SHA384 info=`taskveil/hybrid-dek-wrap-key/v1`で32-byte wrap keyを導出し、XChaCha20-Poly1305でDEKをwrapしてtranscript全体をAADとする。端末・member除名は新generationを発行し、除名対象をrecipient集合から除く。除名前に取得済みの旧dataを遠隔消去できるとは表現しない。
14. device certificate canonical payloadはASCII `TDC1`、suite `u16be`、account UUID、device UUID、valid-from / expiry `i64be`、signing public key、X25519 public key、ML-KEM-768 public keyを各固定長または`u32be length + bytes`で持つ。account rootのEd25519とML-DSA-65両signatureを固定順に付け、fingerprintはpayloadのSHA-384とする。server challenge proofはchallenge 32 bytes、device UUID、certificate fingerprintを同じdevice signing keyで署名する。
    device失効はASCII `TDR1`、suite、account / device UUID、certificate fingerprint、account単位の単調revision、発行時刻、previous statement SHA-256、root fingerprintへaccount rootの両signatureを付ける。serverはrevision / previous hash CASで保存し、対象accountが所属する全Organizationを同一transactionでrotation-requiredにする。clientは完全なrevocation hash chainと観測済みminimum revision / head hashをSQLCipherへpinして失効certificate replay、rollback、同revision forkを拒否する。key transparencyなしでは未観測の最新statementをserverが抑止するfreshness問題は残るため、availability境界として明示する。
15. key manifest canonical payloadはASCII `TKM1`、scope kind `u8`、tenant / list UUID、suite `u16be`、generation `u64be`、status `u8`、minimum write generation `u64be`、previous manifest SHA-256 32 bytes、recipient count `u32be`、fingerprint昇順の32-byte recipient fingerprint列とする。personal scopeはHKDF-SHA256(MK, `taskveil/personal-key-manifest-auth/v1`)によるHMAC-SHA256、Organization scopeはaccount rootのEd25519 + ML-DSA-65両署名で認証する。generation、previous hash、status遷移を検証してreplay / forkを拒否する。
    serverはopaque blob内のList IDを知らないため、record envelopeのminimum-write generationはtenant単位の同期epochとする。List単独rotationでもtenant epochを進め、非対象ListのDEKは同一鍵素材を新generationへrewrapし、対象Listのrecordだけを新DEKで再暗号化する。この同期epochは暗号scopeを統合するものではなく、serverにtask-to-list linkageを開示せずstale writeを拒否するためのtransport境界である。
16. FIPS 203 / 204 primitiveは`aws-lc-rs = 1.17.1`へ正確に固定し、test gateから対応する`aws-lc-sys = 0.42.0`のACVP由来known-answer self-testも実行する。iOS / Android / macOS cross-buildとbinary size、公式vectorを受け入れ条件とする。strict mode、key transparency log、PQC TLSは今回の範囲外とする。
17. 個人利用の外部配布はbaseline hardening、versioned rotation、crash-safe DK / production secret storeの完了後に許可する。Organization共有はdevice certificate、Safety number、hybrid PQC配送の完了後に許可する。一般リリース前に実装担当と独立した暗号reviewを必須とし、外部review完了前に「監査済み」と表示しない。
18. release automationはdirect crypto / platform dependencyのexact pin、`cargo audit --deny warnings`、tracked / untracked secret pattern scan、16 KiBのenvelope / manifest / certificate / identity parser fuzzを必須とする。fuzz smokeは長時間campaignや外部暗号reviewの代替にしない。platform実機、内部独立review、外部reviewは別々に記録し、未実施gateをcross-buildや自動testで合格扱いにしない。
    Safety number確認済みowner / member rootとdigest、両device roster revision / chain head hash、active generationはclientのSQLCipherへdurable pinする。server応答の`verified`だけをtrust anchorにせず、device roster再取得時のSafety rootもcallerが渡すlocal pinへ束縛する。root変更は再確認、roster / generation rollbackまたはforkはfail closedとする。Organization active bundleのproduct経路は両verified rosterとroot署名manifestのrecipient集合完全一致、現在device membershipを必ず検証し、roster chainが進んだ場合はprevious active generationより新しいgenerationまで配送を再開しない。

### 帰結

- suite / generation / purpose / identityが暗号文とmanifestへ認証され、downgrade、cross-scope substitution、stale writeをfail closedにできる。
- rotationは新規writeの切替と既存live headのmigrationを分離し、crashとoffline端末がいてもretirement条件を明示できる。
- 個人利用は現在の対称暗号を維持しつつ、Organization共有だけをserver公開鍵差し替えと量子計算機による将来復号へ備えたhybrid方式にできる。
- schema、wire、Recovery、local capsuleはbreaking changeとなり、既存開発profile / server DBを再作成する。
- 実装は5件のcritical work itemへ分割し、WIP 1と各段階の独立検証を守る。

### 却下した代替案

- **定期的に全contentを再暗号化する案**: 大きなcrash surfaceと帯域負荷を増やす一方、侵害eventのない鍵変更に実効的利益が乏しいため不採用。
- **旧generationでoffline writeを続ける案**: 除名・侵害後の新規contentを旧鍵保持者が読めるため不採用。
- **account単位の共有秘密鍵を全deviceへ複製する案**: 端末単位失効とrecipient真正性を弱めるため不採用。
- **PQCだけへ即時置換する案**: 実装・運用リスクを増やすため、classical + PQのhybridを採用する。
- **server提供公開鍵をそのまま信用する案**: server侵害時の初回key substitutionを防げないため不採用。
- **key transparency logを同時導入する案**: 有力な将来強化だが、初期は別経路Safety numberを正本とし今回のscopeから外す。
