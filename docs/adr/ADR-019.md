# ADR-019: foreground realtime syncは欠落可能なWebSocket通知で起動する

**日付**: 2026-07-15
**状態**: Accepted（2026-07-15 プロダクトオーナー承認）

### 背景

現在のFlutter clientは、ログイン中に30秒周期で既存のHTTPS同期を実行する。同期runは変更がなくてもprotocol / continuity preflight、pull、closure ACKを行うため、foreground利用時間とdevice数に比例してrequestとPostgresアクセスが増える。一方、両方の端末を開いている場合でもlocal mutationのpushとremote pullがpoll周期まで遅れ、ユーザーからは同期が即時に見えない。

Taskveilの同期correctnessは、Postgres上のcurrent stateとserver採番`seq`、server-trusted device continuity、clientのpreflight / pull / reconcile / push state machineが担う。WebSocket providerへこの責務を移すと、通知欠落やprovider障害がデータ欠落へ直結し、ADR-002〜ADR-005、ADR-012、ADR-014、ADR-016で確立した境界を二重化する。

Cloudflare Durable Objectsはtenant単位のconnection coordinationとWebSocket Hibernationを提供し、idle中にclient接続を維持したままobjectをmemoryから退避できる。これは低頻度な変更通知に適するが、同期DBやdelivery保証付きqueueとして採用する理由にはならない。

### 決定

1. PostgreSQLをserver-side同期状態の唯一の正本、SQLCipherをdevice-side local stateの正本として維持する。同期内容の転送、merge、CAS、continuity、cursor更新、outbox ACKは既存HTTPS APIだけで行う。WebSocketは同期protocolの外側に置く、重複・順序逆転・欠落を許容したwake-up hintである。
2. realtime通知の初期providerはCloudflare Worker + Durable Objectsとする。1 tenantを1 Durable Object channelへ対応させ、Hibernation WebSocket APIを用いる。Cloudflare固有処理は通知adapterへ閉じ込め、core sync engine、encrypted envelope、Postgres schemaへprovider概念を持ち込まない。
3. foregroundでsync対象のlocal mutation transactionがcommitされたら、Flutter lifecycle層の共通schedulerは250ms debounce後に既存`TaskveilClient::sync_now`を起動する。連続triggerはcoalesceし、syncはsingle-flightとする。run中に追加mutationまたはremote notificationを受けた場合はdirty flagを残し、triggerなしで1 runが完了するまで1回ずつ追従runを行う。remote pull適用後は既存`SyncStatusNotifier`がlist / task / Home / Calendar / search / timer providerをinvalidateし、UI refreshまで完了する。WebSocketの追加だけで送信側の30秒poll待ちを残さない。
4. serverがtenantの`seq`を前進させるpush transactionをcommitした後、通知adapterは対象tenant channelへchange hintをpublishする。publishは500ms timeoutのbest effortとし、失敗、timeout、Cloudflare outageを理由にcommit済みpushを失敗へ戻さない。Lambda runtimeでresponse後のfire-and-forgetに依存せず、response前にbounded awaitする。将来queueを追加してもqueue deliveryを同期correctness条件にしない。
5. WebSocket notification frame v1は正確に`{"v":1,"type":"changed"}`だけを持つ。`high_water`、暗号blob、record ID、collection、revision HLC、task/list属性、鍵、session token、user ID、tenant UUID、device IDを含めない。clientは通知を既存sync runのwake-upとしてだけ扱い、continuity proof、trusted cursor、outbox ACK、同期到達状態を通知から更新しない。
6. tenant channelは`HMAC-SHA256(channel_key, "taskveil-realtime-channel-v1\n" || lowercase_hyphenated_tenant_uuid)`、source device tagは`HMAC-SHA256(channel_key, "taskveil-realtime-device-v1\n" || lowercase_hyphenated_tenant_uuid || "\n" || lowercase_hyphenated_device_uuid)`の32 bytesをbase64url no-paddingした43文字とする。UUID文字列はASCIIの小文字・hyphen付き36文字へ正規化する。tenant / device UUIDをDurable Object ID、URL、WebSocket frame、log fieldへ直接置かない。Durable Object namespaceはEU jurisdictionへ制約し、`jurisdiction("eu").getByName(opaque_channel)`でtenant単位objectを選ぶ。
7. clientは`POST /v2/tenants/{tenant_id}/realtime/ticket`を既存session tokenで呼び、`{"websocket_url":"wss://.../v1/connect","ticket":"...","expires_at":"RFC3339 UTC"}`を取得する。ticket endpointはsync endpointと同じrequest-time authorization policyを呼ぶ。現時点ではsession / device失効とtenant membershipを検証し、Billing foundationがsync entitlementを同policyへ追加した時点でticketにも同じ判定が適用される。realtimeだけの仮plan / fallback判定は追加しない。
8. ticket payload UTF-8 bytesはfield順を固定した`{"kid":"<key-id>","aud":"taskveil-realtime","channel":"<43-char>","device":"<43-char>","iat":<unix-seconds>,"exp":<unix-seconds>}`とする。key IDは`[A-Za-z0-9_-]{1,32}`、TTLは300秒、JSONへ空白を入れない。payload segmentはbase64url no-padding、signature inputはASCII `taskveil-realtime-ticket-v1\n`とpayload segmentの連結、signatureはHMAC-SHA256の32 bytesをbase64url no-paddingする。tokenは`<payload-segment>.<signature-segment>`の2 segmentとし、WebSocket Upgradeの`Authorization: Bearer` headerで渡してURLへ含めない。
9. ticket signing key、server-to-Worker publish key、channel derivation keyは別の32-byte secretとする。ticket / publish keyはcurrent / previous key IDによるoverlap rotationを許し、channel keyはv1中固定する。publish bodyはfield順を固定した空白なしUTF-8 JSON `{"v":1,"channel":"<43-char>","source_device":"<43-char>"}`、上限512 bytesとする。headerは`X-Taskveil-Realtime-Key-Id`、`X-Taskveil-Realtime-Timestamp`（Unix seconds）、`X-Taskveil-Realtime-Signature`（base64url no-padding）とする。key IDは検証鍵の選択だけに使い、signature inputはASCII `taskveil-realtime-publish-v1\n<timestamp>\n`とraw body bytesの連結である。Workerはconstant-time HMAC比較を行い、±30秒を超えるtimestamp、未知key ID、不正signature、過大bodyを拒否する。
10. Durable ObjectはHibernation WebSocket APIを使い、ticket expiryとopaque device tagをconnection attachmentへ保存する。1 device tagにつき最新1接続、1 tenantにつき最大128接続とする。publish時は期限切れconnectionを送信せずcloseし、source device tagをfan-outから除外する。clientからapplication messageを受理せず、通知はserver publish経路だけから生成する。
11. clientはforeground / loginで接続し、background / logout / disposeで切断する。ticketは期限30秒前に更新し、再接続は1、2、4、8、16、30秒を上限とするexponential backoff + jitterで行う。追加のnetwork reachability dependencyは導入せず、socket failureのretryとforeground復帰で回復する。foreground復帰ではWebSocket状態にかかわらず即時syncする。
12. WebSocket接続中は5分間隔のsafety pullを維持し、切断・接続失敗中は既存の30秒pollingへ戻る。duplicate、out-of-order、遅延、再接続後の古いnotificationはいずれも250ms coalesce後の通常syncを起動するだけであり、回数や順序を同期stateへ反映しない。
13. mobile OSがbackground中のWebSocket維持やprocess実行を保証しないため、即時性のSLOはforegroundに限定する。background同期やuser notificationが必要になった場合はAPNs / FCMを別設計とし、WebSocketをbackground delivery保証として扱わない。
14. Cloudflare通知レイヤーが停止してもlocal CRUDと明示的HTTPS syncは継続し、remote反映はresume syncまたはfallback pollingまで遅延するだけとする。運用監視はconnection成功率、ticket発行失敗、publish失敗/timeout、通知からpull開始までの遅延、WebSocket接続中/切断中のsync run数を集計するが、tenant UUID、record metadata、暗号blobをlogへ含めない。
15. 初期実装は、(a) notification contract / test double、(b) Cloudflare Worker + Durable Object、(c) server ticket / publish adapter、(d) client foreground connectionとsync scheduler、(e) multi-device / outage / reconnect統合テストとobservabilityへ分割する。各段階で通知providerなしでも既存HTTPS同期が成立する状態を維持する。

### 帰結

- 両端末がforegroundかつonlineなら、local mutation後のpushとremote pullがpoll周期を待たず通常数秒以内に開始される。
- WebSocket、Cloudflare、通知publishの障害は同期遅延に縮退し、データの正しさやlocal編集可用性へ波及しない。
- 接続中の定期polling頻度を下げられ、変更がないforeground時間のLambda / Postgres requestを削減できる。
- Lambda + PostgresとCloudflareのmulti-provider運用になるが、Cloudflare依存はopaqueな通知adapterに限定される。
- 即時push scheduler、ticket endpoint、server publish、Worker / Durable Object、Flutter lifecycle、統合テストが必要になる。WebSocketだけを追加しても即時同期は成立しない。
- ticket有効期限中の失効deviceには最大300秒のchange timing metadata露出窓が残るが、frameは固定hintのみで`high_water`やentity metadataを持たない。暗号dataへのアクセスは既存HTTPS APIのrequest-time認可で即時拒否する。

### 却下する代替案

- **Durable Objectsを同期DBの正本にする案**: relationalな認証・membership・RLS、CAS/history、continuityをprovider固有objectへ再実装し、Postgresとの二重正本または全面移行になるため不採用。
- **WebSocket frameで暗号recordを配送してpullを省く案**: delivery欠落、順序、再接続、ACK、GC continuityを別protocolとして再実装することになり、既存HTTPS同期と二重化するため不採用。
- **30秒pollingだけを維持する案**: 構成は単純だが、foregroundでの体感遅延と無変更requestが利用時間・device数に比例するため不採用。
- **AWS AppSync Eventsへ統一する案**: managed Pub/Subとして成立するが、AWS統一は要件ではなく、notification adapterとしての交換可能性を保てるため初期選択にはしない。
- **AWS IoT Core / API Gateway WebSocketを使う案**: MQTT機能またはconnection registry自作が必要になる一方、Taskveilの通知はtenant単位の単純fan-outだけなので初期構成としては複雑である。
- **常駐Rust WebSocket serverをECS等で運用する案**: 既存Rust資産は再利用できるが、初期規模で常時稼働、HA、connection drain、cross-task fan-outを運用する固定費に見合わない。
- **notification publishをPostgres commitの成功条件にする案**: provider障害で正しい同期writeまで失敗し、local-firstの可用性を下げるため不採用。
- **clientが認証なしでtenant channelへpublishする案**: channel enumerationとpull amplificationを許すため不採用。
