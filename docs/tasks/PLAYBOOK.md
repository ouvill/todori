# Codex運用プレイブック ── セッション/サブエージェントの使い方

## 原則（冒頭に記載）

- 1セッション = 1役割 = 1タスク。長引いたセッションで別のタスクを始めない（文脈汚染を防ぐ）
- 状態はチャットに置かない。すべて docs/tasks/ の指示書・完了報告・BACKLOG.md・git履歴に残す。セッションはいつ捨てても再開できる状態を保つ
- 実装した本人に合否判定をさせない。検証は必ず別セッション（または人間）が行う
- 迷ったら止まる: 仕様書(docs/01〜04)と矛盾する事実を見つけたら、書き換えずに完了報告の未解決事項へ
- バックログは自動では増えない。補充セッション（種別6）を定期的に回し、出典のあるタスクだけを積む
- ビジュアル/デザイン系タスクは docs/tasks/DESIGN_PLAYBOOK.md の方式（スクリーンショット必須のループ）に従う
- 検証可能なものしか守られない。新しい品質軸（見た目、操作感、性能等）に着手する前に、その品質を観測する手段（スクリーンショット、数値、ログ）を先に整備する（実例: task-25/28/29は文章基準だけで「完了」したが目的未達だった。task-30はスクショ基盤整備後に1回で到達した）
- 完了≠達成。完了報告は事実の記録であり、品質の達成判定は検証者（親・verifier・人間）だけが行う

## オーケストレーター運用（親Codexがサブエージェントを使う場合）

人間が毎回セッションを開かなくても、親Codexがオーケストレーターとして振る舞い、下記の役割をサブエージェントへ分担できる。ただし、原則は上と同じである。**1エージェント = 1役割 = 1タスク** とし、実装者に自分の実装を合否判定させない。

### 親Codexの責務

- `AGENTS.md` / `docs/tasks/README.md` / `docs/tasks/BACKLOG.md` / 対象タスク指示書を読み、次に進める役割を決める
- サブエージェントへ渡す作業範囲、変更してよいファイル、変更してはいけないファイル、完了条件を明示する
- サブエージェント同士の書き込み範囲が重ならないようにする
- サブエージェントの報告を鵜呑みにせず、必要な差分・ログ・完了報告・git状態を親Codexが確認する
- 状態をチャットに閉じ込めず、`docs/tasks/` の指示書・完了報告・`BACKLOG.md`・git履歴へ残す
- public/private境界を監督し、private詳細をpublic repoへ転記しない

### 推奨フロー

1. **選定**: 親Codexが `BACKLOG.md` と計画書を読み、次タスク候補を1つ選ぶ。判断材料が足りない場合だけ explorer に調査させる。
2. **指示書作成**: planner 役のサブエージェントに `docs/tasks/task-NN-<slug>.md` と `docs/tasks/README.md` 更新を担当させる。実装は禁止する。
3. **親レビュー**: 親Codexが指示書のスコープ、やらないこと、受け入れ基準、public/private境界を確認する。必要なら修正する。
4. **実装**: worker 役のサブエージェントに対象指示書だけを完遂させる。指示書外の変更は禁止する。
5. **親確認**: 親Codexが変更ファイル、完了報告、git状態を確認する。実装者の「通った」という報告だけで進めない。
6. **独立検証**: reviewer/verifier 役の別サブエージェント、または親Codexが別役割として品質ゲートとコードレビューを行う。実装workerには合否判定させない。
7. **合否処理**: 合格なら `BACKLOG.md` の現在地更新を行う。不合格なら修正セッション用の指摘を作り、別workerへ渡す。

### 親Codexの終了チェック

サブエージェントの作業が終わったら、親Codexは次へ進む前に以下を確認する。

- `git -C todori status --short` で意図しない未コミット差分がない
- 対象タスク指示書に `## 9. 完了報告` があり、検証結果と未解決事項が記録されている
- `docs/tasks/README.md` と `docs/tasks/BACKLOG.md` の更新要否が判断済みである
- 実行できなかった品質ゲートがある場合、環境制約かコード起因の失敗かが区別されている
- public repoにprivate情報が混入していない
- 合格判定は実装worker以外が行っている

### サブエージェント種別の使い分け

- **explorer**: 読み取り専用の調査。仕様、既存実装、影響範囲、既知の罠を短く報告させる。ファイル編集は禁止。
- **planner**: 指示書作成専任。`docs/tasks/task-NN-<slug>.md` と必要な `docs/tasks/README.md` 更新だけを担当させる。実装は禁止。
- **worker**: 実装専任。対象タスク指示書で許可されたファイルだけを変更させる。品質ゲート実行と完了報告追記まで担当するが、最終合否は出させない。
- **verifier**: 検証専任。直近コミットまたはworker差分を読み、品質ゲートを再実行し、受け入れ基準・スコープ逸脱・セキュリティ問題を確認する。修正はさせない。
- **fixer**: 検証で不合格になった場合の修正専任。verifierの指摘と対象指示書だけを入力にし、追記形式で完了報告を更新する。

### 並列化してよい場面 / だめな場面

- よい: explorer に複数の独立した調査を並列依頼する（例: Flutter UI影響範囲、Rust bridge影響範囲、既存テスト）。
- よい: 変更ファイルが明確に分離できる場合に、worker を分ける（例: Rust API と Flutter UI）。親Codexが統合責任を持つ。
- よい: worker が実装中に、別explorerへ検証観点や既存規約の確認を依頼する。
- だめ: 同じファイル群を複数workerへ同時に編集させる。
- だめ: 計画、実装、検証を同じサブエージェントに連続して任せる。
- だめ: 依存追加、暗号・鍵階層、公開/非公開境界、仕様変更を人間確認なしで広げる。依存パッケージの追加は人間承認が必要だが、**提案を躊躇してはならない**（2026-07-07人間指示）。実装品質・開発効率が上がるパッケージがあれば、planner/workerは完了報告や指示書で積極的に採用提案し、親が人間へ確認する。
- だめ: private repoの詳細をpublic repoの指示書や完了報告へ貼り付ける。public側には公開可能な抽象化済み要約だけを書く。

### 親Codexからサブエージェントへ渡す共通文

```
あなたはTodori repo内の1役割専任サブエージェントです。親Codexが別途オーケストレーションしています。
作業前に todori/AGENTS.md と todori/docs/tasks/README.md を読み、対象タスクで指定されたファイルだけを扱ってください。
あなたはこのコードベースで単独ではありません。他者の変更を戻さず、git状態を確認し、無関係な差分は触らないでください。
public repoである todori/ には private repo の詳細（課金、収益、法務、監査、公開前ロードマップ等）を転記しないでください。
docs/01_企画書.md / docs/02_機能仕様書.md / docs/03_技術仕様書.md は変更禁止です。矛盾を見つけたら完了報告の未解決事項へ記録してください。
最後に、変更したファイル、実行した検証、未解決事項を短く報告してください。
```

### 役割別プロンプト雛形

#### planner

```
AGENTS.md、docs/tasks/README.md、docs/tasks/BACKLOG.md、docs/07_Phase1計画書.md を読め。
次に着手すべきタスク候補として [候補名] を扱い、docs/tasks/task-NN-<slug>.md を既存task-05〜10の体裁で作成せよ。
docs/tasks/README.md の表にも行を追加せよ。
実装はするな。指示書には「やること」「やらないこと」「受け入れ基準」「完了報告に含めるべき内容」を具体的に書け。
```

#### worker

```
AGENTS.md と docs/tasks/README.md の共通規約を読め。
次に docs/tasks/task-NN-<slug>.md を読み、その指示書だけを完遂せよ。スコープ外の変更はするな。
品質ゲート（AGENTS.md記載）を可能な限り実行し、環境制約で実行不能なものはコード起因の失敗と区別して完了報告に明記せよ。
指示書末尾に「## 9. 完了報告」を追記し、Conventional Commitsでコミットせよ。
最終合否判定は行わず、親Codexへ変更ファイル、検証結果、未解決事項だけを報告せよ。
完了報告には事実（変更ファイル、実行したコマンド、結果の生データ、スクショのパス）だけを書け。「プロダクト品質になった」「十分整った」等の品質評価語を書くな。品質判定は検証者が行う。
```

#### verifier

```
AGENTS.md、docs/tasks/README.md、docs/tasks/task-NN-<slug>.md を読め。
直近コミットまたは指定差分が受け入れ基準を満たすか検証せよ。
品質ゲートを自分で再実行し、変更されたコードとドキュメントを読み、スコープ逸脱・規約違反・セキュリティ問題・public/private境界違反がないか確認せよ。
合格/不合格、根拠、再現コマンド、修正が必要な場合の具体的な指摘を報告せよ。自分で修正するな。
品質達成の判定はお前（または人間）だけが記録できる。完了報告に品質評価語が混入していたら指摘せよ。
```

## 軽量レーン（指示書を省略できる小修正）

すべての変更にplanner→worker→verifierのフルセレモニーを回すのはコスト過剰である。以下の**全条件**を満たす場合、指示書なしの親直轄ワンセッション（実装+品質ゲート+親レビュー）で処理してよい。

- 変更が概ね1〜3ファイル・50行未満
- 仕様変更・新機能・新規依存・DB/FRB/暗号/鍵/public-private境界に触れない
- 既存テストの期待を弱めない（finder追従程度の修正は可）

軽量レーンでも品質ゲートは全て通す。記録はgit履歴（Conventional Commits）で足り、BACKLOGや指示書は作らない。

同じ箇所で2回続けて手戻りが出たら軽量レーンをやめ、通常レーン（指示書方式）へ昇格させる。

判断に迷ったら通常レーンを使う。

## セッション種別と定型プロンプト

### 1. 計画セッション（タスク選定+指示書作成のみ）

```
AGENTS.md と docs/tasks/BACKLOG.md を読み、次に着手すべきタスクを1つ選んで理由を述べよ。
承認したら docs/tasks/task-NN-<slug>.md を task-05〜10 の体裁（1〜8章、9章は作らない）で作成し、
docs/tasks/README.md の表に行を追加せよ。実装はするな。
指示書では「やらないこと」と「受け入れ基準」を特に具体的に書くこと。
完了したら指示書をコミットせよ（docs(tasks): プレフィックス）。
```

### 2. 実装セッション（指示書の完遂のみ）

```
AGENTS.md と docs/tasks/README.md の共通規約を読め。
次に docs/tasks/task-NN-<slug>.md を読み、その指示書だけを完遂せよ。スコープ外の変更はするな。
品質ゲート（AGENTS.md記載の6点）を全て通し、指示書末尾に「## 9. 完了報告」を追記してから
Conventional Commits でコミットせよ。
ゲートが環境制約で実行できない場合は、その旨を完了報告に明記せよ（コード起因の失敗と区別すること）。
完了報告には事実（変更ファイル、実行したコマンド、結果の生データ、スクショのパス）だけを書け。「プロダクト品質になった」「十分整った」等の品質評価語を書くな。品質判定は検証者が行う。
```

### 3. 検証セッション（実装セッションとは別に開く）

```
直近のコミット（git log で確認）が docs/tasks/task-NN-<slug>.md の受け入れ基準を満たすかレビューせよ。
1) 品質ゲート6点を自分で再実行する（完了報告の記載を信用しない）
2) 変更されたコードを全て読み、指示書のスコープ逸脱・規約違反・セキュリティ問題（秘密情報のログ出力等）がないか確認する
3) 合格なら BACKLOG.md の現在地を更新してコミット。不合格なら問題点を列挙し、修正内容を指示書の形式で提示せよ（自分で修正するな）
```

### 4. 修正セッション（検証で不合格になった場合）

```
docs/tasks/task-NN-<slug>.md と、検証セッションが列挙した問題点（以下に貼る）を読み、修正せよ。
修正後、品質ゲートを再実行し、完了報告に追記の形で修正内容を記録してコミットせよ。
---
[検証セッションの指摘をここに貼る]
```

### 5. 知見記録（どのセッションでも、新しい罠を踏んだら）

```
今回判明した環境固有の問題と対処法を AGENTS.md の「重要な設計制約・ハマりどころ」に追記してコミットせよ。
既存項目と重複しないこと。1項目2〜3行で簡潔に。
```

### 6. バックログ補充セッション（棚卸し。週1回程度、またはBACKLOGの残りが3件を切ったら）

```
以下を読み、docs/tasks/BACKLOG.md を棚卸しせよ:
1) docs/07_Phase1計画書.md のマイルストーン表（M3/M4以降）で、BACKLOG未掲載かつ未実装のタスク
2) docs/tasks/task-*.md の全「## 9. 完了報告」の未解決事項欄で、まだタスク化されていないもの
3) docs/07_Phase1計画書.md の「リスクと先送り判断」表で、対応時期が来たもの
これらをBACKLOGの表に優先度付きで追加し、完了済みの行を「現在地」へ移し、
各追加行に出典（計画書のID or 由来タスクの完了報告）を備考に記載してコミットせよ。
新しい機能の発明・仕様の拡大はするな。出典のないタスクを追加してはならない。
判断に迷うもの（仕様変更を伴う等）は表に入れず、「要人間判断」セクションに列挙せよ。
```

### 7. ドッグフーディングセッション（人間の体験レビュー。マイルストーン境界、または2週間に1回）

品質ゲートは感触を検証できない。人間が実機（macOS/iOSシミュレータ）でアプリを5〜10分実際に使い、気になった点を箇条書きでメモする（体裁自由。「〜が遅い」「〜が分かりにくい」で構わない）。

そのメモを以下の定型プロンプトでBACKLOG化する。

```
以下は人間がYYYY-MM-DDに実機でアプリを触ったメモである。
各項目を docs/tasks/BACKLOG.md の表へ、出典「ドッグフーディングYYYY-MM-DD」として優先度付きで追加せよ。
仕様変更を伴うもの・判断が割れるものは表に入れず「要人間判断」セクションへ列挙せよ。
新しい機能の発明はするな。メモにない項目を追加してはならない。
---
[メモを貼る]
```

## 人間の関与ポイント

- 計画セッションのタスク選定の承認（数分）
- 検証セッション合格後のpush判断（まとめてでよい）
- 依存追加・仕様変更・鍵/暗号まわりの変更は必ず人間が目を通す。依存パッケージの追加は人間承認が必要だが、実装品質・開発効率が上がる場合はplanner/workerが完了報告や指示書で積極的に採用提案し、親が人間へ確認する（2026-07-07人間指示）
- 同一タスクで修正セッションが2回失敗したら、それ以上リトライさせず人間が介入する
- 補充セッションが「要人間判断」に積んだ項目の裁定（仕様変更・新機能の追加はここでのみ決まる）
- マイルストーン境界（または2週間ごと）のドッグフーディング（5〜10分実機で触る。種別7参照）

## アンチパターン

- 1つのセッションで計画→実装→検証を全部やらせる（自己採点になる）
- チャットでの口頭指示だけでタスクを進める（指示書がないと検証基準が消える）
- 完了報告の「成功しました」を読んでゲートを再実行せずマージする
- 長寿命セッションの使い回し（古い文脈が新しい規約を上書きする）
- 受け入れ基準を「落ち着いた」「整える」等の検証不能な形容詞で書く（スクショ・数値・テストで確認できる文に落とす。ビジュアル系は DESIGN_PLAYBOOK.md 参照）
- 完了報告に品質評価語（「プロダクト品質へ引き上げた」等）を書く（次セッションがそれを事実として読み込み、達成済みバイアスが再生産される）
