## ADR-001: E2EEを製品の核として厳格適用する

**日付**: 2026-07-03
**状態**: 採用

### 決定

サーバーはDEKで暗号化されたblobと最小限の同期メタデータのみを保持する。サーバー側でコンテンツの復号・解釈・マージは一切行わない。

### 理由

E2EEはこの製品の譲れない価値定義そのものである（Proton等の思想への共感）。

### 帰結

- サーバー側マージ不可のため、クライアント側マージが必須となる。
- 検索・ソートのため平文のローカルDB（SQLCipher保護）が必須となる。
- サーバー起点プッシュ通知は不可であり、リマインダーはローカル通知で実現する。

### 却下した代替案

該当なし（本ADRは前提となる原則の確認であり、対案の比較検討を伴わない）。

## ADR-002: 同期データプレーンはサーバー経由APIとする（Turso不採用）

**日付**: 2026-07-03
**状態**: 採用

### 決定

同期はサーバー（Lambda上のaxum）が提供する小さなpush/pull APIを経由する。旧設計の「クライアント⇄Tursoテナント DB直結+短命スコープJWT」は廃止し、Turso/libSQLは採用しない。

### 理由

1. E2EEでは同期APIは実質2エンドポイント（push/pull）で極小であり、直結モデルの「サーバー実装削減」の利点が薄い。むしろテナントDB発行・スコープJWT発行・トークン更新の管理の方がコード量が多い。
2. 直結モデルのトークンは当該DBへの任意SQL実行権となり、Org共有では悪意ある（または故障した）メンバー端末がDBを物理破壊できる。サーバー経由ならappend-onlyセマンティクス（物理DELETE不可・削除はtombstoneのみ）を強制できる。
3. デバイス失効・課金失効が即時になる（直結はトークンTTL 15〜60分の失効遅延を構造的に抱える）。
4. クライアントからlibsqlクライアント依存が消え、rusqlite+SQLCipherとのSQLite実装二重リンク問題（特にiOS/Androidでのシンボル衝突リスク）が消滅する。
5. libsql（C系フォーク）とturso（Rustリライト、旧Limbo）の二系統エコシステムリスクを回避できる。
6. TursoはORM/sqlx非対応でlibsql crateの生SQLのみだが、Postgresならsqlx（コンパイル時クエリ検証・成熟したマイグレーション）が使える。
7. オフラインファースト設計により同期はバックグラウンド非同期処理であり、同期レイテンシは体感品質に影響しないため、直結・エッジ配置の性能利点が活きない。

### 却下した代替案

- **Turso直結モデル**（旧設計）: 上記理由により不採用とした。
- **Tursoネイティブ同期の利用**: embedded replicaはオフライン書込不可（書込はプライマリへ転送）であり、オフラインファースト要件と衝突する。新offline sync（push/pull）はベータである。またE2EEではテナントDBの中身は暗号blobであり、複製してもクエリ不能で平文の作業用DBが別途必要になるため、複製は「トランスポートのキャッシュ」にしかならない。フィールドレベルマージはどのみちアプリ層実装が必要である。

### コスト認識

Lambdaがデータパスに入る（リクエスト毎の実行コスト、コールドスタート100〜300ms）が、暗号blobは数KB・同期は低頻度非同期のため実害はない。

## ADR-003: データストアはPostgres一本に統合する

**日付**: 2026-07-03
**状態**: 採用

### 決定

コントロールプレーン（users/devices/ラップ済み鍵/organizations/subscriptions/OPAQUE中間状態）と同期ストア（暗号blob）を単一のPostgresに統合する。第一候補はサーバーレスPostgres（Neon等、scale-to-zeroでLambdaと相性が良い）とする（プロバイダと接続方式の最終決定はADR-008参照。当初想定していたHTTPドライバはJavaScript向けでありRust/sqlxには適用できないことが後に判明したため、接続はプーラー経由TCPとする）。データアクセスはsqlxを全面採用する。

### 理由

- スタック一本化（アクセス層・マイグレーション・CI・ローカル開発が1系統）が実現できる。
- コントロールプレーンのデータ（users/devices/orgs/subscriptions）は典型的なリレーショナルデータでJOINが自然に発生する。
- Lambda+サーバーレスPostgresはアイドルコストゼロである。

### 却下した代替案

- **DynamoDB**: リレーショナル不一致でアクセスパターンを先決めする必要がある、最強のAWSロックイン、スタック2系統化、DynamoDB Localでのテスト環境負担、という理由で不採用。
- **管理用Turso DB**: ADR-002でTurso自体を不採用としたため消滅。採用していた場合もsqlx非対応で生SQLのみだった。
- **RDS/Aurora**: 最小構成の月額固定費と接続管理（RDS Proxy等）が初期フェーズに不適合。Aurora Serverless v2は再開レイテンシが認証APIに不適。
- **MySQL**: この用途でPostgresに勝る点がない。

### トレードオフ

テナント分離が物理DB分離から論理分離（tenant_idスコープ）になる。対策として、全クエリにtenant_idを必須とするリポジトリ層設計 + Postgres RLSの多層防御 + 認可の網羅テストを行う。E2EEにより万一の漏洩時も被害は暗号blobとメタデータに限定される。

## ADR-004: 同期エンジンは自前実装とする（フィールドレベルLWW + HLC + fractional indexing）

**日付**: 2026-07-03
**状態**: 採用

**補正**: 2026-07-10のADR-012により、field HLCとrecord revisionの分離、placementの原子的同期、production経路を通る収束テストを必須化した。

### 決定

既存CRDTライブラリは採用せず、フィールドレベルLast-Write-Wins + Hybrid Logical Clock + fractional indexing（並び順）+ tombstone（削除）を自前実装する。proptestによるproperty-based testingで「任意の操作列が全端末で同一状態に収束すること」を検証する。

### 理由

本製品のデータモデルはCRDTの「簡単な端」にある。必要なのはフィールド単位後勝ち・並び順・削除のみであり、CRDTで本当に難しい共同リアルタイムテキスト編集は要件にない。メモ欄はフィールド丸ごとLWWで実用上十分である。セマンティクスを最小に絞れば自前実装は枯れたパターンである。

### 却下した代替案

- **cr-sqlite**: メンテナンス停滞気味。SQLCipher入りbundledビルドへの拡張ロードの複雑さ。changesetとE2EE暗号境界の摩擦がある。
- **Automerge/Loro**: 活発で優秀だがドキュメント型CRDTであり、「SQLiteを正本としたフィールドLWW」にはオーバーキルである。履歴保持のストレージコストもある。

### 将来

リッチテキスト共同編集が要件化した場合、そのフィールドのみAutomerge/Loro等のドキュメントを暗号blobに載せる局所導入が可能である。

## ADR-005: 同期プロトコルは「最新状態方式 + クライアント再push」、カーソルはサーバー採番seq

**日付**: 2026-07-03
**状態**: 採用

**補正**: 2026-07-10のADR-012により、top-level `revision_hlc` と暗号payload内 `field_hlcs` を分離し、再pushで未変更fieldのclockを更新しない方式へ改訂した。

### 決定

- サーバーはレコード（tenant_id, record_id）ごとに最新の暗号blobをupsert保持する（完全append-onlyのopログ方式は不採用）。
- 暗号blobの中にフィールドごとのHLCマップを内包する（`{fields, field_hlcs}` を丸ごと暗号化し、サーバーには不可視とする）。クライアントはpull時にフィールド単位でマージし、ローカルが勝ったフィールドがあればマージ済みblobを再pushする。2026-07-10のADR-012により、再push用 `revision_hlc`、domain変更の `mutation_hlc`、削除の `delete_hlc` をfield HLCから分離した。
- pullカーソルはHLCではなく、サーバーが採番するテナント単位の単調増加seqを用いる。
- HLCはノードID（デバイス識別子由来）と端末単調カウンタを内包し全順序・一意とする。push retryのno-op判定には同一 `revision_hlc` を用いる。merge再pushはremote revisionをobserveしてからより大きいrevisionを採番する（ADR-012）。
- sync_recordsにdevice_idカラムは持たない（HLCのノード成分で足りる。メタデータ最小化のためである）。書き込み主体の帰属はhistoryテーブルのauthor_user_idにのみ持つ。
- サーバーが強制する不変条件: クライアントによる物理DELETE不可（削除はtombstoneフラグのみ）、blobサイズ上限（初期値64KB）、バッチ上限、HLC物理時刻がサーバー時刻+5分を超えるpushは拒否（時計異常・「未来HLCで永久勝ち」攻撃の抑止）、リクエスト毎の課金・メンバーシップ検証。
- エッジケース対策: 上書き時に旧行を`sync_records_history`へ退避し30日保持する（Orgメンバーの破壊行為からの復旧手段）。ただし削除対象recordのhistoryは即時破棄し、contentを持たないtombstoneは180日でGCする。ADR-010 / ADR-016により、`gc_horizon_seq`とserver-trusted device continuityからexpired端末を判定し、full resync closure後にserver-seen / never-synced / dependencyを分類するrebaseへ進める。

### 理由（カーソル設計）

HLCは「編集の論理順序」でありサーバー到着順ではないため、pullカーソルに使うと時計が遅れた端末の後出しpush（古いHLC）を他端末が永久に取り逃す。カーソルには「サーバーで可視になった順」が必要である。Postgresのbigserialや時刻ベースは並行トランザクションのコミット順逆転により可視性の欠番が生じるため不採用とした。`tenant_seq`行の`UPDATE ... RETURNING`による採番は行ロックがコミットまで保持されるためテナント単位でpushが直列化され、「seqの順＝可視になる順」が厳密に保証される。直列化の競合範囲はテナント内（1ユーザーの数端末〜小規模Org）のみで実害はない。状態はDB側にありLambdaはステートレスなため、サーバーレスと完全に両立する。

### 理由（最新状態方式）

E2EEではサーバーがopログをコンパクションできない（中身が読めない）ため、完全append-onlyログは無限成長する。最新状態+短期historyが実質的な均衡点である。

### 却下した代替案

- **完全append-onlyのopログ方式**: 上記「理由（最新状態方式）」のとおり、E2EE下ではコンパクション不能で無限成長するため不採用。
- **pullカーソルにHLCを使う方式**: 上記「理由（カーソル設計）」のとおり、時計遅延端末のpushを取り逃すため不採用。
- **pullカーソルにPostgresのbigserial/時刻ベースを使う方式**: 並行トランザクションのコミット順逆転により可視性の欠番が生じるため不採用。

### 参考スキーマ

```sql
CREATE TABLE sync_records (
    tenant_id      UUID    NOT NULL,
    record_id      UUID    NOT NULL,  -- UUIDv7
    collection     TEXT    NOT NULL,
    seq            BIGINT  NOT NULL,  -- pullカーソル用（サーバー到着順）
    hlc            TEXT    NOT NULL,  -- 競合解決用（固定幅ソート可能エンコード、ノードID内包）
    encrypted_blob BYTEA   NOT NULL,
    deleted        BOOLEAN NOT NULL DEFAULT false,
    updated_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (tenant_id, record_id),
    UNIQUE (tenant_id, seq)
);
CREATE TABLE tenant_seq (tenant_id UUID PRIMARY KEY, last_seq BIGINT NOT NULL);
CREATE TABLE sync_records_history (
    tenant_id UUID NOT NULL, record_id UUID NOT NULL, seq BIGINT NOT NULL,
    hlc TEXT NOT NULL, encrypted_blob BYTEA NOT NULL, deleted BOOLEAN NOT NULL,
    author_user_id UUID NOT NULL, overwritten_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### API

当初のprotocol v1は `POST /v1/tenants/{tenant_id}/push`（body: ops配列 `{record_id, collection, hlc, deleted, blob}`）、`GET /v1/tenants/{tenant_id}/pull?since={seq}&limit={n}` とした。ADR-012により、protocol v2ではop ID、`revision_hlc`、`mutation_hlc`、`delete_hlc`、continuity preflight、fuzzy scan + delta resyncを追加する。

pullで自デバイスのエコーは除外しない（除外すると他端末による上書きを見逃す。マージが冪等なので実害はない）。当初の初回同期はsince=0ページングとしたが、ADR-012によりstable-key fuzzy scan + delta catch-upへ置換する。転送エンコードは初期JSON+base64とし、将来CBOR化に備えプロトコルバージョンをヘッダに持たせる。準リアルタイム反映は初期は「フォアグラウンド復帰時+数分間隔のpull」とし、SSE/long-pollは将来拡張とする。セッションは初期はDB照合のopaqueトークンとする（コントロールプレーンと同一Postgresのため1クエリで済む）。

## ADR-006: 単一リージョンで開始し、リージョン拡張可能性のみ温存する

**日付**: 2026-07-03
**状態**: 採用

### 決定

データストア・サーバーは単一リージョンで開始する。ただし以下の3点を初日から仕込む。

1. regionをテナントの第一級属性としてスキーマに持つ。
2. APIエンドポイントは最初からリージョンプレフィックス形式（例: `eu.api.…`）とする。
3. サインアップのプロビジョニングフローにリージョン選択を組み込む（初期は選択肢1つ）。

あわせて、CloudFrontを前段に置きTLS終端をユーザー近傍化する。

### 理由

マルチリージョン検討の動機はレイテンシだったが、オフラインファースト設計では通常操作はローカルDB完結・同期はバックグラウンド非同期であり、遠距離RTTが体感品質に現れない（ローカルDBこそが最強のエッジレプリカである）。ログイン（OPAQUE複数往復）は低頻度、初回フル同期は一度きりでバッチ設計が支配的である。マルチリージョン化してもユーザーが体感できる改善はほぼない。

### 却下した代替案

- **初期からのマルチリージョン展開**: 上記理由のとおり、体感できる改善がほぼない一方で運用・実装コストが増大するため、初期段階では不採用とした。

### 将来

データレジデンシー（EU訴求等のコンプライアンス/マーケティング）要件が生じた場合は、グローバルコントロールプレーン+リージョナルセル（Lambda+Postgres per region。サーバーレスのためセル増設の固定費はほぼゼロ）へ拡張する。上記の仕込みによりID設計・URL構造への単一リージョン前提の焼き付きを防ぐ。

## ADR-007: テナント単位のリソース分離（データ・鍵・ローカルDB・UI）

**日付**: 2026-07-03
**状態**: 採用

### 決定

**すべての同期対象エンティティは正確に1つのテナント（個人テナント or Orgテナント）に属し、テナントをまたぐ参照は存在しない。**

- tags / templates / schedules / timer_sessions はテナントスコープとする。個人テナントは自分のタグ等を持ち、各Orgは自前のタグ等を持つ。個人タグをOrgタスクに付けることはできず、Org側でタグを作成する。task_tags等の参照は常に同一テナント内で閉じる。
- 鍵構造として**テナントルートDEK**を導入する。テナントスコープのエンティティ（tags/templates/schedules/timer_sessions）はテナントルートDEKで暗号化する。個人テナントは `wrap(TenantDEK, MK)` でサーバー保管し、Orgテナント（Phase 3）はメンバー全員の公開鍵でsealする（Orgのタグは全メンバー可視というセマンティクスになる）。リスト配下のエンティティ（tasks/comments/reminders）は従来通りList DEKで暗号化する。
- タスクの同一テナント内でのリスト間移動は、クライアントで復号→移動先List DEKで再暗号化→pushにより実現する。
- テナントをまたぐタスクの移動は「移動」ではなく**明示的なコピー**操作とする（新レコードIDを発行する。鍵もHLC系譜も混ざらない）。
- ローカルDBはテナントごとにSQLiteファイルを分離する（SQLCipher鍵は従来通りDK由来で共通）。Org脱退・除名時は当該テナントのファイル削除で端末からのデータ消去が完結する。FTS5索引もテナントごとに閉じる。
- UIはワークスペース切替モデル（Slack/Notion方式）を採用する。個人と各Orgは別空間としてスイッチャーで切り替える。テナント横断ビュー（全ワークスペース統合の「今日」画面等）は初期スコープ外とする。ただしローカルDBは平文のため、将来の統合ビューは暗号・同期・サーバーに一切触れず純粋にクライアント側の読み取り機能として後付け可能であり、UI分離はソフトな制約である。

### 理由

1. テナント境界をまたぐ参照（共有リスト上のタスクへの個人タグ付け等）が生む鍵・認可・同期のエッジケースを構造的に排除できる。
2. 「リストに属さないエンティティ（tags/templates等）の対応DEKが未定義」という仕様の穴がテナントルートDEKで解決する。
3. サーバー側の認可検証が「リクエストのテナントIDとリソースの所属の一致確認」で完全に閉じる。
4. ローカルDBファイル分離により、サーバーの論理分離とローカルの構造が対称になり、除名・脱退時のデータ消去がファイル削除で完結する（除名後の端末残存データ問題への部分的な対処にもなる）。
5. UI分離はソフトな制約であり、将来の横断ビューはクライアント側機能として後付け可能なため、UX面の選択肢を失わない。

### 却下した代替案

- **テナント横断参照の許容**（個人タグをOrgタスクへ付与する等）: 鍵の帰属・除名時の挙動・認可が複雑化するため不採用。
- **ローカルDBの単一ファイル + tenant_idカラムによる分離**: テナント境界バグのリスクをローカルにも持ち込む。UI分離により横断クエリの需要がなく、ファイル分離のデメリットが実質ないため不採用。

## ADR-008: サーバーレスPostgresプロバイダはNeon、初期リージョンはeu-central-1とする

**日付**: 2026-07-04
**状態**: 採用

### 決定

- プロバイダは **Neon** とする。
- 初期リージョンは **eu-central-1（フランクフルト）** とし、Lambdaも同一リージョンに同居させる。
- 接続方式は **Neon組み込みの接続プーラー経由のTCP接続 + sqlx** とする（PostgresワイヤプロトコルによるTCP接続。HTTPドライバはJavaScript向けでありRust/sqlxには適用できない）。
- プーラーはトランザクションモードで動作するため、セッション依存機能（advisory lock、LISTEN/NOTIFY等）は使用しない設計とする（現行設計は行ロック+通常トランザクションのみであり抵触しない）。
- 実装開始時の要確認事項: Neonの最新料金プラン、プーラーのprepared statement対応（sqlxはprepared statementを使用する）、リージョン提供状況。

### 理由

1. コスト構造の一致: 真のscale-to-zeroによりユーザーゼロ期のアイドルコストが実質ゼロで、Lambda側と合わせて全層で「使われなければ払わない」が成立する。
2. ブランチ機能によるCI/テストDX: CIで統合テストごとに実Postgresのブランチを短時間で作成・破棄でき、同期プロトコルのproperty-based testingや認可の網羅テストを実DBで回せる。
3. 出口の安さ: 素のPostgresであり、pg_dump/論理レプリケーションで他プロバイダへ移行可能（ADR-003の出口戦略と整合）。
4. リージョン選定: E2EE/プライバシー訴求のコア顧客層は欧米であり、「暗号化データはEU域内に保存」はこの層への訴求として機能する（Protonの「スイス保管」と同じ構図）。EUユーザー最速、米国東海岸〜90ms・西海岸〜150ms・日本〜230msだが、オフラインファースト設計（同期はバックグラウンド非同期）のため全て許容圏内。NeonはAWS東京リージョン非提供だが、本設計では実害がない（重要なのはLambda⇄DBの同一リージョン同居であり、ユーザー⇄Lambdaの距離は非クリティカル）。
5. 誠実性の注記: AWSは米国企業でありCLOUD Actの射程は理論上EUリージョンにも及ぶ。EU配置は法的な完全防御ではなく姿勢とシグナリングであり、本当の防御はE2EEそのもの（サーバーが平文を持たない）である。マーケティング文言の作成時もこの区別を維持する。

### 補足

Neonは2025年にDatabricksに買収されている。サービス継続性への直接の懸念はない（資金面はむしろ盤石化）が、ロードマップがAI/Databricks統合へ傾く可能性はリスクとして認識する。素のPostgresゆえの出口の安さ（理由3）をヘッジとする。

### 却下した代替案

- **Supabase**: 素のPostgres+プーラー内蔵で堅実だが、無料枠は非アクティブ時にプロジェクトが一時停止し本番利用不可。有料プランは常時起動インスタンス課金でscale-to-zeroではない。Auth/Realtime/Storage等の付加機能は自前実装（OPAQUE認証等）と重複し不要。東京リージョンが絶対要件の場合の次点だったが、東京の価値が月額固定費とブランチCI喪失に見合わないと判断。
- **Aurora Serverless v2**: scale-to-zero対応だが復帰に十数秒を要し、認証・同期APIに不適（ADR-003でも却下済み）。
- **Aurora DSQL**: Postgres「互換」だが楽観的並行制御であり、tenant_seq行ロックによる直列化パターン（ADR-005）の前提が崩れる。拡張・機能制限も多い。
- **Nile / Prisma Postgres / PlanetScale for Postgres**: それぞれ特色はある（Nileはテナント分離ネイティブ等）が、若い・実績薄・またはハイエンド志向であり、認証基盤を賭けるには時期尚早。継続ウォッチとする。
- **米国東部リージョン（us-east-1等）での開始**: 英語圏市場の重心としては合理的だが、プライバシー志向層への訴求力とCLOUD Act忌避の議論回避の観点でEU配置が優る。将来のリージョナルセル拡張（ADR-006）で最初に増やす候補はUSセルとなる。

## ADR-009: ゴミ箱を廃止し、削除は恒久削除+アーカイブモデルへ一本化する

**日付**: 2026-07-07
**状態**: 採用

### 背景

2026-07-07人間裁定（データ保持原則、`docs/design/ui-spec.md`参照）は、完了済みタスク（done/wont_do）を振り返りのための記録資産と位置付け、リスト削除を含むいかなる整理操作でも暗黙に失われてはならないとした。この原則を運用する保全経路として、Phase 1でリストのアーカイブ機能を導入することが決まった。一方、実装済みのゴミ箱（task-23、論理削除+30日後物理削除+復元UI）は「消したいが戻したいかもしれない」という中間的な削除需要を満たすために設けたものだったが、done/wont_do/アーカイブという正当な「やめ方」の記録手段がすでに用意されたことで、削除という操作に残る用途は実質的に「ミス操作やノイズの掃除」のみとなった。E2EEを製品の核とする本製品（ADR-001）にとって、「削除」という言葉がユーザーに与える期待は「本当に消える」であるべきであり、実際には一定期間サーバー/端末に残り続けるゴミ箱付き削除はこの期待との乖離（誠実さの毀損）を生む。

### 決定

1. ゴミ箱（trash画面・論理削除保持期間・復元機能）は廃止する。実装済みのゴミ箱UI・復元機能（task-23）は後続タスクで撤去する。
2. 削除は恒久削除とする。削除操作は一覧からの即時操作ではなく、詳細画面のサブメニューから起動する導線とし、「元に戻せません／永久に失われます」という追加確認ダイアログを必須とする。導線を意図的に深くすることで誤操作を防ぐ。
3. 削除に対するUndoは設けない（起動導線の深さ + 不可逆警告確認の二段防御で誤削除リスクを十分に抑制できるため）。
4. 完了（チェック）操作のUndoは維持する。編集操作のUndoも既存どおり維持する。
5. 履歴の保全経路はアーカイブとする（Phase 1導入決定済み）。リスト削除の確認ダイアログでは、削除対象に含まれる完了済みタスクを含む影響件数を明示し、「履歴を残したい場合はアーカイブを使う」旨へ誘導する。
6. E2EEプロダクトとして「削除＝本当に消える」という原則を厳格に守る。Undo履歴テーブル（`task_undo_entries`等）についても、Undo対象外である削除操作の内部データは残さない。

### 帰結

- `docs/02_機能仕様書.md` F-07（ゴミ箱・Undo）の仕様を本ADRの内容へ書き換える。
- 実装済みゴミ箱（task-23、`/trash` route・`trashedTasksProvider`等）の撤去タスクを新設する。
- Undoの削除復元経路（もしあれば）を廃止する。
- `tasks.deleted_at` カラムは将来的にマイグレーション経由で廃止する（撤去タスクの一部として計画する。恒久削除への移行期は物理削除で代替する）。
- リスト削除は、配下タスク（完了済みを含む）を恒久削除する操作となる（従来の「未完了はゴミ箱送り+完了済みは保全」という論理削除モデルは採用しない）。

### 却下した代替案

- **リストの論理削除により完了履歴を裏で保持し続ける案**（2026-07-07データ保持原則の当初運用案）: リストという単位に限れば履歴保全は達成できるが、ユーザーが「削除」を実行したにもかかわらずデータが裏で保持され続ける点で、「削除」という言葉の意味に対して不誠実であり、プライバシー製品としての信頼性訴求と矛盾するため不採用とした。
- **ゴミ箱を維持しつつ、別途「完全削除」操作を追加する案**: 「論理削除（ゴミ箱）」「完全削除」「アーカイブ」の3概念が併存することになり、calm/シンプル原則（`docs/design/ui-spec.md`）に反し概念過多となる。また主要な競合製品（Todoist、Microsoft To Do、Apple リマインダー）はいずれもゴミ箱を持たず削除は原則即時的であり、市場慣習からみても本製品がゴミ箱を持つ必然性は薄いと判断し不採用とした。

プロダクトオーナー（2026-07-07）

## ADR-010: 削除同期は空blob tombstoneとして表現する

**日付**: 2026-07-08
**状態**: Accepted（2026-07-08 プロダクトオーナー承認）

**補正**: 2026-07-10のADR-012により、既知recordのcascade tombstone、pull前preflight、stable-key fuzzy scan + delta full resyncを必須化した。ADR-014によりwire/storage表現は`deleted=true`+空byte列から、`delete_hlc`を持ちblobを持たないtagged tombstoneへ置換した。2026-07-11のADR-016により、aggregate scope / epochは導入せず、180日のtombstone保持とserver-enforced device continuity、expired-device rebase、復号後のclient-side descendant cascadeで削除へ収束する方針に確定した。

### 背景

ADR-009はローカル削除の意味論を「恒久削除・削除Undoなし」へ改訂した。一方、Phase 2の同期プロトコルでは、オフライン端末へ削除を伝播するために一定期間のサーバー側tombstoneが必要である。ここで削除済みレコードの暗号blobを保持し続けると、ユーザーに対する「削除=本当に消える」という説明と、サーバーに削除済みコンテンツ暗号文が残る実態の間にズレが残る。

E2EEではサーバーは暗号blobを復号できないが、プライバシープロダクトとしては「サーバーに内容を読めない形で残す」ことと「内容を即時消滅させ、同期に必要な最小メタデータだけ残す」ことを区別して設計する。

### 決定

1. 削除は同期ストア上で `deleted=true` のtombstoneとして表現する。
2. tombstone行の `encrypted_blob` は空byte列とする。クライアントが削除pushで非空blobを送っても、サーバーは保存時に空byte列へ正規化する。
3. サーバーが削除後に保持する同期情報は、`tenant_id`、`record_id`、`collection`、`seq`、`hlc`、`deleted=true`、`updated_at` に限定する。削除済みコンテンツ本文やフィールドHLC入り暗号blobは保持しない。
4. tombstoneは180日保持し、定期ジョブでGCする。180日という窓はADR-005の従来案を踏襲し、通常の長期オフライン端末復帰を吸収するための保守的な初期値とする。
5. tombstone GC実行時、テナントごとに `gc_horizon_seq` を記録する。これは「この値以下のseqのtombstoneは削除済みであり、サーバーは削除伝播の完全性を保証しない」境界を表す。
6. continuity preflight時、serverが保持するdevice別continuity cursor / generationが `gc_horizon_seq` 未満ならexpiredとし、フル再同期を要求する。client申告の `since` だけでpush可否を決めない。新規profileの `since=0` はexpired対象外だが、全件取得はADR-012のfuzzy scan + delta resyncを用いる（2026-07-11改訂: ADR-016）。
7. GC窓超過の検知は `devices.last_pull_at` ではなく、serverが自身の発行したpull / full-resync closureのACKから単調更新するdevice continuityと `gc_horizon_seq` の比較で行う。`devices.last_pull_at` は長期未使用デバイスの失効候補検出などのテレメトリ用途に限定する（2026-07-11改訂: ADR-016）。
8. expired-deviceフル再同期では未ACK outboxを一律保護しない。server currentにあるlocal mutationはtyped merge / rebaseし、server ACK済みなのにcurrentに不在のrecordはGC済み削除として破棄する。never-synced creation originを持つrecordは、pending key bundleとdependency closureを検証できた場合だけ新規seedとして保護する（2026-07-11改訂: ADR-016）。
9. liveからtombstoneへのCAS過程ではADR-012のsemantic clockを用いる。serverがtombstoneを受理した後はterminal deletionとし、同一record IDの`tombstone → live`をHLCの大小にかかわらず拒否する（2026-07-11改訂: ADR-016）。
10. 削除Undo導線は作らない。ユーザーが同等の内容を再作成する場合は新しいrecord IDを発行する（2026-07-11改訂: ADR-016）。

用語補足: 「outboxに未ACKエントリを持つ」とは、具体的にはクライアントの `sync_outbox` テーブルに当該record_idの行が残存していることを指す（ACKはフィールドではなくpush受理時の行削除操作である）。

補足（2026-07-11）: 2026-07-08の「GC済み削除recordの未push編集を復活させる」トレードオフはADR-016で撤回した。現行規約ではserver-seen but absent recordをrebaseで破棄し、never-synced local-new recordだけを別条件で保護する。

補足（2026-07-08 オーナーレビュー第2弾由来）: `devices.last_pull_at` はリクエスト受信時に更新され得る一方、クライアントのpullカーソルは「取得データをローカルへコミットした後」に前進する。そのため、pull直後かつローカル反映前にクラッシュした端末では、`last_pull_at` が新しく見えても実際の継続性は古いカーソルに留まる。`last_pull_at` 基準の検知はこのケースを偽陰性にし、GC済みtombstoneの取り逃がしを見落とし得る。カーソル自体を `gc_horizon_seq` と比較すれば、端末簿記や時計に依存せず、サーバーが保証できるseq範囲との関係だけで正確に判定できる。

この決定は、`docs/03_技術仕様書.md` §6.6 に残っていた「deviceの最終pull時刻から検知」という従来案との差分である。ADR-010承認に伴い`gc_horizon_seq`基準へ改訂し、ADR-016でclient申告cursorではなくserver-trusted device continuityとの比較へ強化した。

### 実装範囲

task-73時点で実装するのは、削除pushのtombstone blob空化と、180日cutoffを受け取るGC関数のみとする。`deleted=true` の転送と受信側ローカル恒久削除はtask-72で実装済みである。`gc_horizon_seq`の永続化とfuzzy full resyncはtask-95で実装し、server-trusted continuity、terminal tombstone、expired-device rebase、late descendant cascadeはADR-016の後続実装taskへ送る（2026-07-11改訂）。

将来のフル再同期実装タスクでは、受け入れ基準として少なくとも以下の自動テストを必須とする。

- server-trusted device continuityが`gc_horizon_seq`未満の端末は、full resync closureとlocal rebaseを完了するまでkey bundle / entity pushを拒否されること。
- server-seen but absent recordの未push編集は破棄され、tombstone GC後も復活しないこと。
- never-synced origin、pending List DEK bundle、dependency closureを持つ長期offline中の新規list / taskは、full resync後も保護されること。
- server不在の既存list / parentへ依存するrecordはmaterialize / pushされないこと。
- pull直後・local反映前クラッシュではserver側continuityが進まず、次回runが削除reconcileを再実行すること。

### 帰結

- 削除後、サーバーからコンテンツ暗号blobは即時に消える。E2EEの「サーバーは読めない」に加えて、削除操作への説明も「同期に必要な最小メタデータだけが最大180日残る」と誠実に表現できる。
- 180日以内に復帰した端末はtombstoneをpullしてローカル恒久削除できる。
- 180日を超えて復帰した端末、または保持期限内でもserver-trusted device continuityが `gc_horizon_seq` 未満の端末は、tombstoneを取り逃がす可能性があるためフル再同期とlocal rebaseが必須になる（2026-07-11改訂: ADR-016）。
- GC窓超過判定は端末の最終pull時刻やclient申告cursorではなく、serverがclosure ACKから更新したdevice continuityで決める。これにより、pull後クラッシュや端末時計のずれによる誤判定を避けられる。
- tombstoneにはfield HLC mapが残らないため、liveからtombstoneへの競合はfield単位ではなくtop-level `delete_hlc` と `mutation_hlc` で扱う。tombstone受理後はterminalとし、後着編集で同一record IDを復活させない（2026-07-11改訂: ADR-016）。

### 却下した代替案

- **削除tombstoneにも暗号blobを残す案**: オフライン端末とのマージ材料は増えるが、削除済み内容の暗号文がサーバーに残り続けるため、ADR-009の「削除=本当に消える」という設計意図と相性が悪い。
- **削除時にサーバー行を即時物理DELETEする案**: ほかの端末が削除を知る手段を失い、オフライン端末復帰時に削除済みレコードが再pushされて復活しやすい。
- **tombstoneを無期限保持する案**: フル再同期の必要性は減るが、削除メタデータが永続化し、同期ストアが単調増加する。
- **tombstone後も新しいHLCのliveを受理する案**: 別端末の編集を保全できる一方、ユーザーが不可逆と確認した永久削除を復活させる。archive-firstで整理し、永久削除は例外的なterminal操作とするADR-016と相反するため不採用（2026-07-11改訂）。

## ADR-011: Flutterデスクトップ版とCLIは同一ローカルプロファイルを共有する

**日付**: 2026-07-10
**状態**: Accepted（2026-07-10 プロダクトオーナー承認）

### 背景

TodoriはFlutterデスクトップ版に加えてWindows / macOS / Linuxで動作するRust CLIを提供する。`core/domain`、`core/storage`、`core/crypto`、`core/sync` には再利用可能な実装がある一方、現状のクライアント初期化、repository操作、domain処理と同期outbox登録の組み合わせ、アカウント実行状態は `app/rust` に残っている。このままCLIを実装すると、Flutter bridgeとCLIの双方へアプリケーションサービスが複製され、操作結果や同期enqueueの差異が生じる。

また、SQLCipher DBを開くにはDBファイルだけでなく同じDevice Keyから導出したDB鍵が必要である。現状はApple platformのみData Protection Keychainが本番経路であり、Windows / Linuxは開発用の平文ファイルfallbackである。Flutterデスクトップ版とCLIが同じDBを利用するためには、共通データディレクトリ、OS secret store、SQLite多重プロセスアクセスを一体として設計する必要がある。

### 決定

1. 同一PC上のFlutterデスクトップ版とCLIは、同じTodori local profileを共有する。profileはSQLCipher DB、profile識別情報、OS secret store上のDevice Key参照を一単位として扱う。
2. macOS / Windows / Linuxの別PC間ではDBファイルまたはDevice Keyをコピーして共有しない。各PCが独立したlocal profileを持ち、端末間の状態は既存のE2EE同期で収束させる。
3. Flutter bridge、CLI、将来のMCP serverから共有するアプリケーションサービスcrateを `core/client`（Cargo package `todori-client`、Rust crate `todori_client`）として設ける。DB open、repository + domain + outbox enqueue、アカウント状態、同期実行の共通入口をここへ置く。公開runtime facadeは`TodoriClient`、local profileを開く設定は`LocalProfileConfig`とする。`app/rust` はFRB公開関数とDTO変換へ限定し、下位のcrypto/domain/storage/sync crateへ直接依存しない。`core/`はcrate群の配置ディレクトリであり、bare `core` crateは作らない（2026-07-10 task-91で境界を確定、task-94で公開型名を明確化）。
4. macOSのDevice Key / account secretはData Protection Keychainへ保存する。Flutter appとCLIは同一Apple Teamで署名し、同じKeychain access groupを持つ。未署名 `cargo install` バイナリは、同一profileへアクセスする正規配布形態としない。
5. WindowsのDevice Key / account secretはcurrent-user scopeのDPAPIで保護する。同一Windowsユーザー・同一PCのFlutter appとCLIが同じ保護済みsecretを復号する。
6. LinuxのDevice Key / account secretはSecret Service APIを使用する。Secret Service、session D-Bus、またはunlock手段が利用できない場合は明示的に失敗し、本番動作で平文ファイルへ自動降格しない。
7. 非Apple platformに残る `FileDeviceKeyStore` / file account secret storeは開発・テストfallbackに限定し、Windows / Linuxの本番Flutter desktopおよびCLIでは使用しない。
8. profileの既定パス解決は共通client/profile層へ集約する。明示的なprofile / data-dir指定を許可する場合も、OS secret storeの認可を迂回したり、Device Keyをコマンドライン引数・環境変数・ログへ露出したりしない。
9. 同一DBへのFlutter / CLI多重接続に備え、connectionごとの `busy_timeout`、短い書込transaction、migration排他を導入する。domain更新と同期outbox登録は同一transactionで確定する。
10. 同一profileで同期runを同時に複数プロセスから実行しない。DB-backed lease等のプロセス間調停を設け、lease取得失敗は構造化されたbusy状態として返す。
11. 本ADRの共有対象はWindows / macOS / Linuxのデスクトップfrontendである。iOS / AndroidアプリとデスクトップCLIが同じ物理DBを共有することは対象外とし、モバイルとのデータ共有はE2EE同期で行う。

### 帰結

- CLI実接続より先に、共通client/profile層の抽出、Windows DPAPI / Linux Secret Service実装、SQLite多重プロセス対策を完了させる。
- Flutter appとCLIのCRUDは同じdomain処理、repository更新、outbox enqueueを通り、frontend間の挙動差を抑制できる。
- macOS CLIの配布にはApple署名とentitlementが必要になる。単一実行ファイルを未署名で配るだけではFlutter appのData Protection Keychain itemへアクセスできない。
- Linux headless / SSH / systemd環境ではSecret Serviceが利用できない場合がある。初期実装は安全側に失敗し、必要な対話的unlockや専用headless profileは別途裁定する。
- SQLiteは複数processから開けるが、writer競合と同期多重実行をアプリケーション層で処理する必要がある。

### 却下した代替案

- **Flutter appとCLIが同一PC上でも別DBを持ち、常にサーバー同期する案**: secret storeとSQLite排他は単純になるが、オフライン時に同じPC内で状態が即時一致せず、CLI / MCPが「同じローカルTodoを操作する」という仕様に反する。
- **CLIがFlutter appへローカルIPCで接続する案**: DBと鍵の所有者を単一化できる一方、Flutter app停止中にCLIが動作できず、オフラインで独立動作するCLI要件を満たさない。将来、常駐desktop daemonを正式導入する場合に再検討する。
- **全OSで平文 `device.key` をDB横へ置く案**: 実装は単純だが、SQLCipher鍵を保護するDevice Keyの機密性を失い、暗号化DBの防御を実質的に弱めるため不採用とする。
- **DBファイルとDevice KeyをOS間でコピーする案**: Device Keyのnon-exportable / current-user保護方針と矛盾し、端末失効・監査・同期のデバイスモデルも崩すため不採用とする。

## ADR-012: 同期correctnessはfield clock・placement・transaction・snapshotを一体で保証する

**日付**: 2026-07-10
**状態**: Accepted（2026-07-10 プロダクトオーナー承認）

**補正**: 2026-07-10のADR-014により、latest-one encrypted blobのlive更新は`revision_hlc` LWWではなく`base_revision_hlc`付きCASとし、stale pushをclient merge/retryへ戻す方式へ改訂した。2026-07-11のADR-016により、aggregate削除scope / epochの後続要求は、server-visible hierarchy metadataを追加しないbounded tombstone + device continuity + expired-device rebaseへ置換した。

### 背景

ADR-004 / ADR-005は、暗号payload内のfield-level LWW、最新状態方式、client再push、server seq cursorを採用した。ADR-010は空blob tombstoneとGC horizonを採用し、ADR-011はdomain更新とoutbox enqueueを同一transactionへ置く共通client層を決定した。

一方、production実装のレビューで、通常CRUDが全fieldへ同一HLCを再付与するためfield-level LWWがrecord-level LWWへ退化すること、`sort_order` が同期されないこと、domain更新とoutboxが別commitであること、階層削除がroot/listのtombstoneしか生成しないこと、復号不能recordをskipした後にcursorを進めることが確認された。また、ADR-010の `since=0` だけでは、GC後の全件同期を有限かつ連続なcursorへ接続するためのsnapshot/high-waterが不足する。

これらは個別bugではなく、同期correctnessを成立させる不変条件の欠落である。GCとフル再同期を先に実装すると、不完全なlocal stateと削除意味論を固定化するため、correctness基盤を先に再定義する。

### 決定

1. 暗号payload内の `field_hlcs` は各field値の競合解決だけに使う。local mutationは実際に変更したfieldのHLCだけを更新し、未変更fieldのHLCを保持する。
2. serverがlive blob同士のlatest envelopeを採用するためのtransport clockを `revision_hlc` としてfield clockから分離する。local mutationとmerge後の再pushは `revision_hlc` をtickするが、再pushだけを理由に全field HLCを引き上げない。pullで受信したremote revisionをlocal HLCへobserveし、merge再pushのrevisionは受信revisionより必ず大きくする。observe、採番、HLC永続化、repush用outbox登録は同一local transactionで確定する。wire protocol v2で現行top-level `hlc` の名称と意味を移行する。
3. live recordは最後のdomain/user変更を表す `mutation_hlc`、tombstoneは削除操作を表す `delete_hlc` をtop-level semantic clockとして持つ。merge再pushは `mutation_hlc` を更新しない。live同士のenvelope採用は `revision_hlc`、liveからtombstoneへのCAS過程は `mutation_hlc` と `delete_hlc` を用いる。ADR-016に従い、tombstone受理後の同一record IDはliveへ復活させない。
4. taskの配置は `list_id`、`parent_task_id`、`sort_order` を分離fieldとして競合させず、単一の `placement` 値として1つのfield HLCで原子的に更新する。listの `sort_order` も同期対象のplacementとする。
5. fractional rankは同期payloadへ含める。同一gapへの同時insertでrankが衝突しても全端末が同じ順序へ収束し、衝突後も新しいrankを生成できるtie-break / collision repairを実装要件とする。具体的な符号化と依存選定は後続taskで確定する。
6. domain row、対応するoutbox entry、同期用plaintext state、local HLC永続化は、共通client層が所有する同一SQLite transactionで確定する。失敗時は全体をrollbackし、local-only commitまたはoutbox-only commitを作らない。
7. pull pageは、復号と検証に成功した適用内容、sync record state、必要なmerge repush用outbox、cursorを同一local transactionで確定する。HTTP、key refresh等のnetwork I/Oはtransaction外で行い、cursorだけを先行commitしない。
8. continuity / protocol preflightをpayload applyから分離する。missing DEKはkey bundleをrefreshして同一recordを再試行し、なお不足する場合は暗号blobと理由をdurable quarantineへ保存してcursorを進め、key取得後に再適用する。認証復号失敗もdurable quarantineと明示的なcorruption状態を必須とする。未知protocol versionはupgrade-requiredとしてpayload取得とpushを停止する。silent skipは行わない。
9. task subtree/list削除時、削除端末が把握しているrootとdescendantについては、削除前にrecordを列挙し、各recordの空blob tombstoneをdomain削除と同じlocal transactionでoutboxへ登録する。別端末の未同期・同時作成recordは、ADR-016に従い復号後のancestor検証でdomainへmaterializeせず、client-side cascade tombstoneへ収束させる。
10. aggregate削除のためのserver-visible scope ID / epochは追加しない。tombstone保持期間とoffline復帰保証期間を180日で接続し、expired deviceはfull resync closureとserver-seen / local-new / dependency rebaseを完了するまでpushできない。同一record IDのtombstoneはliveへ復活せず、明示的な再作成は新IDを使う。List DEK bundleはADR-016のkey retirement条件が実装されるまで削除しない。
11. sync runはpayloadを扱う前にcontinuity / protocolをpreflightし、通常のremote pull / delete reconcileまたはexpired-device full resync / rebaseをlocal outbox pushより先に完了する。server-trusted device continuityが`gc_horizon_seq`未満ならkey bundle / entity pushを拒否する。個別recordのquarantineは無関係なlocal変更の送信を永久停止させない（2026-07-11改訂: ADR-016）。
12. full resyncは厳密な過去snapshotではなく、stable-keyによるcurrent-state fuzzy scan + delta catch-upとして定義する。開始時にserver transaction内で `base_seq = tenant_seq.last_seq` を取得し、base tableを `seq <= base_seq` に限定せず、`(collection, record_id)` 等の更新で移動しないstable keyで走査する。base中にpage cursorの後方へ作成されたrecordは `seq > base_seq` のdeltaで回収する。
13. delta pageはpage rowsと同じserver transactionで `high_water = tenant_seq.last_seq` を取得する。`has_more=false` かつclient cursorがそのpageの `high_water` に到達した時だけcatch-up closureとする。closure後の更新は通常deltaで取得する。
14. clientはresync generationをlocalに作り、base scanとdelta catch-upで確認したrecordをmarkする。expired-device rebaseでは、server currentにあるrecord、server-seen but absent record、never-synced local-new record、missing/deleted dependencyを分類し、未ACK outboxを一律保護しない。適用・rebase/sweep・最終cursor確定は失敗時に再試行可能なtransaction境界を持つ。最終cursorはclosure時のserver high-waterとする（2026-07-11改訂: ADR-016）。
15. 既存local dataを持つprofileを新規tenantへ登録するときは、tenant/profile紐付け後、全local recordをnever-synced creation origin付きtransactional seed generationへ登録する。その後にfull resync closureを行い、pending List DEK bundle → list → taskの依存順でpushする。各境界のcrash recoveryをテストする（2026-07-11改訂: ADR-016）。
16. login成功だけを理由に同一tenantのinitial backfillとpull cursorを無条件resetしない。local DB/profileをtenant identityへ結び付け、同一accountへの再login、account切替、新規profileを区別する。
17. push responseはrequest順を暗黙に信用しない。protocol op ID、結果件数、record IDを照合してからoutboxをACKする。collectionはprotocol enumで制限し、既存recordのcollection変更を拒否する。
18. property testと2-client統合testはproduction CRUD / common client経路を通す。テスト専用clientがfield clockやmergeを本番より正しく実装する状態を許容しない。
19. serverのtenant分離はqueryの `tenant_id` と認可だけでなく、non-owner application role、実際のRLS policy、必要な `FORCE ROW LEVEL SECURITY`、cross-tenant testで検証する。RLSを有効化しただけの状態を多層防御完了と表現しない。

### 実装順

1. production経路を通る失敗テストと、ADR-011の共通client transactionを実装する。
2. `revision_hlc` / `field_hlcs`、placement、push response / collection validationを実装する。
3. typed pull failure、durable quarantine、cursor transactionを実装する。
4. fuzzy base scan + delta full resyncとGC horizonを実装する。
5. ADR-016のserver-trusted device continuity、expired-device rebase、terminal tombstone、client-side descendant cascade、key retirementを実装する。
6. non-owner DB roleとRLS policyを実装・検証する。

前段が未完了のまま後段をリリース対象へ昇格しない。

### 帰結

- ADR-004の自前同期、ADR-005の最新状態+server seq、ADR-010の空blob tombstone、ADR-011の共通profileは維持する。
- wire protocol、local schema、application serviceの変更を伴うため、既存server/client間の互換境界をprotocol v2として明示する必要がある。protocol v2は少なくとも `revision_hlc`、`mutation_hlc`、`delete_hlc` を区別する。
- 現行2-client testと `cargo test -p todori-sync` の成功だけではproduction correctnessの証拠にならない。
- full resyncとGC horizonはcorrectness基盤の後続になり、直近着手候補から一段後ろへ移る。

### 却下した代替案

- **現行実装へGC/full resyncだけ追加する案**: field clock、outbox、削除cascade、cursor skipの欠陥を温存したままabsence sweepを導入し、データ損失経路を増やすため不採用。
- **再push時に全field HLCを新revisionへ揃える案**: 未確認の同時編集より古いfield値を新HLCで上書きでき、field-level LWWの意味を失うため不採用。
- **transport revisionで削除競合も判定する案**: user mutationを伴わないmerge再pushで削除を復活させ得るため不採用。
- **fractional rankを同期せず各端末で再計算する案**: 同じtask集合でもユーザーが行ったreorderを復元できず、端末間の表示順が収束しないため不採用。
- **復号不能recordを計数してskipする案**: 一時的なkey取得失敗と恒久corruptionを区別せず、cursor前進によって回復不能になるため不採用。
- **full resyncを動的なseq pageだけで実装する案**: updateでrowのseqが移動し、空結果時のhigh-waterも得られず、有限で一貫したabsence sweepを定義できないため不採用。
- **clientが列挙したdescendantのACKだけでList DEK bundleを削除する案**: 別端末の未知recordとlate descendantを覆えず、復号・依存分類の前に鍵を失うため不採用。

## ADR-013: local crypto contextをremote sessionから分離する

**日付**: 2026-07-10
**状態**: Accepted（2026-07-10 プロダクトオーナー承認）

### 背景

task-83でaccount-boundなtask editをtransactional clientへ移したが、再起動時の鍵復元は有効なsession tokenを前提とし、List DEKをRAMへ復元しない。そのためnetwork障害やsession期限切れだけでlocal mutationが不能になる。一方、鍵復元不能を匿名利用とみなすと、account dataをoutboxなしで更新するsilent local-only commitが起きる。

### 決定

1. local mutationに必要な`LocalCryptoContext`をremote認証sessionから分離する。session tokenは同期HTTPの認可にだけ使い、期限切れや一時的なnetwork不通をlocal編集不可の理由にしない。
2. Master Keyは従来どおり`wrap(MK, Device Key)`だけをOS secret storeへ保存し、平文を永続化しない。
3. local DBはtenant / user / deviceのprofile bindingをsingletonとして持つ。同じDBへ異なるtenantを上書きloginせず、別profileを要求する。
4. List DEKは平文を追加保存せず、list IDをAADへ含めたlocal専用`wrap(List DEK, MK)` bundleをtenant ID / list ID単位でSQLCipher DBへ永続cacheする。cacheからの復元はprofile bindingと一致する行だけを一括検証し、欠落・破損・ciphertextのlist間入替・別tenant混入を部分成功として扱わない。
5. login、register、key refreshで全bundleの復号検証に成功してからlocal cacheをtransactionalに置換し、その後runtimeをReadyにする。refresh失敗時は既存cacheを維持する。
6. profile状態は`Anonymous`、`AccountBoundReady`、`AccountBoundUnavailable`を区別する。account bindingがあるのにMKまたは必須List DEKを復元できない場合は、すべてのmutationを永続化前にfail closedする。
7. session期限切れはaccount bindingとlocal key cacheを保持し、local mutationを許可する。明示logoutはremote session secretを削除するが、account bindingを匿名へ降格させない。local keyを端末から除去する操作は、local dataの扱いを確認する別の明示的な「この端末からアカウントを削除」として将来設計する。
8. list削除時にcached List DEK bundleを削除しない。ADR-016のexpired-device rebase、late descendant収束、key retirement条件が実装されるまで保持する。
9. 将来のoffline list作成は、List DEK生成、MK-wrap cache、domain row、entity outbox、key-bundle upload queueを同一local transactionで確定する。key bundleをentity blobより先にidempotent uploadする。network I/Oをwrite transaction内で行わない。
10. serverのlist-key upsertはstale clientによる既存bundle上書きを無条件に許可しない。offline list作成を実装するtaskでcreate-if-absentまたはkey generation条件をprotocolとして固定する。

### 帰結

- app再起動、session期限切れ、network不通でも既存List配下のlocal編集とtransactional outbox生成を継続できる。
- OS secret storeのMK wrapまたはSQLCipher内のwrapped DEK cacheを失ったprofileは、匿名fallbackせず明示的に編集不能になる。
- task/list CRUDのtransaction移行と、offline list作成のkey-bundle upload queueは分けて実装できる。
- session復元とlocal crypto復元を同じruntime stateへ押し込めず、共通client/profile層へ段階的に移す必要がある。

### 却下した代替案

- **List DEK平文をsettingsへ保存する案**: SQLCipherで保護されるが、既存のMK key hierarchyを迂回する不要な平文永続化になるため不採用。
- **起動ごとにserverからList DEKを取得する案**: session期限・network可用性をlocal編集へ結合し、offline-first要件を満たさない。
- **鍵復元不能時に匿名mutationへfallbackする案**: account-bound dataへoutboxなしの変更を作るため不採用。
- **明示logoutでaccount bindingも消す案**: 同じDB内のaccount dataが匿名dataへ見える状態を作るため不採用。

## ADR-014: protocol v2はbase revision CASでACK済み変更を保全する

**日付**: 2026-07-10
**状態**: Accepted（2026-07-10 プロダクトオーナー承認）

### 背景

ADR-012はfield clockとtransport用`revision_hlc`を分離した。しかしserverは暗号blobを復号・field mergeできないため、同じbaseからAがtitle、Bがnoteを編集し、AのACK後にBの大きいrevisionがblind overwriteすると、Aが停止した時点でtitle変更を知るclientが消える。field clockは失われたblobを復元できない。

Todoriはまだrelease前であり、既存wire、開発server data、local sync metadataとの互換を維持する必要はない。互換層を足すより、correctnessを満たすv2へ直接置換する。

### 決定

1. live/tombstoneを問わずpush opは`base_revision_hlc`を持ち、server current revisionとのCASで更新する。新規recordだけbaseを`None`とする。
2. currentとbaseが不一致ならserverはblobを上書きせず`conflict`とcurrent envelopeを返す。clientはcurrentを復号・mergeし、remote revisionをobserveして新しいrevisionで再pushする。
3. `revision_hlc`はCAS revision兼transport順序であり、blind LWWの採用キーにしない。retryのexact envelopeはno-op、同revisionで内容が異なる場合はcorruption/conflictとする。
4. wire stateはnullable fieldの組合せではなく、`Live { mutation_hlc, blob }`と`Tombstone { delete_hlc }`のtagged unionにする。tombstoneはblobを持てない。
5. live/live merge後の`mutation_hlc`は両payloadの最大値かつmutable field clockの最大値とする。merge repushはfield clockとmutation clockを変えず、revisionだけを進める。
6. liveからtombstoneへのCAS過程は`mutation_hlc`と`delete_hlc`で裁定し、revisionだけが大きいrepushで削除を復活させない。ADR-016に従い、tombstone受理後の同一record IDに対する`Live`は拒否する。accepted revisionは許可されたstate遷移で単調増加させる（2026-07-11改訂: ADR-016）。
7. local outboxはrecordごとに未送信headを1件へcoalesceし、`op_id`、base/current revision、semantic clock、stateを永続化する。送信中に新mutationが入った場合、古いresponseは`op_id`不一致なら新headをACKしない。
8. push responseは順序を契約にせず、件数、`op_id`一意性、record ID、collectionを全件検証してからACKする。pullのinvalid base64を空blobへ変換せず、response failureとする。
9. collectionはprotocol enumとし、既存recordのcollection変更を拒否する。server schemaはlive/tombstone clockとblob形状をCHECK制約で強制する。
10. taskのcompletionは`status/completed_at/closed_reason`、placementは`list_id/parent_task_id/rank`をそれぞれcompound fieldとして同期する。list placementもrankを1 fieldとして持つ。任意mapより型付きplaintextを正本とする。
11. rankは固定幅128-bitの32桁lowercase hexへ置換し、表示順は`(rank, record_id)`で決定する。同rankまたは隣接rank間への挿入時だけ、同一scopeを現在順のままtransactionalに再配番する。
12. protocol v1互換は実装しない。sync HTTPは`/v2`、encrypted envelopeもv2へ切り替え、v1 dual-read/writeやdeserialize aliasを持たない。server sync tablesは開発環境で再作成し、local migrationはdomain rowを保持してv1 outbox/stateを破棄しv2 seedを再生成する。

### 実装分割

1. task-86でshared wire、CAS server、op ID照合、local v2 sync schema/outbox coalesce、conflict merge/rebaseを実装する。
2. 後続taskで型付きfield clock、completion/placement、固定幅rank、transactional reorder/rebalanceを実装する。
3. production common clientを通る2-client testで、先にACKしたclientを停止しても別field変更が消えないことをrelease gateにする。

### 帰結

- CASなしの`revision_hlc` LWWを採用しない。
- serverはplaintextを知らないまま、ACK済みheadをstale clientがblind overwriteすることを防げる。
- v2移行中は開発DB/server dataの再作成を許容し、互換コードを保守しない。
- task-86だけではchanged-field clockとplacementは未完了であり、後続task完了まで同期correctness全体をrelease-readyとしない。

### 却下した代替案

- **大きいrevisionを常に採用する案**: ACK済みの別field変更を含まないblobで上書きできるため不採用。
- **server historyから後で復元する案**: 通常同期のcorrectnessを運用上の復旧へ委ね、端末が収束しないため不採用。
- **v1/v2 dual write案**: release済みclientがなく、複雑性だけを増やすため不採用。

### 裁定者

プロダクトオーナー（2026-07-10、release前の破壊的設計変更を許可）

## ADR-015: default Inboxは認証済み候補のUUID最小へ収束する

**日付**: 2026-07-10
**状態**: Accepted（2026-07-10 プロダクトオーナー承認）

### 背景

各端末はlocal DB初期化時にdefault Inboxを作るため、同じaccountへ複数端末を接続すると異なるIDの`is_default=true` listが同期され得る。task-79では同期失敗を避けるため、先にlocalへ存在したdefaultを残し、後からpullしたdefaultのdomain rowだけを`is_default=false`へdemoteした。一方、認証済みsync plaintextは`is_default=true`のまま保持している。

この暫定処理は受信順によって勝者が変わり、重複listを通常listとして表示し得る。ユーザー体験としてはaccount内にInboxを1つだけ表示し、すべてのdefault Inbox配下taskを同じInboxへ収束させる必要がある。serverは暗号blobを解釈せず、Inboxというsemantic metadataも追加で保持しない。

### 決定

1. canonical候補は、liveでstrict typed検証に成功したlist sync plaintextのうち、認証済み`is_default.value == true`であるrecordだけとする。list名、受信順、local domain rowのdemotion状態は判定に使わない。
2. 候補のUUIDをbyte順で比較し、最小のlist IDをcanonical Inboxとする。UUIDはrecordの不変identityであり、全端末が同じ候補集合を得れば同じ結果になる。`created_at`やfield HLCを選挙条件にしない。
3. 選挙とmergeは、initial pull / full resyncがclosureまで完了し、そのclosureに含まれるserver current headがlocalで分類済みである場合だけ行う。`collection=lists`かつlive headのunresolved quarantineが1件でもあれば、そのrecordがdefault候補か判定できないため選挙を保留する。以前の有効plaintextがlocalにあっても、より新しいlive headがquarantine中なら代用しない。unsupported version、pull hard failure、key refresh failureでも永続mergeを開始せず、次の成功したsyncで再試行する。
4. local domainではcanonicalだけを`is_default=true`としてmaterializeする。その他の候補はcanonicalへのlocal aliasとしてdurableに記録して通常list一覧から隠すが、認証済みsync plaintextの`is_default=true`は候補identityとして保持する。serverへcanonical pointer、alias、Inbox名を平文metadataとして追加しない。
5. canonical切替は1つのowned `BEGIN IMMEDIATE`で行う。同じtransactionに、旧canonicalを含むloser domain rowのdemotion、新canonicalのpromotion、durable alias集合の置換、全既知alias taskのtyped placement更新、canonical List DEKでの再暗号化、task record state、HLC、outboxを含める。aliasをUIから隠すのはcommit後とし、鍵不足またはwrite失敗時はcanonical表示を含む全変更をrollbackする。
6. mergeはlogin後のbootstrapと通常sync後に冪等に再評価する。遅れて到着した、またはoffline端末から追加されたalias配下taskも同じ規則でcanonicalへ移す。後からより小さいdefault UUIDが到着した場合は候補集合から再選挙し、新しいcanonicalへ収束する。
7. alias listとList DEK bundleは、fuzzy full resyncとADR-016のexpired-device rebase、late descendant収束、key retirement条件が実装されるまで削除しない。UIから隠すことをserver recordの安全な削除完了とみなさない。
8. 新規accountは登録時にdefault InboxとList DEKを1組だけ作り、追加端末はそれを取得する。通常listが名前を`Inbox`または`インボックス`へ変更されても`is_default=false`なら候補に含めない。

### 帰結

- 正常なbootstrap後は各端末のUIにInboxが1つだけ表示され、既知taskはcanonical切替transactionのcommit後に同時にcanonicalへ見える。
- 他端末の永続状態はmerge outboxのpush / pull後に収束する。network断やkey不足時は既存recordを保持して再試行する。
- task-79のarrival-order demotionは、認証済み候補全体からUUID最小を選ぶmaterializationへ置き換える。
- 恒久実装はoffline list key-bundle queueと、既存profileを安全に列挙するfuzzy full resyncの後続とする。

### 却下した代替案

- **最初にlocalへ存在したInboxを採用する案**: pull順序で端末ごとの勝者が変わるため不採用。
- **list名がInboxのrecordをまとめる案**: ユーザー作成listを誤ってmergeし、localeやrenameにも依存するため不採用。
- **serverへcanonical Inbox IDを平文保存する案**: server-visibleなsemantic metadataを増やさずclientだけで決定できるため不採用。
- **aliasを直ちにtombstone化してList DEKを削除する案**: offline端末の未知taskを復号不能または復活状態にし得るため不採用。

## ADR-016: Archive-first削除同期はbounded tombstoneとexpired-device rebaseで行う

**日付**: 2026-07-11
**状態**: Accepted（2026-07-11 プロダクトオーナー承認）

### 背景

ADR-009は、完了済みtaskを振り返りのための記録資産とし、通常の整理を`done` / `wont_do`とリストのアーカイブ、永久削除をミス操作やノイズを本当に消すための例外操作と定めた。ADR-010は削除済みコンテンツblobを残さないtagged tombstoneを180日保持し、GC horizonを超えた端末をfull resyncへ送る方針を採用した。

ADR-012は、別端末の未知descendantをserverが即時拒否する完全なaggregate削除のため、scope deletion intent / epochの別裁定を後続要求とした。しかし、serverが暗号payload内の`list_id` / `parent_task_id`を知らず、offline端末が未知recordを後から送れる条件で、serverがそのrecordを削除済みaggregateの一部と即時判定するには、scopeまたはhierarchyに相当するserver-visible metadataが必要になる。これは、削除が例外操作であるTodoriに対して、E2EE metadataとprotocol complexityを恒常的に増やす。

一方、tombstoneを永久保持すればscope metadataなしで同一record IDの復活を防げるが、削除済みIDが無期限に増え、削除痕跡をserverが永久保持する。Todoriでは、tombstone保持期間をoffline deviceの通常復帰保証期間と一致させ、期間を超えた端末はserver current stateへrebaseする方が、archiveの長期保全と削除metadataの有限保持を分離できる。

### 決定

1. 製品上の通常整理は、taskの`done` / `wont_do`とリストのアーカイブとする。永久削除は、誤作成または残したくない情報を本当に消すための例外的・不可逆な操作とし、ADR-009の深い導線と追加確認を維持する。
2. Phase 2の個人tenantでは、aggregate scope ID、scope epoch、list ID、parent ID、ancestor setをserver-visible同期metadataとして追加しない。serverはrecord間の親子・所属関係を解釈しない。
3. 削除はADR-014の`Tombstone { delete_hlc }`として表現し、content blobを持たない。tombstone受理時は対象recordの旧contentを`sync_records_history`へ新規退避せず、既存historyも削除する。serverが保持するのは削除伝播に必要な最小metadataだけとする。
4. tombstone保持中、同一record IDの`tombstone → live`遷移はHLCの大小にかかわらず拒否する。永久削除した内容を再作成する場合は新しいrecord IDを発行する。
5. task subtree / list削除時は、削除端末が知るrootとdescendantを同一local transactionで物理削除し、各recordのtombstoneをoutboxへ登録する。遅れて到着した未知taskはclientが復号後に`placement.list_id` / `parent_task_id`とlocal tombstone stateを検証し、削除済みancestorへ到達する場合はdomainへmaterializeせず、自身のtombstoneをtransactionalにenqueueする。missing dependencyは通常taskとして表示・pushしない。
6. 通常syncは、protocol / continuity preflight、remote pullと削除reconcileをentity outbox pushより先に行う。これにより、offline端末が受信済み削除を知らずにstale descendantを先行pushする経路を閉じる。必要なkey refreshは復号reconcileの前、local-new List DEK bundle uploadはrebase後かつentity push前に行う。
7. tombstoneの初期保持期間と通常offline復帰保証期間は180日とする。GC時は`gc_horizon_seq`を単調前進させる。保持期間を超え、server-trusted device cursorがhorizon未満の端末はexpired continuity状態とする。
8. push endpointはclient申告だけを信用せず、認証済みdeviceについてserverが確定したcontinuity cursor / full-resync generationが`gc_horizon_seq`以上であることを検証する。expired deviceはfull resync closureとlocal rebaseを完了するまでkey bundle / entity pushを行えない。対応前protocolのclientはversion gateで拒否する。
9. expired-device full resyncはremote current stateをclosureまで取得した後、local recordを次のように分類する。
   - server currentがliveのrecord: local mutationがあればcurrent revisionへtyped merge / rebaseし、成功したものだけを新outbox headとして保持する。local mutationがなければremote currentを適用する。
   - server currentがtombstoneのrecord: local live domain rowとoutboxを破棄し、remote tombstone stateを保持して同一record IDを復活させない。
   - 一度serverへ存在したがcurrent stateに不在のrecord: tombstoneがGC済みの削除recordとみなし、local domain row、record state、outboxを削除して再pushしない。
   - 一度もserverへ存在しないlocal-new record: durable local origin、未ACK creation、必要なpending key bundle、dependency closureを検証し、正当なものだけを新規seedとして保持する。
   - serverに不在の既存list / parentへ依存するrecord、またはdependency chainを解決できないrecord: domainへmaterializeせず、outboxから除外して削除へ収束させる。
10. local-new判定を`base_revision_hlc = null`だけから推測しない。local sync stateは、server ACK済みrevisionまたはnever-synced creation originをdurableに区別する。outbox coalesce、再起動、crash recovery後もこの区別を失わない。
11. 長期offline中に新規作成したリストは、never-synced local origin、local List DEK、pending wrapped List DEK bundleを持ち、IDがremote current / tombstoneと衝突せず、配下recordのdependency closureがそのlocal-new集合内で成立する場合、full resync後も保持してbundle → list → taskの順に再seedする。server不在だけを理由に削除しない。
12. List DEK bundleは、list tombstone保持中、late descendantの復号・分類中、またはexpired-device rebase / key retirement条件が未実装の間は削除しない。retirementは、tombstoneがGC可能で、active deviceが削除seqを通過またはexpiredとなり、full resync closureで既知current recordが分類済みで、expired deviceのstale pushがserverで拒否される場合に限る。詳細なtransactionとAPIは後続実装taskで固定する。
13. offline端末は接続するまで遠隔消去できない。Todoriが保証するのは、次回syncでcontinuity / protocol gateを通り、削除済みancestor配下recordを表示・再pushせず、server current stateへ収束することである。
14. Organization共有で悪意あるmember clientをserverがscope単位に遮断する要件が生じた場合は、Phase 3の認可・脅威モデルとしてopaque capability / scope metadataを別ADRで再評価する。個人tenantの削除例外のために先行導入しない。

### 帰結

- ADR-012のaggregate scope deletion intent / epoch後続要求は撤回し、server-visible hierarchy metadataを追加しない。
- task-95のstable-key fuzzy full resync、high-water closure、GC horizonは維持するが、未ACK outboxを一律保護する規則を、server-seen / local-new / dependency closureを区別するexpired-device rebaseへ拡張する。
- normal sync順、server-trusted device continuity、terminal tombstone、history purge、local origin永続化、client-side late descendant cascadeは後続の重要変更taskで実装する。
- archive recordはユーザーの長期資産として保持される一方、削除伝播用tombstoneは180日とdevice continuity条件で有限にGCできる。
- ADR-015のcanonical Inbox alias / List DEK bundleは、aggregate scopeではなく本ADRのlate descendant収束とkey retirement条件へ依存する。

### 却下した代替案

- **aggregate scope ID / epochを全recordへ追加する案**: serverによる即時拒否は可能だが、例外的な削除のために所属metadataとprotocol complexityを全recordへ恒常的に追加するため不採用。
- **tombstoneをtenant存続中永久保持する案**: 実装は単純だが、削除済みIDと削除痕跡が無期限に蓄積するため不採用。
- **expired deviceのlocal dataを全件破棄する案**: 削除復活は防げるが、長期offline中に正当に作成した新規リストとtaskまで失うため不採用。
- **full resyncで未ACK outboxをすべて保護する案**: local-newは救済できるが、GC済み削除recordと削除済みlist配下taskを再pushし得るため不採用。
- **List DEK bundleをlist削除と同時に削除する案**: late descendantを復号・分類できず、暗号blobを回復不能なquarantineへ送るため不採用。
- **entity pushを通常pullより先に行う案**: offline端末がserver上の削除を受信する前にstale recordを送るため、archive-first削除同期では不採用。

## ADR-017: タスク期限は日付のみと日時指定を排他的に区別する

**日付**: 2026-07-12
**状態**: Accepted（2026-07-12 プロダクトオーナー承認）

### 背景

現行のtaskは`due_at: Option<i64>`を持ち、日付だけを選んだ場合も端末ローカル日の開始をUTC epoch millisecondsへ変換して保存している。そのため、値の型だけでは「その日中に終えればよい」のか「その瞬間までに終える」のかを判別できない。タイムゾーンを跨いだ表示やToday分類でも、日付というユーザー意図が別の日へずれ得る。

一方、`scheduled_at`は作業を始める予定日時であり、完了期限ではない。通知の`remind_at`もユーザーへ知らせる瞬間であり、期限そのものではない。この3つを流用せず、期限の意味をdomainで明示する必要がある。

### 決定案

1. taskの期限は`Option<TaskDue>`とし、`TaskDue`を次の排他的なtagged unionとする。
   - `Date { due_on: CivilDate }`: 「この日までに」。`CivilDate`はproleptic Gregorian calendarの`YYYY-MM-DD`であり、時刻・offset・timezoneを持たない。
   - `DateTime { due_at: UtcInstant, time_zone: IanaTimeZone }`: 「この時刻までに」。`due_at`は一意な瞬間、`time_zone`は入力・編集・曖昧性表示に使うIANA timezone IDである。
2. `Date`と`DateTime`を同時に持たせない。日時指定には日付も含まれるため、独立した2期限ではなく1つの期限モードとして切り替える。
3. raw epoch値をdomain / client / FRB / Dartの期限APIとして公開しない。`CivilDate`、`UtcInstant`、`IanaTimeZone`を検証済み型として扱う。local DBはrange query用に`UtcInstant`をepoch millisecondsで保持してよいが、これはstorage表現に閉じ込める。
4. local DBは`due_kind`、`due_on`、`due_at_ms`、`due_time_zone`を持ち、CHECK constraintで次だけを許す。
   - 期限なし: 全列NULL。
   - 日付のみ: `due_kind='date'`、`due_on`だけ非NULL。
   - 日時指定: `due_kind='datetime'`、`due_at_ms`と`due_time_zone`だけ非NULL。
5. 日付のみの期限切れ判定は`due_on < viewer_local_today`とする。`23:59:59.999`等の擬似instantへ変換しない。日時指定の期限切れ判定は`now >= due_at`とする。
6. Today分類は次とする。
   - 日付のみ: `due_on == viewer_local_today`。
   - 日時指定: `due_at`をviewerの現在timezoneへ変換したlocal dateがtoday。期限切れは時刻単位で別途明示する。
   - `scheduled_at`がTodayへ含まれる既存要件は期限とは別条件として維持し、後続UI実装時に重複表示を防ぐ。
7. 同じlocal dateに属する期限は、未来の日時指定を`due_at`昇順、その後に日付のみを置く。日付のみは「その日中」を意味するため、同日の特定時刻より前に期限切れ扱いしない。期限切れ同士は期限切れになった順が古いものから並べる。
8. sync plaintextは`due: Clocked<Option<TaskDue>>`を1 fieldとして暗号化する。tag、値、timezoneへ別々のHLCを付けず、1回の期限編集をatomicにmergeする。
9. `scheduled_at`は開始予定instant、`remind_at`は通知instantとして残し、`TaskDue`へ統合しない。日付のみ期限へ通知を付ける場合、通知側が明示的な`remind_at`を持つ。
10. UIは期限設定時に「日付のみ」と「日時指定」を明示する。日付のみは時刻を表示せず、日時指定は時刻を必ず表示し、viewer timezoneと入力timezoneが異なる場合はtimezoneまたは換算後時刻を曖昧にならない形で示す。
11. 一般リリース前のため、旧`due_at`の値を推測変換しない。local DB schema、FRB API、sync plaintext / protocolをfinal designへbreaking変更し、既存開発profileとserver開発データは再作成する。dual read/write、legacy fallback、midnight値からの種別推測は実装しない。

### 帰結

- 日付だけの期限は端末timezone変更やDSTで別日へずれない。
- 時刻までの締切は一意なinstantとして通知・期限切れ判定でき、入力時timezoneも失わない。
- storage、sync、UIは期限種別を推測せず、domainのtagを正本にできる。
- DB schema、同期plaintext、FRB、Dart provider、Today / sort、作成・編集UIを同時にbreaking更新する重要変更taskが必要になる。
- `docs/01_企画書.md`と`docs/02_機能仕様書.md`の「期限」「締切日時」「Today」表現は、裁定後に人間承認付きで本契約へ同期する必要がある。

### 却下する代替案

- **既存`due_at`の00:00を日付のみと推測する案**: 本当に00:00締切のtaskと区別できず、timezone変換後は00:00でなくなるため不採用。
- **`due_on`と`due_at`を独立nullable fieldにする案**: 両方が同時に残る無効状態と、別端末編集時のfield-clock競合を生むため不採用。
- **日付のみをその日の23:59:59.999へ変換する案**: timezoneとDSTに依存し、日付という意図をinstantへ不可逆に潰すため不採用。
- **`scheduled_at`を時刻付き期限として流用する案**: 開始予定と完了期限が同じ意味になり、Today・通知・所要時間計画を正しく表せないため不採用。
- **日時指定にtimezoneを保存しない案**: instantとしての判定はできるが、入力した現地時刻の編集と旅行時の表示意図が失われるため不採用。

## ADR-018: Timer計測状態とtask statusを分離し、完了work sessionだけを同期する

**日付**: 2026-07-13
**状態**: Accepted（2026-07-13 プロダクトオーナー承認）

### 背景

Timer / Pomodoro / Focusの実行状態と、Kanbanでユーザーが管理する`todo` / `in_progress` / `done` / `wont_do`は異なる概念である。Timer開始を`in_progress`へ結合すると、計測を試しただけでtask workflowが変わり、別端末の明示的なstatus操作とも競合する。また、秒単位で変化するactive sessionを同期すると、background復帰、clock drift、同時利用の分散排他が同期correctnessへ侵入する。

一方、完了した作業時間はユーザーの成果であり、複数端末で見積対実績を振り返れる必要がある。ADR-007は`timer_sessions`をtenant-scope entity、Tenant Root DEKをその暗号鍵と定めたが、local cache / runtime供給と同期recordの厳密な形は未実装である。

### 決定

1. task statusとTimerは直交する。Focus / Pomodoro / Stopwatchのstart / pause / resume / finish / discardはtask statusを変更せず、特に`in_progress`を自動設定しない。Focus中にユーザーがtaskを完了する場合は現在のwork sessionを保存終了してから既存task status APIを明示的に呼ぶ。task completionのUndoはTimerを自動再開しない。
2. active sessionは端末ローカルのSQLCipher DBだけへdurable保存し、1 device 1 activeをsingleton制約で保証する。stateは`running` / `paused`、phaseは`work` / `short_break` / `long_break`、modeは`pomodoro` / `stopwatch`を持つ。workだけtask IDを必須とし、breakはtask IDを持たない。
3. active durationの正本は`accumulated_active_ms`と`last_resumed_at`である。running復元時は`accumulated_active_ms + max(0, now_wall_ms - last_resumed_at)`、paused時は`accumulated_active_ms`とする。wall clock逆行分は0へclampし、固定tick回数やprocess uptimeを永続値にしない。1回のactive spanと累積値は0以上、最大7日とする。
4. 同期対象はwork phaseを完了した、またはユーザーが明示的に「中断を保存」したimmutable `CompletedTimerSession`だけとする。fieldは`id`、`task_id`、`mode`（pomodoro / stopwatch）、`finish_kind`（completed / interrupted）、`started_at`、`ended_at`、`active_duration_ms`、`created_at`である。`started_at <= ended_at <= created_at`、`0 < active_duration_ms <= ended_at - started_at <= 7日`、各timestampが表現可能なUTC instantであることを必須とする。activeの完了確定はID、task、mode、work phase、開始時刻がcompleted値と一致する場合に限り、completed row・同期outbox/state・active clearを同一transactionで行う。Pomodoroのtarget / break設定は端末ローカル設定でありsessionへ保存せず、short / long breakの計測も実績同期しない。
5. completed sessionは作成後immutableとする。同record IDの同一plaintextは冪等no-op、異なるplaintextはUUID collisionまたはcorruptionとしてmergeせずquarantineする。編集は旧sessionをtombstone化し、新UUIDのsessionを作る。live / tombstone競合はADR-014 / ADR-016に従い、serverがtombstoneを受理したrecord IDはterminalでlive復活を拒否する。
6. task / list永久削除では、削除前にlocalで関連completed sessionを列挙し、各session tombstone、task/list tombstone、domain物理削除、record state、outbox、HLCを同一`BEGIN IMMEDIATE`で確定する。serverはtask_idを知らないためcascadeを行わない。削除端末が認識していないlate sessionは、復号したclientがlocal task tombstoneを検出した時点でmaterializeせずsession tombstoneをenqueueする。
7. `timer_sessions`は新しい`SyncCollection`であり、plaintextは`kind=timer_session`と上記immutable fieldsを持つ。task status、task title、list ID、Pomodoro設定を保持しない。serverはcollection、record ID、opaque encrypted blob / tombstone、CAS / continuity metadataだけを扱う。
8. timer session blobはTenant Root DEKでXChaCha20-Poly1305暗号化する。既存envelope v3とAAD`todori-sync-envelope/v2\ncollection:{collection}\nrecord_id:{record_id}`はtask / listと同じwire契約として維持し、collection / record ID swap、wrong Tenant Root DEKを拒否する。Timer plaintext variant追加だけを理由にenvelopeを変更・再暗号化しない。Tenant Root DEKのkey versionは1とする。
9. 個人tenantのTenant Root DEKは既存どおりserverへ`wrap(TenantDEK, MK)`で保存する。端末ではMKで再wrapした`wrapped_tenant_root_dek`をSQLCipher内へ保存し、AADを`todori/wrap/local-tenant-root-dek-by-mk/v1/{tenant_id}`とする。平文Tenant Root DEKはaccount runtimeの`Zeroizing` memoryだけに置く。binding済みprofileでcacheを復元できない場合は匿名fallbackせずTimer同期mutationを拒否する。
10. local schemaをv18、sync protocolをv5へ更新し、encrypted envelopeはv3のまま維持する。v18はactive singleton、completed timer sessions、local Tenant Root DEK cacheを作り、local sync metadata tableをCHECK拡張のためtransactionalにrebuild-copyする。既存outbox、record state、cursor、quarantine、full-resync state / marks、origin、terminal tombstoneを失わない。server migrationはcollection CHECKとstable cursor CHECKへ`timer_sessions`を追加する。旧protocolのdual read/write、deserialize alias、fallbackは実装しない。
11. active state、session row、key cache、record state、outbox、HLCのwrite failureは全rollbackする。秘密鍵、plaintext session、task title、durationをlogへ出さず、errorは型とrecord IDまでに限定する。

### 帰結

- ユーザーはKanban statusをTimerから独立して管理できる。
- background / process restartはlocal durable wall-clock stateだけで復元でき、秒ごとの同期やdistributed lockが不要になる。
- 完了work sessionはtenant内でE2EE同期され、serverはtask linkageや実績時間を解釈できない。
- Tenant Root DEKのlocal供給が、将来のtags / templates / schedulesにも再利用できる。
- v18 / protocol v5はcollection enumのbreaking cutoverだが、envelope v3と既存server head / local tombstone系譜は保全される。

### 却下した代替案

- **Timer開始時にtaskを`in_progress`へ変更する案**: Kanban上の明示状態と一時的な計測を混同するため不採用。
- **active sessionを同期する案**: 秒単位更新、clock drift、multi-device leader electionが必要になり、offline-firstの単純性を失うため不採用。
- **breakを作業実績として同期する案**: taskへ費やしたactive workを過大計上するため不採用。
- **completed sessionをmutable LWW recordにする案**: 成果時間が別端末編集で静かに変わるため不採用。訂正はtombstone + 新recordとする。
- **timer sessionをList DEKで暗号化する案**: list移動で再暗号化が必要になり、tenant-scope集計とADR-007に反するため不採用。
- **Tenant Root DEKをSQLCipher内へ平文保存する案**: MK hierarchyを迂回するため不採用。
