## 1. 背景とコンテキスト

Todoriの開発は4フェーズのロードマップに分けられている（`docs/01_企画書.md` §8）。Phase 1は「ローカル専用で提供。シンプルUI、Todo・リスト・サブタスク・完了・Won't Do機能、デバイスキーによる保存時暗号化を実装する」MVPフェーズと定義されている。

しかし企画書のこの記述は概要レベルであり、実装可能な粒度（マイルストーン、タスク、依存関係、完了条件）には分解されていない。このタスクは、Phase 1の範囲を機能仕様書（`docs/02_機能仕様書.md`）の該当F番号と突き合わせながら実装可能な粒度に分解し、`docs/07_Phase1計画書.md` として文書化する。

**このタスクはコード変更を一切伴わない、ドキュメント作成のみのタスクである。**

## 2. 事前に読むべきファイル

- `docs/01_企画書.md` §8「ロードマップ」（Phase 1〜4の区分と、Phase 1の概要記述）
- `docs/01_企画書.md` §9「リスクと対策」（計画書のリスク欄の参考にする）
- `docs/02_機能仕様書.md` のうち、Phase 1関連と想定される以下のF番号: F-01〜F-11（UIモード、タスク管理、リスト・タグ・全文検索）、F-24〜F-26（ローカル通知）、F-43（アプリロック）、F-48〜F-53（多言語対応、アクセシビリティ、非機能要件）
- `docs/03_技術仕様書.md` §3（データモデル）、§5（ローカルストレージ）
- `docs/tasks/task-01-opaque-poc.md`、`docs/tasks/task-02-sqlcipher-poc.md`、`docs/tasks/task-03-flutter-rust-bridge.md`（これらPoCの結果がPhase 1計画のどこに影響するかを明示するため。実行済みであれば結果を、未実行であれば前提条件として扱う）

## 4. スコープ

### やること

`docs/07_Phase1計画書.md` に以下の内容を含める。既存の `docs/01〜04` と同様、章立てのMarkdown文書として作成すること（既存ドキュメントの体裁・見出しレベルの付け方に倣う）。

1. **Phase 1スコープの確定**
   - 対象機能を明記する: シンプルUI（F-02）、リスト・タスクのCRUD（F-05, F-09）、無制限階層のサブタスク（F-06）、done/wont_do のステータス遷移（F-06）、ゴミ箱・Undo（F-07）、並び替え（F-08）、ローカル通知（F-24〜F-26）、Device Key + SQLCipherによる保存時暗号化（`docs/03_技術仕様書.md` §5, §7.1）、i18n基盤（en/ja、F-48）。
   - 対象プラットフォームは**iOS先行**（決定済み、docs/06_事業・法務方針.md §6 参照。開発ホストがLinuxのためiOSビルドはmacOSランナー/実機が必要である点を計画に織り込むこと。開発中のdogfooding用にLinuxデスクトップビルドを維持する提案は歓迎）。
   - 全文検索（F-11）・タグ（F-10）・アプリロック（F-43）・アクセシビリティ（F-49）・クラッシュレポート（F-53）など、対象F番号に含まれるが企画書のPhase 1概要記述には直接現れない項目については、Phase 1に含めるか次フェーズに送るかを明確に判断し、判断理由を記載する。
2. **スコープ外の明示**
   - 同期・アカウント機能（F-27〜F-31）、Organization共有（F-32〜F-35）、課金（`docs/04_課金設計書.md` 全体）、カレンダー表示（F-13）、Pomodoro/タイマー（F-16〜F-18）、テンプレート・繰り返し（F-19〜F-21）、ローカルAI（F-36〜F-39）、MCP/CLIの実機能（F-40, F-41。ただし雛形crate自体は既に存在する点に触れる）が明確にPhase 1のスコープ外であることを明記する。
3. **マイルストーン分割**（3〜5段階の例。実装可能な粒度になるよう調整してよい）
   - M1: コア層完成 ── `core/domain` のエンティティ・ユースケース、`core/storage` のSQLCipher統合とリポジトリ実装（task-02の成果を前提とする）
   - M2: ブリッジとUI骨格 ── `flutter_rust_bridge` 統合（task-03の成果を前提とする）、画面遷移の骨格、状態管理方針の確立
   - M3: 機能完成 ── リスト/タスクCRUD、サブタスク、ゴミ箱/Undo、並び替え、シンプルUIの本実装
   - M4: 通知・暗号化・磨き込み ── ローカル通知の実装、DK生成・SQLCipher鍵管理の本実装、i18n基盤の適用、非機能要件（起動性能F-50、描画性能F-51、オフライン動作F-52）の検証
   - （必要であればM5として「リリース準備」等を追加してよい）
4. **各マイルストーン内のタスク表**
   - 各マイルストーンごとに、`ID / 内容 / 依存タスクID / 完了条件` の4列を持つ表を作成する。
   - 完了条件は「〜のテストが通ること」「〜画面から〜操作ができ、DBに反映されること」のようにテスト可能な粒度で記述する。曖昧な表現（「適切に動作する」等）を避ける。
5. **PoC結果への依存関係の明示**
   - task-01（OPAQUE認証PoC）は同期・アカウント機能に必要なものでありPhase 1（ローカル専用）では直接使わない旨を明記する。ただし将来のPhase 2移行を見据えた `core/crypto` の設計に影響しうる場合はその旨を記載する。
   - task-02（SQLCipherビルド検証）の結果は M1 の前提条件であることを明記する。
   - task-03（flutter_rust_bridge統合）の結果は M2 の前提条件であることを明記する。
   - 各PoCが未実施・未完了の場合は、計画書内に「前提: task-0Xの完了待ち」と明記し、実施状況に応じて計画書を更新すべき旨の注記を入れる。
6. **リスクと先送り判断の記録欄**
   - `docs/01_企画書.md` §9のリスク表を参考に、Phase 1固有のリスク（例: Flutter Desktopの通知APIの成熟度、SQLCipherのプラットフォーム別ビルド、UIモード切替の設計負債化等）を列挙し、対策または「Phase 1では対策せず先送りする」という判断を明記する表を作成する。

- `docs/tasks/` 配下の既存ファイルの変更。
