# 技術仕様書

> Technical baseline: 2026-07-23 product owner承認。ADR-023参照。
>
> 本書はTaskveilの技術的な唯一の真実源である。詳細な不変条件はリンクする`docs/redesign/`文書を本書のnormative supplementとして扱う。本書とsupplementが矛盾する場合は本書を優先し、同じwork itemで矛盾を解消する。
>
> 既存implementation、wire protocol、local/server schema、development dataとの互換性は要件にしない。実装は本baselineへbreaking置換する。

## 1. Architecture

### 1.1 System boundary

```text
┌─────────────────────────────────────────────────────┐
│ Official client                                     │
│ UI / local index / notification / timer             │
│ application core / crypto / sync / encrypted DB     │
└───────────────────┬─────────────────────────────────┘
                    │ TLS + signed E2EE envelopes
┌───────────────────▼─────────────────────────────────┐
│ Taskveil server                                     │
│ OPAQUE/session / RBAC / key envelopes / opaque      │
│ Record CAS / cursor / quota / generic wake-up       │
└───────────────────┬─────────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────────┐
│ Database / backup / push provider                   │
│ No Task plaintext or plaintext content key          │
└─────────────────────────────────────────────────────┘
```

Official clientだけがdomain plaintext、search index、notification content、Account/Space keyを扱う。Serverはauthentication、authorization、availability、opaque storageを提供するがE2EE trust boundaryではない。

### 1.2 Client architecture

現在のFlutter UI + Rust shared core方向を維持する。

- Flutter: UI、platform integration、accessibility、local notification。
- Rust application core: typed domain、crypto、local storage、sync、Account/Space state machine。
- Flutter/CLI/将来clientは同じapplication-core contractを使い、frontendがkey/repository/sync coordinatorを直接保持しない。
- platform secure storage adapterだけをOSごとに分離する。
- official client間で同じwire encoding、test vector、domain validationを共有する。

具体的crate/module名はimplementation refactorで決める。本baselineは既存module互換を要求しない。

### 1.3 Server architecture

- Stateless API processとtransactional relational databaseを基本とする。
- authentication/session、Space RBAC、membership transition、Record CAS、quotaをdatabase transactionで強制する。
- server process、database、backup、operatorのいずれもcontent decryption keyを持たない。
- deployment provider、region、realtime transportはE2EE protocolから分離する。
- generic wake-upは欠落可能なhintであり、sync正本ではない。

## 2. Core terminology and scope

| Term | Technical meaning |
|---|---|
| Account | login、recovery、device/session、stable cryptographic identityの単位 |
| Space | membership、Space Key generations、sync cursor、quota attributionの境界 |
| Personal Space | Account ownerだけがmemberであるprivate Space。Accountごとに1つ |
| Shared Space | owner/editor/viewerが参加するSpace。初期UIでは1共有list |
| Record | serverがkind/plaintextを解釈せずcurrent headを保存する暗号化同期単位 |
| Revision | 1 Record headのversion。random IDとbase revisionを持つ |
| Operation | retry idempotencyの単位 |
| Key generation | membership/security changeで更新するSpace Key世代 |
| Tombstone | contentなしのterminal permanent-delete Record |
| Completion Record | 1回のTask completeを残すimmutable Record |
| Work Session | 完了した作業計測を残すimmutable Personal Space Record |
| Conflict Record | same-field conflictの両candidateをE2EE保存するRecord |

Organization、tenant、workspace root key、attachment objectはinitial modelに存在しない。

## 3. Domain model

### 3.1 Record kinds

Record kindはciphertext内のtyped discriminantである。

- `list`
- `section`
- `task`
- `tag`
- `comment`
- `completion`
- `work_session`
- `template`
- `saved_filter`
- `space_setting`
- `conflict`

Unknown kind/schema versionはdomainへmaterializeせずupgrade-requiredまたはquarantineとする。任意JSON fallbackを作らない。

### 3.2 Task

Task plaintextは少なくとも次を持つ。

```text
id
title
note
url
status: todo | in_progress | done | wont_do
priority
due: none | date_only | instant(timezone)
scheduled_start
duration_estimate
list_id
section_id
parent_task_id
rank
tag_refs
reminders
recurrence
assignee_ref
created_at
updated_at
```

Rules:

- Relationは同じSpace内に限定する。
- parent cycle/self-parentを拒否する。
- due variantをatomic fieldとして扱う。
- status/completion reference、placement、recurrenceをそれぞれcompound fieldとしてmergeする。
- `done`/`wont_do`をdeleteと解釈しない。

### 3.3 Completion Record

```text
id
task_id
command_id
completed_at
actor_account_id | local_actor
display_snapshot
reversal_or_replacement_ref | null
created_at
```

- immutableである。
- complete commandはTask status revisionと同一local transaction/server batchに入れる。
- reopenはCompletion Recordを削除しない。
- immediate unsynced undoを除き、訂正はrelationを持つnew Recordで表す。

### 3.4 Work Session

```text
id
task_ref | null:
  space_id
  record_id
task_display_snapshot | null
mode: stopwatch | pomodoro | manual
started_at
ended_at
active_duration
finish_kind
note
correction_ref | null
created_at
```

- Personal Spaceだけに保存する。
- immutableで、訂正はreplacement/correction relationとする。
- Personal Taskへの参照は同じSpaceのrelation、Shared Taskへの参照は`(space_id, record_id)`のsoft external referenceとする。
- Shared Taskの削除、member removal、access喪失ではcascade deleteせず、encrypted display snapshotとunavailable stateで個人成果を残す。
- active timer、pause、remaining、breakはRecordではなくdevice-local stateとする。

### 3.5 Mutable and immutable

| Kind | Mutation |
|---|---|
| list/section/task/tag/template/saved_filter/space_setting | mutable current head |
| comment | editable current head、edit marker |
| completion/work_session | immutable、correction/reversalはnew Record |
| conflict | candidate immutable、resolution stateのみ更新 |
| Tombstone | terminal、liveへ戻さない |

## 4. Server data model

Server schemaはdomain tableではなくopaque storageとして設計する。

### 4.1 Account/session

```text
accounts
  account_id
  email
  opaque_registration/config
  public_identity_bundle/version
  wrapped_account_root_key
  recovery_wrapped_account_root_key
  wrapped_identity_private_bundle
  wrapped_personal_space_keyring
  account_security_revision
  plan/quota state

sessions
  session_id
  account_id
  device_id
  push_token
  created/last_seen/revoked

account_security_transitions
  account_id
  transition_id
  one_time_challenge
  expected_security_revision
  pending_object_digests
  account_signature
  expires/state
```

Password、Recovery Key、ARK、private identity keyは保存しない。

### 4.2 Space/membership

```text
spaces
  space_id
  type: personal | shared
  owner_account_id
  membership_revision
  current_key_generation
  server_seq
  encrypted_byte_usage
  state

memberships
  space_id
  account_id
  role: owner | editor | viewer
  identity_version
  state
  joined/removed time

space_key_envelopes
  space_id
  generation
  recipient_account_id
  recipient_identity_version
  hpke_ciphertext
  owner_signature

membership_transitions
  space_id
  membership_revision
  canonical_manifest
  manifest_version/digest
  previous_manifest_digest
  transition_kind
  key_envelope_set_digest
  owner_signature
  target_acceptance_signature | null
  committed_at/stored_bytes

invitations
  invite_id
  space_id
  created_by_account_id
  expiry/state
  package_version
  encrypted_invitation_package
  accepting_account_id | null
  stored_bytes
```

Personal Spaceはowner Account以外のmembershipを拒否する。Shared Spaceはexactly-one-ownerをdatabase constraint/transactionで維持する。
Signed membership transition historyは過去Recordのauthor/role検証とmanifest chain検証のためSpace存続中保持し、quotaへ算入する。Serverはcanonical manifest/transition bytesとsignatureを検証・保存できるが、Shared Space表示名やRecord relationは得ない。

### 4.3 Record head

```text
record_heads
  space_id
  record_id
  revision_id
  base_revision_id
  operation_id
  protocol_version
  crypto_suite_id
  key_generation
  author_account_id
  author_identity_version
  nonce
  ciphertext
  padding
  author_signature
  tombstone
  server_seq
  stored_bytes
```

- primary identityは`(space_id, record_id)`。
- current headまたはTombstoneを1件だけ持つ。
- Record kind、domain field、relation、user timestampのcolumnを作らない。
- every-edit ciphertext historyを無期限保持しない。

### 4.4 Delta log and idempotency

```text
space_delta_log
  space_id
  server_seq
  record_id
  revision_id
  tombstone

operation_results
  space_id
  operation_id
  request_digest
  result
```

Delta logはboundedでよい。cursorがlower boundより古いclientはcurrent headsのfull resyncを行う。Tombstone自体はSpace存続中Record headとして保持する。

## 5. Cryptography and key management

Normative detail: `docs/redesign/cryptography-and-key-management.md`。

### 5.1 Protocol families

| Purpose | Baseline |
|---|---|
| Password authentication | RFC 9807 OPAQUE-3DH候補 |
| Password hardening | Argon2id KSF、mobile benchmark後にparameter固定 |
| Key derivation | HKDF、purpose/domain separation |
| Content/wrap AEAD | XChaCha20-Poly1305候補 |
| Member key delivery | RFC 9180 HPKE、X25519 family候補 |
| Signature | RFC 8032 Ed25519 |
| Encoding | deterministic CBOR profile候補 |
| Random | OS CSPRNG、random key/ID/nonce |

Exact ciphersuite、parameter、library、binary formatはimplementation前のADR/test vectorで固定する。Protocol familyをad-hoc raw ECDH/custom cipherへ置換しない。

### 5.2 Key hierarchy

```text
Password
  -> OPAQUE export key
  -> Account Unlock Key (AUK)
  -> wraps Account Root Key (ARK)

Recovery Key
  -> Recovery Wrap Key
  -> wraps same ARK

ARK
  -> wraps Account Identity private bundle
  -> wraps Personal Space Key generations

Shared Space Key generation N
  -> HPKE envelope per active Account

Space Key + Space/Record/purpose context
  -> HKDF Record Encryption Key

Device Local Wrapping Key
  -> protects local DB key and cached ARK
```

Password/email/Account IDからARK/Space Keyを導出しない。Space generationは互いに独立したrandom keyとし、hash chainにしない。

### 5.3 Account identity

Account identity bundle:

- Account ID、identity version。
- Ed25519 signing public key。
- X25519 HPKE recipient public key。
- previous identity statement、self/old-key signature。

Private bundleはARKでwrapする。Normal rotationはold keyがnew bundleを署名する。Old keyを失ったresetはunsigned security resetとしてShared Spaceでremove/reinvite/rekeyを要求する。

Initial modelはAccount-level identityでありdevice certificate treeを持たない。1 deviceのAccount private key compromiseはAccount compromiseとして扱う。

### 5.4 Record envelope

Conceptual outer envelope:

```text
protocol_version
crypto_suite_id
space_id
record_id
revision_id
base_revision_id
operation_id
key_generation
author_account_id
author_identity_version
nonce
ciphertext
padding
author_signature
```

- Record kind/domain fieldはciphertext内に置く。
- Header canonical bytesをAADに含める。
- Header、nonce、ciphertext全体をauthorが署名する。
- clientはsignature、AAD/AEAD、canonical encoding、schema、domain invariantの順で検証する。
- unknown suite/version、signature/AEAD failureをfail closedする。

### 5.5 Password/recovery

- Registration/loginでpasswordをserverへplaintext送信しない。
- OPAQUE export keyからAUKをdomain-separated deriveする。
- Password changeは同じARKをnew AUKでrewrapし、new OPAQUE record/wrapped ARKのdigestとexpected account security revisionをAccount signing keyで認可する。
- Recovery KeyでARKとAccount signing keyを復元し、new OPAQUE record、new wrapped ARK、new recovery-wrapped ARKのdigest、Account ID、one-time challenge、expected security revisionをbindしたsigned recovery transitionを作る。
- Serverはcurrent public identity keyでtransition signatureを検証し、authentication record、wrapper、security revision、challenge消費、既存session revokeをatomic CASする。署名のないregistration/wrapper置換とpartial updateを拒否する。
- Password、Recovery Key、unlock済みdeviceをすべて失った場合、supportも旧dataを復旧しない。

## 6. Local storage

### 6.1 Requirements

- domain table、outbox、base plaintext、Space key cache、search index、active timerをencrypted local DBへ保存する。
- Device Local Wrapping KeyをOS secure storageで保護する。
- ARK/Space Key plaintextをDBへ直接保存しない。
- log、analytics、crash report、clipboardへsecret/plaintextを出さない。

SQLCipherはcurrent implementation candidateとして維持できるが、schema/key capsuleは本baselineへbreaking更新する。Exact local-store designは別ADRで固定する。

### 6.2 Transaction boundary

Domain mutation、materialized state、encrypted candidate、outbox、local logical counterを1 local transactionでcommitする。Network I/Oをtransaction内で行わない。

Account-bound keyを復元できないprofileはanonymous/local-onlyへfallbackせず、account-bound mutationをfail closedする。

### 6.3 Search and notification

- Search indexはlocal encrypted DB内に置く。
- Reminder schedulingとnotification body generationはclientで行う。
- extension/processへAccount/Space keyを広く共有しない。

## 7. Sync protocol

Normative detail: `docs/redesign/sync-model.md`。

### 7.1 Server validation

Serverはmutationごとに次を検証する。

1. authenticated sessionとactive Space membership。
2. owner/editor role。Viewer write拒否。
3. request Accountとauthor Accountの一致。
4. author public keyによるouter signature。
5. current key generation。
6. supported protocol/suite/header/size。
7. operation idempotency。
8. base revision CAS。
9. Tombstone terminal rule。
10. atomic quota。
11. head/delta/cursor/quota/resultのsingle transaction commit。

ServerはAEAD plaintext、Record kind、relation、Task statusを検証しない。

### 7.2 Sync cycle

1. Session/account security revision確認。
2. Membership/current generation/key envelope refresh。
3. Cursor delta pull。範囲外ならfull resync。
4. Signature/decrypt/schema/domain verify。
5. Transactional materialization。
6. Local outbox rebase/merge/current-generation encryption。
7. Dependency-ordered CAS push。
8. CAS conflictのtyped 3-way merge。
9. Materialization commit後のcursor advance。

Membership/key refreshをentity pushより前に行う。

### 7.3 Typed three-way merge

`base`、`local`、`remote current`を使う。

- 片側だけが変えたfieldはその値。
- 両側が同じ値へ変えたfieldはその値。
- 両側が異なる値へ変え、domain ruleがなければremote currentをprovisional表示し、両candidateをConflict Recordへ保存する。
- status/completion reference、due、placement、recurrenceはcompound fieldとしてmergeする。
- tag/assignee collectionはelement operationで独立変更をmergeし、same-element add/removeはremove-wins + Conflict Recordをbaselineとする。
- arbitrary JSON field clockとgeneric LWWを使わない。

### 7.4 Atomic batch

次は1 Space内のall-or-nothing server batchとする。

- Task complete + Completion Record。
- 同じSpaceのRelated Recordを含むpermanent delete。
- Recurring occurrence settlement。
- Membership/owner/key-generation transition。

Cross-Space moveはdistributed transactionにせず、copy confirmation後にsource archive/deleteするresumable workflowとする。

### 7.5 Tombstone and full resync

- Tombstoneはcontentなし、terminal、Space存続中保持、quota対象。
- Same record IDをliveへ戻さない。
- Delta log範囲外/new device/corruptionはcurrent-head full resyncを行う。
- Staging verify後にlocal stateをatomic swapする。
- Never-synced local Recordを黙って捨てない。
- Server-acknowledged rowの不可解な欠落を自動re-uploadしない。

### 7.6 Referential integrity

- Task parent/list/section/tag/assignee等のstructural relationはSpace内に限定し、clientがcycle/missing targetを検証する。
- Personal Work SessionからShared Taskへの参照だけをsoft external referenceとして許可し、通常のdependency/cascade対象にしない。
- Shared Taskが削除済み、参照不能、またはaccess喪失ならWork Sessionのencrypted display snapshotを表示し、linkをunavailableとする。Work Session自体は削除しない。

## 8. Sharing protocol

Normative detail: `docs/redesign/sharing-model.md`。

### 8.1 Roles and manifest

- Roleは`owner`、`editor`、`viewer`。
- Shared Spaceにはexactly one owner。
- Ownerだけがmembership、role、key generation、transfer、deleteを変更する。
- Signed membership manifestはSpace、revision、generation、owner、members/role/identity fingerprint、previous digest、transitionを含む。
- Owner transferはtarget acceptance signatureも含む。
- Server CASでsame revisionから1 transitionだけをcommitする。

### 8.2 Invitation

- Ownerはrandom invite IDと256-bit invitation secretを生成する。
- ServerへはsecretでAEAD暗号化したinvitation packageだけを置く。
- SecretはURL fragmentまたはQRに置きHTTP requestへ送らない。
- Recipientはowner identityをpinし、自身のsigned identity acceptanceをsecretで返す。
- Ownerはrecipient identityへhistorical keyringをHPKE encryptする。
- Inviteはone-time、expiry、cancel、rate limitを持つ。

Exact message format/security proofはimplementation前ADRで固定する。

### 8.3 New member historical access

- New memberへ未削除Recordに必要な全historical Space Key generationsを配布する。
- Join時刻でserver-side content filteringしない。
- New memberはjoin前を含むcurrent headsをdecryptできる。
- Tombstoneはcontentを持たないためpermanent deleted dataは読めない。

### 8.4 Removal and progressive re-encryption

Removal transaction:

1. Fresh random Space Key generation `N+1`。
2. Removed memberを除くactive memberへのHPKE envelopes。
3. Owner-signed manifest transition。
4. Serverによるaccess revoke、manifest/generation/envelopeのatomic commit。
5. Removed Account read/writeと全old-generation write拒否。

Old Recordをremoval時にbulk re-encryptしない。Readだけではrewriteせず、次回edit時にwhole Recordをcurrent generationで暗号化する。Historical key generationsはold Recordとfuture new-member accessのため保持する。

### 8.5 Leave and owner transfer

- Non-owner leaveではread/writeを止め、owner rekey完了までShared Space writeをfreezeできる。
- Ownerはtransfer/deleteなしにleaveできない。
- Owner transferはold owner proposal + target editor acceptance。
- Old owner -> editor、target -> owner、quota attribution、manifest revisionをsingle transactionで更新する。

## 9. Timer and recurrence

### 9.1 Active timer

- Device-local encrypted stateとして保存する。
- 同一local profileで1 active timer。
- Tick、pause、break、remainingをsyncしない。
- Cross-device distributed lock/handoffをinitial protocolへ入れない。

### 9.2 Completed work

- Completed Work SessionだけをPersonal Spaceへ同期する。
- Shared Task relationを持てるがWork Session自体をShared Spaceへ置かない。Relationはsoft external referenceであり、Task削除やaccess喪失後もsnapshot付き成果記録として残す。
- Immutable + correction/replacement relationとする。

### 9.3 Recurrence

- Rule、timezone/anchor、exceptionはencrypted plaintext内に置く。
- Clientがoccurrenceを決定的に生成/deduplicateする。
- Serverはschedule、next time、Task kindを知らない。
- Same occurrenceのretry/concurrent generationはidempotent operation/secret-derived identifierで収束させる。Exact constructionはtest vectorで固定する。

## 10. Server-visible metadata and push

Normative inventory: `docs/redesign/threat-model.md`。

Server-visible:

- Account/email、OPAQUE registration/config、account security revision、MFA/plan/quota。
- Public identity bundle/version/signatureと、wrapped ARK/recovery-wrapped ARK/wrapped private bundle/wrapped Personal Space keyringの存在・version・size。
- Password/recovery transition ID、challenge、expected revision、pending object digest、Account signature、result。
- Device/session、IP、push token、created/last-seen/revoked time。
- Space type/ID/state、membership/role/owner/identity version、membership revision、generation、key envelope、server sequence/delta lower bound。
- Membership manifest/transitionのcurrent/history、manifest/previous digest、transition kind、key-envelope-set digest、owner/target acceptance signature、version/size/retention。
- Record/revision/base revision/operation ID、protocol/suite、author Account/key version、nonce、ciphertext/padding/signatureとsize、server sequence、stored bytes、Tombstone flag。
- Delta/idempotencyのrecord/revision/operation ID、request digest、result、retention state。
- Invitation ID、Space ID、creator Account、state/expiry、package version、encrypted package/size、accepting Account。
- Request timing/frequency/result、abuse telemetry、encrypted byte usage、quota result、generic push target/timing。

Not visible:

- Record kind、private list count。
- Task/list/tag/comment plaintext。
- status、date、reminder、recurrence、hierarchy、assignee。
- timer/task relation、duration、time。
- search term、filter、notification body。

Push payloadはopaque wake-up token/coarse reasonだけを持つ。Push欠落はdelayだけを生み、polling/manual/full syncで収束する。

## 11. Quota and retention

- 課金Accountのencrypted structured dataは約1 GiBを安全上限目安とする。
- Stored ciphertext、signature、key material、metadata、Tombstoneをactual bytesで計上する。
- Shared Space bytesはowner Accountへ1回だけ計上する。
- Batch commit前にquotaをatomic検査し、partial acceptしない。
- Quota exceededでもpull、local read/edit、storage-reducing delete、exportを許可する。
- Outboxをdurable保持し、retry stormをbackoffする。
- Completed dataをserverが自動purgeしない。
- Every-edit ciphertext historyは無期限保持しない。
- TombstoneはSpace存続中保持する。

Backup expiry、logical deletion SLA、free quota、billing detailは別の公開/非公開運用設計で固定する。

## 12. Threat model and non-guarantees

Normative detail: `docs/redesign/threat-model.md`。

Protect:

- Network interception、server snapshot/operator/server compromiseからのcontent confidentiality。
- AEAD/signatureによるtamper/author forgery detection。
- Honest server RBACとnew generationによるpost-removal new-content confidentiality。
- Offline local correctnessとconflict preservation。

Do not guarantee:

- Compromised endpoint、fake client/update。
- 正規memberがcopyしたplaintextのcontrol。
- Removed memberからpast plaintext/old keyのremote erase。
- Availability、traffic-analysis resistance、membership hiding。
- New clientを含むcomplete global fork/transparency detection。
- Message-by-message forward secrecy/post-compromise security。
- Single-server OPAQUE credential compromise後のlow-entropy password guessing完全防止。
- Initial post-quantum security。

## 13. Versioning and migration

独立version空間:

- OPAQUE configuration。
- Account/key envelope。
- Record protocol。
- Crypto suite。
- Plaintext schema。
- Local DB schema。
- Export format。

Rules:

- Clientはminimum supported versionとobserved highest security versionをdurable保持する。
- Server-provided downgradeをsilent acceptしない。
- Unknown suite/schemaをtrial-and-error fallbackで開かない。
- Algorithm migrationはnew suite/generationをnew/edited Recordへ適用する。
- Existing implementationからのdual read/write、compatibility alias、legacy fallbackは明示要求なしに作らない。
- Development dataのrecreateを許容する。

## 14. Quality and release gates

Implementation前後に次を必須とする。

- Maintained crypto library、license、audit、platform supportのreview。
- RFC/official vector + Taskveil cross-platform vector。
- Wrong key/AAD/ID/generation、swap、truncation、unknown version、invalid signatureのnegative test。
- OPAQUE/Argon2 mobile benchmarkとDoS evaluation。
- Invitation、add/remove/leave/owner-transferのmodel/state-machine test。
- Sync CAS、same-field conflict、quota、full resync、Tombstoneのfailure injection。
- Secret logging/zeroization review。
- Official client interoperability。
- External cryptographic/security design review。

Independent reviewが完了するまで本baselineをrelease-readyとしない。
