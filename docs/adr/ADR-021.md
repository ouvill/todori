# ADR-021: テンプレートと繰り返し予定はE2EE同期し、予定回を端末で決定的に生成する

**日付**: 2026-07-17
**状態**: Accepted（2026-07-17 プロダクトオーナー承認）

### 背景

P2-M8ではtask subtreeの再利用とRRULEによる繰り返し起票を追加する。local-firstとE2EEを保つには、serverへ平文のテンプレート・予定を開示せず、offline端末でも欠落なく生成できなければならない。一方、複数端末が同じ予定回を生成すると、random IDでは同一内容のtask treeが重複する。端末ごとにcursorが競合する場合、古い端末が進捗を後退させない契約も必要である。

一般配布前でschemaとprotocolを直接更新できるため、旧形式のcompatibility reader / writerは導入しない。

### 決定

1. `Template`は名前、既定list ID、snapshot revision、作成・更新日時を持つ。snapshotはschema revision付きcanonical JSONで、最大100 nodeかつ49,152 UTF-8 bytesとする。各nodeはtemplate内で安定したnode key、nullableなparent node key、sibling order、title、note、priority、estimated minutesだけを持つ。status、due、scheduled time、reminder、timer、assignee、task ID、rank、未実装tagは保存しない。sync plaintextではsnapshotをJSON文字列へ二重serializeせず、typedな1 atomic fieldとして暗号化する。作成・置換transactionは最終record envelopeが64 KiB以下であることも暗号化後に検証する。
2. 任意の既存task subtreeをtemplate化でき、別task subtreeからsnapshot全体を置換できる。template編集は以後の起票だけへ反映し、生成済みtaskを変更しない。template削除は確認後に参照scheduleも同一local operationでtombstone化するが、生成済みtaskを変更・削除しない。
3. `Schedule`はtemplate ID、canonicalに正規化したRRULE、開始instant、IANA timezone、nullableなnext occurrence instant、enabled、config revision、作成・更新日時を持つ。終了済み規則のnext occurrenceはNULLとする。snapshot revisionとconfig revisionは既存HLC形式のtotal-order identityであり、各revisionはparent revisionとeffective-from instantを持つ。template snapshotとschedule configはそれぞれrevisionを含む1 atomic clocked groupとしてmergeし、fieldごとのhybrid configを作らない。
4. recurrence parserは`rrule = 0.14.0`を用い、DAILY / WEEKLY / MONTHLY / YEARLY、INTERVAL、BYDAY、BYMONTHDAY、COUNT、UNTILだけを受理する。SECONDLY / MINUTELY / HOURLY、複数RRULE、RDATE / EXDATE / EXRULEはv1で拒否し、crateのvalidation limitを常時有効にする。開始instantとIANA timezoneはRRULE文字列とは別のtyped fieldとする。
5. templateとscheduleはtask/listと同じ既存record envelopeを使い、Tenant Root DEKで暗号化して同期する。local schemaはv20、sync protocolはv6とし、`templates`と`schedules` collectionを追加する。serverは暗号blob、head、history、stable cursor、tombstone、quarantineだけを扱い、平文予定、list linkage、生成処理を持たない。
6. task rootには内部recurrence provenanceとしてschedule ID、schedule config revision、template snapshot revision、occurrence instantを保存する。子nodeにも同じ予定回へ属することを識別できるprovenanceを保存する。既存taskの各fieldはmigrationでNULLとし、dual read / writeやfallbackを設けない。
7. scheduled task IDはschedule IDをnamespaceとし、schedule config revision、template snapshot revision、occurrence instant、snapshot node keyをcanonical bytesで連結してUUIDv5を生成する。同じrevision組の同一予定回を複数端末が生成しても同じtreeへ収束させる。手動起票は通常のUUIDv7を使う。
8. 同一config revisionのcursorは`Pending(instant) < Exhausted`というlatticeでmergeする。`Pending`同士は遅いinstantが勝ち、終了規則の`next_run_at=NULL`は`Exhausted`という終端最大値としてstaleな非NULL cursorで復活しない。異なるrevisionのcursorは比較せず、winning config revisionのcursorだけを採用する。
9. future-only編集を端末間で一意にするため、scheduleとtemplateはwinning revisionまでのancestor revision、parent、effective-from境界からなるlineageを同期する。schedule設定変更またはtemplate snapshot置換は、同じ`TaskveilClient` operation内で編集instant以下の全due occurrenceを旧revisionで100件ずつsettleし終えることを必須の前処理とし、途中失敗時はconfig / snapshotを変更しない。effective-fromは編集instantとし、新revisionの最初のoccurrenceはそれよりstrictly futureに置く。このためcutover前に旧payloadを必要とする未生成occurrenceは残らず、lineageには旧RRULE / snapshot payloadを保持しない。disabled scheduleは停止期間を補完せず、template置換時のsettlement対象にも含めない。concurrent child revisionはHLC total orderでwinnerを選び、lineage外のbranchが生成したtreeは全てsupersededとする。ancestor revisionのtreeは次のchild revisionのeffective-from未満だけを正当とし、境界以後にoffline旧端末から届いたtreeはsupersededとする。参照taskが残るlineage metadataは削除しない。
10. pull / merge後のreconciliationは、supersededなrootと全子nodeをlocal domain、record state、outbox上でtransactionalにtombstone化してからwinning revisionをsettleする。これによりschedule / template revisionをUUIDv5へ含めてもlive treeを1件へ収束させる。schedule設定変更より前に正当に生成されたinstanceは保持するが、境界後に旧revisionで生成されたinstanceは端末上で編集されていてもwinning scheduleを優先する。
11. 1 occurrenceのtask tree生成、outbox追加、schedule cursor更新を1 local transactionで確定する。settlement 1回は最大100 occurrenceを処理し、残件があればevent loopへyieldした後に再実行して、90日等の上限なく全件補完する。起動、foreground復帰、sync pull適用後にsettlementを行い、生成でoutboxが増えた場合は既存single-flight sync schedulerへ再実行を要求する。匿名local profileも起動・復帰時に生成する。
12. schedule無効化中は生成しない。再開時はconfig revisionを増やし、再開instantより前の停止期間を補完せず、再開後最初のoccurrenceへcursorを進める。
13. 手動起票はtemplateの既定listへ1 actionで生成する。既定listが削除済みまたはarchive済みならcanonical Inboxへ生成する。自動起票はroot taskだけにoccurrence instantを`scheduled_at`として設定し、dueは設定しない。子taskに日時を設定せず、全nodeを`todo`で生成する。
14. streakは連続する予定回を単位にする。各root taskが次の予定回より前に`done`になれば達成し、`wont_do`、未完了、次回以後の完了は途切れとする。reopenは達成を取り消す。進行中の最新回が次回前なら直前までの値を維持する。COUNT / UNTILの最終回は終了条件を除いた同じcadenceからvirtual next occurrenceを1件だけ算出し、それを達成期限とする。終了済みscheduleはその期限を過ぎるか最終回が閉じた後に最終値を表示する。config revision変更後も同一scheduleのcanonical lineageを時系列として評価する。
15. FlutterはLists画面にTemplates導線と`/templates`専用画面を追加し、task詳細からのsnapshot保存、manual起票、名前・既定list・snapshot置換、schedule作成・編集・停止・削除を提供する。GUI presetは毎日、曜日指定の毎週、毎月特定日とし、詳細入力は検証済みRRULE文字列を扱う。英日ARBとsemanticsを備え、schedule行には控えめなstreakだけを表示する。

### 帰結

- server侵害時にもtemplate内容と予定規則のplaintextを開示せず、offlineだけで予定回を生成できる。
- UUIDv5とatomic settlementにより、同時生成、retry、full resyncが同じtask treeへ収束する。
- cursorの単調mergeとrevision境界により、古い端末が生成進捗を後退させず、設定変更と停止期間skipを明確に扱える。
- template snapshotはcontent-onlyとなり、状態や日時を誤って複製しない。
- schema v20、protocol v6、domain、storage、sync、server、FRB、Flutterにbreaking changeが広がる。

### 却下した代替案

- **serverで予定回を生成する案**: 平文RRULEまたは復号能力をserverへ渡し、E2EE境界を破るため不採用。
- **random task IDで端末ごとに生成する案**: offline同時生成後に同じ予定回のtreeを安全に同定・統合できないため不採用。
- **生成履歴をscheduleとは別に全件同期する案**: occurrenceごとのmetadataを増やし、task provenanceと決定的IDで得られる冪等性を重複実装するため不採用。
- **90日等でcatch-upを打ち切る案**: offline期間によって予定回を黙って欠落させるため不採用。
- **停止期間を再開後に補完する案**: 利用者が明示的に停止した期間のtaskを大量生成するため不採用。
- **snapshotへtask全fieldを保存する案**: 完了状態、日時、reminder、timer、identityを新規taskへ誤って複製するため不採用。
