## 1. 技術スタック選定

### 1.1 全体構成

本アプリケーションは、クライアントUIとコアロジックを明確に分離した「Flutter + Rust」構成を採用する。

| レイヤー | 技術 | 役割 |
|---|---|---|
| クライアントUI | Flutter | iOS / Android / Windows / macOS / Linux 向けUI描画、ユーザー操作 |
| コアロジック | Rust | 暗号処理、同期エンジン、ドメインロジック、ローカルDBアクセス |
| ブリッジ | flutter_rust_bridge | Flutter(Dart) ↔ Rust のFFI境界を自動生成 |
| サーバー | Rust (axum) on AWS Lambda | OPAQUE認証、テナントプロビジョニング、同期API（push/pull）提供、課金Webhook（サーバーレス・常駐プロセスなし） |
| CLI | Rust (clap) | コマンドラインからのタスク操作 |
| MCPサーバー | Rust (rmcp 公式SDK) | LLMエージェント向けTodo操作インターフェース |
| ローカルAI | llama.cpp系Rustバインディング / candle | オンデバイス推論（自然言語パース、要約等） |
| 課金 | App Store / Play IAP、外部課金プロバイダ（Desktop / Org） | サブスクリプション課金 |

### 1.2 選定理由

- **Flutter**: 単一コードベースでモバイル・デスクトップの全対象OSをカバーでき、UI実装の生産性が高い。マテリアル/Cupertino双方の表現力があり、Todoアプリのような情報密度が高いUIとの親和性も良い。
- **Rust**: 暗号ライブラリ（鍵導出・AEAD・OPAQUE等）のエコシステムが充実しており、メモリ安全性の観点からもE2EEの実装言語として妥当である。加えて、コアロジックをRust crateとして切り出すことで、CLI・MCPサーバー・（将来的な）サーバーサイド機能から同一の暗号/ドメインロジックを再利用でき、実装の重複と検証対象の増加を避けられる。
- **flutter_rust_bridge**: DartとRust間の型安全なバインディングをコード生成できるため、手書きFFIに比べて保守コストが低い。

### 1.3 Flutter + Rust構成の評価

**妥当性の結論**: UIの生産性・クロスプラットフォーム性（Flutter）と、暗号実装の安全性・コアロジックのCLI/MCPへの再利用性（Rust）を両立できる構成であり、本アプリの要件（マルチプラットフォームE2EEアプリ、かつCLI/MCPを同一データストアに対して提供）に対して妥当と判断する。

**懸念点と対策**:

| 懸念点 | 内容 | 対策 |
|---|---|---|
| ブリッジのビルド複雑性 | flutter_rust_bridgeはRustのクロスコンパイル設定（iOS/Android/各デスクトップOS向けターゲット）をFlutterのビルドパイプラインに統合する必要があり、CI構築コストが高い | プラットフォームごとのビルドスクリプトをmonorepo内に集約し、CIでプラットフォーム別ジョブをキャッシュ付きで並列実行する。`cargokit` 等の既存の統合ツールチェーンの採用を検討する |
| Flutter Desktop（特にLinux）の成熟度 | プラグインエコシステムやネイティブ連携（キーチェーン、通知）がモバイルに比べて発展途上 | Desktop向けの重要機能（キーチェーンアクセス等）はRust側で自前実装し、Flutterプラグインへの依存を最小化する。Linux版は正式リリース時点でモバイル・Windows/macOSに対し機能段階（フィーチャーパリティ）が遅れる可能性を許容する |
| FFI境界でのエラーハンドリング | Rust側のpanicやエラーがDart側に正しく伝播しない場合、UIがクラッシュ/フリーズするリスク | Rustコア関数はすべて `Result` 型で返却し、`catch_unwind` でpanicを捕捉してDart側に構造化エラーとして通知する |

### 1.4 国際化（i18n）方針

顧客ターゲットはグローバルであり（[`01_企画書.md`](./01_企画書.md) §3）、初期リリースの時点から多言語対応を行う（[`02_機能仕様書.md`](./02_機能仕様書.md) F-48）。技術面の方針は以下のとおりとする。

- 基準言語は英語とする。全UI文字列はFlutter標準のl10nツールチェーン（ARBファイル + `intl`、ICU MessageFormat）で外部化し、UI層でのハードコードを禁止する。
- Rustコア側はエラー等を自然言語文字列ではなくエラーコードとして返却し、表示文言への変換はFlutter側のl10n層で行う。コア層に自然言語文字列を持たせない。
- CLI / MCPサーバーのメッセージは英語を基準とする（MCPのtool説明等も英語で記述する）。
- 日付・時刻・数値の書式化はロケールに従う。タイムゾーンは内部的にUTCで保持し、表示時にロケール・端末タイムゾーンへ変換する（§3のデータモデルと整合させる）。
- RTLレイアウトは、Flutter標準のRTLサポートを前提として初期実装の時点から考慮する。

### 1.5 サーバー実行基盤（決定: AWS Lambda）

サーバー（コントロールプレーン）は **AWS Lambda 上で Rust ネイティブバイナリとして稼働**させる。デプロイ形態は**コンテナイメージ + AWS Lambda Web Adapter（LWA）**とする。axum アプリケーションは通常のHTTPサーバーとして実装し（**Lambda 固有コードを一切含まない**）、静的バイナリと LWA を同梱した最小コンテナイメージ（distroless 等）として Lambda のコンテナイメージサポートでデプロイする。LWA が Lambda イベントとHTTPリクエストの変換を担う。常駐サーバーは持たない。

本節および6章の同期プロトコルに関する設計判断の経緯は [`05_設計判断記録.md`](./05_設計判断記録.md)（ADR-002・ADR-003）を参照。

**選定理由**（比較対象: Cloudflare Workers / Cloudflare Containers。2026年7月時点の調査に基づく）

1. **Rustネイティブ実行**: opaque-ke 等の暗号クレートを WASM 変換なしでそのまま利用できる。認証コードをWASM化の不確実性に載せない（Workersはworkers-rsによるWASM変換が必要でPoCを要する）。
2. **バイナリサイズ制約がない**: Workersはスクリプトサイズ上限（圧縮後、Paidで10MB）があり暗号ライブラリ入りWASMは圧迫しやすい。Lambdaはzipで250MB（展開後）、コンテナイメージなら10GBまで。
3. **データストアとの近接性**: サーバーレスPostgres（Neon等）はリージョン単位で稼働しており、Lambdaを同一リージョンに配置することでサーバー⇄Postgres間がリージョン内通信になり、同期API・認証APIのレイテンシと転送コストを抑えられる。
4. **可搬性**: アプリは Lambda 固有コードを含まない通常のHTTPサーバーであるため、**同一コンテナイメージ**を ECS / Fly.io 等へそのまま移設可能で、出口戦略が確保される。
5. **コールドスタート**: Rustの最小コンテナイメージ構成でのコールドスタートは概ね100〜300ms程度で、ログイン・トークン更新という低頻度APIのp99への影響は許容範囲。将来コールドスタート短縮が必要になった場合は、`lambda_http` + cargo-lambda による zip デプロイ方式（数十ms、ただし main 関数に Lambda 固有のエントリポイントが必要）への切替を検討する。

**不採用理由の要約**: Cloudflare Workers は WASM 制約・サイズ上限が課題。Cloudflare Containers（2026年4月GA）は任意のDockerイメージが動くもののコールドスタート1〜3秒・オートスケール未実装で、まばらな認証トラフィックとの相性が悪い（公式推奨もリソース集約型ワークロード向け）。

**周辺構成**: OPAQUEログインの多段プロトコルの中間状態は、DynamoDBではなく**同一Postgres内のephemeralテーブル**（`expires_at` カラムを持ち、状態消費（consume）時に該当行を即時削除するのに加え、定期クリーンアップジョブで期限切れ行を掃除する。OPAQUE中間状態は一度使用したら破棄する性質のものであり、DynamoDB等のネイティブTTL機構は不要である）に短期保存する。永続メタデータ（users/devices/ラップ済み鍵/organizations/subscriptions等）も同一Postgresに保存する（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-003を参照）。定期処理（tombstone GC・sync_records_historyの期限切れ削除・期限切れOPAQUE中間状態の掃除・失効後180日削除）は EventBridge Scheduler で実行する。tombstone GCは、`delete_hlc IS NOT NULL`かつ`encrypted_blob IS NULL`のtagged tombstoneを180日保持した後に削除し、テナントごとに `gc_horizon_seq` を記録する（2026-07-10改訂: ADR-010、ADR-014）。公開エンドポイントの前段は CloudFront + WAF または API Gateway のスロットリング（最終構成は§12未決事項）とする。ビルド・デプロイは Docker イメージビルド（linux/arm64 または x86_64）→ ECR push → Lambda 更新のパイプラインを CI に構築する。サーバーレスPostgresは**Neon**を採用し、初期リージョンは**eu-central-1（フランクフルト）**、Lambdaも同一リージョンに配置する（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-008参照）。接続はNeon組み込みの接続プーラー経由のTCP接続（sqlx）とし、Lambdaの同時実行数増加に伴うコネクション枯渇を回避する。プーラーはトランザクションモードで動作するため、セッション依存機能（advisory lock、LISTEN/NOTIFY等）は使用しない設計とする（本仕様は行ロックと通常トランザクションのみを用いており抵触しない）。

## 2. リポジトリ/モジュール構成

monorepo構成とし、以下のディレクトリ構成を基本とする。

```
todori/
├── app/                  Flutter アプリ本体
│   ├── lib/
│   ├── ios/ android/ windows/ macos/ linux/
│   └── rust_builder/     flutter_rust_bridge 生成物・ビルド設定
│
├── core/                 Rust コアクレート群
│   ├── domain/           エンティティ・ユースケース（純粋ロジック）
│   ├── crypto/           鍵導出・暗号化・OPAQUE クライアント実装
│   ├── sync/             HLC・差分検出・push/pull同期エンジン
│   ├── storage/          SQLite(SQLCipher) アクセス層、マイグレーション、FTS5
│   └── client/           Flutter / CLI / MCP共通のclient/profile・アプリケーションサービス
│
├── cli/                  Rust CLI (clap)。todori-client を入口に利用
│
├── mcp-server/           Rust MCP サーバー (rmcp)。todori-client を入口に利用
│
├── server/               Rust API サーバー (axum)。AWS Lambda 上で稼働
│   ├── auth/             OPAQUE サーバー実装
│   ├── sync/             push/pull同期API、テナント認可、seq採番
│   └── billing/          IAP / 外部課金プロバイダのWebhook処理
│
└── docs/                 設計ドキュメント
```

- `core/` はcrate群の配置ディレクトリであり、`core`というcrateではない。実packageは`todori-client`等、Rust crate名は`todori_client`等とする。暗号・同期・ドメインロジックの単一の実装源泉（Single Source of Truth）をこのcrate群へ置く。`core/client`（package `todori-client`）はDB open、repository + domain + outbox enqueue、アカウント状態、同期実行をまとめるfrontend非依存の唯一の共通入口とし、Flutter bridgeはFRB公開関数とDTO変換、CLI / MCPは入出力変換に限定する。frontend adapterから下位のcrypto/domain/storage/sync crateへ直接依存しない（2026-07-10改訂: ADR-011、task-91）。
- `server/` は**コントロールプレーン**（認証・ラップ済み鍵の保管・テナント管理・課金Webhook）と**同期データプレーン**（push/pull同期API）を提供する。ただしE2EEデータの中身は一切解釈せず、暗号blobの中継・保管に徹する（§6.1）。ドメインロジック（domain / sync / storage）には依存しないが、OPAQUE等の暗号プロトコル実装を共有するため `core/crypto` には依存しうる（クライアントとサーバーで同一実装系を用い、プロトコル互換を保証するため）。

## 3. データモデル

ID はすべて **UUIDv7** を採用する（生成時刻の情報を含み、ソート性とグローバル一意性を両立するため）。時刻はすべて **UTC epoch milliseconds（int64）** で保持する。以下は論理スキーマであり、実データはレコード単位で暗号化されてサーバーへ同期される（詳細は「4. 暗号設計」「6. 同期プロトコル」を参照）。

すべての同期対象エンティティは正確に1つのテナント（個人テナント or Orgテナント）に属し、テナントをまたぐ参照は存在しない。tags / templates / schedules / timer_sessions はテナントスコープである（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-007参照）。

ローカルDB（SQLite/SQLCipher）のスキーマバージョンは `PRAGMA user_version` で管理し、`core/storage` のmigration runnerがDB open時に未適用migrationを順次適用する。現行latestはv18であり、v17はADR-017に従ってtask期限をtyped dueへ置換し、v18はADR-018に従ってactive / completed timer session、local Tenant Root DEK cache、protocol v5同期metadataを追加する。旧versionのdual read/writeは行わない（2026-07-13改訂: ADR-017、ADR-018 / task-101、task-105）。

ローカルDB専用のアプリ設定値は、SQLCipher内の `settings` テーブルへ保存する。`settings` は端末ローカルのkey/valueストアであり、ローカルDBファイル全体の保存時暗号化により保護される。Phase 1ではF-01 UIモード保存口として `ui_mode` キーを予約し、未設定時はDart側helperが既定値 `simple` を返す。切替UIと初回選択UIはPhase 3以降のスコープとする（2026-07-08追記: task-63）。同期対象recordには、server ACK済みrevisionとnever-synced creation originを再起動後も識別できる永続状態を持たせ、`base_revision_hlc = null`だけでlocal-newと推測しない（2026-07-11追記: ADR-016）。

| key | TEXT PRIMARY KEY | 設定キー |
| value | TEXT NOT NULL | 設定値 |
| updated_at | INTEGER NOT NULL | 更新日時（UTC epoch milliseconds） |

| created_at | int64 | 作成日時 |
| updated_at | int64 | 更新日時 |

期限4列はCHECK constraintで排他的にする。期限なしは全列NULL、日付のみは`due_kind='date'`かつ`due_on`だけ非NULL、日時指定は`due_kind='datetime'`かつ`due_at_ms` / `due_time_zone`だけ非NULLでなければならない。domain / client / FRBはraw epoch単独で期限を公開せず、`Option<TaskDue>`を正本にする。FRB / Dart境界の`TaskDueInput` / `TaskDueDto`は`Date { due_on }` / `DateTime { due_at: DateTime<Utc>, time_zone }`のsealed tagged unionとし、`kind: String` + nullable payloadや`dueAtMs`を公開しない。日付のみは`due_on < viewer_local_today`、日時指定は`now >= due_at`で期限切れとする。Today queryは期限を持つtaskに加え、`scheduled_at`がviewer local todayのtaskを別条件で対象にし、UI分類時も期限なしのscheduled taskをTodayへ残す（2026-07-12追記: ADR-017 / task-101）。

Calendar期間取得は、viewer timezoneから構成したcivil-dateの`[start_on, end_on)`とUTC instantの`[start_at, end_at)`を組にした`CalendarRange`を正本とする。日付のみ期限はcivil-date範囲、日時期限・`scheduled_at`・`completed_at`はinstant範囲で判定し、DST境界で固定24時間を加算しない。`todo` / `in_progress`はdueとscheduledを別occurrenceとして返すため、同一taskが両方を持つ場合は2件になる。`done` / `wont_do`は`completed_at` occurrenceだけを返し、通常agendaとCompletedへ二重表示しない。archive済みlistはlist名とarchive状態を伴って含め、`deleted_at IS NOT NULL`のtaskは除外する。FRB / Dart境界はtyped `CalendarRangeInput`とsealed `CalendarOccurrenceKindDto`（date due / datetime due / scheduled / completed）を用い、task本体とlist contextを返す。occurrenceの日付移動は既存task updateとfield-level LWWへ委譲し、local schema・sync plaintext・protocolを変更しない（2026-07-13追記: task-104）。

`parent_task_id` による自己参照で無制限階層のサブタスクを表現する。ローカル削除は物理削除とし、タスク詳細画面のサブメニューから起動し、不可逆警告の追加確認を必須とする。ゴミ箱は設けず、削除Undoも設けない（完了/編集Undoは維持する）。サブタスクを持つ親タスクを削除する場合は子孫も物理削除する。履歴保全は削除ではなくリストのアーカイブが担う。`tasks.deleted_at` は移行互換のため残置する非推奨列であり、将来のv3マイグレーションで廃止予定である。同期上の削除は`delete_hlc`だけを持ち暗号blobを持たないterminal tagged tombstoneで伝播し、同一record IDのlive復活を許さない（2026-07-11改訂: ADR-010、ADR-014、ADR-016）。

## 4. 暗号設計

本章はE2EE設計の中核であり、最も厳密な実装・レビューを要する領域である。

### 4.1 設計方針

- サーバーはいかなる場合もユーザーのパスワード、および各種コンテンツの平文を保持・観測できない。
- 認証（本人確認）と暗号鍵の保護（Master Keyの保護）は独立した仕組みとし、パスワード変更が発生してもデータ本体の再暗号化を必要としない設計とする。
- 共有（Organization）機能は公開鍵暗号（X25519 sealed box）によって実現し、共有単位（List）ごとに独立した対称鍵（DEK）を用いる。

### 4.2 鍵階層

```
                       ┌────────────────────┐
                       │   User Password    │  (サーバーへは非送信)
                       └─────────┬──────────┘
                                 │ OPAQUE (opaque-ke)
                                 ▼
                       ┌────────────────────┐
                       │     exportKey      │  (OPAQUE成功時にクライアントのみ導出)
                       └─────────┬──────────┘
                                 │ HKDF-SHA256
                                 ▼
                       ┌────────────────────┐
                       │      KEK_pw        │  (パスワード由来 鍵暗号鍵)
                       └─────────┬──────────┘
                                 │ wrap
                                 ▼
   ┌─────────────┐     ┌────────────────────┐     ┌─────────────────┐
   │  Device Key │──wrap→│    Master Key      │←wrap──│  Recovery Key   │
   │    (DK)     │     │       (MK)          │     │ (BIP39風 人間可読) │
   └─────────────┘     └─────────┬──────────┘     └─────────────────┘
    OSキーチェーン保存           │ HKDF / wrap                サーバー保存(wrap(MK,RecoveryKey))
    未登録時はDKで               ▼                            ユーザーのみ保持(平文)
    ローカルDB保護    ┌────────────────────────┐
                      │  User X25519 KeyPair    │ 秘密鍵は wrap(SK, MK) でサーバー保管
                      └───────────┬────────────┘
                                  │ sealed box (org共有時)
                                  ▼
              ┌───────────────────────────────────┐
              │            List DEK (32byte)        │
              │  個人リスト: wrap(DEK, MK)           │
              │  共有リスト: 各メンバー公開鍵でsealed  │
              └───────────────┬─────────────────────┘
                              │ XChaCha20-Poly1305
                              ▼
                 タスク/リスト/タグ等のコンテンツblob
```

List DEKと並列に、テナントごとの **Tenant Root DEK** が存在する。個人テナントは `wrap(TenantDEK, MK)` としてサーバー保管し、Orgテナントはメンバー全員の公開鍵でsealされる。tags / templates / schedules / timer_sessions 等のテナントスコープのエンティティ（3章参照）はTenant Root DEKで暗号化され、List配下のエンティティ（tasks/comments/reminders）とは異なる鍵系譜に属する（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-007参照）。

### 4.3 各鍵の定義

| 鍵 | サイズ/形式 | 生成タイミング | 保存場所 |
|---|---|---|---|
| Device Key (DK) | 32byte 乱数 | 初回起動時 | iOS: Keychain(Secure Enclave裏付け) / Android: Keystore / Desktop: OSキーチェーン |
| Master Key (MK) | 32byte 乱数 | アカウント登録時（既存ローカルデータがあればMK配下に移行） | 平文はメモリ上のみ。`wrap(MK, KEK_pw)`をサーバー、`wrap(MK, DK)`をローカル、`wrap(MK, RecoveryKey)`をサーバーに保存 |
| KEK_pw | 32byte（HKDF出力） | ログイン成功（OPAQUE成功）ごとに導出 | 保存しない（都度導出） |
| Recovery Key | 24単語程度のBIP39風ニーモニック | アカウント登録時に1回生成 | ユーザーのみが保持（サーバー・端末には保存しない） |
| User X25519 KeyPair | 秘密鍵32byte / 公開鍵32byte | アカウント登録時 | 公開鍵はサーバーに平文保存。秘密鍵は`wrap(SK, MK)`でサーバー保存（Org共有のため他デバイスからも取得できる必要がある） |
| List DEK | 32byte 乱数 | リスト作成時 | 個人: `wrap(DEK, MK)`。共有: メンバーごとに`seal(DEK, member_pubkey)` |
| Tenant Root DEK | 32byte 乱数 | テナント作成時（個人テナントは初回アカウント登録時、Orgは組織作成時） | 個人: `wrap(TenantDEK, MK)`。Org: メンバーごとに`seal(TenantDEK, member_pubkey)` |

端末上ではremote sessionとlocal crypto contextを分離する。`wrap(MK, DK)`はOS secret store、local cache用の`wrap(List DEK, MK)`はlist IDをAADへ含めた専用形式でtenant ID / list ID単位にSQLCipher DBへ保存し、List DEK平文は永続化しない。Tenant Root DEKもMKで再wrapし、AAD`todori/wrap/local-tenant-root-dek-by-mk/v1/{tenant_id}`へtenant IDを束縛してSQLCipher内にcacheする。key versionは1、平文Tenant Root DEKは`Zeroizing` runtime memoryだけに置く。session tokenの期限切れやnetwork不通でも、これらからlocal crypto contextを復元できれば既存dataのmutationを許可する。account bindingがあるのにMK、必要なList DEK、またはTenant Root DEKを復元できない場合は匿名利用へfallbackせず、対象mutation前に明示的に失敗する（2026-07-13改訂: ADR-013、ADR-018）。account-boundのoffline list作成では、List DEK、list IDをAADに持つlocal cache bundle、domain list、typed record state、entity outbox、server用MK-wrapを持つ`pending_list_key_bundles`を同一`BEGIN IMMEDIATE`で確定する。pending rowはtenant/listごとにimmutableとし、同じciphertextの再登録だけをno-opとして許可する。server用bundleはentity recordより先にuploadし、成功ACK前に別DEKへ生成し直さない。

Apple platformのDKはgeneric password itemとして保存し、serviceは`dev.todori.todori.device-key`、accountは`default`、synchronizableは`false`、access groupは指定しない。iOSではData Protection Keychainと`AfterFirstUnlockThisDeviceOnly`相当を必須とする。macOSでは同じData Protection Keychainを最初に試みるが、非署名テストバイナリやad-hoc署名ビルドでは`kSecUseDataProtectionKeychain`がentitlement不足により`errSecMissingEntitlement (-34018)`を返すため、このエラーの場合のみlegacy login keychain（`kSecUseDataProtectionKeychain`なし、synchronizable `false`、同一service/account）へフォールバックする。フォールバック使用時のログは秘密情報を含まない1行に限定する（2026-07-08追記: task-64）。

### 4.4 未登録利用時の扱い

アカウント未登録の状態では、ローカルDBはDevice Keyのみで保護される。アカウント登録時、既存のローカルデータ（個人リスト・タスク等）はMaster Key配下に移行（再暗号化）され、以後DKはMKのラップ用途に限定される。

### 4.5 パスワード変更

パスワード変更時に必要な処理は以下のみである。

1. OPAQUEレジストレーションの再実行（新パスワードでの再登録）。
2. 新しいexportKeyからKEK_pw'を導出し、`wrap(MK, KEK_pw')`を再計算してサーバー側の値を更新。

Master Key自体は不変であるため、**タスク等のデータ本体の再暗号化は一切不要**である。これが、exportKeyを直接Master Keyとして採用せず、間接的にKEK_pwを介してMKをラップする理由である（exportKeyを直接MKとして使うと、パスワード変更のたびに全データの再暗号化が必要になる）。

### 4.6 Organization共有

- 各ユーザーはアカウントごとにX25519鍵ペアを1組保持する（複数デバイス間ではMKで秘密鍵を共有）。
- 共有リストのDEKは、メンバー各自の公開鍵に対してsealed box（NaCl方式、送信者匿名の公開鍵暗号）でラップし、`org_members.sealed_dek_map` 相当の形でサーバーに保存する。
- **メンバー除名時**は当該リストのDEKをローテーション（新DEK生成）し、既存コンテンツを新DEKで再暗号化した上で、残存メンバーの公開鍵で再ラップする。除名されたメンバーは新DEKを取得できないため、以後の更新内容を復号できない（ただし除名以前にローカルへ同期済みのデータは端末側に残存し得る点に留意する）。

### 4.7 暗号アルゴリズム・ライブラリ

| 用途 | 方式 | ライブラリ |
|---|---|---|
| 認証 | OPAQUE (PAKE) | `opaque-ke` |
| パスワードストレッチング（OPAQUE内KSF） | Argon2 | `argon2`（opaque-ke の argon2 feature 経由） |
| 鍵導出 | HKDF-SHA256 | RustCrypto `hkdf` |
| 対称暗号（AEAD） | XChaCha20-Poly1305 | RustCrypto `chacha20poly1305` |
| 鍵交換/sealed box | X25519 | RustCrypto系 `x25519-dalek` |

### 4.8 レコード暗号化方式

各エンティティ（task, list, tag, comment等）は、暗号化対象フィールドをまとめたJSONオブジェクトとしてシリアライズし、対応するDEKでXChaCha20-Poly1305暗号化した上でblobとして同期する。

- **AAD（Associated Data）**はenvelope v4で`todori-sync-envelope/v4\ncollection:{collection}\nrecord_id:{record_id}`とし、レコードID・collection名を含めて暗号文の入れ替えを検知・防止する。
- ステータス・優先度・並び順などUI表示上不可欠だが機微性が低いフィールドも、原則としてすべて暗号化対象に含める（サーバー側解析面の最小化を優先する）。同期メタデータとして必要な最小限の情報（後述）のみ平文でサーバーに保持される。

**エンティティとDEKの対応**（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-007参照）

- tasks / comments / reminders → 所属リストの List DEK。
- lists自体のメタデータ（name / color / icon） → 当該 List DEK。
- tags / templates / schedules / timer_sessions → 所属テナントの Tenant Root DEK。

タスクの同一テナント内でのリスト間移動は、クライアントで復号した上で移動先List DEKによる再暗号化を要する。テナントをまたぐタスクの移動は「移動」ではなく**明示的なコピー**操作とし、新規レコードIDを発行する（鍵もHLC系譜も混ざらない）。

### 4.9 サーバーが知り得る情報 / 知り得ない情報

**サーバーが知り得る情報**

| 項目 | 内容 |
|---|---|
| アカウントID | ユーザー識別子、メールアドレス |
| OPAQUEレジストレーションレコード | パスワード検証用データ（パスワード自体は含まない） |
| ラップ済み鍵 | `wrap(MK, KEK_pw)`、`wrap(MK, RecoveryKey)`、`wrap(SK, MK)` |
| デバイスリスト | デバイス名、公開鍵、登録/失効日時 |
| 課金状態 | プラン種別、サブスクリプション有効期限 |
| Organizationメンバーシップ | 所属関係、ロール、sealed DEK |
| 暗号blob + 同期メタデータ | レコードID、collection種別、revision HLC（ノードID成分を含む）、seq、blobサイズ、サーバー書込み時刻 |

**サーバーが知り得ない情報**

- ユーザーのパスワード（平文はサーバーに一切送信されない）
- タスクのタイトル・メモ・タグ名・コメント本文などのコンテンツ平文
- リマインダー日時、期限日時などの時刻情報（暗号化対象のためサーバーからは不可視）

### 4.10 帰結: プッシュ通知の制約

リマインダー日時はコンテンツとして暗号化されているため、サーバーはリマインド対象・タイミングを把握できず、**サーバー起点のプッシュ通知でリマインドを送信することはできない**。したがって、リマインド通知はすべて**クライアント側のローカルスケジューリング**（OS標準のローカル通知API）によって実現する。この制約により、アプリが完全に終了・OSにより強制停止されている状況での通知信頼性は、プッシュ通知方式に比べて劣る可能性がある点を設計上の既知の制約として明記する。

## 5. ローカルストレージ

### 5.1 採用技術

- ローカルDBは素のSQLite + **SQLCipher** を採用する（Rustからは `rusqlite` の bundled-sqlcipher 系featureを想定）。
- libSQL/Tursoは採用しない（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-002）。同期はサーバー経由のpush/pull API（6章）であり、クライアントはlibsqlクライアントを持たず、`rusqlite` + SQLCipherと通常のHTTPSクライアントのみを持つ。これにより`rusqlite`+SQLCipherとlibsqlクライアントのSQLite実装二重リンク問題（特にiOS/Androidでのシンボル衝突リスク）も発生しない。
- サーバー（Postgres）に保存されるのはDEKで暗号化済みのblobのみであるため、E2EEの保証はストレージ層の保存時暗号化には依存しない（Postgres側の保存時暗号化を多層防御として有効化してよいが、必須ではない）。
- ローカルDBは**テナントごとにSQLiteファイルを分離**する（個人テナント用ファイル1つ + 所属Orgテナントごとに1ファイル）。SQLCipherの鍵はいずれも5.3節のとおりDevice Key (DK) 由来で共通であり、ファイル分離自体が鍵構造に影響することはない。Org脱退・除名時は当該テナントのファイルを削除するだけで端末上のデータ消去が完結する。FTS5索引もテナントごとのファイル内に閉じる（5.4節参照）。詳細な採用理由は[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-007を参照。

### 5.2 暗号化の2レイヤー構造

「二重暗号化」ではなく、適用場所・脅威モデルの異なる2レイヤーであることを明確にする。

| レイヤー | 方式 | 適用場所 | 脅威モデル |
|---|---|---|---|
| コンテンツ暗号化（E2EE本体） | DEKによるXChaCha20-Poly1305 | 同期境界（push時にblob生成、pull時に復号） | サーバー・ネットワーク上の第三者 |
| 保存時暗号化 | SQLCipher（ページレベル） | ローカルDBファイル全体 | 端末紛失・盗難、他プロセスからのファイル読み取り |

- ローカルDB内のカラムは**平文**で保持する。SQLクエリ・インデックス・ソート・FTS5を機能させるためであり、DEK暗号化されたblobをローカルの正本として保持する設計は採らない。
- DEKによる暗号化は同期境界でのみ適用される。outboxに一時保持される送信待ちblobがSQLCipher内に置かれることはあるが、これは設計上意図した多層であり冗長な二重暗号化ではない。
- local crypto cacheにはList DEK平文ではなく`wrap(List DEK, MK)` bundleだけを保存する。list削除時も、late descendantの復号・分類、expired-device rebase、tombstone GC、server側stale push拒否の安全条件が揃うまでcache rowを削除しない（ADR-012 / ADR-013 / ADR-016）。
- local schema v13の`sync_quarantine`は、適用不能なremote recordの最新headをrecord ID単位で保持する。保存対象はcollection、server seq、revision HLC、live/tombstone tagとsemantic HLC、live時の暗号blob、型付きreason（`missing_dek` / `no_matching_dek` / `authentication_failed` / `corrupt_envelope` / `invalid_plaintext`）、必要な場合のlist ID、初回/最終失敗時刻、attempt countである。plaintext、DEK、Master Key、session token、crypto libraryの詳細errorは保存しない。record IDに対応するcollectionはrecord state / outbox / quarantine間で不変とし、同seq / revisionの再試行は暗号stateを置換せずattemptだけを加算する。古いseqはlatest headを上書きせず、同seqでrevisionが異なる入力は不正として拒否し、新しいseqだけがheadを置換する。同recordのquarantine中もlocal outbox head自体は保持し、送信列挙だけを停止する。新revisionの適用またはquarantine再適用が成功したtransactionでのみrowを削除する（2026-07-10改訂: ADR-012、task-88）。

### 5.3 ローカルDB鍵

- SQLCipherの鍵は**常にDevice Key (DK) からHKDFで導出**する（登録状態によらず一貫）。
- MK由来の鍵に切り替えない理由: `wrap(MK, DK)` が同一端末に保存される以上、MK由来にしてもDKが破られれば等しく破られるため、防御力が向上しない。鍵を一貫させることでアカウント登録時のDBリキー（PRAGMA rekey）も不要になる。
- 将来オプション（strict mode）: `wrap(MK, DK)` をローカルに保存せず、起動ごとにパスワード入力（OPAQUE）でMKを取得する運用。アプリ起動にパスワードが必須になる代わりに、端末単独ではデータを復号できなくなる。初期リリースのスコープ外とし、12章の未決事項に含める。

### 5.4 全文検索

- FTS5はSQLCipher上でそのまま動作する（SQLCipherはページレベル暗号化のためSQLite機能に透過的である）。
- FTS5の転置インデックスは平文を含むが、暗号化DBファイル内に格納されるため、脅威モデルはレイヤー2（DK保護）に帰着する。
- `tasks_fts` は `tasks` tableの `INSERT` / `UPDATE` / `DELETE` トリガーで同期する。`title` / `note` 更新、移行互換の `deleted_at` 付き行、物理削除はFTS5索引へ即時反映する（2026-07-08追記: task-62）。
- tokenizerは追加依存なしのFTS5標準 `unicode61` を用いる。検索APIは空白区切り語を前方一致queryへ変換するため、英語単語や日本語連続文字列の先頭一致は扱えるが、任意部分一致や日本語の形態素単位分割は提供しない。検索UI設計時はこの制約を前提に、必要ならtokenizer/インデックス設計を再評価する（2026-07-08追記: task-62）。

## 6. 同期プロトコル

本章の設計判断の経緯は [`05_設計判断記録.md`](./05_設計判断記録.md)（ADR-002〜ADR-005、ADR-010、ADR-012、ADR-014〜ADR-016）を参照。

### 6.1 構成

同期はクライアント⇄サーバー直結ではなく、サーバー（Lambda上のaxum）が提供する**push/pull API**を経由する（ADR-002）。テナント（ユーザーごとの個人テナント、Organizationごとの Orgテナント）は物理DB分離ではなく、単一Postgres内で `tenant_id` により論理分離する（ADR-003）。

論理分離の担保は以下の多層防御による。

- 全クエリに `tenant_id` を必須とするリポジトリ層設計（クエリ条件漏れを型・レビューで検出する）。
- Postgres RLS（Row Level Security）による多層防御。
- 認可の網羅テスト。

認可・課金の有効性検証はAPIリクエストごとにサーバーが実施する。これにより、デバイス失効・課金失効は（旧設計のトークンTTLに伴う遅延なく）即時に同期アクセスへ反映される。

### 6.2 サーバースキーマ

サーバー側Postgresはコンテンツを解釈しない汎用的な同期テーブルのみを保持する。

```sql
CREATE TABLE sync_records (
    tenant_id      UUID    NOT NULL,
    record_id      UUID    NOT NULL,  -- UUIDv7
    collection     TEXT    NOT NULL CHECK (collection IN ('lists', 'tasks', 'timer_sessions')),
    seq            BIGINT  NOT NULL,  -- pullカーソル用（サーバー到着順）
    revision_hlc   TEXT    NOT NULL,  -- base revision CAS用（固定幅ソート可能エンコード、ノードID内包）
    mutation_hlc   TEXT,              -- live domain変更のsemantic clock
    delete_hlc     TEXT,              -- tombstone削除操作のsemantic clock
    encrypted_blob BYTEA,
    updated_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (tenant_id, record_id),
    UNIQUE (tenant_id, seq)
);
CREATE TABLE tenant_seq (tenant_id UUID PRIMARY KEY, last_seq BIGINT NOT NULL);
CREATE TABLE sync_records_history (
    tenant_id UUID NOT NULL, record_id UUID NOT NULL, seq BIGINT NOT NULL,
    collection TEXT NOT NULL, revision_hlc TEXT NOT NULL,
    mutation_hlc TEXT, delete_hlc TEXT, encrypted_blob BYTEA,
    author_user_id UUID NOT NULL, overwritten_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE TABLE tenant_device_continuity (
    tenant_id UUID NOT NULL, device_id UUID NOT NULL,
    continuity_seq BIGINT NOT NULL, continuity_generation BIGINT NOT NULL,
    required_generation BIGINT NOT NULL, initialized BOOLEAN NOT NULL,
    PRIMARY KEY (tenant_id, device_id)
);
-- device_resync_sessionsはstable-key base cursorをgeneration単位に保持する。
-- continuity_closure_proofsはproof IDをtenant/device/high-water/generationへ束縛する。
-- live: mutation_hlc + non-empty encrypted_blob、tombstone: delete_hlcのみ。
-- CHECK制約で両stateのnullable shapeと revision_hlc >= semantic clock を強制する。
```

- `sync_records` はレコード（`tenant_id`, `record_id`）ごとに**最新の暗号blobのみ**を保持する（完全append-onlyのopログ方式は不採用。理由はADR-005参照）。
- `seq` はpullカーソル専用であり、「サーバーへの到着順（＝可視になった順）」を表す。`tenant_seq` 行の `UPDATE ... RETURNING` で採番する。この採番は行ロックがコミットまで保持されるため、テナント単位でpushが直列化され「seqの順＝可視になる順」が厳密に保証される（直列化の競合範囲はテナント内、すなわち1ユーザーの数端末〜小規模Orgに限られ実害はない）。
- top-level `revision_hlc` はCAS revision、`mutation_hlc` は最後のdomain/user変更、`delete_hlc` は削除操作を表す。live blobをrevisionの大小だけでblind overwriteせず、pushの`base_revision_hlc`がserver current revisionと一致した場合だけ更新する。stale pushはcurrent envelopeとのclient merge/retryへ戻す（2026-07-10改訂: ADR-012、ADR-014）。
- `sync_records` は `device_id` カラムを持たない。HLCのノードID成分で更新元デバイスの識別は足りており、あえて別カラムを設けないことでメタデータを最小化する（ADR-005）。書き込み主体（ユーザー）の帰属は `sync_records_history.author_user_id` にのみ持たせる。
- `sync_records_history` は上書き時の旧行を退避するテーブルであり、Orgメンバーの破壊的上書きからの復旧手段として30日保持する（詳細は6.6節）。
- tombstone受理時は削除対象の旧contentを`sync_records_history`へ新規退避せず、同record IDの既存historyも削除する。tombstoneには削除伝播に必要なrecord ID、collection、revision / delete clock、seq、時刻以外のcontentを持たせない（2026-07-11追記: ADR-016）。
- `timer_sessions`も同じopaque record table、CAS、continuity、history、terminal tombstoneを使う。serverは暗号blob内のtask ID、mode、実績時間を知らず、初回に受理したlive ciphertextをimmutable headとする。同IDへの異なるlive ciphertextはbaseが一致しても更新せずconflictとして拒否し、exact retryだけをno-opとする（2026-07-13追記: ADR-018、task-105）。
- `list_key_bundles` の`(tenant_id, list_id)`はgeneration 0のimmutable createとして扱う。初回insertと同一ciphertextのretryは成功し、異なるciphertextは409 conflictとして既存rowを変更しない。List DEK rotation / generationは別ADRで追加し、既存rowの無条件overwriteとして実装しない（2026-07-10追記: ADR-013、task-90）。
- `tenant_device_continuity`、`device_resync_sessions`、`continuity_closure_proofs`はopaqueなtransport continuityだけを保持し、list ID、parent ID、ancestor、aggregate scope / epochを追加しない。closure proofはlocal page / cursorまたはfull-resync rebaseのcommit後にACKされ、そのACK transactionだけが`continuity_seq` / `continuity_generation`を単調更新する（2026-07-11追記: ADR-016、task-98）。
- `list_key_bundles.deletion_seq`は同じlist record IDのtombstone受理seqだけを保持する。bundle retirementはtombstone GC済み、全deviceがdelete seq通過またはexpired、current-state分類済み、未ACK closure proofなし、caller continuity currentを同一transactionで確認した場合だけ成功する（2026-07-11追記: ADR-016、task-98）。

### 6.3 変更追跡

- 各デバイスは **Hybrid Logical Clock (HLC)** を保持し、local mutationとmerge後の再pushでインクリメントする。HLCは「物理時刻（ミリ秒）+ 論理カウンタ + ノードID（デバイス識別子由来）」の3要素からなり、固定幅のソート可能な文字列としてエンコードする。pullでremote revisionをlocal HLCへobserveし、再push revisionは必ず受信revisionより大きくする。observe、採番、HLC永続化、repush用outbox登録は同一local transactionで確定する（2026-07-10改訂: ADR-012）。
- マージはフィールドレベルの **Last-Write-Wins (LWW)** を基本とする。local mutationは実際に変更したfieldのHLCだけを更新し、未変更fieldのHLCを保持する。merge後の再pushもtop-level `revision_hlc` だけを新規採番し、payload内の未変更field HLCを引き上げない（2026-07-10改訂: ADR-012）。
- 暗号blobは`kind=task|list|timer_session`で厳密にtag付けしたcollection別plaintextを持つ。task / listのmutable field groupは`Clocked<T> { value, hlc }`として丸ごと暗号化する。taskは`title`、`note`、`priority`、`due: Option<TaskDue>`、`scheduled_at`、`estimated_minutes`、`assignee`、`created_at`、`updated_at`に加え、`completion { status, completed_at, closed_reason }`と`placement { list_id, parent_task_id, rank }`を各1 clockのcompound fieldとして持つ。`due`はtagged union全体へ1 clockを付ける。listは`name`、`color`、`icon`、`org_id`、`is_default`、`archived_at`、`created_at`、`updated_at`と`placement { rank }`を持つ。timer sessionはADR-018の確定実績全体を1つのimmutable valueとして持ち、同ID・異内容をfield mergeしない。未知field、欠落field、collection/kind不一致、live payload内`deleted_at`は受理しない。競合解決情報とtimerのtask linkage / durationは暗号境界内にあり、サーバーからは不可視である（2026-07-13改訂: ADR-012、ADR-014、ADR-017、ADR-018）。
- default Inboxが複数ある場合は、liveかつtyped検証済みで認証済み`is_default=true`のlist sync stateを候補とし、UUID最小のrecordをcanonicalとする。initial pull / full resync closure後も、`collection=lists`のlive current headにunresolved quarantineが1件でもあれば候補集合不明として選挙を保留する。canonicalのpromotion、loserのdemotion、durable alias集合、全既知alias taskのtyped placement更新・canonical List DEKでの再暗号化・record state・HLC・outboxを単一`BEGIN IMMEDIATE`で確定し、commit後にだけaliasを非表示にする。通常sync後も冪等に再評価し、alias recordとList DEK bundleはfull resync、late descendant収束、expired-device rebase、key retirementの安全条件が整うまで削除しない（2026-07-11改訂: ADR-015、ADR-016）。
- rankは`00000000000000000000000000000000`〜`ffffffffffffffffffffffffffffffff`の固定幅lowercase hex unsigned 128-bitとし、通常挿入は隣接rankの数値中点を使う。taskの表示scopeは`(list_id, parent_task_id)`、listの表示scopeはlocal profile全体で、全経路を`(rank, record_id)`順へ統一する。同rank衝突または隣接値間の空間枯渇時だけ対象scopeを現順のまま十分な間隔へ再配番し、変更されたdomain row、typed field clock、record state、outbox head、local HLCを同一`BEGIN IMMEDIATE`で確定する（2026-07-10改訂: ADR-012、ADR-014）。
- 削除の同期表現は、`state.kind=tombstone`と`delete_hlc`だけを持つtagged stateとする。tombstoneは`encrypted_blob`を持てず、liveは`mutation_hlc`と非空blobを必須とする。liveからtombstoneへのCAS過程ではsemantic clockを用いるが、tombstone受理後の同一record IDに対するtombstone → live遷移はHLCの大小にかかわらず拒否する。再作成は新しいrecord IDを使う（2026-07-11改訂: ADR-010、ADR-012、ADR-014、ADR-016）。
- `seq`、`revision_hlc`、`mutation_hlc` / `delete_hlc`、`field_hlcs` は役割を分離する。`seq` はpull cursor、`revision_hlc` はbase revision CAS、`mutation_hlc` / `delete_hlc` はlive同士のmergeとlive → tombstone遷移のsemantic clock、`field_hlcs` はfield値の競合解決キーである。tombstone受理後のterminal性はclock比較で覆さない（詳細はADR-005、ADR-012、ADR-014、ADR-016）。

### 6.4 同期フロー

1. ローカルでの書込は、domain row、`sync_outbox`、暗号payload生成元となるsync record state、local HLCを共通client層の同一SQLite transactionで確定する。account-bound list作成ではlocal List DEK cacheと`pending_list_key_bundles`も同じtransactionへ含める。途中で失敗した場合は全体をrollbackし、local-only / outbox-only / key-only commitを作らない（2026-07-10改訂: ADR-011〜ADR-013、task-90）。
2. **protocol / continuity preflight**: entity outboxと`pending_list_key_bundles`を読む前に`GET /v2/tenants/{tenant_id}/preflight`をprotocol v5 header付きで呼び、responseの`protocol_version`とenvelope v4がclientの対応値と一致することを確認する。unsupported値は必要versionをSQLCipher設定へdurable blockとして保存し、outbox読取、key bundle / entity push、payload pull、ACK、cursor更新を行わない。serverは認証済みdeviceのcontinuityと`gc_horizon_seq`を比較し、normal pullかexpired-device full resyncかを裁定する。client申告のcursorだけでpush可否を決めず、旧protocol clientがpreflightを迂回してwrite endpointを直接呼んでも拒否する（2026-07-13改訂: ADR-012、ADR-014、ADR-016〜018、task-105）。
3. **pull / delete reconcile**: normal deviceは `GET /v2/tenants/{tenant_id}/pull?since={seq}&limit={n}`、expired deviceはstable-key current-state full resyncを行い、どちらもentity pushより先にremote stateを復号・検証する。pull failureは`MissingDek`、key identity未確定taskの`NoMatchingDek`、期待DEKでの`AuthenticationFailed`、既知versionの`CorruptEnvelope` / `InvalidPlaintext`、`UnsupportedEnvelopeVersion`、`UnsupportedProtocolVersion`、既知protocol response不正へ分類する。missing系を検出したpageの初回`BEGIN IMMEDIATE`はcommitせず、transactionをdropした後にkey bundleをnetwork refreshし、runtime/local cache更新後に同じpage全体を1回だけ再実行する。refresh失敗はcorruptionへ降格せずcursorを保持し、成功後もmissingのrecordと既知version corruptionだけをdurable quarantineへ保存する。task復号candidateは`(list ID, DEK)`として扱い、認証済みplaintextの`placement.list_id`がそのcandidate list IDと一致した場合だけ採用する。新規taskが全candidate DEKで認証不一致でもlist linkageをserverへ追加せず`NoMatchingDek`とし、corruptionへ断定しない。quarantine再適用はkey refreshで回復可能な`missing_dek` / `no_matching_dek`だけを`(seq, record_id)`の安定順でbounded paginationし、known corruptionは毎run再試行せず新revisionによるsupersedeを待つ。page内のdomain apply/delete、remote HLC observe、typed record state、merge repush outbox、quarantine upsert/resolve、page末尾cursorを1つのtransactionで確定し、どのwrite failureでも全rollbackする。unsupported envelopeはpage全rollback後にupgrade-required blockへ昇格する（2026-07-11改訂: ADR-012、ADR-014、ADR-016、task-88）。
4. **closure / local rebase**: normal pullではremote deletionをlocal tombstone stateへ反映し、後着のtaskが削除済みlist / ancestorへ到達する場合はdomainへmaterializeせず自身のtombstoneをtransactionalにenqueueする。expired-device full resyncではclosure後、server currentがliveのrecordはlocal mutationがあればcurrent revisionへtyped merge / rebaseし、なければremote currentを適用する。server currentがtombstoneのrecordはlocal live domain row / outboxを破棄し、remote tombstone stateを保持する。server ACK済みなのにcurrentから消えたrecordはGC済み削除としてdomain / state / outboxから破棄する。never-synced creation originのrecordは、pending key bundleとdependency closureが整いID衝突がない場合だけlocal-new seedとして保護する。serverに不在の既存list / parentへ依存するrecordは表示・再pushしない。local transactionが完了した後にだけ、server発行のclosure proofをACKし、serverがdevice continuityを単調更新する（2026-07-11追記: ADR-016）。
5. **pending key bundle push**: continuity closure / rebase完了後、clientは保護されたlocal-new listの`pending_list_key_bundles`を作成順にdrainする。`POST /v2/tenants/{tenant_id}/list-keys`の初回createまたはsame-ciphertext retryが成功したときだけ、送信したtenant/list/ciphertextと一致するlocal rowを短いtransactionでACKする。bundle uploadまたはlocal ACKが失敗したrunではentity outboxを読まない。server成功後・local ACK前の停止は同じqueued ciphertextを再送して回復する（2026-07-11改訂: ADR-013、ADR-016、task-90）。
6. **entity push**: pending key bundleが空になった後、clientはdependency順にlocal entity outboxをpushできる。`POST /v2/tenants/{tenant_id}/push` は`{op_id, record_id, collection, base_revision_hlc, revision_hlc, state}`を受け取る。`state`は`Live { mutation_hlc, blob }`または`Tombstone { delete_hlc }`である。serverは各pushでdevice continuityを再検証し、expired / old-generation deviceとtombstone済みrecordの`Live`を拒否する。base不一致はcurrent envelope付きconflictとし、clientはmerge/rebase後に再pushする。結果件数、`op_id`、record ID、collectionを照合してからoutboxをACKする。個別recordのmissing key/corruptionは無関係なlocal変更を永久停止させないが、未知protocol versionとcontinuity不適合はpush全体を停止する（2026-07-11改訂: ADR-014、ADR-016）。
7. マージの結果local値が勝ったfieldがあれば、field HLCと `mutation_hlc` を保持したマージ済みblobを**再push**する。remote revisionをobserveし、それより大きい `revision_hlc` を採番する。tombstone済みの同一record IDはマージ対象にせず、再作成は新IDで行う。push後はecho pullを行う。
8. マージ完了後、ローカルUIへ変更を反映する（リアクティブなクエリ購読を想定）。

初回バックフィルはtenant identityへ結び付いたlocal profileごとに一度だけ実行する。既存local dataを持つprofileを新規tenantへ登録するときは、tenant/profile紐付け後、各recordをnever-synced creation origin付きtransactional seed generationへ登録し、remote current-state scanとclosure後にkey bundle → list → task → timer sessionの依存順でpushする。同一tenantへの再loginだけを理由にcursorやserver-seen stateを削除しない（2026-07-13改訂: ADR-012、ADR-016、ADR-018）。

初回同期およびGC horizon超過後のfull resyncは、厳密な過去snapshotではなくstable-key current-state fuzzy scan + delta catch-upとする。開始transactionで `base_seq` を取得し、base rowを `seq <= base_seq` に限定せずstable keyで走査する。続いて `seq > base_seq` のdeltaを取得し、各delta pageと同じserver transactionで返す `high_water` に `has_more=false` のcursorが到達した時だけclosureとする。expired deviceは未ACK outboxを一律保護せず、server current / server-seen but absent / never-synced local-new / missing dependencyを分類してlocal rebaseを完了する。長期offline中の新規listは、durable origin、local List DEK、pending wrapped bundle、local-new集合内のdependency closureを検証できれば保護し、server不在だけでsweepしない（2026-07-11改訂: ADR-012、ADR-016）。

### 6.5 競合ケースと解決方針

| ケース | 内容 | 解決方針 |
|---|---|---|
| サブタスクの親子同時編集 | デバイスAが親タスクのstatusをdoneに、デバイスBが同じ親の子タスクを新規追加、というように異なるフィールド/異なる階層を同時編集 | 親タスクと子タスクは別レコードのため、レコード単位でのマージにより両方の変更が独立に反映される。親の完了と子の追加は互いを妨げず、UI側で「完了済み親タスクに未完了の子タスクが存在する」状態を許容し警告表示するのみとする |
| 同一タスクのstatus同時更新 | デバイスAがdoneに、デバイスBがin_progressに、ほぼ同時に更新 | フィールドレベルLWWにより、HLCがより新しい方の値を採用する。ユーザー体験上の取りこぼしを避けるため、直近で上書きされた側の変更を「変更履歴」としてローカルに一定期間保持し、必要であれば手動で復元できるようにする（将来拡張。初期リリースではLWWの結果をそのまま採用） |
| 削除と同時編集の競合 | デバイスAがタスクを削除し、デバイスBが同時に当該タスクの内容を編集 | serverがtombstoneを受理した同一record IDはterminal deletionとし、後着の編集HLCが大きくてもliveへ戻さない。編集側は先行pull / rebaseで削除を適用し、stale outboxを破棄する。ユーザーが内容を再作成する場合は新しいrecord IDを発行する（2026-07-11改訂: ADR-016） |
| 同一レコードの異なるフィールドを同時編集 | デバイスAがtitleを、デバイスBがnoteを、同じbaseから編集 | 先着pushだけがbase revision CASに成功する。後着pushはblind overwriteせずcurrent envelope付きconflictとなり、clientがfield mergeして新base/new revisionで再pushする。先着clientがACK直後に停止しても変更は失われない（ADR-014） |

### 6.6 サーバーが強制する不変条件とエッジケース

サーバーはE2EEデータの中身を解釈しない一方で、以下の不変条件をリクエスト単位で強制する（ADR-005）。

- クライアントによるサーバー同期ストアの物理DELETEは不可とし、削除はblobを持たないtagged tombstoneのみで表現する。live/tombstoneの不正なfield組合せはwire deserializeまたはDB CHECK制約で拒否する（2026-07-10改訂: ADR-010、ADR-014）。
- task subtree/list削除では、削除端末が把握しているrootとdescendantを削除前に列挙し、各recordの空blob tombstoneをdomain物理削除と同じlocal transactionでoutboxへ登録する。別端末の未知recordは、遅れてpullしたclientが復号済み`placement.list_id` / `parent_task_id`とlocal tombstone stateから削除済みancestorを検出し、domainへmaterializeせず自身のtombstoneをenqueueして収束させる。server-visibleなaggregate scope / epochは追加しない（2026-07-11改訂: ADR-012、ADR-016）。
- task subtree/list永久削除では関連する既知timer sessionも削除前に列挙し、session / task / list tombstone、domain物理削除、record state、outbox、HLCを同じlocal transactionへ含める。別端末から遅着したtimer sessionは、復号したtask IDのlocal taskがterminal tombstone済みならmaterializeせずsession tombstoneをenqueueする。task completionのUndoはsessionやactive Timerを復元しない（2026-07-13追記: ADR-018、task-105）。
- push endpointは、serverが保持するdevice別`continuity_seq` / `continuity_generation`が`gc_horizon_seq`以上の有効世代であることをkey bundle / entityの書込ごとに検証する。expired deviceはserver current-state closureとlocal rebaseが完了するまでpushできない（2026-07-11追記: ADR-016）。
- normal / full-resync pull closureで発行するproofはtenant / device / high-water / generationへ束縛し、別tenant / deviceからのACK、古いgeneration、改変high-waterを拒否する。ACK前の停止ではcontinuityを進めず、同じclosureの再取得と冪等ACKで回復する（2026-07-11追記: ADR-016、task-98）。
- 暗号blobのサイズには上限を設ける（初期値64KB）。
- push/pull 1リクエストあたりのレコード件数（バッチ）にも上限を設ける。
- `collection` はprotocolで列挙した値だけを許可し、既存 `(tenant_id, record_id)` のcollection変更を拒否する。push結果はrequestの順序だけで対応付けず、`op_id`、件数、record IDを検証してからACKする（2026-07-10改訂: ADR-012）。
- pushされたレコードのHLC物理時刻がサーバー時刻より5分を超えて未来である場合、当該pushは拒否する（端末の時計異常や「未来HLCによる永久勝ち」攻撃の抑止のため）。
- 課金の有効性・Orgメンバーシップの検証はAPIリクエストごとにサーバー側で実施する。
- tenant分離の多層防御は、non-owner application role、実際のRLS policy、必要な `FORCE ROW LEVEL SECURITY`、cross-tenant testが揃った状態を指す。`ENABLE ROW LEVEL SECURITY` だけでは完了扱いにしない（2026-07-10改訂: ADR-012）。

エッジケースへの対策は以下のとおりである。

- **Orgメンバーによる破壊的上書き**: 共有リストにおいて悪意ある（または誤操作の）メンバーが他者の変更を意図せず/意図的に上書きした場合に備え、上書き時の旧行を `sync_records_history` へ退避し30日保持する。これにより復旧手段を確保する。
- **tombstone GCとオフライン端末**: tombstoneと通常offline復帰保証は180日とし、GC実行時にテナントごとの `gc_horizon_seq` を単調前進させる。`devices.last_pull_at`やclient申告cursorはGC窓超過判定に使わない。server-trusted continuityがhorizon未満のdeviceはexpiredとし、stable-key fuzzy scan、`base_seq`後delta、high-water closure、server-seen / local-new / dependency closure別のlocal rebaseを完了するまでpushを拒否する。これによりtombstoneを永久保持せず、長期offline中の正当な新規listだけをreseedできる（2026-07-11改訂: ADR-010、ADR-012、ADR-016）。

## 7. アカウントフロー

### 7.1 未登録利用開始

1. アプリ初回起動時、Device Key (DK) を生成しOSキーチェーンへ保存。
2. DK由来の鍵でローカルDB（SQLite + SQLCipher）を初期化・暗号化。
3. アカウント登録なしでリスト・タスクの作成/編集がフル機能で可能（同期・共有・課金プランに紐づく上位機能を除く）。

### 7.2 新規登録（既存ローカルデータの移行を含む）

1. メールアドレス・パスワードを入力し、クライアントでOPAQUEレジストレーションを実行（パスワードはサーバーへ非送信）。
2. exportKeyからHKDFでKEK_pwを導出。
3. Master Key (MK) を新規生成。
4. 既存ローカルデータ（DK配下で暗号化済み）をMK配下へ再暗号化して移行。
5. `wrap(MK, KEK_pw)` と `wrap(MK, DK)` を生成し、それぞれサーバー・ローカルに保存。
6. Recovery Keyを生成し、`wrap(MK, RecoveryKey)` をサーバーに保存。Recovery Key自体はユーザーに提示し、安全な場所への保管を促す（サーバー・端末には平文を残さない）。
7. User X25519鍵ペアを生成し、公開鍵をサーバーへ、秘密鍵を`wrap(SK, MK)`としてサーバーへ保存。
8. サーバー側でPostgres内にテナントレコードを作成する。サインアップフローにはリージョン選択のステップが含まれる（初期はデータレジデンシー未対応のため選択肢は1つのみ。[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-006参照）。

### 7.3 ログイン・新デバイス追加

1. 新デバイスでメールアドレス・パスワードを入力し、OPAQUEログインを実行。
2. exportKeyからKEK_pwを導出し、サーバーから取得した `wrap(MK, KEK_pw)` を復号してMKを得る。
3. MKで `wrap(SK, MK)` を復号し、User X25519秘密鍵を取得（Org共有リストの復号に利用）。
4. 新デバイスのDevice Keyを生成し、`wrap(MK, DK)` をローカルに保存（4.3節の定義どおり、DKによるMKラップはローカル保存とする）。サーバーのデバイスリストへは新規デバイスのメタデータ（デバイス名・登録日時等）のみを登録。
5. セッション認証（初期はDB照合のopaqueトークン）を確立した上で同期API（6章）のstable-key fuzzy scan + delta full resyncを実行し、ローカルDBを構築。

### 7.4 パスワード変更

1. 現在のパスワードでOPAQUE認証し、MKを取得済みの状態にする。
2. 新パスワードでOPAQUEレジストレーションを再実行。
3. 新exportKeyからKEK_pw'を導出し、`wrap(MK, KEK_pw')` を再計算してサーバー側の値を更新。
4. データ本体の再暗号化は不要（4.5節参照）。

### 7.5 リカバリーキーでの復旧

1. パスワードを忘れた場合、ユーザーが保管しているRecovery Keyを入力。
2. サーバーから `wrap(MK, RecoveryKey)` を取得し、Recovery KeyでMKを復号。
3. MKが得られたら、新しいパスワードを設定しOPAQUE再登録を実行、`wrap(MK, KEK_pw_new)` を生成してサーバーへ保存。
4. 新しいRecovery Keyを再発行し、旧Recovery Keyを無効化（サーバー側の `wrap(MK, RecoveryKey)` を新しい値で上書き）。

### 7.6 デバイス失効

1. ユーザーが既存デバイス一覧から対象デバイスを選択し失効操作を実行。
2. サーバー側で該当デバイスの `devices.revoked_at` を設定する。同期API（push/pull）はリクエストごとに `devices.revoked_at` を検証するため、**失効は即時に実効化される**（旧設計のような発行済みトークンのTTL経過待ちは発生せず、失効反映の遅延はない）。
3. Organizationの共有リストに参加していた場合、当該デバイスに紐づくユーザーの鍵自体は失効しない（ユーザー単位の鍵のため）。デバイス単位の失効はあくまで同期アクセスの遮断であり、より強固に締め出す必要がある場合（デバイス盗難等）は、別途アカウントパスワード変更・Org DEKローテーションを併用することを推奨する運用とする。

## 8. MCP / CLI

### 8.1 共通方針

CLIおよびMCPサーバーは、いずれもfrontend共通入口の`todori-client` crateにある`TodoriClient`を通じてローカルの暗号化DBへ直接アクセスし、**サーバーAPIを経由しない**。`TodoriClient::open`へはDB directoryとbootstrap値だけを持つ`LocalProfileConfig`を渡す。これにより、オフライン環境でもCLI/MCP双方がフル機能で動作し、同期は`todori-client`のcoordinatorが`todori-sync`のstate machineを用いて非同期に処理する。crate境界の詳細は[`dev/client-profile-architecture.md`](./dev/client-profile-architecture.md)を参照する（2026-07-10改訂: task-91、task-94）。

Windows / macOS / Linuxでは、同一PC上のFlutterデスクトップ版、CLI、MCPサーバーが同じTodori local profile（SQLCipher DB + OS secret store上のDevice Key参照）を共有する。異なるPCやOS間でDBファイルやDevice Keyをコピーせず、各端末のprofile間はE2EE同期で収束させる（ADR-011）。

OS別の本番secret storeは、macOSが同一Apple Team / Keychain access groupのData Protection Keychain、Windowsがcurrent-user scopeのDPAPI、LinuxがSecret Service APIとする。Windows / Linuxでは平文 `device.key` fallbackを開発・テスト用途に限定し、Secret Service等が利用できない場合に本番CLIが平文保存へ自動降格してはならない。

同一DBへの多重process接続では、connectionごとの `busy_timeout`、短い書込transaction、migration排他を必須とする。domain更新と同期outbox登録は同一transactionで確定し、同期runはDB-backed lease等で同一profileにつき1つへ制限する。

### 8.2 MCPサーバー

- トランスポートは **stdio** を採用する（ローカルプロセス間通信であり、ネットワーク経由でのE2EEデータ露出を避けるため）。
- 提供するtoolsの例:

| tool名 | 概要 |
|---|---|
| `list_tasks` | 指定リスト・条件（status、期限等）でタスク一覧を取得 |
| `create_task` | 新規タスクを作成（親タスク指定によるサブタスク作成も可） |
| `complete_task` | タスクを完了状態に更新 |
| `search_tasks` | 全文検索（ローカルFTS5インデックス経由） |

- 初回のtool呼び出し時、ローカルDBへのアクセスに対するユーザーの明示的な許可プロンプトを表示し、以後のセッションでの利用可否を確認する。

### 8.3 CLI

Rustバイナリとして `clap` を用いてサブコマンド体系を構築する。

```
todori add "牛乳を買う" --due tomorrow
todori add "資料作成" --list work --priority 2 --parent <task-id>
todori list --list work --status todo
todori done <task-id>
todori search "牛乳"
```

CLI・MCPサーバーともに、初回起動時に対象アカウントのローカルDB（未登録時はDK保護のローカルのみのDB、登録済みならMK保護のDB）へ接続するための鍵取得フロー（OSキーチェーンからのDK取得、必要であればパスワード入力によるMK取得）を実行する。

macOS CLIはFlutter appと同じKeychain access groupを持つ署名済みバイナリとして配布する。Windows CLIは同一WindowsユーザーのDPAPI保護secret、Linux CLIは同一ログインセッションのSecret Service itemを取得する。profile / data-dirを明示指定できる場合もDevice Keyを引数、環境変数、ログへ露出してはならない。

## 9. ローカルAI

### 9.1 方針

推論は**オンデバイスのみ**で実行し、タスク内容がサーバーはもとより外部のAI APIへ送信されることも一切ない。これによりE2EEの前提（サーバーがコンテンツを知り得ない）をAI機能導入後も維持する。

### 9.2 実装

- 推論エンジンには llama.cpp系のRustバインディング、または `candle` を用いる。
- モデルはGGUF量子化形式、2〜4Bパラメータクラスの軽量モデルを対象とし、ユーザーが任意のタイミングでダウンロード・削除できるモデル管理UIを提供する（初期状態ではモデル未同梱とし、アプリサイズの肥大化を避ける）。

### 9.3 対応機能

| 機能 | 概要 |
|---|---|
| 自然言語パース | 「明日の15時に資料作成」のような自然文からタイトル・期限・リストを抽出しタスク化 |
| タスク分解 | 大きなタスクをサブタスクの候補に分解して提案 |
| 週次要約 | 完了/未完了タスクの傾向をローカルで要約しレポート表示 |

### 9.4 低スペック端末フォールバック

推論に必要な最低メモリ/CPU要件を満たさない端末では、AI機能そのものをUI上非表示にし、モデルダウンロードや推論呼び出しを行わないようにする。判定は端末のRAM容量・CPUコア数等の静的情報に基づき、アプリ起動時に一度だけ実施する。

## 10. 課金・プラン制御

本章は概要である。公開向けの課金方針は `billing_overview.md` を参照する。具体価格、収益試算、provider別の運用判断はprivate repo側で管理する。

### 10.1 プラン体系

Free / Pro / Org の3プランを設け、機能をプラン単位でゲートする。

- **Free**: ローカルで完結する全機能（リスト・タスク・タグ・リマインダー・テンプレート・スケジュール自動起票・タイマー・統計・ローカルAI・CLI/MCP を含む）。単一端末利用。同期・共有は不可。
- **Pro**: E2EE同期（マルチデバイス）、暗号化クラウドバックアップ。
- **Org**: Organization共有、担当者アサイン・コメント、メンバー管理、監査ログ。

機能ゲートはサーバー依存機能（同期・共有）に限定され、ローカル機能はプランによらず制限しない。

### 10.2 実装方針

- レシート検証（App Store / Play / 外部課金プロバイダ）はすべて**サーバー側**で実施し、検証結果に基づき機能フラグをアカウント（`users.plan`等）に紐づける。
- クライアントはサーバーから取得した機能フラグに従いUI上の機能出し分けを行う。フラグの正当性はサーバー起点であり、クライアント側での改ざんに対しては、機能ゲートが必要な操作（例: Org作成、同期API（push/pull）へのアクセス）についてはサーバー側APIでリクエストごとに再検証する（6.1節参照）。トークン発行時点の一括判定ではなく、リクエスト単位の検証であるため課金失効は同期アクセスへ即時に反映される。
- サブスクリプション管理サービスは実装時点の要件に応じて選定する。エンタイトルメント判定の唯一の真実はあくまで自社サーバー側とし、外部課金サービスはその更新ソースとして抽象化する。

### 10.3 課金失効時の挙動

課金が失効（サブスクリプション解約・支払い失敗等）した場合でも、**ローカルデータは全機能で編集を継続できる**ものとする。失効により制限されるのはサーバーとの同期（およびOrg共有等のサーバー依存機能）のみであり、ユーザーのデータそのものを人質に取らない方針を明確にする。同期停止中に行われたローカル変更は、再度課金が有効化された時点で通常のpush/pullフローにより同期される。

## 11. テスト・品質

### 11.1 Rustコア

- **暗号処理**: OPAQUE、HKDF、XChaCha20-Poly1305、X25519 sealed boxについて、既知のテストベクタ（RFC/参照実装由来）を用いた単体テストを整備する。
- **同期エンジン**: property testと2-client統合testはproduction CRUD / common client経路を通す。異field同時編集、remote revision observe、terminal tombstone済みrecordの同一ID復活拒否、遅着descendantのclient-side cascade、task/list reorder、placement原子性、rank衝突、local write/outbox/record-state/HLC/cursor各crash window、missing DEK / corrupt blobのdurable quarantine、unknown protocolのupgrade-required、push response照合を自動テストする（2026-07-11改訂: ADR-012、ADR-016）。
- **フル再同期 / GC**: 新規tenantへのnever-synced seed、server-trusted continuityがexpiredの場合のkey bundle / entity push拒否、server-seen but absent recordの破棄、長期offline中の新規listとtaskの保護、missing/deleted dependencyの除外、空server、resync中の同時更新、stable-key fuzzy scan、`base_seq` 後deltaとpage transactionのhigh-water closureを自動テストする（2026-07-11改訂: ADR-010、ADR-012、ADR-016）。

### 11.2 Flutter

- 主要画面のwidget testと、実DBを用いたシナリオベースのintegration test（タスク作成〜同期〜他デバイス反映等）を整備する。

### 11.3 脆弱性対応

- 依存クレートの脆弱性監査に `cargo audit` をCIパイプラインへ組み込み、定期的（週次等）に実行する。
- E2EE設計・実装については、正式リリース前に**外部の第三者機関によるセキュリティ監査**を実施することを推奨する。監査範囲には鍵階層、OPAQUE実装、Org共有のsealed box実装、サーバー保存データの解析可能性を含める。

- 翻訳管理プラットフォーム（Crowdin / Weblate 等）の選定、および追加言語の優先順位（市場規模・ストア地域統計に基づく）。
- Lambda公開エンドポイントの前段構成（CloudFront + WAF / API Gateway スロットリング）の選定。
- 同期APIのクォータ・blobサイズ上限（6.6節、初期64KB）・バッチ上限の最終値。
- `sync_records_history` 保持期間（初期30日）の最終値。削除対象のhistoryは即時削除し、contentを持たないtombstoneを180日保持後にGCする。`gc_horizon_seq`、server-trusted device continuity、expired-device rebaseで復帰を制御する方針は確定済み（2026-07-11改訂: ADR-010、ADR-016承認）。
- 転送エンコードのCBOR化（6.4節）の実施時期。
- リージョン拡張（データレジデンシー対応、[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-006参照）の実施条件。
- テナント間コピー操作の詳細仕様（サブタスク・コメント・リマインダーを含めるか、コピー時の帰属の扱い。[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-007参照）。
- Neonの最新料金プラン・接続プーラーのprepared statement対応（sqlxが使用）・リージョン提供状況の確認（実装開始時に実施。ADR-008の前提検証）。

なお、CRDTライブラリの選定（自前実装 vs 既存CRDT拡張の利用）はADR-004にて自前実装（フィールドレベルLWW + HLC + fractional indexing）で決定済みである。同期トークンのTTLの最終値は、同期プロトコルがトークンTTLに依存しない設計（ADR-002・ADR-005）となったため概念自体が消滅した。コントロールプレーンの永続メタデータ保存先はADR-003にてPostgres一本化で決定済みである。サーバーレスPostgresプロバイダと接続方式はADR-008にてNeon（eu-central-1、プーラー経由TCP+sqlx）で決定済みである。
