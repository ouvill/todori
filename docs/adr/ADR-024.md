# ADR-024: TemplateとTask Seriesを独立させ、Task Blueprintを値コピーする

**日付**: 2026-07-24
**状態**: Accepted（2026-07-24 プロダクトオーナー承認）

### 背景

ADR-021では不定期に再利用するtask subtreeを`TaskTemplate`、RRULEに従って自動起票する規則を`RecurrenceSchedule`として実装した。ScheduleはTemplate IDを必須参照し、Template削除時にScheduleも削除する。この構造では繰り返しだけを設定したい利用者もTemplateを先に作る必要があり、Template編集は参照Scheduleのsettlement、schedule revisionとtemplate revisionの組み合わせ、2本のlineageを必要とする。

Templateの利用意図は人間が必要なときに起票する不定期の定型作業であり、Task Seriesの利用意図は規則に従う自動起票である。両者が共有すべきものはライフサイクルではなく、新しいTaskへコピーできる内容とsubtree構造だけである。

### 決定

1. 再利用可能な内容を`TaskContent`、subtree内の位置を加えたnodeを`TaskBlueprintNode`、1 rootのtree全体を`TaskBlueprint`とする。TaskContentはtitle、note、priority、estimated minutesだけを持つ。BlueprintNodeは安定node key、nullableなparent node key、sibling order、TaskContentを持つ。Blueprintはschema revision、最大100 node、49,152 UTF-8 bytesの既存制約を維持する。
2. `TaskTemplate`はID、表示名、nullableな既定list、TaskBlueprint、同期revision、作成・更新日時を持つ独立Entityとする。空のBlueprintから直接作成・編集でき、既存Task subtreeからの保存と別Templateの複製も同じBlueprint入力へ正規化する。
3. `RecurrenceSchedule`を`TaskSeries`へ置き換える。TaskSeriesはTaskBlueprint、nullableな起票先list、正規化RRULE、開始instant、IANA timezone、enabledを1つのatomic configとして所有し、config revision、parent revision、effective-from、lineage、cursor、作成・更新日時を持つ。
4. TemplateからTaskまたはTaskSeriesを作る操作はBlueprintの値コピーとする。TaskSeriesはTemplate IDを保存せず、Templateの編集・削除は既存Task / TaskSeriesへ作用しない。Task subtreeからTemplateまたはTaskSeriesを直接作ることもできる。
5. TaskTemplate削除はTemplateだけをtombstone化する。TaskSeries削除はSeriesだけをtombstone化し、生成済みTaskを変更しない。生成済みTaskの編集・完了も元のTemplate / Series configへ逆伝播しない。
6. Series configとcursorは別の同期field groupとする。同一config revisionのcursorは`Pending(instant) < Exhausted`で単調mergeし、異なるrevisionのcursorを比較しない。config編集は従来どおり編集instant以下を旧revisionで全settleした後にfuture-onlyでcutoverする。
7. Seriesから生成したTask treeは全nodeにseries ID、series config revision、occurrence instant、blueprint node keyからなるprovenanceを持つ。rootだけがoccurrence instantを`scheduled_at`へ持つ。手動TaskとTemplateからの手動起票はseries provenanceを持たない。
8. Series生成Task IDはseries IDをnamespaceとし、series config revision、occurrence instant、blueprint node keyをcanonical bytesで連結したUUIDv5とする。Template revisionをID入力から除く。
9. streakは同一TaskSeriesのroot occurrenceを時系列評価する。既存のdone期限、wont-do、reopen、有限規則のvirtual next occurrence契約を維持する。
10. local schemaはv22、sync protocolはv8とする。`schedules` collectionを`task_series`へ置き換え、plaintext kindも`task_series`とする。暗号化された旧Template / Schedule / Task plaintextはserver側で変換できないため、一般配布前の互換性方針に従ってserverの旧`tasks` / `templates` / `schedules` recordを削除する。local v22 migrationは通常Taskを保持して旧recurrence provenanceだけを外し、同期metadataを初期化して再seedする。旧Template / Scheduleはlocalでも再作成し、dual read / writeは追加しない。Template、TaskSeries、Taskを含む全RecordはADR-023のTenant Record Key境界を使い、serverは平文Blueprint、RRULE、list linkage、生成処理を扱わない。
11. full resyncと初回backfillは`templates`と`task_series`の間に依存順を持たない。Taskはseries recordが未取得でもprovenanceをtypedに保持できるが、settlementとstreakはSeries取得後に行う。
12. FlutterではTemplateとTask Seriesを別の管理対象として表示する。Template editorはTask editorのBlueprint modeを再利用し、status、due、scheduled time、reminder、timer、assignee、Task identityを表示・保存しない。

### 帰結

- 不定期の定型作業と周期的な自動起票を独立して利用できる。
- Templateを削除・編集してもTask Seriesが停止または変化しない。
- Seriesのfuture-only編集は1本のconfig revisionだけを扱い、schedule revisionとtemplate revisionの直積を廃止できる。
- Template editorとTask editorはTaskContentの語彙を共有しつつ、実行状態をBlueprintへ混入させない。
- schema v22、protocol v8、domain、storage、sync、server、FRB、Flutterへbreaking changeが広がる。

### 却下した代替案

- **TemplateをTask Seriesの必須親として維持する案**: 手動再利用と自動起票のライフサイクルを結合し続けるため不採用。
- **TaskへRRULEとcursorを直接持たせる案**: 1回分の実行状態と将来の生成定義が混在し、「この回だけ」と「今後すべて」の境界が曖昧になるため不採用。
- **Template IDをsource provenanceとしてSeriesへ残す案**: 利用者が期待するlive linkと単なる出自情報を区別できず、削除・同期依存を再導入するため不採用。
- **Template更新を既存Seriesへ自動伝播する案**: offline concurrent編集時に明示されないfuture-only変更を起こすため不採用。必要なら利用者がSeriesへ明示的に新Blueprintをコピーする。
- **Blueprintを独立同期Entityにする案**: TemplateとSeriesのライフサイクルを再結合し、単独削除と権限境界を複雑にするため不採用。
