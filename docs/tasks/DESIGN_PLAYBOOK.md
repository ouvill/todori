# デザイン実装プレイブック ── codexに「雰囲気」を実装させる方法

## 1. なぜ従来のpolishタスクは参照画像の雰囲気に近づかなかったか

task-25/28/29では「落ち着いた」「整える」「プロダクト品質へ引き上げる」といった形容詞ベースの指示を出していた。この種の指示は実装エージェントには検証不能であり、実装側の自己申告のpolishで完了扱いになってしまう。

実装エージェントはスクリーンショットを見ずに作業していた。task-28では目視QA用のテストがその場限りで作られ、資産化されずに使い捨てられていたため、次のタスクへ引き継がれなかった。

配色トークンが仕様どおりに合っていても、タイポグラフィと行密度が伴わなければ参照画像の雰囲気は再現されない。task-30で、ブランドフォント（Lora/Inter）の導入とメタデータチップの削減だけで第一印象が一変したことが実証されている。

## 2. 原則

1. **見ずに描かせない**: ビジュアル系タスクでは必ず `sh app/tool/visual_qa.sh`（`app/build/visual_qa/` にPNG出力）でbefore/afterスクリーンショットを生成させ、完了報告に保存パスを記録させる。親/検証者はPNGを自分の目で見て判定する。
2. **形容詞ではなく差分で指示する**: 「メタデータを控えめに」ではなく「Statusチップを削除する」「`Due: 2026-07-05` を相対表記（Today/Tomorrow/MMMd）に変える」のように書く。受け入れ基準は「スクリーンショットで確認できる文」で書く。
3. **「雰囲気」を5変数に分解してギャップ分析する**: ①タイポグラフィ ②行密度・余白 ③チップ/装飾の量 ④配色トークン ⑤角丸/border。参照画像と現状スクショをこの5軸で比較し、1タスクで扱う軸を明示する。
4. **参照画像は採用リストに翻訳してから渡す**: 画像を「これに近づけて」と直接渡さない。`docs/design/visual-direction.md` のAdopt/Do not adopt形式に翻訳し、指示書からはそのルールを参照させる。
5. **禁止事項の再発を受け入れ基準で検出する**: 一度削除した要素（例: Local protectionチップ）が別画面に残っていないかをチェック項目にする（task-30でdetail画面に残存していた実例がある）。
6. **実装と判定を分離する**: workerに合否を出させない。検証は親または別セッションがスクショを見て行い、不合格は「画面名+具体差分」の形で修正指示にする。

## 3. ビジュアルタスクの標準ループ

1. 親/人間: `visual_qa.sh` を実行し、現状スクショと参照画像（`assets/brand/generated/`）を5軸で比較してギャップを列挙する。
2. planner: ギャップを指示書化する（`docs/tasks/task-NN-*.md`。受け入れ基準はスクショ検証可能な文で書く）。
3. worker: `app/build/visual_qa/` を `visual_qa_before/` へ退避してから実装し、`visual_qa.sh` でafterを生成し、PNGを自分でも目視し、品質ゲートを通し、完了報告にbefore/afterのパスを記録する。
4. 親/verifier: afterスクショを目視し、受け入れ基準と参照画像に照らして合否判定する。
5. 不合格時: 「`home_tasks.png` でシェブロンがまだ最下行にある。行右端に垂直センター配置へ」のような画面単位の具体指摘で修正セッションへ渡す。

## 4. スクショレビューのチェックリスト

- 見出しにディスプレイ書体（Lora）が効いているか / 本文がInterか。
- タスク行1件の高さ、チップの数と文言の長さ（プレフィックス付き冗長ラベルの再発）。
- 同じ情報（pending数等）の重複表示。
- visual-direction.mdのトークン外の色の混入。
- 禁止事項の混入（ロック/暗号化マーク、マスコット常駐、confetti、card-in-card、heavy shadow）。
- 日本語・長文タイトル・狭幅での折返し破綻。
- デバッグバナー以外の描画異常（豆腐文字はハーネスのフォント登録漏れ）。

## 5. worker/verifier向け定型プロンプト追加句

PLAYBOOK.mdの役割別雛形に追加して使う。

worker追加句:

```
作業前に app/build/visual_qa/ を app/build/visual_qa_before/ へ退避し、実装後に sh app/tool/visual_qa.sh を実行し、
生成された全PNGを自分で目視確認し、完了報告にbefore/afterのパスと確認結果を書け。
```

verifier追加句:

```
sh app/tool/visual_qa.sh を自分で再実行し、PNGを自分の目で確認して受け入れ基準のスクリーンショット項目を判定せよ。
完了報告の記述を信用するな。
```

## 6. 参照資産

- 参照画像: `assets/brand/generated/`。ただし正本は `docs/design/visual-direction.md` の採用/不採用ルールであり、画像はピクセル基準ではない。
- スクショ基盤: `app/test/visual_qa/visual_qa_screenshots_test.dart` + `app/tool/visual_qa.sh`（`TODORI_VISUAL_QA=1` のときだけ実行する。CIには影響しない）。
- シードデータ: `app/test/support/fake_bridge_service.dart`。優先度・期日・完了・サブタスク・日英混在を含む現実的なデータにしておくこと。スクショの説得力はシードデータで決まる。
- 実績: task-30（この方式で参照画像の雰囲気へ最初に到達したタスク。before/after比較の実例）。
