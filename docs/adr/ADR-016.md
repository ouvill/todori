# ADR-016: Archive-first削除同期はbounded tombstoneとexpired-device rebaseで行う

**日付**: 2026-07-11
**状態**: Accepted（2026-07-11 プロダクトオーナー承認）

### 背景

ADR-009は、完了済みtaskを振り返りのための記録資産とし、通常の整理を`done` / `wont_do`とリストのアーカイブ、永久削除をミス操作やノイズを本当に消すための例外操作と定めた。ADR-010は削除済みコンテンツblobを残さないtagged tombstoneを180日保持し、GC horizonを超えた端末をfull resyncへ送る方針を採用した。

ADR-012は、別端末の未知descendantをserverが即時拒否する完全なaggregate削除のため、scope deletion intent / epochの別裁定を後続要求とした。しかし、serverが暗号payload内の`list_id` / `parent_task_id`を知らず、offline端末が未知recordを後から送れる条件で、serverがそのrecordを削除済みaggregateの一部と即時判定するには、scopeまたはhierarchyに相当するserver-visible metadataが必要になる。これは、削除が例外操作であるTaskveilに対して、E2EE metadataとprotocol complexityを恒常的に増やす。

一方、tombstoneを永久保持すればscope metadataなしで同一record IDの復活を防げるが、削除済みIDが無期限に増え、削除痕跡をserverが永久保持する。Taskveilでは、tombstone保持期間をoffline deviceの通常復帰保証期間と一致させ、期間を超えた端末はserver current stateへrebaseする方が、archiveの長期保全と削除metadataの有限保持を分離できる。

### 決定

1. 製品上の通常整理は、taskの`done` / `wont_do`とリストのアーカイブとする。永久削除は、誤作成または残したくない情報を本当に消すための例外的・不可逆な操作とし、ADR-009の深い導線と追加確認を維持する。
2. Phase 2の個人tenantでは、aggregate scope ID、scope epoch、list ID、parent ID、ancestor setをserver-visible同期metadataとして追加しない。serverはrecord間の親子・所属関係を解釈しない。
3. 削除はADR-014の`Tombstone { delete_hlc }`として表現し、content blobを持たない。tombstone受理時は対象recordの旧contentを`sync_records_history`へ新規退避せず、既存historyも削除する。serverが保持するのは削除伝播に必要な最小metadataだけとする。
4. tombstone保持中、同一record IDの`tombstone → live`遷移はHLCの大小にかかわらず拒否する。永久削除した内容を再作成する場合は新しいrecord IDを発行する。
5. task subtree / list削除時は、削除端末が知るrootとdescendantを同一local transactionで物理削除し、各recordのtombstoneをoutboxへ登録する。遅れて到着した未知taskはclientが復号後に`placement.list_id` / `parent_task_id`とlocal tombstone stateを検証し、削除済みancestorへ到達する場合はdomainへmaterializeせず、自身のtombstoneをtransactionalにenqueueする。missing dependencyは通常taskとして表示・pushしない。
6. 通常syncは、protocol / continuity preflight、remote pullと削除reconcileをentity outbox pushより先に行う。これにより、offline端末が受信済み削除を知らずにstale descendantを先行pushする経路を閉じる。必要なkey refreshは復号reconcileの前、local-new List DEK bundle uploadはrebase後かつentity push前に行う。
7. tombstoneの初期保持期間と通常offline復帰保証期間は180日とする。GC時は`gc_horizon_seq`を単調前進させる。保持期間を超え、server-trusted device cursorがhorizon未満の端末はexpired continuity状態とする。
8. push endpointはclient申告だけを信用せず、認証済みdeviceについてserverが確定したcontinuity cursor / full-resync generationが`gc_horizon_seq`以上であることを検証する。expired deviceはfull resync closureとlocal rebaseを完了するまでkey bundle / entity pushを行えない。対応前protocolのclientはversion gateで拒否する。
9. expired-device full resyncはremote current stateをclosureまで取得した後、local recordを次のように分類する。
   - server currentがliveのrecord: local mutationがあればcurrent revisionへtyped merge / rebaseし、成功したものだけを新outbox headとして保持する。local mutationがなければremote currentを適用する。
   - server currentがtombstoneのrecord: local live domain rowとoutboxを破棄し、remote tombstone stateを保持して同一record IDを復活させない。
   - 一度serverへ存在したがcurrent stateに不在のrecord: tombstoneがGC済みの削除recordとみなし、local domain row、record state、outboxを削除して再pushしない。
   - 一度もserverへ存在しないlocal-new record: durable local origin、未ACK creation、必要なpending key bundle、dependency closureを検証し、正当なものだけを新規seedとして保持する。
   - serverに不在の既存list / parentへ依存するrecord、またはdependency chainを解決できないrecord: domainへmaterializeせず、outboxから除外して削除へ収束させる。
10. local-new判定を`base_revision_hlc = null`だけから推測しない。local sync stateは、server ACK済みrevisionまたはnever-synced creation originをdurableに区別する。outbox coalesce、再起動、crash recovery後もこの区別を失わない。
11. 長期offline中に新規作成したリストは、never-synced local origin、local List DEK、pending wrapped List DEK bundleを持ち、IDがremote current / tombstoneと衝突せず、配下recordのdependency closureがそのlocal-new集合内で成立する場合、full resync後も保持してbundle → list → taskの順に再seedする。server不在だけを理由に削除しない。
12. List DEK bundleは、list tombstone保持中、late descendantの復号・分類中、またはexpired-device rebase / key retirement条件が未実装の間は削除しない。retirementは、tombstoneがGC可能で、active deviceが削除seqを通過またはexpiredとなり、full resync closureで既知current recordが分類済みで、expired deviceのstale pushがserverで拒否される場合に限る。詳細なtransactionとAPIは後続実装taskで固定する。
13. offline端末は接続するまで遠隔消去できない。Taskveilが保証するのは、次回syncでcontinuity / protocol gateを通り、削除済みancestor配下recordを表示・再pushせず、server current stateへ収束することである。
14. Organization共有で悪意あるmember clientをserverがscope単位に遮断する要件が生じた場合は、Phase 3の認可・脅威モデルとしてopaque capability / scope metadataを別ADRで再評価する。個人tenantの削除例外のために先行導入しない。

### 帰結

- ADR-012のaggregate scope deletion intent / epoch後続要求は撤回し、server-visible hierarchy metadataを追加しない。
- task-95のstable-key fuzzy full resync、high-water closure、GC horizonは維持するが、未ACK outboxを一律保護する規則を、server-seen / local-new / dependency closureを区別するexpired-device rebaseへ拡張する。
- normal sync順、server-trusted device continuity、terminal tombstone、history purge、local origin永続化、client-side late descendant cascadeは後続の重要変更taskで実装する。
- archive recordはユーザーの長期資産として保持される一方、削除伝播用tombstoneは180日とdevice continuity条件で有限にGCできる。
- ADR-015のcanonical Inbox alias / List DEK bundleは、aggregate scopeではなく本ADRのlate descendant収束とkey retirement条件へ依存する。

### 却下した代替案

- **aggregate scope ID / epochを全recordへ追加する案**: serverによる即時拒否は可能だが、例外的な削除のために所属metadataとprotocol complexityを全recordへ恒常的に追加するため不採用。
- **tombstoneをtenant存続中永久保持する案**: 実装は単純だが、削除済みIDと削除痕跡が無期限に蓄積するため不採用。
- **expired deviceのlocal dataを全件破棄する案**: 削除復活は防げるが、長期offline中に正当に作成した新規リストとtaskまで失うため不採用。
- **full resyncで未ACK outboxをすべて保護する案**: local-newは救済できるが、GC済み削除recordと削除済みlist配下taskを再pushし得るため不採用。
- **List DEK bundleをlist削除と同時に削除する案**: late descendantを復号・分類できず、暗号blobを回復不能なquarantineへ送るため不採用。
- **entity pushを通常pullより先に行う案**: offline端末がserver上の削除を受信する前にstale recordを送るため、archive-first削除同期では不採用。
