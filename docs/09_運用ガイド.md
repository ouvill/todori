## 2. 全体像

TodoriはローカルファーストのE2EE Todoアプリである。クライアントはSQLCipherで暗号化されたローカルDBを正本として動作し、同期が利用可能な場合だけサーバーのpush/pull APIへ暗号blobを送受信する。

サーバー構成の正は [`docs/03_技術仕様書.md`](./03_技術仕様書.md) §1.5 と §6である。サーバーはAWS Lambda上のRust axumサーバーとして動かし、コンテナイメージをECRへpushしてLambdaへ更新する想定である。DBはNeon Postgresを採用し、初期リージョンは `eu-central-1`、Lambdaも同一リージョンに置く。設計判断は [`docs/05_設計判断記録.md`](./05_設計判断記録.md) ADR-008を参照する。

2026-07-08時点では実AWS/Neon環境へは未デプロイである。運用済みの環境はローカルPostgres + `todori-server` のみで、起動は [`tool/dev_server.sh`](../tool/dev_server.sh) が担う。

## 3. リリース状態と互換性

Todoriは現在、一般リリース前である。プロダクトオーナーがこの状態を変更するまで、既存client、wire protocol、API、local DB schema、server schema、開発データとの後方互換性は要件としない。

correctness・security・設計の一貫性を優先し、breaking change、破壊的migration、開発データの再作成を許容する。互換レイヤ、dual read/write、旧形式fallbackは、taskで明示的に要求されない限り追加しない。局所的な互換修正より、taskまたはADRで判断を記録したうえで正しい最終設計へ直接置き換える。

最初のbetaを含む外部配布を開始する前に、`AGENTS.md`と本節を更新し、配布済みclient、local DB、server API、保存済みデータに対する互換性境界を明示する。それ以降のDB/API運用には5〜7章のリリース後方針を適用する。

最初の一般リリースは課金基盤のrelease gate合格後に行う。iOS sandbox購入・復元、server-side billing event検証、subscription / entitlement集約、同期APIのrequest-time認可、失効・返金・再有効化がend-to-endで確認できるまで、store提出、release tag、公開告知を行わない。具体価格、provider契約、launch offer、実credentialはpublic文書へ記録しない。

## 4. 開発運用

通常の変更は `docs/tasks/` の指示書単位で行う。作業前に [`AGENTS.md`](../AGENTS.md)、対象タスク、必要な仕様書を読み、完了時は対象タスクへ `## 9. 完了報告` を追記する。

コミット前品質ゲートは以下を基準にする。

```sh
cargo fmt --all -- --check
cargo clippy --workspace -- -D warnings
cargo test --workspace
cd app && flutter analyze
cd app/rust && env CARGO_TARGET_DIR=target cargo build --release
cd app && flutter test
sh app/tool/check_hardcoded_strings.sh
git diff --check
```

Flutter UIの見た目を変える場合は、通常のwidget testに加えてvisual QAを使う。visual QAはCIの常時ゲートではなく、必要時に次を実行する。

```sh
app/tool/visual_qa.sh
```

ローカル同期サーバーは次で起動する。

```sh
./tool/dev_server.sh
```

このスクリプトは `todori-dev-postgres` コンテナを作成または再利用し、`server/migrations/*.sql` を適用してから `cargo run -p todori-server` を `http://localhost:8080` で起動する。2台同期確認は [`docs/dev/two-device-sync-test.md`](./dev/two-device-sync-test.md) の手順を使う。

## 5. サーバー運用

サーバーはE2EEデータを復号しない。保存するのは認証・デバイス・テナント・セッション等のコントロールプレーン情報、同期用の暗号blob、同期メタデータである。同期APIは `POST /v2/tenants/{tenant_id}/push` と `GET /v2/tenants/{tenant_id}/pull` を中心に構成する。

本番想定の構成は次の通りである。

- AWS Lambda: Rust axum HTTPサーバーをコンテナイメージとして実行する。
- AWS Lambda Web Adapter: LambdaイベントとHTTPリクエストを変換する。
- ECR: `server/Dockerfile` で作成したコンテナイメージを保管する。
- Neon Postgres: 同期ストアとコントロールプレーンDB。接続はNeon組み込み接続プーラー経由のTCP接続を使う。
- 前段: CloudFront + WAF または API Gateway throttling。最終構成は人間判断とする。

実AWS/NeonのアカウントID、ECRリポジトリ名、Lambda関数名、Neon connection string、WAF/API Gateway設定値はprivate側または人間管理で保持する。

### 5.1 foreground realtime通知

foreground realtime通知はCloudflare Worker + Durable Objectsを初期providerとするが、同期の正本ではない。Cloudflare停止、WebSocket切断、通知欠落、server publish timeout時も、local CRUD、明示sync、foreground復帰sync、30秒fallback pollingを継続する。通知障害を理由にcommit済みpushを失敗扱いしない（ADR-019）。

serverとWorkerには用途別のchannel derivation、ticket signing、publish signing secretを設定する。ticket / publish signing keyはcurrent / previous key IDを使ってoverlap rotationし、channel derivation keyはv1中固定する。全realtime環境変数がないserverは通知disabledで起動できるが、一部だけ設定された状態はstartup errorとする。secret、ticket、opaque channel / device tag、tenant / device UUIDをlog、完了報告、public issueへ出してはならない。

serverのrealtime設定は次の11変数を1組として扱う。URLはそれぞれ`wss://.../v1/connect`と`https://.../v1/publish`、keyは32 bytesのbase64url no-padding、key IDは`[A-Za-z0-9_-]{1,32}`とする。current / previous IDと、channel / ticket / publishのkey materialは重複させない。11変数がすべて未設定ならdisabled、1つでも設定した場合は全変数を必須とする。実値はsecret storeで管理し、public文書やdeploy commandへ記載しない。

- `TODORI_REALTIME_WEBSOCKET_URL`
- `TODORI_REALTIME_PUBLISH_URL`
- `TODORI_REALTIME_CHANNEL_KEY`
- `TODORI_REALTIME_TICKET_KEY_CURRENT_ID` / `TODORI_REALTIME_TICKET_KEY_CURRENT`
- `TODORI_REALTIME_TICKET_KEY_PREVIOUS_ID` / `TODORI_REALTIME_TICKET_KEY_PREVIOUS`
- `TODORI_REALTIME_PUBLISH_KEY_CURRENT_ID` / `TODORI_REALTIME_PUBLISH_KEY_CURRENT`
- `TODORI_REALTIME_PUBLISH_KEY_PREVIOUS_ID` / `TODORI_REALTIME_PUBLISH_KEY_PREVIOUS`

初期監視項目は、ticket発行成否、WebSocket接続成否、publish成功 / timeout / provider error、通知受信からsync開始までの遅延、接続中 / 切断中のsync run数とする。logはerror kindと集計値だけを持ち、tenant別識別子やrecord metadataを含めない。Cloudflare本番resource作成、secret投入、data jurisdiction確認、費用上限設定はcredentialを持つ人間のrelease作業として扱う。

## 6. DBマイグレーション運用

サーバーDBのマイグレーションSQLは [`server/migrations/`](../server/migrations/) に置く。現在の実装では `server/src/db.rs` の `run_migrations` が`DATABASE_MIGRATION_URL`のowner接続でSQLファイルを適用し、通常queryは`DATABASE_URL`のnon-owner runtime loginで実行する。runtime loginはNOLOGIN group role `todori_app`を`INHERIT`し、session-level `SET ROLE`に依存しないためNeon transaction poolと両立する。tenant所有tableはtransaction-local user / tenant context、RLS policy、`FORCE ROW LEVEL SECURITY`で保護する。ローカル開発では [`tool/dev_server.sh`](../tool/dev_server.sh) が同じSQLファイルを `psql` でも適用する。

一般リリース後のDB変更は原則として前方のみのマイグレーションにする。互換性が必要な列・テーブル変更はexpand-contractを基本にし、即時の破壊的変更、既存列の意味変更、データ削除を避ける。現在の一般リリース前期間は3章を優先する。

詳細手順は [`docs/ops/runbook-db-migration.md`](./ops/runbook-db-migration.md) を参照する。

## 7. クライアント運用

ローカルDBはSQLCipherで暗号化され、スキーマバージョンはSQLiteの `PRAGMA user_version` で管理する。現行の `core/storage` はDB open時に未適用migrationを順次適用する。

一般リリース後の互換性方針は次の通りである。現在の一般リリース前期間は3章を優先する。

- 新しいアプリが古いDBを開く場合、実装済みmigrationを順に適用して最新スキーマへ上げる。
- 古いアプリが新しいDBを開く場合、既知の最新 `user_version` を超えるため `UnsupportedSchemaVersion` として開かない。これはDB破損や誤った旧スキーマ書込を避けるための挙動である。
- クライアントリリースでは、DB migrationを含むバージョンをロールバックしない前提で扱う。ストア公開後の取り下げや差し戻しが必要な場合も、DB schemaを戻すのではなく修正版を前方migrationとして出す。
- 前方互換が必要な変更は、旧アプリが未知列を無視できる追加に留める。削除・型変更・NOT NULL追加は段階的に行う。

## 8. セキュリティ運用

秘密情報はログ、Debug出力、スクリーンショット、完了報告、public issue、public PRに出してはならない。該当する秘密情報には、パスワード、OPAQUE中間状態、session token、Device Key、Master Key、DEK、Recovery Key、exportKey、SQLCipher鍵、Neon connection string、AWS credentialが含まれる。

鍵の取り扱い禁忌は次の通りである。

- Device Key、Master Key、DEK、Recovery Key、exportKeyを平文で永続化しない。
- 復号済みタスク本文をサーバーログ、分析ログ、クラッシュレポートへ送らない。
- E2EE暗号blobの復号をサーバー側で試みない。Todoriの前提は「サーバーはユーザーデータを復号できない」である。
- 開発用の `FileDeviceKeyStore` を本番primary storeとして扱わない。

依存脆弱性の確認として、`cargo audit` を定期実行することを推奨する。未導入環境では次を実行する。

```sh
cargo install cargo-audit
cargo audit
```

脆弱性を発見した場合は [`SECURITY.md`](../SECURITY.md) の非公開報告導線を使う。public issueに技術詳細や再現手順を書かない。
