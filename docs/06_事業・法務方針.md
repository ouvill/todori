## 1. 事業主体

個人事業として開始する（決定）。法人化は事業の発展後に改めて判断する。

これに伴う含意は以下のとおりである。

- Apple Developer Program は個人登録（$99/年）で申請する。
- Google Play Console は個人登録（$25 買切り）で申請する。
- Stripe Managed Payments の利用にあたっては、個人事業主としての適格性確認が必要である（詳細は [`04_課金設計書.md`](./04_課金設計書.md) §7.3 の実装前検証事項に含まれる）。

個人事業主は無限責任であるため、事業上のリスクを完全には遮断できない。このため、利用規約における責任制限条項（§4）に加え、そもそも重要データをE2EEにより開発者側が預からない設計そのものを保険的なリスク緩和策として位置づける。

## 2. OSS方針

リポジトリを公開し、ソースコードを第三者が検証可能な状態にする（決定）。目的は以下のとおりである。

- E2EEアプリにおける信頼の裏付け（実装が主張通りであることを利用者・第三者が確認できる）
- セキュリティ監査可能性の確保
- コミュニティ形成による品質向上・機能提案の受け入れ

### ライセンス（未決、GitHub公開前に決定）

- **第一候補: AGPL-3.0 + CLA**。Signal・Bitwarden等、プライバシー系プロダクトの定石であり、商用フォークやSaaSとしてのタダ乗りに対する防御力が強い。
- **対案: MPL-2.0**。ファイル単位の弱いコピーレフトであり、App Store配布との互換性が明快でCLAも不要という利点がある。

### 注意点

- GPL系ライセンスとApple App Store配布の間には一般に相性問題が指摘されるが、**単独著作権者である間はこの問題は生じない**（著作権者自身がストア配布を許諾できるため）。
- 外部コントリビューションを受け入れる段階になった場合は、CLA（またはApp Store配布に関する例外条項）の整備が必要になる。

### 現状の実装との整合

現状 `Cargo.toml` の `license` フィールドは `UNLICENSED` というプレースホルダである。ライセンス決定時に、`LICENSE` ファイルの追加とあわせて更新する。

## 3. 暗号輸出規制への対応

E2EEアプリは暗号輸出規制の対象となる。以下をリリース前チェックリストとする。いずれの項目も、実際の提出時点で最新の規制を再確認すること。

1. **米国 EAR**: 標準暗号アルゴリズムを用いるマスマーケットソフトウェアとして自己分類し、年次の自己分類レポートを BIS / NSA に提出する。ソースコードを公開する場合は、公開暗号ソースコードの所在URL通知（EAR §740.13(e) 相当のメール通知）により手続が軽くなる可能性が高い（§2のOSS方針との相乗効果）。
2. **App Store Connect**: 輸出コンプライアンス質問への回答、および `ITSAppUsesNonExemptEncryption` キーの適切な設定（標準アルゴリズム使用の申告）を行う。
3. **フランス**: ANSSI への暗号機能申告の要否を、フランスを配信対象国に含める前に確認する。
4. 本節は2026年7月時点の一般的な整理であり、ストア提出前に必ず最新の規制・手続を再確認する（必要に応じて専門家に相談する）。

## 4. プライバシーポリシー・利用規約の方針

### データ最小化

収集する個人データは、アカウント識別に必要な最小限（メールアドレス、課金状態等）に限定する。サーバーが技術的に知り得る情報の完全な列挙は [`03_技術仕様書.md`](./03_技術仕様書.md) §4.9 を正とし、プライバシーポリシーはこれをユーザー向けに平易に記述したものとする。

### 責任制限

「いかなる場合も免責」という完全免責条項は、日本の消費者契約法やEU・米国の消費者保護法制の下で無効となりうるため採用しない。業界標準の構成を採る。

1. 現状有姿（AS IS）での提供
2. 間接損害・逸失利益の免責
3. 賠償総額の上限を、直近12ヶ月にユーザーが支払った金額とする
4. 故意・重過失は制限の対象外と明記する（法制上の要請）

### E2EE特有のリスクの明示

パスワードとリカバリーキーの両方を喪失した場合にデータ復旧が不可能であること（[`02_機能仕様書.md`](./02_機能仕様書.md) F-31）を、利用規約・UIの双方で明示し、同意を取得する。

### GDPR等への対応

データ管理者は個人事業主たる開発者とする。E2EEを採用していても、メールアドレス等は個人データに該当するため、アクセス・削除請求への対応手順（アカウント削除機能）を設ける。

### ドラフト・レビュー

ドラフトは標準テンプレートを基に作成し、正式リリース前に可能な範囲で専門家レビューを受ける。

## 5. プロダクト名（検討中）

現仮称は Cotori。以下を候補とする。

- **Todori**: ToDo + 鳥。エメラルド色の小鳥「メジロ」をマスコットにできる。
- **Verdo**: エスペラント語 verde（緑）+ Do。
- Sprig
- todoenc（記述的で発音しづらく、ブランド方向と不整合のため非推奨）

### 決定前チェックリスト

- 商標データベース検索（J-PlatPat / USPTO / EUIPO）
- App Store / Google Play における既存アプリ名の重複確認
- ドメイン（`.app` 等）の取得可否
- SNSハンドルの空き確認
- 主要言語圏における発音・意味の確認

### 決定期限

ストア申請・GitHubリポジトリ公開・ドメイン取得のいずれよりも前に決定する。現在のBundle ID `dev.cotori` は暫定であり、名称確定時に変更する。

## 6. リポジトリ運用

- ホスティングは GitHub とする（決定）。当面は private で開始し、ライセンス決定（§2）後に public 化する。
- CI は GitHub Actions を利用する。iOSビルドには macOS ランナーが必要である（開発ホストがLinuxのため、iOSの署名付きビルドはCIまたはmacOS実機で行う）。
