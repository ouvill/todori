# ADR-015: default Inboxは認証済み候補のUUID最小へ収束する

**日付**: 2026-07-10
**状態**: Accepted（2026-07-10 プロダクトオーナー承認）

### 背景

各端末はlocal DB初期化時にdefault Inboxを作るため、同じaccountへ複数端末を接続すると異なるIDの`is_default=true` listが同期され得る。task-79では同期失敗を避けるため、先にlocalへ存在したdefaultを残し、後からpullしたdefaultのdomain rowだけを`is_default=false`へdemoteした。一方、認証済みsync plaintextは`is_default=true`のまま保持している。

この暫定処理は受信順によって勝者が変わり、重複listを通常listとして表示し得る。ユーザー体験としてはaccount内にInboxを1つだけ表示し、すべてのdefault Inbox配下taskを同じInboxへ収束させる必要がある。serverは暗号blobを解釈せず、Inboxというsemantic metadataも追加で保持しない。

### 決定

1. canonical候補は、liveでstrict typed検証に成功したlist sync plaintextのうち、認証済み`is_default.value == true`であるrecordだけとする。list名、受信順、local domain rowのdemotion状態は判定に使わない。
2. 候補のUUIDをbyte順で比較し、最小のlist IDをcanonical Inboxとする。UUIDはrecordの不変identityであり、全端末が同じ候補集合を得れば同じ結果になる。`created_at`やfield HLCを選挙条件にしない。
3. 選挙とmergeは、initial pull / full resyncがclosureまで完了し、そのclosureに含まれるserver current headがlocalで分類済みである場合だけ行う。`collection=lists`かつlive headのunresolved quarantineが1件でもあれば、そのrecordがdefault候補か判定できないため選挙を保留する。以前の有効plaintextがlocalにあっても、より新しいlive headがquarantine中なら代用しない。unsupported version、pull hard failure、key refresh failureでも永続mergeを開始せず、次の成功したsyncで再試行する。
4. local domainではcanonicalだけを`is_default=true`としてmaterializeする。その他の候補はcanonicalへのlocal aliasとしてdurableに記録して通常list一覧から隠すが、認証済みsync plaintextの`is_default=true`は候補identityとして保持する。serverへcanonical pointer、alias、Inbox名を平文metadataとして追加しない。
5. canonical切替は1つのowned `BEGIN IMMEDIATE`で行う。同じtransactionに、旧canonicalを含むloser domain rowのdemotion、新canonicalのpromotion、durable alias集合の置換、全既知alias taskのtyped placement更新、canonical List DEKでの再暗号化、task record state、HLC、outboxを含める。aliasをUIから隠すのはcommit後とし、鍵不足またはwrite失敗時はcanonical表示を含む全変更をrollbackする。
6. mergeはlogin後のbootstrapと通常sync後に冪等に再評価する。遅れて到着した、またはoffline端末から追加されたalias配下taskも同じ規則でcanonicalへ移す。後からより小さいdefault UUIDが到着した場合は候補集合から再選挙し、新しいcanonicalへ収束する。
7. alias listとList DEK bundleは、fuzzy full resyncとADR-016のexpired-device rebase、late descendant収束、key retirement条件が実装されるまで削除しない。UIから隠すことをserver recordの安全な削除完了とみなさない。
8. 新規accountは登録時にdefault InboxとList DEKを1組だけ作り、追加端末はそれを取得する。通常listが名前を`Inbox`または`インボックス`へ変更されても`is_default=false`なら候補に含めない。

### 帰結

- 正常なbootstrap後は各端末のUIにInboxが1つだけ表示され、既知taskはcanonical切替transactionのcommit後に同時にcanonicalへ見える。
- 他端末の永続状態はmerge outboxのpush / pull後に収束する。network断やkey不足時は既存recordを保持して再試行する。
- task-79のarrival-order demotionは、認証済み候補全体からUUID最小を選ぶmaterializationへ置き換える。
- 恒久実装はoffline list key-bundle queueと、既存profileを安全に列挙するfuzzy full resyncの後続とする。

### 却下した代替案

- **最初にlocalへ存在したInboxを採用する案**: pull順序で端末ごとの勝者が変わるため不採用。
- **list名がInboxのrecordをまとめる案**: ユーザー作成listを誤ってmergeし、localeやrenameにも依存するため不採用。
- **serverへcanonical Inbox IDを平文保存する案**: server-visibleなsemantic metadataを増やさずclientだけで決定できるため不採用。
- **aliasを直ちにtombstone化してList DEKを削除する案**: offline端末の未知taskを復号不能または復活状態にし得るため不採用。
