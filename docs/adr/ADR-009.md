# ADR-009: ゴミ箱を廃止し、削除は恒久削除+アーカイブモデルへ一本化する

**日付**: 2026-07-07
**状態**: 採用

### 背景

2026-07-07人間裁定（データ保持原則、`docs/design/ui-spec.md`参照）は、完了済みタスク（done/wont_do）を振り返りのための記録資産と位置付け、リスト削除を含むいかなる整理操作でも暗黙に失われてはならないとした。この原則を運用する保全経路として、Phase 1でリストのアーカイブ機能を導入することが決まった。一方、実装済みのゴミ箱（task-23、論理削除+30日後物理削除+復元UI）は「消したいが戻したいかもしれない」という中間的な削除需要を満たすために設けたものだったが、done/wont_do/アーカイブという正当な「やめ方」の記録手段がすでに用意されたことで、削除という操作に残る用途は実質的に「ミス操作やノイズの掃除」のみとなった。E2EEを製品の核とする本製品（ADR-001）にとって、「削除」という言葉がユーザーに与える期待は「本当に消える」であるべきであり、実際には一定期間サーバー/端末に残り続けるゴミ箱付き削除はこの期待との乖離（誠実さの毀損）を生む。

### 決定

1. ゴミ箱（trash画面・論理削除保持期間・復元機能）は廃止する。実装済みのゴミ箱UI・復元機能（task-23）は後続タスクで撤去する。
2. 削除は恒久削除とする。削除操作は一覧からの即時操作ではなく、詳細画面のサブメニューから起動する導線とし、「元に戻せません／永久に失われます」という追加確認ダイアログを必須とする。導線を意図的に深くすることで誤操作を防ぐ。
3. 削除に対するUndoは設けない（起動導線の深さ + 不可逆警告確認の二段防御で誤削除リスクを十分に抑制できるため）。
4. 完了（チェック）操作のUndoは維持する。編集操作のUndoも既存どおり維持する。
5. 履歴の保全経路はアーカイブとする（Phase 1導入決定済み）。リスト削除の確認ダイアログでは、削除対象に含まれる完了済みタスクを含む影響件数を明示し、「履歴を残したい場合はアーカイブを使う」旨へ誘導する。
6. E2EEプロダクトとして「削除＝本当に消える」という原則を厳格に守る。Undo履歴テーブル（`task_undo_entries`等）についても、Undo対象外である削除操作の内部データは残さない。

### 帰結

- `docs/02_機能仕様書.md` F-07（ゴミ箱・Undo）の仕様を本ADRの内容へ書き換える。
- 実装済みゴミ箱（task-23、`/trash` route・`trashedTasksProvider`等）の撤去タスクを新設する。
- Undoの削除復元経路（もしあれば）を廃止する。
- `tasks.deleted_at` カラムは将来的にマイグレーション経由で廃止する（撤去タスクの一部として計画する。恒久削除への移行期は物理削除で代替する）。
- リスト削除は、配下タスク（完了済みを含む）を恒久削除する操作となる（従来の「未完了はゴミ箱送り+完了済みは保全」という論理削除モデルは採用しない）。

### 却下した代替案

- **リストの論理削除により完了履歴を裏で保持し続ける案**（2026-07-07データ保持原則の当初運用案）: リストという単位に限れば履歴保全は達成できるが、ユーザーが「削除」を実行したにもかかわらずデータが裏で保持され続ける点で、「削除」という言葉の意味に対して不誠実であり、プライバシー製品としての信頼性訴求と矛盾するため不採用とした。
- **ゴミ箱を維持しつつ、別途「完全削除」操作を追加する案**: 「論理削除（ゴミ箱）」「完全削除」「アーカイブ」の3概念が併存することになり、calm/シンプル原則（`docs/design/ui-spec.md`）に反し概念過多となる。また主要な競合製品（Todoist、Microsoft To Do、Apple リマインダー）はいずれもゴミ箱を持たず削除は原則即時的であり、市場慣習からみても本製品がゴミ箱を持つ必然性は薄いと判断し不採用とした。
