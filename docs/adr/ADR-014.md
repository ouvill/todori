# ADR-014: protocol v2はbase revision CASでACK済み変更を保全する

**日付**: 2026-07-10
**状態**: Accepted（2026-07-10 プロダクトオーナー承認）

### 背景

ADR-012はfield clockとtransport用`revision_hlc`を分離した。しかしserverは暗号blobを復号・field mergeできないため、同じbaseからAがtitle、Bがnoteを編集し、AのACK後にBの大きいrevisionがblind overwriteすると、Aが停止した時点でtitle変更を知るclientが消える。field clockは失われたblobを復元できない。

Taskveilはまだrelease前であり、既存wire、開発server data、local sync metadataとの互換を維持する必要はない。互換層を足すより、correctnessを満たすv2へ直接置換する。

### 決定

1. live/tombstoneを問わずpush opは`base_revision_hlc`を持ち、server current revisionとのCASで更新する。新規recordだけbaseを`None`とする。
2. currentとbaseが不一致ならserverはblobを上書きせず`conflict`とcurrent envelopeを返す。clientはcurrentを復号・mergeし、remote revisionをobserveして新しいrevisionで再pushする。
3. `revision_hlc`はCAS revision兼transport順序であり、blind LWWの採用キーにしない。retryのexact envelopeはno-op、同revisionで内容が異なる場合はcorruption/conflictとする。
4. wire stateはnullable fieldの組合せではなく、`Live { mutation_hlc, blob }`と`Tombstone { delete_hlc }`のtagged unionにする。tombstoneはblobを持てない。
5. live/live merge後の`mutation_hlc`は両payloadの最大値かつmutable field clockの最大値とする。merge repushはfield clockとmutation clockを変えず、revisionだけを進める。
6. liveからtombstoneへのCAS過程は`mutation_hlc`と`delete_hlc`で裁定し、revisionだけが大きいrepushで削除を復活させない。ADR-016に従い、tombstone受理後の同一record IDに対する`Live`は拒否する。accepted revisionは許可されたstate遷移で単調増加させる（2026-07-11改訂: ADR-016）。
7. local outboxはrecordごとに未送信headを1件へcoalesceし、`op_id`、base/current revision、semantic clock、stateを永続化する。送信中に新mutationが入った場合、古いresponseは`op_id`不一致なら新headをACKしない。
8. push responseは順序を契約にせず、件数、`op_id`一意性、record ID、collectionを全件検証してからACKする。pullのinvalid base64を空blobへ変換せず、response failureとする。
9. collectionはprotocol enumとし、既存recordのcollection変更を拒否する。server schemaはlive/tombstone clockとblob形状をCHECK制約で強制する。
10. taskのcompletionは`status/completed_at/closed_reason`、placementは`list_id/parent_task_id/rank`をそれぞれcompound fieldとして同期する。list placementもrankを1 fieldとして持つ。任意mapより型付きplaintextを正本とする。
11. rankは固定幅128-bitの32桁lowercase hexへ置換し、表示順は`(rank, record_id)`で決定する。同rankまたは隣接rank間への挿入時だけ、同一scopeを現在順のままtransactionalに再配番する。
12. protocol v1互換は実装しない。sync HTTPは`/v2`、encrypted envelopeもv2へ切り替え、v1 dual-read/writeやdeserialize aliasを持たない。server sync tablesは開発環境で再作成し、local migrationはdomain rowを保持してv1 outbox/stateを破棄しv2 seedを再生成する。

### 実装分割

1. task-86でshared wire、CAS server、op ID照合、local v2 sync schema/outbox coalesce、conflict merge/rebaseを実装する。
2. 後続taskで型付きfield clock、completion/placement、固定幅rank、transactional reorder/rebalanceを実装する。
3. production common clientを通る2-client testで、先にACKしたclientを停止しても別field変更が消えないことをrelease gateにする。

### 帰結

- CASなしの`revision_hlc` LWWを採用しない。
- serverはplaintextを知らないまま、ACK済みheadをstale clientがblind overwriteすることを防げる。
- v2移行中は開発DB/server dataの再作成を許容し、互換コードを保守しない。
- task-86だけではchanged-field clockとplacementは未完了であり、後続task完了まで同期correctness全体をrelease-readyとしない。

### 却下した代替案

- **大きいrevisionを常に採用する案**: ACK済みの別field変更を含まないblobで上書きできるため不採用。
- **server historyから後で復元する案**: 通常同期のcorrectnessを運用上の復旧へ委ね、端末が収束しないため不採用。
- **v1/v2 dual write案**: release済みclientがなく、複雑性だけを増やすため不採用。
