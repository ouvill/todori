# 機能仕様書

> Baseline: 2026-07-23 product owner承認。ADR-023参照。
>
> 本書はTaskveilの機能仕様の正本である。旧版のF番号を参照する完了taskは当時の履歴であり、本baselineのF番号へ読み替えない。

## 1. Product mode and onboarding

### F-01 Local-only開始

- Accountを作らずtask管理を開始できる。
- local databaseとsearch indexはdefaultで暗号化する。
- server通信なしでtask、view、search、recurrence、notification、template、timer、exportを利用できる。
- 端末とencrypted exportを両方失った場合は復旧不能であることを説明する。

### F-02 Account-backed移行

- Local-only profileからAccountを作り、local dataをPersonal Spaceへ明示的に取り込める。
- 取り込み前に件数とscopeを表示し、完了までlocal dataを削除しない。
- network失敗時はlocal dataと再開可能なmigration stateを保持する。

### F-03 UI complexity

- 初期状態は日常操作を優先した単一UIとする。
- advanced filter、bulk operation、security detail等は必要な画面で段階的に開示する。
- 「simple/high-function」の別product modeで機能を分断しない。

### F-04 Design and accessibility

- 落ち着いた視覚tone、明確なinformation hierarchy、一貫したinteractionを使う。
- dynamic type、screen reader、keyboard、contrast、touch target、色以外のstate表現を提供する。
- E2EE状態を常時badgeで騒がしく表示せず、recovery/key/member transitionで結果を明示する。

## 2. Task

### F-05 Task fields

Taskは少なくとも次を持つ。

- title、plain text note、URL。
- status: `todo`、`in_progress`、`done`、`wont_do`。
- priority、created/updated timestamp。
- date-only due、time-specific due、scheduled start、duration estimate。
- list、section、multiple tag、parent task、manual rank。
- multiple reminder、recurrence rule。
- Shared Spaceではassigneeとcompletion actor。

添付file、image、audio、thumbnailは持たない。URLは文字列としてE2EEにする。

### F-06 Create and edit

- quick add、detail edit、inline edit、duplicateを提供する。
- title以外のfieldは必要に応じて後から追加できる。
- local DB transaction成功時に操作完了としてUIへ反映し、networkを待たない。
- server upload前後を問わず同じediting capabilityを保つ。

### F-07 Status transition

- `todo <-> in_progress`、`todo/in_progress -> done/wont_do`、`done/wont_do -> todo`を許可する。
- complete時にCompletion Recordを作る。
- reopenしても過去Completion Recordを保持する。
- `wont_do`は「実施した成果」と混同せず、理由を任意記録できる。

### F-08 Subtask

- Taskは同じSpace内のTaskをparentにできる。
- protocolは固定depth上限を設けない。
- cycle、self-parent、cross-Space parentを拒否する。
- subtreeのmove、collapse、進捗表示、一括操作を提供する。
- 未完了descendantへ影響するcomplete/deleteでは件数と結果を表示する。

### F-09 Move, copy, batch

- Task/subtreeをlist/section間でmove/copyできる。
- batchでstatus、priority、tag、due、assignee、archiveを変更できる。
- Personal SpaceとShared Space間のmove/copyは暗号scope変更として明示する。
- cross-Space moveはcopy成功後にsource archive/deleteする再開可能workflowとし、偽のatomicityを表示しない。

### F-10 Ordering

- drag-and-drop/manual rankを提供する。
- due、priority、created、updated、completed、title等でsortできる。
- concurrent rank collisionでもstable表示し、必要時にclientがrebalanceする。

### F-11 Undo and conflict

- complete、reopen、edit、move、tag等の安全に反転できる操作をundoできる。
- undoは同期済み操作を隠れて消すのではなくinverse mutationとして扱える。
- same-field sync conflictでは片方を黙って捨てず、Conflict Recordからcandidateを確認・解決できる。
- permanent deleteはundo対象にしない。

## 3. List, section, tag, template

### F-12 Inbox and list

- Personal Spaceにcanonical Inboxを1つ持つ。
- 複数listを作成、rename、manual reorder、archiveできる。
- archiveしてもTask、Completion Record、Work Sessionを保持する。
- Shared Spaceは初期UIで1共有listとして表示する。

### F-13 Section

- list内にsectionを作成、rename、reorder、archiveできる。
- Taskをsection間でmoveできる。
- section delete時は配下Taskの移動先または同時削除を確認する。

### F-14 Tag

- Taskに複数tagを付与できる。
- tagはSpace内だけで再利用する。
- tag rename/color/orderはE2EE同期する。
- Spaceをまたぐ同一tag objectは作らない。

### F-15 Template

- Task、subtree、list構造をtemplateとして保存できる。
- templateから作るinstanceは新しいrandom IDを持つ。
- template自体とparameterはE2EE同期する。

### F-16 Archive

- Task、list、section、Shared Spaceを通常の整理としてarchiveできる。
- archiveはpermanent deleteではなく、search/filter/exportから参照可能である。
- completed historyを容量整理のため自動archive/deleteしない。

## 4. View and search

### F-17 Standard views

- Inbox、Today、Upcoming、Calendar、All、Completed、Won't Doを提供する。
- viewはlocal decrypted data/indexから生成する。
- Todayはdate-only dueとlocal timezoneを正しく扱う。

### F-18 Board and grouping

- status、section、priority、assignee等でBoard/Kanban表示できる。
- list、tag、date、status等でgroupingできる。
- drag操作がdomain field変更を伴う場合は結果を明示する。

### F-19 Search

- title、note、URL、tag、list、commentをlocal full-text searchできる。
- queryをserverへ送らない。
- large completed historyでもincremental indexとpaginationで操作できる。

### F-20 Filter and saved view

- field、date range、status、tag、list、assignee、has-note、estimated/actual timeを組み合わせてfilterできる。
- filterをSaved FilterとしてE2EE同期できる。
- server-side smart listやserver-visible due indexを作らない。

### F-21 Calendar

- due/scheduled Taskをday/week/month等で表示できる。
- date-onlyとtime-specificを区別する。
- platform calendar integrationはclient permissionと明示mappingで行い、Task plaintextをTaskveil serverへ渡さない。

## 5. Completion, history, deletion

### F-22 Completion Record

- completeごとにimmutableなCompletion Recordを作る。
- task ID、completed time、actor、必要最小限のdisplay snapshotをE2EE保存する。
- 同じTaskをreopenして再completeすると別Completion Recordを作る。
- concurrent completeはidempotent commandをdeduplicateし、別actor/commandは両方を履歴に残す。

### F-23 History and review

- completed TaskとCompletion Recordをdate/list/tag/actorで閲覧・検索できる。
- weekly/monthly reviewと実績集計はclientで行う。
- edit history全体を永久保存するevent sourcingは行わない。
- historyを通常画面へ過剰表示せず、振り返り時にaccessできる。

### F-24 Permanent delete

- permanent deleteは残したくない情報を消す例外的操作とする。
- deep menu、不可逆警告、対象件数、再確認を必須にする。
- Task deleteでは同じSpaceのknown comment、Completion Record等の関連dataもTombstone化する。
- Personal Work SessionからShared Taskへのsoft external referenceはcascade deleteせず、成果記録として残す。
- Tombstoneはcontentを持たず、同じrecord IDの復活を拒否する。
- list/Shared Space deleteでは未完了・completedを含む影響件数を表示する。

## 6. Timer and time tracking

### F-25 Stopwatch

- Taskあり/なしでstopwatchを開始できる。
- start、pause、resume、finish、discardを提供する。
- active timerはdevice-localにdurable保存し、app restart、background、reboot後に復元する。
- 同一local profileのactive timerは1つだけとする。

### F-26 Pomodoro

- work、short break、long break、cycleを提供する。
- durationとlong-break intervalを利用者が設定できる。
- breakはdefaultで作業実績へ算入しない。
- active state/tick/breakをserver同期しない。

### F-27 Work Session

- finishしたwork intervalまたはmanual entryをimmutable Work SessionとしてPersonal Spaceへ保存・同期する。
- task reference、encrypted display snapshot、mode、started/ended time、active duration、finish kind、noteを持てる。
- Shared Space Taskを計測してもdefaultで共有しない。
- Shared Task relationはsoft external referenceとし、Task削除・member removal・access喪失後もsnapshotとunavailable表示でWork Sessionを残す。
- correction/replacement relationで訂正履歴を保ち、単純な誤記録はpermanent deleteできる。

### F-28 Estimate and actual

- task、day、week、tag、listごとにactual timeをclient集計する。
- Task estimateとactualを比較できる。
- manager approval、billing、timesheet submissionは初期scope外とする。

## 7. Recurrence and notification

### F-29 Recurrence

- daily/weekly/monthly/yearly、weekday、interval、count/until等のcommon ruleを表現する。
- fixed-scheduleとcompletion-based recurrenceを区別する。
- this occurrence、following、series全体のeditを提供する。
- occurrence generation/deduplicationはclientが決定的に行い、serverはscheduleを知らない。

### F-30 Recurrence exception

- occurrenceのskip、reschedule、overrideを記録する。
- offline複数deviceでも同じlogical occurrenceをduplicate表示しない。
- timezone/DST変化時のpolicyをseriesに保持する。

### F-31 Reminder

- 1 Taskに複数のabsolute/relative reminderを持てる。
- snooze、dismiss、permission/error状態を提供する。
- reminder schedulingとnotification body generationはclientで行う。
- serverとpush providerへTask title、due、notification bodyを送らない。

### F-32 Generic sync wake-up

- server pushは「変更がある」というopaque wake-upだけを送る。
- push欠落はsync遅延だけを生み、data lossを生まない。
- foreground、background polling、manual refreshで同じstateへ収束する。

## 8. Account, device, recovery

### F-33 Account registration and login

- emailとpasswordでAccountを作成・loginできる。
- passwordはOPAQUE候補protocolへだけ入力し、serverへplaintext送信しない。
- optional MFAを提供できる。MFAはserver sessionの追加認証でありcontent keyの代わりではない。

### F-34 Device and session

- device/session一覧、created/last-used概略、revokeを提供する。
- session revokeはserver accessを止めるが、既にcopyされたplaintext/keyのremote eraseを保証しない。
- device名をserver-visibleにする場合は必要性を別途判断し、defaultでencrypted metadataを優先する。

### F-35 Password change

- unlock済みARKを持つ正規clientがnew passwordへ変更できる。
- Account Root Keyをnew Account Unlock Keyでrewrapし、Recordを再暗号化しない。
- new OPAQUE record/wrapped ARK digestとexpected security revisionをAccount signing keyで認可する。
- Account security revisionとserver authentication stateをatomicに更新する。

### F-36 Recovery Key

- clientがrandom Recovery Keyを生成し、一度表示、copy/print/encrypted exportを案内する。
- Recovery Keyをserverへplaintext送信しない。
- Recovery Keyで同じAccount Root Keyを復旧し、new passwordを登録できる。
- 復元したAccount signing keyでnew OPAQUE record/wrapper digest、Account ID、one-time challenge、expected security revisionを署名する。
- serverはsignatureを検証し、OPAQUE record、wrapped ARK、rotated recovery wrapper、security revision、既存session revokeをatomic CASする。
- recovery完了時にnew Recovery Keyを表示し、旧Recovery Keyを無効化する。

### F-37 Irrecoverable reset

- Password、Recovery Key、unlock済みdeviceをすべて失った場合、旧dataをsupportも復旧できない。
- resetで同一identityを装わず、new identity/security resetとして共有先へ警告する。
- 旧encrypted data削除は復旧不能を確認する別操作にする。

## 9. Sharing

### F-38 Shared Space

- Shared Spaceを暗号鍵、membership、role、sync、quota attributionの境界とする。
- 初期UIでは1 Shared Spaceを1共有listとして扱う。
- Shared Space表示名/list名/contentはE2EEにする。
- serverはmembership、role、owner、key generationを知る。

### F-39 Invitation

- ownerがone-time linkまたはQRを作る。
- serverへ送られないrandom invitation secretでinitial identityをbindする。
- expiry、cancel、one-time use、rate limit、recipient Account確認を提供する。
- public/guest/anonymous content accessは提供しない。

### F-40 Role

- roleは`owner`、`editor`、`viewer`の3つだけとする。
- owner/editorはcontent read/write、viewerはread-only。
- editorは全Shared contentを改変・permanent deleteできる。招待/role変更前にこの破壊権限をowner/targetへ表示する。
- signatureはactorを識別するがcurrent-head-only storageから旧contentを復元しない。Shared Spaceをbackupとして表示せず、user-controlled exportを提供する。
- ownerだけがinvite、role/member変更、key rotation、owner transfer、Space deleteを行う。
- Shared Spaceには常に正確に1 ownerがいる。

### F-41 Historical access

- new memberへ未削除Recordに必要な全historical Space Key generationを配布する。
- new memberはjoin前を含む現在・過去の未削除dataを閲覧できる。
- invitation確定前に、このpast accessをownerへ明示する。
- permanent deleted contentはTombstoneだけなので閲覧できない。

### F-42 Member removal

- ownerがactive memberをremoveできる。
- access revoke、new Space Key generation、remaining memberへのkey envelope、manifest更新をatomic commitする。
- removed Accountのread/writeと全Accountのold-generation writeをserverが拒否する。
- removed memberはnew/edited Recordをdecryptできない。
- removal前に取得したdataのremote eraseは保証しない。

### F-43 Progressive re-encryption

- removal時にold Recordをbulk re-encryptしない。
- old-generation Recordはreadだけではrewriteしない。
- old Recordをeditするとwhole Recordをcurrent generationで暗号化する。
- historical keysはold Recordとfuture new-member historical accessのため保持する。

### F-44 Owner transfer

- current ownerがactive editorへtransfer proposalを出す。
- target acceptanceを必須にする。
- outgoing ownerはdefaultでeditorになる。
- 2 role、owner field、quota attribution、membership revisionをserver transactionでatomic更新する。
- transfer前後のいずれもowner 0人/2人を許さない。

### F-45 Leave and Space deletion

- editor/viewerはleaveできる。
- ownerによるnew key generation完了までShared Space writeを一時freezeできる。
- ownerはtransferまたはSpace deleteなしにleaveできない。
- Space deleteはstrong confirmationを必要とし、archiveを通常整理として案内する。

## 10. Export, platform, security UX

### F-46 Export/import

- versioned encrypted exportを作成・検証・importできる。
- plaintext JSON/CSV/Markdown exportはE2EE境界外になる警告を出す。
- exportにkey、integrity/authenticity、schema versionを含める。
- service状態やsubscriptionでexportを禁止しない。

### F-47 Natural language input

- date、time、recurrence、priority、tag等をquick addで解釈できる。
- baselineはdevice上のdeterministic parserとする。
- external AI送信を必須にしない。
- plaintextを第三者serviceへ送る機能は明示的なE2EE境界外integrationとする。

### F-48 App lock and local protection

- OS secure storage、device lock、encrypted local DBをbaselineにする。
- biometric/app PINはkey releaseを強化するoptionとして提供できる。
- rooted/jailbroken/compromised endpointへの完全なsecurity boundaryとは表示しない。

### F-49 Platform integration

- widget、share sheet、shortcut、desktop notification等を段階提供できる。
- extension/processへAccount/Space keyを広く共有しない。
- notification previewやclipboardへsecret/plaintextを不要に残さない。

### F-50 Internationalization

- initial localeはEnglish/Japaneseとし、全user-facing stringをlocalize可能にする。
- date/time/timezone、plural、sort、calendarをlocaleに従って表示する。
- protocol/storageにはlocalized labelを識別子として保存しない。

### F-51 Security-visible failure

- signature/AEAD/schema failureを通常dataとして適用しない。
- affected Recordをquarantineし、秘密を含まないdiagnostic IDを表示する。
- identity unsigned reset、unexpected key change、cursor rollbackをsilent acceptしない。

## 11. Quota and non-functional requirements

### F-52 Quota

- 課金Accountのserver-side encrypted structured dataは約1 GiBを安全上限目安とする。
- ciphertext、signature、key material、Record metadata、Tombstoneをactual stored bytesで計上する。
- 通常画面へ常時meterを出さず、warning threshold/上限到達時に知らせる。
- serverはmutation batchを部分受理しない。
- old completed dataを自動削除しない。

### F-53 Quota/subscription failure

- quota/subscriptionでupload不能でもlocal read/edit/search/exportを許可する。
- outboxをdurable保持し、retry stateを表示する。
- pull、storageを減らすpermanent delete、Account exportを許可する。
- Account dataをservice状態で人質にしない。

### F-54 Performance

- local mutationを通常100ms以内にUIへ反映することを目標にする。
- large historyはpagination、virtualization、incremental indexで扱う。
- sync、crypto、migrationはUI threadをblockしない。
- exact budgetはimplementation benchmarkでplatform別に固定する。

### F-55 Offline correctness

- networkなしでlocal capabilityを維持する。
- domain mutation、encrypted candidate、outboxを同じlocal transactionでcommitする。
- retryをidempotentにし、timeoutだけでduplicateを作らない。
- full resyncでもnever-synced local dataを黙って捨てない。

### F-56 Privacy-safe diagnostics

- secret、plaintext content、password、Recovery Key、Account/Space Key、session tokenをlog/analytics/crash reportへ出さない。
- record ID等のpseudonymous identifierも必要最小限、retention付きで扱う。
- privacy-sensitive telemetryはopt-inまたは集計化を検討し、product content analyticsをserverで行わない。

## 12. Initial exclusions

- 添付file、file preview/object storage。
- enterprise Organization、admin recovery、SSO/SCIM、監査、legal hold。
- server-side plaintext search/AI/reminder scheduling。
- public/guest/anonymous share。
- realtime chat/presence/live cursor。
- active timer cross-device control。
- supportによるsecret recovery。
- custom cryptographic primitive、custom consensus、blockchain。
