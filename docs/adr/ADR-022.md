# ADR-022: staging deploymentはAWS・Neon・Cloudflareを環境分離して自動化する

**日付**: 2026-07-19
**状態**: Accepted（2026-07-19 プロダクトオーナー承認）

### 背景

AWS / Cloudflare / Neonの実環境が未構築の段階で、credentialなしで検証できるruntime境界、IaC、CDを先に固定する。stagingとproductionが同じprovider credentialやDBを共有すると、sandbox課金状態の混入、owner DB credentialのruntime流入、誤applyの影響拡大を招く。

### 決定

1. stagingはAWS Lambda + API Gateway HTTP API + Neon + Cloudflare Worker / Durable Objectsで構築し、AWS / Neonはproductionと別account / projectにする。リージョンは`eu-central-1`とする。
2. OpenTofuで再利用moduleとstaging / production rootを定義する。productionは別backend / account / Neon projectを必須とし、apply workflowを持たない。
3. Secrets Managerはruntime、migration、deployment-providerの3secretに分ける。Lambdaはruntimeだけ、staging deploy roleは3件、通常CIはどれも読めない。secret valueはOpenTofu stateに入れない。
4. stagingはRevenueCat sandboxだけ、productionはproductionだけのcredentialを保持する。通常serverはpooled non-owner DB credentialだけを読み、migration owner credentialは専用binaryだけが使う。
5. deployはまずGitHub staging Environmentの承認付き手動実行とする。同一commit SHAからWorker / serverを作り、scan、migration、version / alias切替、smokeの順を固定する。DBはrollbackせず、失敗時はLambda aliasとWorker deploymentだけを直前versionへ戻す。
6. 手動deploy 3回連続成功と2端末同期、realtime、provider停止時HTTPS fallbackの人間検証後だけ、`STAGING_AUTO_DEPLOY_ENABLED=true`でmainからの自動deployを有効化する。

### 帰結

- account作成、MFA、課金情報、state / OIDC bootstrap、secret投入、初回applyは人間承認の停止点とする。
- production apply、WAF / CloudFront、store配布、PR preview環境、release tagは本決定の対象外とする。
- 実domain、account ID、zone ID、Neon project ID、予算通知先はprivate repoだけで管理し、secret valueはprivate Gitにも保存しない。
