# ADR-018: Timer計測状態とtask statusを分離し、完了work sessionだけを同期する

**日付**: 2026-07-13
**状態**: Accepted（2026-07-13 プロダクトオーナー承認）

### 背景

Timer / Pomodoro / Focusの実行状態と、Kanbanでユーザーが管理する`todo` / `in_progress` / `done` / `wont_do`は異なる概念である。Timer開始を`in_progress`へ結合すると、計測を試しただけでtask workflowが変わり、別端末の明示的なstatus操作とも競合する。また、秒単位で変化するactive sessionを同期すると、background復帰、clock drift、同時利用の分散排他が同期correctnessへ侵入する。

一方、完了した作業時間はユーザーの成果であり、複数端末で見積対実績を振り返れる必要がある。ADR-007は`timer_sessions`をtenant-scope entity、Tenant Root DEKをその暗号鍵と定めたが、local cache / runtime供給と同期recordの厳密な形は未実装である。

### 決定

1. task statusとTimerは直交する。Focus / Pomodoro / Stopwatchのstart / pause / resume / finish / discardはtask statusを変更せず、特に`in_progress`を自動設定しない。Focus中にユーザーがtaskを完了する場合は現在のwork sessionを保存終了してから既存task status APIを明示的に呼ぶ。task completionのUndoはTimerを自動再開しない。
2. active sessionは端末ローカルのSQLCipher DBだけへdurable保存し、1 device 1 activeをsingleton制約で保証する。stateは`running` / `paused`、phaseは`work` / `short_break` / `long_break`、modeは`pomodoro` / `stopwatch`を持つ。workだけtask IDを必須とし、breakはtask IDを持たない。
3. active durationの正本は`accumulated_active_ms`と`last_resumed_at`である。running復元時は`accumulated_active_ms + max(0, now_wall_ms - last_resumed_at)`、paused時は`accumulated_active_ms`とする。wall clock逆行分は0へclampし、固定tick回数やprocess uptimeを永続値にしない。1回のactive spanと累積値は0以上、最大7日とする。
4. 同期対象はwork phaseを完了した、またはユーザーが明示的に「中断を保存」したimmutable `CompletedTimerSession`だけとする。fieldは`id`、`task_id`、`mode`（pomodoro / stopwatch）、`finish_kind`（completed / interrupted）、`started_at`、`ended_at`、`active_duration_ms`、`created_at`である。`started_at <= ended_at <= created_at`、`0 < active_duration_ms <= ended_at - started_at <= 7日`、各timestampが表現可能なUTC instantであることを必須とする。activeの完了確定はID、task、mode、work phase、開始時刻がcompleted値と一致する場合に限り、completed row・同期outbox/state・active clearを同一transactionで行う。Pomodoroのtarget / break設定は端末ローカル設定でありsessionへ保存せず、short / long breakの計測も実績同期しない。
5. completed sessionは作成後immutableとする。同record IDの同一plaintextは冪等no-op、異なるplaintextはUUID collisionまたはcorruptionとしてmergeせずquarantineする。編集は旧sessionをtombstone化し、新UUIDのsessionを作る。live / tombstone競合はADR-014 / ADR-016に従い、serverがtombstoneを受理したrecord IDはterminalでlive復活を拒否する。
6. task / list永久削除では、削除前にlocalで関連completed sessionを列挙し、各session tombstone、task/list tombstone、domain物理削除、record state、outbox、HLCを同一`BEGIN IMMEDIATE`で確定する。serverはtask_idを知らないためcascadeを行わない。削除端末が認識していないlate sessionは、復号したclientがlocal task tombstoneを検出した時点でmaterializeせずsession tombstoneをenqueueする。
7. `timer_sessions`は新しい`SyncCollection`であり、plaintextは`kind=timer_session`と上記immutable fieldsを持つ。task status、task title、list ID、Pomodoro設定を保持しない。serverはcollection、record ID、opaque encrypted blob / tombstone、CAS / continuity metadataだけを扱う。
8. timer session blobはTenant Root DEKでXChaCha20-Poly1305暗号化する。既存envelope v3とAAD`taskveil-sync-envelope/v2\ncollection:{collection}\nrecord_id:{record_id}`はtask / listと同じwire契約として維持し、collection / record ID swap、wrong Tenant Root DEKを拒否する。Timer plaintext variant追加だけを理由にenvelopeを変更・再暗号化しない。Tenant Root DEKのkey versionは1とする。
9. 個人tenantのTenant Root DEKは既存どおりserverへ`wrap(TenantDEK, MK)`で保存する。端末ではMKで再wrapした`wrapped_tenant_root_dek`をSQLCipher内へ保存し、AADを`taskveil/wrap/local-tenant-root-dek-by-mk/v1/{tenant_id}`とする。平文Tenant Root DEKはaccount runtimeの`Zeroizing` memoryだけに置く。binding済みprofileでcacheを復元できない場合は匿名fallbackせずTimer同期mutationを拒否する。
10. local schemaをv18、sync protocolをv5へ更新し、encrypted envelopeはv3のまま維持する。v18はactive singleton、completed timer sessions、local Tenant Root DEK cacheを作り、local sync metadata tableをCHECK拡張のためtransactionalにrebuild-copyする。既存outbox、record state、cursor、quarantine、full-resync state / marks、origin、terminal tombstoneを失わない。server migrationはcollection CHECKとstable cursor CHECKへ`timer_sessions`を追加する。旧protocolのdual read/write、deserialize alias、fallbackは実装しない。
11. active state、session row、key cache、record state、outbox、HLCのwrite failureは全rollbackする。秘密鍵、plaintext session、task title、durationをlogへ出さず、errorは型とrecord IDまでに限定する。

### 帰結

- ユーザーはKanban statusをTimerから独立して管理できる。
- background / process restartはlocal durable wall-clock stateだけで復元でき、秒ごとの同期やdistributed lockが不要になる。
- 完了work sessionはtenant内でE2EE同期され、serverはtask linkageや実績時間を解釈できない。
- Tenant Root DEKのlocal供給が、将来のtags / templates / schedulesにも再利用できる。
- v18 / protocol v5はcollection enumのbreaking cutoverだが、envelope v3と既存server head / local tombstone系譜は保全される。

### 却下した代替案

- **Timer開始時にtaskを`in_progress`へ変更する案**: Kanban上の明示状態と一時的な計測を混同するため不採用。
- **active sessionを同期する案**: 秒単位更新、clock drift、multi-device leader electionが必要になり、offline-firstの単純性を失うため不採用。
- **breakを作業実績として同期する案**: taskへ費やしたactive workを過大計上するため不採用。
- **completed sessionをmutable LWW recordにする案**: 成果時間が別端末編集で静かに変わるため不採用。訂正はtombstone + 新recordとする。
- **timer sessionをList DEKで暗号化する案**: list移動で再暗号化が必要になり、tenant-scope集計とADR-007に反するため不採用。
- **Tenant Root DEKをSQLCipher内へ平文保存する案**: MK hierarchyを迂回するため不採用。
