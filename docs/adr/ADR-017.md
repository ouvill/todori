# ADR-017: タスク期限は日付のみと日時指定を排他的に区別する

**日付**: 2026-07-12
**状態**: Accepted（2026-07-12 プロダクトオーナー承認）

### 背景

現行のtaskは`due_at: Option<i64>`を持ち、日付だけを選んだ場合も端末ローカル日の開始をUTC epoch millisecondsへ変換して保存している。そのため、値の型だけでは「その日中に終えればよい」のか「その瞬間までに終える」のかを判別できない。タイムゾーンを跨いだ表示やToday分類でも、日付というユーザー意図が別の日へずれ得る。

一方、`scheduled_at`は作業を始める予定日時であり、完了期限ではない。通知の`remind_at`もユーザーへ知らせる瞬間であり、期限そのものではない。この3つを流用せず、期限の意味をdomainで明示する必要がある。

### 決定案

1. taskの期限は`Option<TaskDue>`とし、`TaskDue`を次の排他的なtagged unionとする。
   - `Date { due_on: CivilDate }`: 「この日までに」。`CivilDate`はproleptic Gregorian calendarの`YYYY-MM-DD`であり、時刻・offset・timezoneを持たない。
   - `DateTime { due_at: UtcInstant, time_zone: IanaTimeZone }`: 「この時刻までに」。`due_at`は一意な瞬間、`time_zone`は入力・編集・曖昧性表示に使うIANA timezone IDである。
2. `Date`と`DateTime`を同時に持たせない。日時指定には日付も含まれるため、独立した2期限ではなく1つの期限モードとして切り替える。
3. raw epoch値をdomain / client / FRB / Dartの期限APIとして公開しない。`CivilDate`、`UtcInstant`、`IanaTimeZone`を検証済み型として扱う。local DBはrange query用に`UtcInstant`をepoch millisecondsで保持してよいが、これはstorage表現に閉じ込める。
4. local DBは`due_kind`、`due_on`、`due_at_ms`、`due_time_zone`を持ち、CHECK constraintで次だけを許す。
   - 期限なし: 全列NULL。
   - 日付のみ: `due_kind='date'`、`due_on`だけ非NULL。
   - 日時指定: `due_kind='datetime'`、`due_at_ms`と`due_time_zone`だけ非NULL。
5. 日付のみの期限切れ判定は`due_on < viewer_local_today`とする。`23:59:59.999`等の擬似instantへ変換しない。日時指定の期限切れ判定は`now >= due_at`とする。
6. Today分類は次とする。
   - 日付のみ: `due_on == viewer_local_today`。
   - 日時指定: `due_at`をviewerの現在timezoneへ変換したlocal dateがtoday。期限切れは時刻単位で別途明示する。
   - `scheduled_at`がTodayへ含まれる既存要件は期限とは別条件として維持し、後続UI実装時に重複表示を防ぐ。
7. 同じlocal dateに属する期限は、未来の日時指定を`due_at`昇順、その後に日付のみを置く。日付のみは「その日中」を意味するため、同日の特定時刻より前に期限切れ扱いしない。期限切れ同士は期限切れになった順が古いものから並べる。
8. sync plaintextは`due: Clocked<Option<TaskDue>>`を1 fieldとして暗号化する。tag、値、timezoneへ別々のHLCを付けず、1回の期限編集をatomicにmergeする。
9. `scheduled_at`は開始予定instant、`remind_at`は通知instantとして残し、`TaskDue`へ統合しない。日付のみ期限へ通知を付ける場合、通知側が明示的な`remind_at`を持つ。
10. UIは期限設定時に「日付のみ」と「日時指定」を明示する。日付のみは時刻を表示せず、日時指定は時刻を必ず表示し、viewer timezoneと入力timezoneが異なる場合はtimezoneまたは換算後時刻を曖昧にならない形で示す。
11. 一般リリース前のため、旧`due_at`の値を推測変換しない。local DB schema、FRB API、sync plaintext / protocolをfinal designへbreaking変更し、既存開発profileとserver開発データは再作成する。dual read/write、legacy fallback、midnight値からの種別推測は実装しない。

### 帰結

- 日付だけの期限は端末timezone変更やDSTで別日へずれない。
- 時刻までの締切は一意なinstantとして通知・期限切れ判定でき、入力時timezoneも失わない。
- storage、sync、UIは期限種別を推測せず、domainのtagを正本にできる。
- DB schema、同期plaintext、FRB、Dart provider、Today / sort、作成・編集UIを同時にbreaking更新する重要変更taskが必要になる。
- `docs/01_企画書.md`と`docs/02_機能仕様書.md`の「期限」「締切日時」「Today」表現は、裁定後に人間承認付きで本契約へ同期する必要がある。

### 却下する代替案

- **既存`due_at`の00:00を日付のみと推測する案**: 本当に00:00締切のtaskと区別できず、timezone変換後は00:00でなくなるため不採用。
- **`due_on`と`due_at`を独立nullable fieldにする案**: 両方が同時に残る無効状態と、別端末編集時のfield-clock競合を生むため不採用。
- **日付のみをその日の23:59:59.999へ変換する案**: timezoneとDSTに依存し、日付という意図をinstantへ不可逆に潰すため不採用。
- **`scheduled_at`を時刻付き期限として流用する案**: 開始予定と完了期限が同じ意味になり、Today・通知・所要時間計画を正しく表せないため不採用。
- **日時指定にtimezoneを保存しない案**: instantとしての判定はできるが、入力した現地時刻の編集と旅行時の表示意図が失われるため不採用。
