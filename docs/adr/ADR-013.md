# ADR-013: local crypto contextをremote sessionから分離する

**日付**: 2026-07-10
**状態**: Accepted（2026-07-10 プロダクトオーナー承認）

### 背景

task-83でaccount-boundなtask editをtransactional clientへ移したが、再起動時の鍵復元は有効なsession tokenを前提とし、List DEKをRAMへ復元しない。そのためnetwork障害やsession期限切れだけでlocal mutationが不能になる。一方、鍵復元不能を匿名利用とみなすと、account dataをoutboxなしで更新するsilent local-only commitが起きる。

### 決定

1. local mutationに必要な`LocalCryptoContext`をremote認証sessionから分離する。session tokenは同期HTTPの認可にだけ使い、期限切れや一時的なnetwork不通をlocal編集不可の理由にしない。
2. Master Keyは従来どおり`wrap(MK, Device Key)`だけをOS secret storeへ保存し、平文を永続化しない。
3. local DBはtenant / user / deviceのprofile bindingをsingletonとして持つ。同じDBへ異なるtenantを上書きloginせず、別profileを要求する。
4. List DEKは平文を追加保存せず、list IDをAADへ含めたlocal専用`wrap(List DEK, MK)` bundleをtenant ID / list ID単位でSQLCipher DBへ永続cacheする。cacheからの復元はprofile bindingと一致する行だけを一括検証し、欠落・破損・ciphertextのlist間入替・別tenant混入を部分成功として扱わない。
5. login、register、key refreshで全bundleの復号検証に成功してからlocal cacheをtransactionalに置換し、その後runtimeをReadyにする。refresh失敗時は既存cacheを維持する。
6. profile状態は`Anonymous`、`AccountBoundReady`、`AccountBoundUnavailable`を区別する。account bindingがあるのにMKまたは必須List DEKを復元できない場合は、すべてのmutationを永続化前にfail closedする。
7. session期限切れはaccount bindingとlocal key cacheを保持し、local mutationを許可する。明示logoutはremote session secretを削除するが、account bindingを匿名へ降格させない。local keyを端末から除去する操作は、local dataの扱いを確認する別の明示的な「この端末からアカウントを削除」として将来設計する。
8. list削除時にcached List DEK bundleを削除しない。ADR-016のexpired-device rebase、late descendant収束、key retirement条件が実装されるまで保持する。
9. 将来のoffline list作成は、List DEK生成、MK-wrap cache、domain row、entity outbox、key-bundle upload queueを同一local transactionで確定する。key bundleをentity blobより先にidempotent uploadする。network I/Oをwrite transaction内で行わない。
10. serverのlist-key upsertはstale clientによる既存bundle上書きを無条件に許可しない。offline list作成を実装するtaskでcreate-if-absentまたはkey generation条件をprotocolとして固定する。

### 帰結

- app再起動、session期限切れ、network不通でも既存List配下のlocal編集とtransactional outbox生成を継続できる。
- OS secret storeのMK wrapまたはSQLCipher内のwrapped DEK cacheを失ったprofileは、匿名fallbackせず明示的に編集不能になる。
- task/list CRUDのtransaction移行と、offline list作成のkey-bundle upload queueは分けて実装できる。
- session復元とlocal crypto復元を同じruntime stateへ押し込めず、共通client/profile層へ段階的に移す必要がある。

### 却下した代替案

- **List DEK平文をsettingsへ保存する案**: SQLCipherで保護されるが、既存のMK key hierarchyを迂回する不要な平文永続化になるため不採用。
- **起動ごとにserverからList DEKを取得する案**: session期限・network可用性をlocal編集へ結合し、offline-first要件を満たさない。
- **鍵復元不能時に匿名mutationへfallbackする案**: account-bound dataへoutboxなしの変更を作るため不採用。
- **明示logoutでaccount bindingも消す案**: 同じDB内のaccount dataが匿名dataへ見える状態を作るため不採用。
