# 企画書：E2EE TODOアプリ「Taskveil」

> Baseline: 2026-07-23 product owner承認。ADR-023参照。
>
> 本書はproduct scopeと原則の正本である。詳細機能は`docs/02_機能仕様書.md`、技術契約は`docs/03_技術仕様書.md`を正本とする。

## 1. Product definition

Taskveilは、個人が日々のtaskと行ったことの記録を安心して蓄積し、選んだlistを家族・友人と共同管理できるlocal-first E2EE TODOアプリである。

一般的な高機能TODOとしての便利さをE2EEのために大きく削らない。検索、filter、calendar、recurrence、reminder、template、timer等はclient側で実現し、serverへtask plaintextを渡さない。

E2EEは追加optionではなく、Account-backed dataと共有dataに常時適用する基盤である。Local-onlyでもlocal encrypted databaseをdefaultとする。

## 2. Product principles

優先順位は次のとおりとする。

1. **利用者の記録を失わない**: offline、競合、quota、subscription、server障害でlocal dataを黙って捨てない。
2. **内容をserverへ渡さない**: title、note、status、date、reminder、recurrence、hierarchy、timer relationをE2EEにする。
3. **TODOとして妥協しない**: privacyを理由に一般的なtask管理、検索、通知、recurrence、calendar、timerを省かない。
4. **完了を成果として残す**: completed taskと完了したwork sessionを通常の整理で削除しない。
5. **暗号を日常操作へ露出しすぎない**: recovery、identity change、member removal等、結果が重要な場面だけを平易に説明する。
6. **標準と単純さを優先する**: 実績のあるprimitiveを組み合わせ、独自暗号やmessage/enterprise protocolの流用を避ける。
7. **privacyの限界を明示する**: server-visible metadata、endpoint compromise、共有相手によるcopyを隠さない。

## 3. Target users

### 3.1 Initial target

- 私生活と仕事のtaskを1つのappで管理したい個人。
- plaintext cloud TODOへ内容を預けたくない人。
- 複数端末を使い、offlineでも確実に記録したい人。
- 買い物、家事、旅行、育児、event準備等を家族・partner・友人と共有したい人。
- completed taskと費やした時間を後から振り返りたい人。

### 3.2 Initial non-target

- SSO、SCIM、organization policy、admin recovery、監査、法的保全、eDiscoveryを必要とする企業。
- 勤怠承認、請求、manager reportingを必要とするtime tracking。
- 匿名性network、membership hiding、国家規模のtargeted endpoint攻撃を必須要件とする利用者。
- realtime chat、共同文書編集、公開project management。

Account、Space、membership、roleを分離し、将来enterprise layerを追加できる境界は残す。ただし初期UI、server policy、key hierarchyへOrganizationやhidden administratorを先取りしない。

## 4. Product modes

### 4.1 Local-only

- Accountなしで開始できる。
- task、search、view、recurrence、notification、template、timer、exportを利用できる。
- dataとsearch indexをlocal encrypted databaseへ保存する。
- 端末とencrypted exportの両方を失った場合は復旧できない。
- 後からAccountを作ると、local dataをPersonal Spaceへ明示的に取り込める。

### 4.2 Account-backed

- E2EE multi-device sync、encrypted server replica、共有を利用できる。Current-head同期はversioned backupではない。
- serverはlogin、MFA、session、access controlを扱うがcontent decryption keyを得ない。
- offline、subscription expired、quota exceededでもlocal閲覧、編集、search、exportを禁止しない。
- upload不能時のmutationはdurable outboxへ残す。

## 5. Product scope

### 5.1 Personal TODO

- Inbox、複数list、section、tag、template。
- Taskのtitle、note、URL、status、priority、due、scheduled start、estimate、multiple reminder。
- parent/subtask、manual order、move/copy、batch edit、undo。
- recurrence、skip/reschedule、series edit。
- Inbox、Today、Upcoming、Calendar、Board、All、Completed、Won't Do、saved filter。
- local full-text search、sort、group。
- local notification、platform integration、accessible UI。
- encrypted exportと、明示警告付きplaintext export。

### 5.2 Completion and retention

- `done`と`wont_do`は削除ではなく長期保持するstateである。
- Completion Recordによりreopen後も過去の完了を残す。
- 通常の整理はcomplete、won't do、archiveで行う。
- permanent deleteは残したくない情報を消すための例外的・不可逆な操作とする。
- every-edit revision historyを無期限保存するevent sourcingは行わない。

### 5.3 Timer and time tracking

- taskあり/なしのstopwatch。
- configurable Pomodoro work/short break/long break。
- start、pause、resume、finish、discard、manual entry、訂正。
- active timerはdevice-localで復元可能にし、秒単位で同期しない。
- 完了work intervalをWork SessionとしてPersonal Spaceへ同期する。
- Shared Space taskを計測しても、個人のWork Sessionはdefaultで共有しない。
- task/day/week/tag/listごとの実績とestimate比較をclientで集計する。

### 5.4 Family and friends sharing

- Shared Space単位のone-time link/QR invitation。
- `owner`、`editor`、`viewer`の3 role。
- Editorは全共有contentを改変・permanent deleteできるため、招待/role変更時に権限と復旧限界を明示する。
- server access controlとE2EE key possession/signatureを併用する。
- Task、list、section、tag、comment、assignee、completion actorを共同利用する。
- new memberは現在および過去の未削除dataを閲覧できる。
- owner transferはtarget acceptanceを必要とする。
- member removalではserver accessを失効し、fresh Space Key generationを発行する。
- removal時にold dataを一括再暗号化せず、old Recordを次に編集した時にcurrent generationへ移す。

## 6. E2EE promise

serverは次を復号できない。

- Task/list/section/tagの内容、種類、構造。
- status、priority、due、reminder、recurrence。
- comment、assignee、Completion Record。
- timerとtaskのrelation、duration、started/ended time。
- search term、saved filter、notification content。

serverはservice提供のため次を知る。

- Account、email、device/session、IP、push token。
- OPAQUE registration、public identity、wrapped key material、account security/recovery transition。
- Personal/Shared Spaceの存在。
- Shared Space membership、role、owner、revision、key generation、encrypted key envelope、署名付きmanifest/transition履歴。
- opaque record/revision/base revision/operation ID、author Account、protocol version、ciphertext size、Tombstone、delta/idempotency state、change timing。
- encrypted byte usage、plan/quota state。

E2EEはcompromised endpoint、偽client、正規共有相手によるcopy、traffic analysis、serverによるaccess denialを防がない。保証範囲は`docs/redesign/threat-model.md`に記録する。

## 7. Storage quota

- 課金Accountのserver-side encrypted structured dataは約`1 GiB`を安全上限の目安とする。
- 添付fileがないため、通常利用で上限を意識させず、通常画面へ常時meterを表示しない。
- quotaは実際に保存するciphertext、signature、key material、Record metadata、Tombstoneを基準にする。
- quota超過時もpull、local read/edit、permanent delete、exportを許可する。
- serviceは古いcompleted taskを自動削除しない。

価格、free quota、grace period等の詳細はpublic/private境界に従い別文書で管理する。

## 8. Initial exclusions

- 添付file、image/audio upload、file sharing。
- enterprise Organization、admin、SSO/SCIM、監査、retention policy、legal hold。
- server-side plaintext search、analytics、AI、reminder scheduling。
- public/guest/anonymous share。
- realtime chat、presence、typing indicator、live cursor。
- active timerのcross-device handoff/remote control。
- server/supportによるAccount Root Key recovery。
- custom cryptographic primitive、custom consensus、blockchain。

## 9. Business and portability principles

- Local-onlyの基本task管理をservice lock-inにしない。
- Account-backed sync/shared storageにはquotaとplan enforcementを適用できる。
- subscription失効でlocal dataの閲覧、編集、exportを禁止しない。
- Shared Space storageは1回だけowner Accountへquota計上する。
- encrypted/plaintext exportによりserviceから離脱できる。

具体的な価格、収益、契約、内部roadmapはprivate repoを正本とし、本書へ転記しない。

## 10. Delivery order

1. 承認済みproduct/threat/key/sync/sharing baselineを正本文書へ固定する。
2. Crypto suite/encoding、sync protocol、sharing state machineをADR/test vectorへ落とす。
3. Local storage/recovery、server metadata/privacy、quota/retentionを設計する。
4. Implementation work itemをdependency順に分割する。
5. Cross-platform interoperability、failure injection、external security reviewをrelease gateとする。

既存実装、wire protocol、local/server schema、development dataとの後方互換性は要件にしない。互換layerを追加せず、承認済みfinal designへbreaking置換する。

## 11. Major risks

| Risk | Response |
|---|---|
| E2EEでserver-side便利機能が使えない | local index、local scheduling、generic wake-upを使う |
| Password/recovery loss | Recovery Key、device bootstrap、復旧不能条件の事前説明 |
| Shared key/member transition defect | signed manifest、server transaction、state-machine/model test |
| Offline conflict/data loss | local transaction、CAS、typed 3-way merge、Conflict Record |
| Removed memberの旧data保持 | 新dataだけをnew generationで守り、遠隔消去不能を明示 |
| Malicious editorの大量改変・削除 | 招待時の権限説明、署名によるactor識別、viewer選択、user-controlled export。旧content復旧は保証しない |
| Completed historyによる容量増加 | 約1 GiB quota、no attachment、no unbounded edit history |
| 独自protocol complexity | OPAQUE/HPKE/Ed25519/AEAD等の標準primitive、external review |
| Endpoint compromise | OS secure storage、encrypted local DB、正規client/update、明示的非保証 |
