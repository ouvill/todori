## 2. 全体像

TaskveilはローカルファーストのE2EE Todoアプリである。クライアントはSQLCipherで暗号化されたローカルDBを正本として動作し、同期が利用可能な場合だけサーバーのpush/pull APIへ暗号blobを送受信する。

サーバー構成の正は [`docs/03_技術仕様書.md`](./03_技術仕様書.md) §1.5 と §6である。サーバーはAWS Lambda上のRust axumサーバーとして動かし、コンテナイメージをECRへpushしてLambdaへ更新する想定である。DBはNeon Postgresを採用し、初期リージョンは `eu-central-1`、Lambdaも同一リージョンに置く。設計判断は [`docs/05_設計判断記録.md`](./05_設計判断記録.md) ADR-008を参照する。

2026-07-08時点では実AWS/Neon環境へは未デプロイである。運用済みの環境はローカルPostgres + `taskveil-server` のみで、起動は [`tool/dev_server.sh`](../tool/dev_server.sh) が担う。

## 3. リリース状態と互換性

Taskveilは現在、一般リリース前である。プロダクトオーナーがこの状態を変更するまで、既存client、wire protocol、API、local DB schema、server schema、開発データとの後方互換性は要件としない。

correctness・security・設計の一貫性を優先し、breaking change、破壊的migration、開発データの再作成を許容する。互換レイヤ、dual read/write、旧形式fallbackは、taskで明示的に要求されない限り追加しない。局所的な互換修正より、taskまたはADRで判断を記録したうえで正しい最終設計へ直接置き換える。

最初のbetaを含む外部配布を開始する前に、`AGENTS.md`と本節を更新し、配布済みclient、local DB、server API、保存済みデータに対する互換性境界を明示する。それ以降のDB/API運用には5〜7章のリリース後方針を適用する。

最初の一般リリースは、課金基盤に加え、ADR-023のAccount/Space/Record baseline、crypto suite/encoding、sync CAS/merge/Tombstone、invitation/remove/owner-transfer state machine、local storage/recovery、quota/retentionを実装し、cross-platform interoperability、failure injection、外部security reviewを完了した後に行う。旧ADR-020 Organization/PQC gateの合格をnew baselineの代替証拠にしない。具体価格、provider契約、launch offer、実credentialはpublic文書へ記録しない。

## 4. 開発運用

通常の変更は `docs/tasks/` の指示書単位で行う。作業前に [`AGENTS.md`](../AGENTS.md)、対象タスク、必要な仕様書を読み、完了時は対象タスクへ `## 9. 完了報告` を追記する。

コミット前品質ゲートは以下を基準にする。

```sh
cargo fmt --all -- --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
cd app && flutter analyze
cd app/rust && env CARGO_TARGET_DIR=target cargo build --release
cd app && flutter test
sh app/tool/check_hardcoded_strings.sh
sh tool/check_crypto_dependency_pins.sh
sh tool/check_secret_patterns.sh
git diff --check
```

Flutter UIの見た目を変える場合は、通常のwidget testに加えてvisual QAを使う。visual QAはCIの常時ゲートではなく、必要時に次を実行する。

```sh
app/tool/visual_qa.sh
```

ローカル同期サーバーは次で起動する。

```sh
./tool/dev_server.sh
```

このスクリプトは `taskveil-dev-postgres` コンテナを作成または再利用し、`server/migrations/*.sql` を適用してから `cargo run -p taskveil-server` を `http://localhost:8080` で起動する。2台同期確認は [`docs/dev/two-device-sync-test.md`](./dev/two-device-sync-test.md) の手順を使う。

> **Legacy範囲（5〜8.1）**: 以下はADR-023以前の現行実装を安全に運用するための手順である。新baselineのAPI、server schema、同期方式、鍵配布方式として再利用してはならない。新baselineの正本は `docs/03_技術仕様書.md` と `docs/redesign/` である。

## 5. 現行legacyサーバー運用

サーバーはE2EEデータを復号しない。保存するのは認証・デバイス・テナント・セッション等のコントロールプレーン情報、同期用の暗号blob、同期メタデータである。同期APIは `POST /v2/tenants/{tenant_id}/push` と `GET /v2/tenants/{tenant_id}/pull` を中心に構成する。

本番想定の構成は次の通りである。

- AWS Lambda: Rust axum HTTPサーバーをコンテナイメージとして実行する。
- AWS Lambda Web Adapter: LambdaイベントとHTTPリクエストを変換する。
- ECR: `server/Dockerfile` で作成したコンテナイメージを保管する。
- Neon Postgres: 同期ストアとコントロールプレーンDB。接続はNeon組み込み接続プーラー経由のTCP接続を使う。
- 前段: CloudFront + WAF または API Gateway throttling。最終構成は人間判断とする。

実AWS/NeonのアカウントID、ECRリポジトリ名、Lambda関数名、Neon connection string、WAF/API Gateway設定値はprivate側または人間管理で保持する。

### 5.1 foreground realtime通知

foreground realtime通知はCloudflare Worker + Durable Objectsを初期providerとするが、同期の正本ではない。Cloudflare停止、WebSocket切断、通知欠落、server publish timeout時も、local CRUD、明示sync、foreground復帰sync、30秒fallback pollingを継続する。通知障害を理由にcommit済みpushを失敗扱いしない（ADR-019）。

serverとWorkerには用途別のchannel derivation、ticket signing、publish signing secretを設定する。ticket / publish signing keyはcurrent / previous key IDを使ってoverlap rotationし、channel derivation keyはv1中固定する。全realtime環境変数がないserverは通知disabledで起動できるが、一部だけ設定された状態はstartup errorとする。secret、ticket、opaque channel / device tag、tenant / device UUIDをlog、完了報告、public issueへ出してはならない。

serverのrealtime設定は次の11変数を1組として扱う。URLはそれぞれ`wss://.../v1/connect`と`https://.../v1/publish`、keyは32 bytesのbase64url no-padding、key IDは`[A-Za-z0-9_-]{1,32}`とする。current / previous IDと、channel / ticket / publishのkey materialは重複させない。11変数がすべて未設定ならdisabled、1つでも設定した場合は全変数を必須とする。実値はsecret storeで管理し、public文書やdeploy commandへ記載しない。

- `TASKVEIL_REALTIME_WEBSOCKET_URL`
- `TASKVEIL_REALTIME_PUBLISH_URL`
- `TASKVEIL_REALTIME_CHANNEL_KEY`
- `TASKVEIL_REALTIME_TICKET_KEY_CURRENT_ID` / `TASKVEIL_REALTIME_TICKET_KEY_CURRENT`
- `TASKVEIL_REALTIME_TICKET_KEY_PREVIOUS_ID` / `TASKVEIL_REALTIME_TICKET_KEY_PREVIOUS`
- `TASKVEIL_REALTIME_PUBLISH_KEY_CURRENT_ID` / `TASKVEIL_REALTIME_PUBLISH_KEY_CURRENT`
- `TASKVEIL_REALTIME_PUBLISH_KEY_PREVIOUS_ID` / `TASKVEIL_REALTIME_PUBLISH_KEY_PREVIOUS`

初期監視項目は、ticket発行成否、WebSocket接続成否、publish成功 / timeout / provider error、通知受信からsync開始までの遅延、接続中 / 切断中のsync run数とする。logはerror kindと集計値だけを持ち、tenant別識別子やrecord metadataを含めない。Cloudflare本番resource作成、secret投入、data jurisdiction確認、費用上限設定はcredentialを持つ人間のrelease作業として扱う。

実装済みのstructured event field allowlistは、serverが`event`と任意の数値`status_class`、Workerが`event`だけ、Flutterが`event`、`connection_state`、`trigger_kind`、任意の非負`latency_ms`だけである。`trigger_kind=remote_hint`の`realtime_sync_started`にある`latency_ms`を通知受信からHTTPS sync開始までのlocal観測値として集計する。いずれの層でもticket、URL、tenant / device UUID、opaque channel / device tag、record metadataをfieldやmessageへ追加しない。

## 6. 現行legacy DBマイグレーション運用

サーバーDBのマイグレーションSQLは [`server/migrations/`](../server/migrations/) に置く。専用`taskveil-migrate` binaryが`DATABASE_MIGRATION_URL`のowner接続で適用し、通常`taskveil-server`は起動時migrationを行わず`DATABASE_URL`のnon-owner runtime loginだけを使う。runtime loginはNOLOGIN group role `taskveil_app`を`INHERIT`し、session-level `SET ROLE`に依存しないためNeon transaction poolと両立する。tenant所有tableはtransaction-local user / tenant context、RLS policy、`FORCE ROW LEVEL SECURITY`で保護する。ローカル開発では [`tool/dev_server.sh`](../tool/dev_server.sh) が同じSQLファイルを `psql` でも適用する。

一般リリース後のDB変更は原則として前方のみのマイグレーションにする。互換性が必要な列・テーブル変更はexpand-contractを基本にし、即時の破壊的変更、既存列の意味変更、データ削除を避ける。現在の一般リリース前期間は3章を優先する。

詳細手順は [`docs/ops/runbook-db-migration.md`](./ops/runbook-db-migration.md) を参照する。

## 7. 現行legacyクライアント運用

ローカルDBはSQLCipherで暗号化され、スキーマバージョンはSQLiteの `PRAGMA user_version` で管理する。現行の `core/storage` はDB open時に未適用migrationを順次適用する。

一般リリース後の互換性方針は次の通りである。現在の一般リリース前期間は3章を優先する。

- 新しいアプリが古いDBを開く場合、実装済みmigrationを順に適用して最新スキーマへ上げる。
- 古いアプリが新しいDBを開く場合、既知の最新 `user_version` を超えるため `UnsupportedSchemaVersion` として開かない。これはDB破損や誤った旧スキーマ書込を避けるための挙動である。
- クライアントリリースでは、DB migrationを含むバージョンをロールバックしない前提で扱う。ストア公開後の取り下げや差し戻しが必要な場合も、DB schemaを戻すのではなく修正版を前方migrationとして出す。
- 前方互換が必要な変更は、旧アプリが未知列を無視できる追加に留める。削除・型変更・NOT NULL追加は段階的に行う。

## 8. 現行legacy実装のセキュリティ運用

秘密情報はログ、Debug出力、スクリーンショット、完了報告、public issue、public PRに出してはならない。該当する秘密情報には、パスワード、OPAQUE中間状態、session token、Device Key、Master Key、DEK、Recovery Key、exportKey、SQLCipher鍵、Neon connection string、AWS credentialが含まれる。

鍵の取り扱い禁忌は次の通りである。

- Device Key、Master Key、DEK、Recovery Key、exportKeyを平文で永続化しない。
- 復号済みタスク本文をサーバーログ、分析ログ、クラッシュレポートへ送らない。
- E2EE暗号blobの復号をサーバー側で試みない。Taskveilの前提は「サーバーはユーザーデータを復号できない」である。
- 開発用の `FileDeviceKeyStore` を本番primary storeとして扱わない。
- active generation未満の鍵でlive dataを新規暗号化しない。active key / signed manifestを取得できないscopeはfail closedにする。
- `pending` DK capsuleを通常の互換fallbackに使わない。crash recovery時だけactive / pendingの一方へ収束させる。
- Safety number未確認またはaccount root変更後のOrganization membershipへTenant / List DEKを配送しない。
- Safety number確認済みroot / digest、device roster revision / chain head hash、active generationのclient durable pinを消去・巻き戻さない。root変更は再確認し、署名済み`TDR1`失効chainのrollback / forkまたはgenerationがlocal minimumより古い場合はfail closedにする。device失効時はaccountが所属する全Organizationがrotation-requiredになったことを確認する。

### 8.1 鍵ローテーション

- rotation statusは`prepared -> active -> migrating -> retired`の順だけを許す。active切替はminimum write generationと同一transactionで行う。
- password変更とRecovery再発行はMKを変えずwrapper revisionだけを更新する。MK rotationは子鍵rewrapだけを行い、Tenant / List DEK rotationだけがlive contentを再暗号化する。
- member / device除名、鍵侵害疑い、Recovery漏洩、algorithm移行をrotation triggerとする。定期的な全data再暗号化は行わない。
- 旧generation bundleは、全live head移行、全非expired device continuity ACK、30日history保持終了を確認してから削除する。
- incident時も、除名前に端末へ取得済みの旧dataを遠隔消去できるとは案内しない。

#### 8.1.1 Device Key platform確認

release候補ごとに、実機をunlock済みの状態で次を実行する。testは同じprofileを再利用するため、1回目がrotationと新DKでのDB reopen、2回目がprocess再起動後のactive capsule再利用を検証する。

```sh
cd app
flutter drive --driver=test_driver/integration_test.dart --target=integration_test/device_key_rotation_test.dart -d <device-id> --profile
flutter drive --driver=test_driver/integration_test.dart --target=integration_test/device_key_rotation_test.dart -d <device-id> --profile
# 配布形式そのもののbuild成立も別途確認する
flutter build ios --release --no-codesign       # iOS候補の場合
export JAVA_HOME="/absolute/path/to/jdk-21"
flutter build apk --release
```

- iOS / macOS: 両runでKeychain認証promptが0回、generationが単調増加し、`getLists`が成功することを記録する。Keychain itemはData Protection、`synchronizable=false`、`AfterFirstUnlockThisDeviceOnly`であることを確認する。
- Android: 両runに加えて`cd app/android && ./gradlew connectedDebugAndroidTest`で`AndroidCapsuleStoreTest`も実行し、AES keyの`encoded == null`（non-exportable）、active / pending capsule roundtrip、DB reopen成功を記録する。profile buildはCargokitがRust `--release`へ写像するためproduction storeを通り、配布形式の成立は別途release buildで確認する。
- crash recovery確認ではtest-only failure injectionの4境界（pending保存後、rekey後、新DK reopen検証後、active昇格後）が全件activeまたはpendingの一方へ収束することをRust testで確認する。pendingが無い通常起動で別generationを試してはならない。
- `<db_dir>/device.key`、平文capsule、DK / SQLCipher keyが生成されていないことと、platform log / crash reportへ秘密値が出ていないことを確認する。

CIでは非docs変更ごとにiOS arm64 no-codesign release appとAndroid arm64 release APKをbuildし、各packageのRust bridge成果物まで検証する。API 36 / Google APIs / x86_64 Emulatorの`AndroidCapsuleStoreTest`とDevice Key rotation 2回は週次scheduleおよびGitHub Actionsの手動実行で通す。iOS Simulator runtime testは実機のKeychain / 通知 / 購入・復元 / 同期gateを代替しないため定期CIへ追加しない。ローカルのPixel 10 AVD（Android 37.1 / arm64-v8a / 16 KiB page-size）は前方互換確認に使うが、安定版CIや接続実機release gateの代替にしない。

### 8.2 暗号release gate

- RFC 9807 OPAQUE、BIP39、FIPS 203 / 204 vectorとnegative testを実行する。
- rotation境界failure injection、3端末offline / removal / stale push / expired-device scenarioを実行する。
- Apple / Android実機でsecret store、DB rekey、再起動、秘密値のlog非露出を確認する。
- `cargo audit`、dependency pin、envelope / manifest parser fuzz、秘密情報grepを実行する。
- 実装担当と独立した暗号reviewを一般リリース前に行い、外部review完了前は「監査済み」と表示しない。

依存脆弱性、固定version、parser fuzz、秘密情報grepはCIとrelease候補の必須gateである。完全なcommand、scenario対応、platform / review状態は [`docs/ops/crypto-release-gate.md`](./ops/crypto-release-gate.md) を正本とする。未導入環境では固定tool versionを導入する。

```sh
cargo install cargo-audit --version 0.22.2 --locked
cargo install cargo-fuzz --version 0.13.2 --locked
cargo audit --deny warnings
cargo +nightly-2026-07-14 fuzz run crypto_parsers -- -max_total_time=60 -timeout=10 -max_len=16384 -dict=fuzz/crypto_parsers.dict
```

脆弱性を発見した場合は [`SECURITY.md`](../SECURITY.md) の非公開報告導線を使う。public issueに技術詳細や再現手順を書かない。
