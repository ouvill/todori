# ADR-012: 同期correctnessはfield clock・placement・transaction・snapshotを一体で保証する

**日付**: 2026-07-10
**状態**: Accepted（2026-07-10 プロダクトオーナー承認）

**補正**: 2026-07-10のADR-014により、latest-one encrypted blobのlive更新は`revision_hlc` LWWではなく`base_revision_hlc`付きCASとし、stale pushをclient merge/retryへ戻す方式へ改訂した。2026-07-11のADR-016により、aggregate削除scope / epochの後続要求は、server-visible hierarchy metadataを追加しないbounded tombstone + device continuity + expired-device rebaseへ置換した。

### 背景

ADR-004 / ADR-005は、暗号payload内のfield-level LWW、最新状態方式、client再push、server seq cursorを採用した。ADR-010は空blob tombstoneとGC horizonを採用し、ADR-011はdomain更新とoutbox enqueueを同一transactionへ置く共通client層を決定した。

一方、production実装のレビューで、通常CRUDが全fieldへ同一HLCを再付与するためfield-level LWWがrecord-level LWWへ退化すること、`sort_order` が同期されないこと、domain更新とoutboxが別commitであること、階層削除がroot/listのtombstoneしか生成しないこと、復号不能recordをskipした後にcursorを進めることが確認された。また、ADR-010の `since=0` だけでは、GC後の全件同期を有限かつ連続なcursorへ接続するためのsnapshot/high-waterが不足する。

これらは個別bugではなく、同期correctnessを成立させる不変条件の欠落である。GCとフル再同期を先に実装すると、不完全なlocal stateと削除意味論を固定化するため、correctness基盤を先に再定義する。

### 決定

1. 暗号payload内の `field_hlcs` は各field値の競合解決だけに使う。local mutationは実際に変更したfieldのHLCだけを更新し、未変更fieldのHLCを保持する。
2. serverがlive blob同士のlatest envelopeを採用するためのtransport clockを `revision_hlc` としてfield clockから分離する。local mutationとmerge後の再pushは `revision_hlc` をtickするが、再pushだけを理由に全field HLCを引き上げない。pullで受信したremote revisionをlocal HLCへobserveし、merge再pushのrevisionは受信revisionより必ず大きくする。observe、採番、HLC永続化、repush用outbox登録は同一local transactionで確定する。wire protocol v2で現行top-level `hlc` の名称と意味を移行する。
3. live recordは最後のdomain/user変更を表す `mutation_hlc`、tombstoneは削除操作を表す `delete_hlc` をtop-level semantic clockとして持つ。merge再pushは `mutation_hlc` を更新しない。live同士のenvelope採用は `revision_hlc`、liveからtombstoneへのCAS過程は `mutation_hlc` と `delete_hlc` を用いる。ADR-016に従い、tombstone受理後の同一record IDはliveへ復活させない。
4. taskの配置は `list_id`、`parent_task_id`、`sort_order` を分離fieldとして競合させず、単一の `placement` 値として1つのfield HLCで原子的に更新する。listの `sort_order` も同期対象のplacementとする。
5. fractional rankは同期payloadへ含める。同一gapへの同時insertでrankが衝突しても全端末が同じ順序へ収束し、衝突後も新しいrankを生成できるtie-break / collision repairを実装要件とする。具体的な符号化と依存選定は後続taskで確定する。
6. domain row、対応するoutbox entry、同期用plaintext state、local HLC永続化は、共通client層が所有する同一SQLite transactionで確定する。失敗時は全体をrollbackし、local-only commitまたはoutbox-only commitを作らない。
7. pull pageは、復号と検証に成功した適用内容、sync record state、必要なmerge repush用outbox、cursorを同一local transactionで確定する。HTTP、key refresh等のnetwork I/Oはtransaction外で行い、cursorだけを先行commitしない。
8. continuity / protocol preflightをpayload applyから分離する。missing DEKはkey bundleをrefreshして同一recordを再試行し、なお不足する場合は暗号blobと理由をdurable quarantineへ保存してcursorを進め、key取得後に再適用する。認証復号失敗もdurable quarantineと明示的なcorruption状態を必須とする。未知protocol versionはupgrade-requiredとしてpayload取得とpushを停止する。silent skipは行わない。
9. task subtree/list削除時、削除端末が把握しているrootとdescendantについては、削除前にrecordを列挙し、各recordの空blob tombstoneをdomain削除と同じlocal transactionでoutboxへ登録する。別端末の未同期・同時作成recordは、ADR-016に従い復号後のancestor検証でdomainへmaterializeせず、client-side cascade tombstoneへ収束させる。
10. aggregate削除のためのserver-visible scope ID / epochは追加しない。tombstone保持期間とoffline復帰保証期間を180日で接続し、expired deviceはfull resync closureとserver-seen / local-new / dependency rebaseを完了するまでpushできない。同一record IDのtombstoneはliveへ復活せず、明示的な再作成は新IDを使う。List DEK bundleはADR-016のkey retirement条件が実装されるまで削除しない。
11. sync runはpayloadを扱う前にcontinuity / protocolをpreflightし、通常のremote pull / delete reconcileまたはexpired-device full resync / rebaseをlocal outbox pushより先に完了する。server-trusted device continuityが`gc_horizon_seq`未満ならkey bundle / entity pushを拒否する。個別recordのquarantineは無関係なlocal変更の送信を永久停止させない（2026-07-11改訂: ADR-016）。
12. full resyncは厳密な過去snapshotではなく、stable-keyによるcurrent-state fuzzy scan + delta catch-upとして定義する。開始時にserver transaction内で `base_seq = tenant_seq.last_seq` を取得し、base tableを `seq <= base_seq` に限定せず、`(collection, record_id)` 等の更新で移動しないstable keyで走査する。base中にpage cursorの後方へ作成されたrecordは `seq > base_seq` のdeltaで回収する。
13. delta pageはpage rowsと同じserver transactionで `high_water = tenant_seq.last_seq` を取得する。`has_more=false` かつclient cursorがそのpageの `high_water` に到達した時だけcatch-up closureとする。closure後の更新は通常deltaで取得する。
14. clientはresync generationをlocalに作り、base scanとdelta catch-upで確認したrecordをmarkする。expired-device rebaseでは、server currentにあるrecord、server-seen but absent record、never-synced local-new record、missing/deleted dependencyを分類し、未ACK outboxを一律保護しない。適用・rebase/sweep・最終cursor確定は失敗時に再試行可能なtransaction境界を持つ。最終cursorはclosure時のserver high-waterとする（2026-07-11改訂: ADR-016）。
15. 既存local dataを持つprofileを新規tenantへ登録するときは、tenant/profile紐付け後、全local recordをnever-synced creation origin付きtransactional seed generationへ登録する。その後にfull resync closureを行い、pending List DEK bundle → list → taskの依存順でpushする。各境界のcrash recoveryをテストする（2026-07-11改訂: ADR-016）。
16. login成功だけを理由に同一tenantのinitial backfillとpull cursorを無条件resetしない。local DB/profileをtenant identityへ結び付け、同一accountへの再login、account切替、新規profileを区別する。
17. push responseはrequest順を暗黙に信用しない。protocol op ID、結果件数、record IDを照合してからoutboxをACKする。collectionはprotocol enumで制限し、既存recordのcollection変更を拒否する。
18. property testと2-client統合testはproduction CRUD / common client経路を通す。テスト専用clientがfield clockやmergeを本番より正しく実装する状態を許容しない。
19. serverのtenant分離はqueryの `tenant_id` と認可だけでなく、non-owner application role、実際のRLS policy、必要な `FORCE ROW LEVEL SECURITY`、cross-tenant testで検証する。RLSを有効化しただけの状態を多層防御完了と表現しない。

### 実装順

1. production経路を通る失敗テストと、ADR-011の共通client transactionを実装する。
2. `revision_hlc` / `field_hlcs`、placement、push response / collection validationを実装する。
3. typed pull failure、durable quarantine、cursor transactionを実装する。
4. fuzzy base scan + delta full resyncとGC horizonを実装する。
5. ADR-016のserver-trusted device continuity、expired-device rebase、terminal tombstone、client-side descendant cascade、key retirementを実装する。
6. non-owner DB roleとRLS policyを実装・検証する。

前段が未完了のまま後段をリリース対象へ昇格しない。

### 帰結

- ADR-004の自前同期、ADR-005の最新状態+server seq、ADR-010の空blob tombstone、ADR-011の共通profileは維持する。
- wire protocol、local schema、application serviceの変更を伴うため、既存server/client間の互換境界をprotocol v2として明示する必要がある。protocol v2は少なくとも `revision_hlc`、`mutation_hlc`、`delete_hlc` を区別する。
- 現行2-client testと `cargo test -p taskveil-sync` の成功だけではproduction correctnessの証拠にならない。
- full resyncとGC horizonはcorrectness基盤の後続になり、直近着手候補から一段後ろへ移る。

### 却下した代替案

- **現行実装へGC/full resyncだけ追加する案**: field clock、outbox、削除cascade、cursor skipの欠陥を温存したままabsence sweepを導入し、データ損失経路を増やすため不採用。
- **再push時に全field HLCを新revisionへ揃える案**: 未確認の同時編集より古いfield値を新HLCで上書きでき、field-level LWWの意味を失うため不採用。
- **transport revisionで削除競合も判定する案**: user mutationを伴わないmerge再pushで削除を復活させ得るため不採用。
- **fractional rankを同期せず各端末で再計算する案**: 同じtask集合でもユーザーが行ったreorderを復元できず、端末間の表示順が収束しないため不採用。
- **復号不能recordを計数してskipする案**: 一時的なkey取得失敗と恒久corruptionを区別せず、cursor前進によって回復不能になるため不採用。
- **full resyncを動的なseq pageだけで実装する案**: updateでrowのseqが移動し、空結果時のhigh-waterも得られず、有限で一貫したabsence sweepを定義できないため不採用。
- **clientが列挙したdescendantのACKだけでList DEK bundleを削除する案**: 別端末の未知recordとlate descendantを覆えず、復号・依存分類の前に鍵を失うため不採用。
