# 企画書：E2EE Todoアプリ「Todori（トドリ）」

> 本ドキュメントはプロダクト企画の全体像を示すものです。機能の詳細は [`02_機能仕様書.md`](./02_機能仕様書.md)、実装方式の詳細は [`03_技術仕様書.md`](./03_技術仕様書.md) を参照してください。

## 1. プロダクト概要

- **プロダクト名**: Todori（トドリ）
  - ※ 名称は2026年7月に正式決定。名称の由来は ToDo + 鳥（tori）。マスコットにはエメラルド色の小鳥「メジロ」を想定する（ブランドカラーと整合）。公開向けの法務・OSS方針は [`legal_overview.md`](./legal_overview.md) を参照する。
- **コンセプト**: 「プライバシーを一切妥協しない、ふわっと親しみやすいTodoアプリ」

Todoriは、E2EE（エンドツーエンド暗号化）とローカルファーストの設計思想を軸に据えたTodo管理アプリです。iOS / Android / Desktop（Windows・macOS・Linux）のマルチプラットフォームで動作し、ユーザーはアカウント登録や課金を行わなくても、単一端末上でアプリの全コア機能を利用できます。複数端末間の同期やOrganization（組織）でのタスク共有など、サーバーを介した機能のみを有料の対象とすることで、「まずは気軽に使い始められる」体験と「本気で使うほど便利になる」拡張性の両立を目指します。

Todoriは、完了・中止したタスクをユーザーの実績と振り返りのための記録資産として扱います。日常の整理ではタスクを`done` / `wont_do`として閉じ、リストはアーカイブして履歴を保全します。永久削除は、誤って作成した情報や残したくない情報を本当に消すための例外的かつ不可逆な操作として、明示的な確認を伴う導線だけで提供します。

デザイン面では、丸みのあるフォルムや淡い配色、心地よいマイクロインタラクションを取り入れ、「柔らかく・親しみやすく・エレガント」をキーワードに、セキュリティ製品にありがちな硬い印象を払拭します。ブランドカラーにはエメラルドを採用し、信頼感と親しみやすさを両立させます。また、初回起動時にはタスク管理に不慣れなユーザー向けの「シンプルUI」と、高度なタスク管理を求めるユーザー向けの「高機能UI」を選択でき、利用開始後にいつでも切り替えられるようにします。プライバシー保護と使いやすさのどちらも妥協しない、というのがTodoriの根幹にある思想です。

## 2. 背景・課題

現在広く使われているTodoアプリ（Todoist、TickTick、Things、Microsoft To Do など）の多くは、タスクの内容を平文のままサーバーに保存しています。しかし、Todoリストには健康・仕事上の機密事項・家族や人間関係に関する事柄など、非常にセンシティブな情報が日常的に記録されます。にもかかわらず、これらの情報がどのように扱われ、誰がアクセスしうるのかについて、ユーザーが十分に意識・選択できる仕組みは一般的ではありません。

一方で、Standard Notesのように高い暗号化水準を備えたE2EEノートアプリはすでに存在します。しかし、「E2EEでありながら、繰り返しタスクやサブタスク、タイマー機能といった高機能なTodo管理を実現しているプロダクト」は依然として空白地帯です。さらに近年は、ローカルAIやMCP（Model Context Protocol）の普及により、「自分のデータを外部サービスに渡すことなく、手元でAIを活用したい」というニーズが急速に高まっています。Todoriは、この課題領域とニーズの高まりに応えるプロダクトとして企画されました。

## 3. ターゲットユーザー

対象市場は特定の国に限定せず、**グローバル市場**とします。プライバシー意識の高いユーザー層は地域を問わず存在し、E2EEという価値提案は言語・文化圏に依存しないためです。このため、UI・ストア掲載情報ともに基準言語を英語とし、初期リリース時点から多言語対応を行います（詳細は機能仕様書 F-48）。

| ユーザー層 | 想定される利用シーン |
| --- | --- |
| プライバシー意識の高い個人 | エンジニア、士業（弁護士・会計士等）、医療従事者、ジャーナリストなど、業務上・職業倫理上、情報の機密性に強く配慮する必要があるユーザー |
| セキュリティ要件の厳しい小規模チーム・組織 | 顧客情報や機密プロジェクトを扱うため、外部サービスへの平文保存を避けたいチーム |
| シンプルに使いたい一般ユーザー | まずは無料・登録不要という入口から気軽に使い始め、必要に応じて高度な機能や有料プランへ移行するユーザー |

## 4. 差別化ポイント

1. **E2EE + OPAQUE認証**: サーバー側はユーザーのパスワードはもちろん、タスクの内容も一切知り得ない設計とします。
2. **登録不要・オフラインファースト**: アプリをインストールした瞬間から、アカウント作成なしにフル機能を使い始められます。
3. **シンプルUI／高機能UIの二段構え**: 利用者の習熟度や好みに応じてUIを切り替え可能とし、初心者にもパワーユーザーにも対応します。
4. **オープンな拡張性**: MCP・CLIによる連携やローカルAIの活用により、データ主権を手放すことなく最新のAI体験を取り込めます。
5. **Organization対応のE2EEタスク共有**: 組織利用においても、暗号化を維持したままタスクの共同管理を可能にします。

## 5. 機能一覧

Todoriが提供する機能は「ユーザー要求機能」と「追加提案機能」に大別されます。詳細な仕様は [`02_機能仕様書.md`](./02_機能仕様書.md) にまとめます。

### 5.1 ユーザー要求機能

- 単純なTodo管理
- カレンダー表示
- Pomodoro・作業タイマー
- 完了タスクの保存と振り返り
- 「やらない（Won't Do）」によるクローズ
- 無制限に入れ子にできるサブタスク
- タスクテンプレート
- 繰り返しタスクによるスケジュール自動登録
- ローカルAIによるTodo作成
- MCP・CLI連携
- 日付のみ期限、日時指定期限、作業予定時刻、所要時間、通知時刻などの詳細設定
- 複数リストによる管理

### 5.2 追加提案機能（企画チームからの提案）

> 以下は要求機能を補完する目的でチーム側から追加提案する機能です。ユーザー要求機能とは区別してご確認ください。

- タグ・優先度・全文検索／フィルタ（検索処理はローカルで完結させ、E2EEであってもインデックス情報がサーバーに漏れない設計とします）
- Todayビュー／フォーカスモード、かんばん形式の表示
- 自然言語によるクイック入力（例:「明日15時 歯医者」と入力すると期日を自動設定。処理はローカルAIで実行）
- AIによるタスク分解提案（大きなタスクをサブタスク案に自動分解）
- 習慣トラッキング（繰り返しタスクの連続達成streakの可視化）
- 統計・週次レビュー（Pomodoro実績、タスク完了率などのふりかえり機能）
- リカバリーキーの発行（パスワード忘れ対策。E2EEを採用する以上、必須の機能と位置付けます）
- 生体認証によるアプリロック、デバイス管理（リンク済み端末の一覧表示・失効）
- アーカイブと完了・編集Undo、暗号化されたバックアップ／エクスポート（JSON・Markdown・CSV形式）
- ホーム画面ウィジェット、OSの共有シートからのタスク登録
- Organization向け機能: 担当者アサイン、タスクへのコメント、監査ログ
- 将来的な拡張候補: Apple Watch／Wear OS対応、タイムブロッキング機能

## 7. 技術方針（概要）

技術面の詳細は [`03_技術仕様書.md`](./03_技術仕様書.md) にまとめますが、企画上の要点は以下のとおりです。

- UI層はFlutter、暗号処理・同期・ドメインロジックなどのコア部分はRustで実装し、両者を`flutter_rust_bridge`で接続します。
- CLIおよびMCPサーバーも同一のRust製コアを再利用することで、ロジックの二重実装を避けます。
- データベースにはPostgres（Neon等のサーバーレスPostgres）を採用します。サーバーにはエンドツーエンド暗号化された解読不能なデータのみが保存され、同期APIがテナント単位のアクセス制御を強制します。
- 認証にはOPAQUEプロトコルを採用します。OPAQUEの`exportKey`はマスターキーとして直接使うのではなく、鍵導出における一要素として利用します。
- 通知はローカル通知のみで実現します。これはE2EEを採用する以上、サーバー側が通知時刻などのスケジュール情報を把握できないという設計上の帰結です。
- クライアント・コアを含めてソースコードを公開（OSS）します。E2EEアプリにとって、実装が第三者に検証可能であることはそれ自体が信頼の裏付けとなるためです（公開向けの法務・OSS方針は [`legal_overview.md`](./legal_overview.md)）。

## 8. ロードマップ

開発は以下の4フェーズに分けて段階的に進めます。

| フェーズ | 概要 |
| --- | --- |
| Phase 1（MVP） | ローカル基盤。M1〜M4でシンプルUI、Todo・リスト・サブタスク・完了・Won't Do機能、デバイスキーによる保存時暗号化を実装済みです。M5の一般リリース準備は課金基盤完成後へ延期します。 |
| Phase 2 | アカウント機能、OPAQUE認証、E2EE同期、カレンダー表示、Pomodoro、Android Flutter release APK build、Android Keystoreを実装済みです。Android接続実機でのKeystore・Device Key rotation・DB reopen・同期検証、テンプレート・繰り返しタスク、一般リリースを解放する課金基盤が残ります。 |
| Phase 3 | Organizationによるタスク共有、高機能UI、MCP・CLI連携、タグ・統計機能を実装します。全文検索UIはPhase 2後続のtask-103で先行実装済みです。 |
| Phase 4 | ローカルAI、ホーム画面ウィジェット、ウォッチ対応、監査ログなど、さらなる拡張機能を実装します。 |

## 9. リスクと対策

| リスク | 対策 |
| --- | --- |
| Flutter Desktopの成熟度が十分でない可能性 | 開発の早期段階からDesktop版を実際に使い込む（dogfooding）ことで問題を早期に発見・対処します。 |
| OPAQUEライブラリ（opaque-ke等）の保守状況への依存 | 暗号処理層を抽象化し、ライブラリの差し替えが可能な設計とします。 |
| パスワード紛失＝データ喪失というE2EE特有のUXリスク | アカウント登録フローにリカバリーキーの発行を組み込み、パスワード紛失時の救済手段を用意します。 |
| 特定ベンダーへのロックイン | データストアは標準的なPostgresであり、マネージドサービス間の移行は容易です。同期プロトコルは自前の抽象化層として設計し、特定バックエンドに依存しません。 |
| ローカルAIのモデルサイズ・端末性能差による体験差 | ローカルAI機能はオプトイン方式とし、モデルは任意ダウンロードとすることで、端末性能に応じた選択を可能にします。 |
