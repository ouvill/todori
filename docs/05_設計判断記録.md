## ADR-001: E2EEを製品の核として厳格適用する

**日付**: 2026-07-03
**状態**: 採用

### 決定

サーバーはDEKで暗号化されたblobと最小限の同期メタデータのみを保持する。サーバー側でコンテンツの復号・解釈・マージは一切行わない。

### 理由

E2EEはこの製品の譲れない価値定義そのものである（Proton等の思想への共感）。

### 帰結

- サーバー側マージ不可のため、クライアント側マージが必須となる。
- 検索・ソートのため平文のローカルDB（SQLCipher保護）が必須となる。
- サーバー起点プッシュ通知は不可であり、リマインダーはローカル通知で実現する。

### 却下した代替案

該当なし（本ADRは前提となる原則の確認であり、対案の比較検討を伴わない）。

## ADR-002: 同期データプレーンはサーバー経由APIとする（Turso不採用）

**日付**: 2026-07-03
**状態**: 採用

### 決定

同期はサーバー（Lambda上のaxum）が提供する小さなpush/pull APIを経由する。旧設計の「クライアント⇄Tursoテナント DB直結+短命スコープJWT」は廃止し、Turso/libSQLは採用しない。

### 理由

1. E2EEでは同期APIは実質2エンドポイント（push/pull）で極小であり、直結モデルの「サーバー実装削減」の利点が薄い。むしろテナントDB発行・スコープJWT発行・トークン更新の管理の方がコード量が多い。
2. 直結モデルのトークンは当該DBへの任意SQL実行権となり、Org共有では悪意ある（または故障した）メンバー端末がDBを物理破壊できる。サーバー経由ならappend-onlyセマンティクス（物理DELETE不可・削除はtombstoneのみ）を強制できる。
3. デバイス失効・課金失効が即時になる（直結はトークンTTL 15〜60分の失効遅延を構造的に抱える）。
4. クライアントからlibsqlクライアント依存が消え、rusqlite+SQLCipherとのSQLite実装二重リンク問題（特にiOS/Androidでのシンボル衝突リスク）が消滅する。
5. libsql（C系フォーク）とturso（Rustリライト、旧Limbo）の二系統エコシステムリスクを回避できる。
6. TursoはORM/sqlx非対応でlibsql crateの生SQLのみだが、Postgresならsqlx（コンパイル時クエリ検証・成熟したマイグレーション）が使える。
7. オフラインファースト設計により同期はバックグラウンド非同期処理であり、同期レイテンシは体感品質に影響しないため、直結・エッジ配置の性能利点が活きない。

### 却下した代替案

- **Turso直結モデル**（旧設計）: 上記理由により不採用とした。
- **Tursoネイティブ同期の利用**: embedded replicaはオフライン書込不可（書込はプライマリへ転送）であり、オフラインファースト要件と衝突する。新offline sync（push/pull）はベータである。またE2EEではテナントDBの中身は暗号blobであり、複製してもクエリ不能で平文の作業用DBが別途必要になるため、複製は「トランスポートのキャッシュ」にしかならない。フィールドレベルマージはどのみちアプリ層実装が必要である。

### コスト認識

Lambdaがデータパスに入る（リクエスト毎の実行コスト、コールドスタート100〜300ms）が、暗号blobは数KB・同期は低頻度非同期のため実害はない。

## ADR-003: データストアはPostgres一本に統合する

**日付**: 2026-07-03
**状態**: 採用

### 決定

コントロールプレーン（users/devices/ラップ済み鍵/organizations/subscriptions/OPAQUE中間状態）と同期ストア（暗号blob）を単一のPostgresに統合する。第一候補はサーバーレスPostgres（Neon等、scale-to-zero・HTTPドライバでLambdaと相性が良い）とする。データアクセスはsqlxを全面採用する。

### 理由

- スタック一本化（アクセス層・マイグレーション・CI・ローカル開発が1系統）が実現できる。
- コントロールプレーンのデータ（users/devices/orgs/subscriptions）は典型的なリレーショナルデータでJOINが自然に発生する。
- Lambda+サーバーレスPostgresはアイドルコストゼロである。

### 却下した代替案

- **DynamoDB**: リレーショナル不一致でアクセスパターンを先決めする必要がある、最強のAWSロックイン、スタック2系統化、DynamoDB Localでのテスト環境負担、という理由で不採用。
- **管理用Turso DB**: ADR-002でTurso自体を不採用としたため消滅。採用していた場合もsqlx非対応で生SQLのみだった。
- **RDS/Aurora**: 最小構成の月額固定費と接続管理（RDS Proxy等）が初期フェーズに不適合。Aurora Serverless v2は再開レイテンシが認証APIに不適。
- **MySQL**: この用途でPostgresに勝る点がない。

### トレードオフ

テナント分離が物理DB分離から論理分離（tenant_idスコープ）になる。対策として、全クエリにtenant_idを必須とするリポジトリ層設計 + Postgres RLSの多層防御 + 認可の網羅テストを行う。E2EEにより万一の漏洩時も被害は暗号blobとメタデータに限定される。

## ADR-004: 同期エンジンは自前実装とする（フィールドレベルLWW + HLC + fractional indexing）

**日付**: 2026-07-03
**状態**: 採用

### 決定

既存CRDTライブラリは採用せず、フィールドレベルLast-Write-Wins + Hybrid Logical Clock + fractional indexing（並び順）+ tombstone（削除）を自前実装する。proptestによるproperty-based testingで「任意の操作列が全端末で同一状態に収束すること」を検証する。

### 理由

本製品のデータモデルはCRDTの「簡単な端」にある。必要なのはフィールド単位後勝ち・並び順・削除のみであり、CRDTで本当に難しい共同リアルタイムテキスト編集は要件にない。メモ欄はフィールド丸ごとLWWで実用上十分である。セマンティクスを最小に絞れば自前実装は枯れたパターンである。

### 却下した代替案

- **cr-sqlite**: メンテナンス停滞気味。SQLCipher入りbundledビルドへの拡張ロードの複雑さ。changesetとE2EE暗号境界の摩擦がある。
- **Automerge/Loro**: 活発で優秀だがドキュメント型CRDTであり、「SQLiteを正本としたフィールドLWW」にはオーバーキルである。履歴保持のストレージコストもある。

### 将来

リッチテキスト共同編集が要件化した場合、そのフィールドのみAutomerge/Loro等のドキュメントを暗号blobに載せる局所導入が可能である。

## ADR-005: 同期プロトコルは「最新状態方式 + クライアント再push」、カーソルはサーバー採番seq

**日付**: 2026-07-03
**状態**: 採用

### 決定

- サーバーはレコード（tenant_id, record_id）ごとに最新の暗号blobをupsert保持する（完全append-onlyのopログ方式は不採用）。
- 暗号blobの中にフィールドごとのHLCマップを内包する（`{fields, field_hlcs}` を丸ごと暗号化し、サーバーには不可視とする）。クライアントはpull時にフィールド単位でマージし、ローカルが勝ったフィールドがあればマージ済みblobを再pushする（レコードHLC=全フィールドHLCの最大値から採番する）。これにより全端末が収束する。
- pullカーソルはHLCではなく、サーバーが採番するテナント単位の単調増加seqを用いる。
- HLCはノードID（デバイス識別子由来）と端末単調カウンタを内包し全順序・一意とする。push冪等性はここから自然に得られる（リトライは同一HLC→no-op。別途の冪等キー機構は不要）。
- sync_recordsにdevice_idカラムは持たない（HLCのノード成分で足りる。メタデータ最小化のためである）。書き込み主体の帰属はhistoryテーブルのauthor_user_idにのみ持つ。
- サーバーが強制する不変条件: クライアントによる物理DELETE不可（削除はtombstoneフラグのみ）、blobサイズ上限（初期値64KB）、バッチ上限、HLC物理時刻がサーバー時刻+5分を超えるpushは拒否（時計異常・「未来HLCで永久勝ち」攻撃の抑止）、リクエスト毎の課金・メンバーシップ検証。
- エッジケース対策: 上書き時に旧行を`sync_records_history`へ退避し30日保持する（Orgメンバーの破壊行為からの復旧手段）。tombstoneは180日でGCする。GC窓を超えてオフラインだった端末はサーバーがdeviceの最終pull時刻から検知し410 Goneを返してフル再同期させる（サーバーに存在しないpush済みレコードはローカル削除、未pushレコードは保持）。

### 理由（カーソル設計）

HLCは「編集の論理順序」でありサーバー到着順ではないため、pullカーソルに使うと時計が遅れた端末の後出しpush（古いHLC）を他端末が永久に取り逃す。カーソルには「サーバーで可視になった順」が必要である。Postgresのbigserialや時刻ベースは並行トランザクションのコミット順逆転により可視性の欠番が生じるため不採用とした。`tenant_seq`行の`UPDATE ... RETURNING`による採番は行ロックがコミットまで保持されるためテナント単位でpushが直列化され、「seqの順＝可視になる順」が厳密に保証される。直列化の競合範囲はテナント内（1ユーザーの数端末〜小規模Org）のみで実害はない。状態はDB側にありLambdaはステートレスなため、サーバーレスと完全に両立する。

### 理由（最新状態方式）

E2EEではサーバーがopログをコンパクションできない（中身が読めない）ため、完全append-onlyログは無限成長する。最新状態+短期historyが実質的な均衡点である。

### 却下した代替案

- **完全append-onlyのopログ方式**: 上記「理由（最新状態方式）」のとおり、E2EE下ではコンパクション不能で無限成長するため不採用。
- **pullカーソルにHLCを使う方式**: 上記「理由（カーソル設計）」のとおり、時計遅延端末のpushを取り逃すため不採用。
- **pullカーソルにPostgresのbigserial/時刻ベースを使う方式**: 並行トランザクションのコミット順逆転により可視性の欠番が生じるため不採用。

### 参考スキーマ

```sql
CREATE TABLE sync_records (
    tenant_id      UUID    NOT NULL,
    record_id      UUID    NOT NULL,  -- UUIDv7
    collection     TEXT    NOT NULL,
    seq            BIGINT  NOT NULL,  -- pullカーソル用（サーバー到着順）
    hlc            TEXT    NOT NULL,  -- 競合解決用（固定幅ソート可能エンコード、ノードID内包）
    encrypted_blob BYTEA   NOT NULL,
    deleted        BOOLEAN NOT NULL DEFAULT false,
    updated_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (tenant_id, record_id),
    UNIQUE (tenant_id, seq)
);
CREATE TABLE tenant_seq (tenant_id UUID PRIMARY KEY, last_seq BIGINT NOT NULL);
CREATE TABLE sync_records_history (
    tenant_id UUID NOT NULL, record_id UUID NOT NULL, seq BIGINT NOT NULL,
    hlc TEXT NOT NULL, encrypted_blob BYTEA NOT NULL, deleted BOOLEAN NOT NULL,
    author_user_id UUID NOT NULL, overwritten_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### API

`POST /v1/tenants/{tenant_id}/push`（body: ops配列 {record_id, collection, hlc, deleted, blob}）、`GET /v1/tenants/{tenant_id}/pull?since={seq}&limit={n}` → {records, next_since, has_more}。

pullで自デバイスのエコーは除外しない（除外すると他端末による上書きを見逃す。マージが冪等なので実害はない）。初回フル同期はsince=0からのページング+zstd圧縮で行う。転送エンコードは初期JSON+base64とし、将来CBOR化に備えプロトコルバージョンをヘッダに持たせる。準リアルタイム反映は初期は「フォアグラウンド復帰時+数分間隔のpull」とし、SSE/long-pollは将来拡張とする。セッションは初期はDB照合のopaqueトークンとする（コントロールプレーンと同一Postgresのため1クエリで済む）。

## ADR-006: 単一リージョンで開始し、リージョン拡張可能性のみ温存する

**日付**: 2026-07-03
**状態**: 採用

### 決定

データストア・サーバーは単一リージョンで開始する。ただし以下の3点を初日から仕込む。

1. regionをテナントの第一級属性としてスキーマに持つ。
2. APIエンドポイントは最初からリージョンプレフィックス形式（例: `eu.api.…`）とする。
3. サインアップのプロビジョニングフローにリージョン選択を組み込む（初期は選択肢1つ）。

あわせて、CloudFrontを前段に置きTLS終端をユーザー近傍化する。

### 理由

マルチリージョン検討の動機はレイテンシだったが、オフラインファースト設計では通常操作はローカルDB完結・同期はバックグラウンド非同期であり、遠距離RTTが体感品質に現れない（ローカルDBこそが最強のエッジレプリカである）。ログイン（OPAQUE複数往復）は低頻度、初回フル同期は一度きりでバッチ設計が支配的である。マルチリージョン化してもユーザーが体感できる改善はほぼない。

### 却下した代替案

- **初期からのマルチリージョン展開**: 上記理由のとおり、体感できる改善がほぼない一方で運用・実装コストが増大するため、初期段階では不採用とした。

### 将来

データレジデンシー（EU訴求等のコンプライアンス/マーケティング）要件が生じた場合は、グローバルコントロールプレーン+リージョナルセル（Lambda+Postgres per region。サーバーレスのためセル増設の固定費はほぼゼロ）へ拡張する。上記の仕込みによりID設計・URL構造への単一リージョン前提の焼き付きを防ぐ。

## ADR-007: テナント単位のリソース分離（データ・鍵・ローカルDB・UI）

**日付**: 2026-07-03
**状態**: 採用

### 決定

**すべての同期対象エンティティは正確に1つのテナント（個人テナント or Orgテナント）に属し、テナントをまたぐ参照は存在しない。**

- tags / templates / schedules / timer_sessions はテナントスコープとする。個人テナントは自分のタグ等を持ち、各Orgは自前のタグ等を持つ。個人タグをOrgタスクに付けることはできず、Org側でタグを作成する。task_tags等の参照は常に同一テナント内で閉じる。
- 鍵構造として**テナントルートDEK**を導入する。テナントスコープのエンティティ（tags/templates/schedules/timer_sessions）はテナントルートDEKで暗号化する。個人テナントは `wrap(TenantDEK, MK)` でサーバー保管し、Orgテナント（Phase 3）はメンバー全員の公開鍵でsealする（Orgのタグは全メンバー可視というセマンティクスになる）。リスト配下のエンティティ（tasks/comments/reminders）は従来通りList DEKで暗号化する。
- タスクの同一テナント内でのリスト間移動は、クライアントで復号→移動先List DEKで再暗号化→pushにより実現する。
- テナントをまたぐタスクの移動は「移動」ではなく**明示的なコピー**操作とする（新レコードIDを発行する。鍵もHLC系譜も混ざらない）。
- ローカルDBはテナントごとにSQLiteファイルを分離する（SQLCipher鍵は従来通りDK由来で共通）。Org脱退・除名時は当該テナントのファイル削除で端末からのデータ消去が完結する。FTS5索引もテナントごとに閉じる。
- UIはワークスペース切替モデル（Slack/Notion方式）を採用する。個人と各Orgは別空間としてスイッチャーで切り替える。テナント横断ビュー（全ワークスペース統合の「今日」画面等）は初期スコープ外とする。ただしローカルDBは平文のため、将来の統合ビューは暗号・同期・サーバーに一切触れず純粋にクライアント側の読み取り機能として後付け可能であり、UI分離はソフトな制約である。

### 理由

1. テナント境界をまたぐ参照（共有リスト上のタスクへの個人タグ付け等）が生む鍵・認可・同期のエッジケースを構造的に排除できる。
2. 「リストに属さないエンティティ（tags/templates等）の対応DEKが未定義」という仕様の穴がテナントルートDEKで解決する。
3. サーバー側の認可検証が「リクエストのテナントIDとリソースの所属の一致確認」で完全に閉じる。
4. ローカルDBファイル分離により、サーバーの論理分離とローカルの構造が対称になり、除名・脱退時のデータ消去がファイル削除で完結する（除名後の端末残存データ問題への部分的な対処にもなる）。
5. UI分離はソフトな制約であり、将来の横断ビューはクライアント側機能として後付け可能なため、UX面の選択肢を失わない。

### 却下した代替案

- **テナント横断参照の許容**（個人タグをOrgタスクへ付与する等）: 鍵の帰属・除名時の挙動・認可が複雑化するため不採用。
- **ローカルDBの単一ファイル + tenant_idカラムによる分離**: テナント境界バグのリスクをローカルにも持ち込む。UI分離により横断クエリの需要がなく、ファイル分離のデメリットが実質ないため不採用。
