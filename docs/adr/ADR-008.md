# ADR-008: サーバーレスPostgresプロバイダはNeon、初期リージョンはeu-central-1とする

**日付**: 2026-07-04
**状態**: 採用

### 決定

- プロバイダは **Neon** とする。
- 初期リージョンは **eu-central-1（フランクフルト）** とし、Lambdaも同一リージョンに同居させる。
- 接続方式は **Neon組み込みの接続プーラー経由のTCP接続 + sqlx** とする（PostgresワイヤプロトコルによるTCP接続。HTTPドライバはJavaScript向けでありRust/sqlxには適用できない）。
- プーラーはトランザクションモードで動作するため、セッション依存機能（advisory lock、LISTEN/NOTIFY等）は使用しない設計とする（現行設計は行ロック+通常トランザクションのみであり抵触しない）。
- 実装開始時の要確認事項: Neonの最新料金プラン、プーラーのprepared statement対応（sqlxはprepared statementを使用する）、リージョン提供状況。

### 理由

1. コスト構造の一致: 真のscale-to-zeroによりユーザーゼロ期のアイドルコストが実質ゼロで、Lambda側と合わせて全層で「使われなければ払わない」が成立する。
2. ブランチ機能によるCI/テストDX: CIで統合テストごとに実Postgresのブランチを短時間で作成・破棄でき、同期プロトコルのproperty-based testingや認可の網羅テストを実DBで回せる。
3. 出口の安さ: 素のPostgresであり、pg_dump/論理レプリケーションで他プロバイダへ移行可能（ADR-003の出口戦略と整合）。
4. リージョン選定: E2EE/プライバシー訴求のコア顧客層は欧米であり、「暗号化データはEU域内に保存」はこの層への訴求として機能する（Protonの「スイス保管」と同じ構図）。EUユーザー最速、米国東海岸〜90ms・西海岸〜150ms・日本〜230msだが、オフラインファースト設計（同期はバックグラウンド非同期）のため全て許容圏内。NeonはAWS東京リージョン非提供だが、本設計では実害がない（重要なのはLambda⇄DBの同一リージョン同居であり、ユーザー⇄Lambdaの距離は非クリティカル）。
5. 誠実性の注記: AWSは米国企業でありCLOUD Actの射程は理論上EUリージョンにも及ぶ。EU配置は法的な完全防御ではなく姿勢とシグナリングであり、本当の防御はE2EEそのもの（サーバーが平文を持たない）である。マーケティング文言の作成時もこの区別を維持する。

### 補足

Neonは2025年にDatabricksに買収されている。サービス継続性への直接の懸念はない（資金面はむしろ盤石化）が、ロードマップがAI/Databricks統合へ傾く可能性はリスクとして認識する。素のPostgresゆえの出口の安さ（理由3）をヘッジとする。

### 却下した代替案

- **Supabase**: 素のPostgres+プーラー内蔵で堅実だが、無料枠は非アクティブ時にプロジェクトが一時停止し本番利用不可。有料プランは常時起動インスタンス課金でscale-to-zeroではない。Auth/Realtime/Storage等の付加機能は自前実装（OPAQUE認証等）と重複し不要。東京リージョンが絶対要件の場合の次点だったが、東京の価値が月額固定費とブランチCI喪失に見合わないと判断。
- **Aurora Serverless v2**: scale-to-zero対応だが復帰に十数秒を要し、認証・同期APIに不適（ADR-003でも却下済み）。
- **Aurora DSQL**: Postgres「互換」だが楽観的並行制御であり、tenant_seq行ロックによる直列化パターン（ADR-005）の前提が崩れる。拡張・機能制限も多い。
- **Nile / Prisma Postgres / PlanetScale for Postgres**: それぞれ特色はある（Nileはテナント分離ネイティブ等）が、若い・実績薄・またはハイエンド志向であり、認証基盤を賭けるには時期尚早。継続ウォッチとする。
- **米国東部リージョン（us-east-1等）での開始**: 英語圏市場の重心としては合理的だが、プライバシー志向層への訴求力とCLOUD Act忌避の議論回避の観点でEU配置が優る。将来のリージョナルセル拡張（ADR-006）で最初に増やす候補はUSセルとなる。
