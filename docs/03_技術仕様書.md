## 1. 技術スタック選定

### 1.1 全体構成

本アプリケーションは、クライアントUIとコアロジックを明確に分離した「Flutter + Rust」構成を採用する。

| レイヤー | 技術 | 役割 |
|---|---|---|
| クライアントUI | Flutter | iOS / Android / Windows / macOS / Linux 向けUI描画、ユーザー操作 |
| コアロジック | Rust | 暗号処理、同期エンジン、ドメインロジック、ローカルDBアクセス |
| ブリッジ | flutter_rust_bridge | Flutter(Dart) ↔ Rust のFFI境界を自動生成 |
| サーバー | Rust (axum) on AWS Lambda | OPAQUE認証、テナントプロビジョニング、同期API（push/pull）提供、課金Webhook（サーバーレス・常駐プロセスなし） |
| CLI | Rust (clap) | コマンドラインからのタスク操作 |
| MCPサーバー | Rust (rmcp 公式SDK) | LLMエージェント向けTodo操作インターフェース |
| ローカルAI | llama.cpp系Rustバインディング / candle | オンデバイス推論（自然言語パース、要約等） |
| 課金 | App Store / Play IAP、外部課金プロバイダ（Desktop / Org） | サブスクリプション課金 |

### 1.2 選定理由

- **Flutter**: 単一コードベースでモバイル・デスクトップの全対象OSをカバーでき、UI実装の生産性が高い。マテリアル/Cupertino双方の表現力があり、Todoアプリのような情報密度が高いUIとの親和性も良い。
- **Rust**: 暗号ライブラリ（鍵導出・AEAD・OPAQUE等）のエコシステムが充実しており、メモリ安全性の観点からもE2EEの実装言語として妥当である。加えて、コアロジックをRust crateとして切り出すことで、CLI・MCPサーバー・（将来的な）サーバーサイド機能から同一の暗号/ドメインロジックを再利用でき、実装の重複と検証対象の増加を避けられる。
- **flutter_rust_bridge**: DartとRust間の型安全なバインディングをコード生成できるため、手書きFFIに比べて保守コストが低い。

### 1.3 Flutter + Rust構成の評価

**妥当性の結論**: UIの生産性・クロスプラットフォーム性（Flutter）と、暗号実装の安全性・コアロジックのCLI/MCPへの再利用性（Rust）を両立できる構成であり、本アプリの要件（マルチプラットフォームE2EEアプリ、かつCLI/MCPを同一データストアに対して提供）に対して妥当と判断する。

**懸念点と対策**:

| 懸念点 | 内容 | 対策 |
|---|---|---|
| ブリッジのビルド複雑性 | flutter_rust_bridgeはRustのクロスコンパイル設定（iOS/Android/各デスクトップOS向けターゲット）をFlutterのビルドパイプラインに統合する必要があり、CI構築コストが高い | プラットフォームごとのビルドスクリプトをmonorepo内に集約し、CIでプラットフォーム別ジョブをキャッシュ付きで並列実行する。`cargokit` 等の既存の統合ツールチェーンの採用を検討する |
| Flutter Desktop（特にLinux）の成熟度 | プラグインエコシステムやネイティブ連携（キーチェーン、通知）がモバイルに比べて発展途上 | Desktop向けの重要機能（キーチェーンアクセス等）はRust側で自前実装し、Flutterプラグインへの依存を最小化する。Linux版は正式リリース時点でモバイル・Windows/macOSに対し機能段階（フィーチャーパリティ）が遅れる可能性を許容する |
| FFI境界でのエラーハンドリング | Rust側のpanicやエラーがDart側に正しく伝播しない場合、UIがクラッシュ/フリーズするリスク | Rustコア関数はすべて `Result` 型で返却し、`catch_unwind` でpanicを捕捉してDart側に構造化エラーとして通知する |

### 1.4 国際化（i18n）方針

顧客ターゲットはグローバルであり（[`01_企画書.md`](./01_企画書.md) §3）、初期リリースの時点から多言語対応を行う（[`02_機能仕様書.md`](./02_機能仕様書.md) F-48）。技術面の方針は以下のとおりとする。

- 基準言語は英語とする。全UI文字列はFlutter標準のl10nツールチェーン（ARBファイル + `intl`、ICU MessageFormat）で外部化し、UI層でのハードコードを禁止する。
- Rustコア側はエラー等を自然言語文字列ではなくエラーコードとして返却し、表示文言への変換はFlutter側のl10n層で行う。コア層に自然言語文字列を持たせない。
- CLI / MCPサーバーのメッセージは英語を基準とする（MCPのtool説明等も英語で記述する）。
- 日付・時刻・数値の書式化はロケールに従う。タイムゾーンは内部的にUTCで保持し、表示時にロケール・端末タイムゾーンへ変換する（§3のデータモデルと整合させる）。
- RTLレイアウトは、Flutter標準のRTLサポートを前提として初期実装の時点から考慮する。

### 1.5 サーバー実行基盤（決定: AWS Lambda）

サーバー（コントロールプレーン）は **AWS Lambda 上で Rust ネイティブバイナリとして稼働**させる。デプロイ形態は**コンテナイメージ + AWS Lambda Web Adapter（LWA）**とする。axum アプリケーションは通常のHTTPサーバーとして実装し（**Lambda 固有コードを一切含まない**）、静的バイナリと LWA を同梱した最小コンテナイメージ（distroless 等）として Lambda のコンテナイメージサポートでデプロイする。LWA が Lambda イベントとHTTPリクエストの変換を担う。常駐サーバーは持たない。

本節および6章の同期プロトコルに関する設計判断の経緯は [`05_設計判断記録.md`](./05_設計判断記録.md)（ADR-002・ADR-003）を参照。

**選定理由**（比較対象: Cloudflare Workers / Cloudflare Containers。2026年7月時点の調査に基づく）

1. **Rustネイティブ実行**: opaque-ke 等の暗号クレートを WASM 変換なしでそのまま利用できる。認証コードをWASM化の不確実性に載せない（Workersはworkers-rsによるWASM変換が必要でPoCを要する）。
2. **バイナリサイズ制約がない**: Workersはスクリプトサイズ上限（圧縮後、Paidで10MB）があり暗号ライブラリ入りWASMは圧迫しやすい。Lambdaはzipで250MB（展開後）、コンテナイメージなら10GBまで。
3. **データストアとの近接性**: サーバーレスPostgres（Neon等）はリージョン単位で稼働しており、Lambdaを同一リージョンに配置することでサーバー⇄Postgres間がリージョン内通信になり、同期API・認証APIのレイテンシと転送コストを抑えられる。
4. **可搬性**: アプリは Lambda 固有コードを含まない通常のHTTPサーバーであるため、**同一コンテナイメージ**を ECS / Fly.io 等へそのまま移設可能で、出口戦略が確保される。
5. **コールドスタート**: Rustの最小コンテナイメージ構成でのコールドスタートは概ね100〜300ms程度で、ログイン・トークン更新という低頻度APIのp99への影響は許容範囲。将来コールドスタート短縮が必要になった場合は、`lambda_http` + cargo-lambda による zip デプロイ方式（数十ms、ただし main 関数に Lambda 固有のエントリポイントが必要）への切替を検討する。

**不採用理由の要約**: Cloudflare Workers は WASM 制約・サイズ上限が課題。Cloudflare Containers（2026年4月GA）は任意のDockerイメージが動くもののコールドスタート1〜3秒・オートスケール未実装で、まばらな認証トラフィックとの相性が悪い（公式推奨もリソース集約型ワークロード向け）。

**周辺構成**: OPAQUEログインの多段プロトコルの中間状態は、DynamoDBではなく**同一Postgres内のephemeralテーブル**（`expires_at` カラムを持ち、状態消費（consume）時に該当行を即時削除するのに加え、定期クリーンアップジョブで期限切れ行を掃除する。OPAQUE中間状態は一度使用したら破棄する性質のものであり、DynamoDB等のネイティブTTL機構は不要である）に短期保存する。永続メタデータ（users/devices/ラップ済み鍵/organizations/subscriptions等）も同一Postgresに保存する（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-003を参照）。定期処理（tombstone GC・sync_records_historyの期限切れ削除・期限切れOPAQUE中間状態の掃除・失効後180日削除）は EventBridge Scheduler で実行する。tombstone GCはPhase 2同期導入時に再設計する（ADR-009はローカル削除の意味論を定めるもの）。（2026-07-07改訂: docs/05 ADR-009参照）。公開エンドポイントの前段は CloudFront + WAF または API Gateway のスロットリング（最終構成は§12未決事項）とする。ビルド・デプロイは Docker イメージビルド（linux/arm64 または x86_64）→ ECR push → Lambda 更新のパイプラインを CI に構築する。サーバーレスPostgresは**Neon**を採用し、初期リージョンは**eu-central-1（フランクフルト）**、Lambdaも同一リージョンに配置する（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-008参照）。接続はNeon組み込みの接続プーラー経由のTCP接続（sqlx）とし、Lambdaの同時実行数増加に伴うコネクション枯渇を回避する。プーラーはトランザクションモードで動作するため、セッション依存機能（advisory lock、LISTEN/NOTIFY等）は使用しない設計とする（本仕様は行ロックと通常トランザクションのみを用いており抵触しない）。

## 2. リポジトリ/モジュール構成

monorepo構成とし、以下のディレクトリ構成を基本とする。

```
todori/
├── app/                  Flutter アプリ本体
│   ├── lib/
│   ├── ios/ android/ windows/ macos/ linux/
│   └── rust_builder/     flutter_rust_bridge 生成物・ビルド設定
│
├── core/                 Rust コアクレート群
│   ├── domain/           エンティティ・ユースケース（純粋ロジック）
│   ├── crypto/           鍵導出・暗号化・OPAQUE クライアント実装
│   ├── sync/             HLC・差分検出・push/pull同期エンジン
│   └── storage/          SQLite(SQLCipher) アクセス層、マイグレーション、FTS5
│
├── cli/                  Rust CLI (clap)。core を利用
│
├── mcp-server/           Rust MCP サーバー (rmcp)。core を利用
│
├── server/               Rust API サーバー (axum)。AWS Lambda 上で稼働
│   ├── auth/             OPAQUE サーバー実装
│   ├── sync/             push/pull同期API、テナント認可、seq採番
│   └── billing/          IAP / 外部課金プロバイダのWebhook処理
│
└── docs/                 設計ドキュメント
```

- `core/` はFlutterアプリ・CLI・MCPサーバーの共通依存であり、暗号・同期・ドメインロジックの単一の実装源泉（Single Source of Truth）とする。
- `server/` は**コントロールプレーン**（認証・ラップ済み鍵の保管・テナント管理・課金Webhook）と**同期データプレーン**（push/pull同期API）を提供する。ただしE2EEデータの中身は一切解釈せず、暗号blobの中継・保管に徹する（§6.1）。ドメインロジック（domain / sync / storage）には依存しないが、OPAQUE等の暗号プロトコル実装を共有するため `core/crypto` には依存しうる（クライアントとサーバーで同一実装系を用い、プロトコル互換を保証するため）。

## 3. データモデル

ID はすべて **UUIDv7** を採用する（生成時刻の情報を含み、ソート性とグローバル一意性を両立するため）。時刻はすべて **UTC epoch milliseconds（int64）** で保持する。以下は論理スキーマであり、実データはレコード単位で暗号化されてサーバーへ同期される（詳細は「4. 暗号設計」「6. 同期プロトコル」を参照）。

すべての同期対象エンティティは正確に1つのテナント（個人テナント or Orgテナント）に属し、テナントをまたぐ参照は存在しない。tags / templates / schedules / timer_sessions はテナントスコープである（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-007参照）。

## 4. 暗号設計

本章はE2EE設計の中核であり、最も厳密な実装・レビューを要する領域である。

### 4.1 設計方針

- サーバーはいかなる場合もユーザーのパスワード、および各種コンテンツの平文を保持・観測できない。
- 認証（本人確認）と暗号鍵の保護（Master Keyの保護）は独立した仕組みとし、パスワード変更が発生してもデータ本体の再暗号化を必要としない設計とする。
- 共有（Organization）機能は公開鍵暗号（X25519 sealed box）によって実現し、共有単位（List）ごとに独立した対称鍵（DEK）を用いる。

### 4.2 鍵階層

```
                       ┌────────────────────┐
                       │   User Password    │  (サーバーへは非送信)
                       └─────────┬──────────┘
                                 │ OPAQUE (opaque-ke)
                                 ▼
                       ┌────────────────────┐
                       │     exportKey      │  (OPAQUE成功時にクライアントのみ導出)
                       └─────────┬──────────┘
                                 │ HKDF-SHA256
                                 ▼
                       ┌────────────────────┐
                       │      KEK_pw        │  (パスワード由来 鍵暗号鍵)
                       └─────────┬──────────┘
                                 │ wrap
                                 ▼
   ┌─────────────┐     ┌────────────────────┐     ┌─────────────────┐
   │  Device Key │──wrap→│    Master Key      │←wrap──│  Recovery Key   │
   │    (DK)     │     │       (MK)          │     │ (BIP39風 人間可読) │
   └─────────────┘     └─────────┬──────────┘     └─────────────────┘
    OSキーチェーン保存           │ HKDF / wrap                サーバー保存(wrap(MK,RecoveryKey))
    未登録時はDKで               ▼                            ユーザーのみ保持(平文)
    ローカルDB保護    ┌────────────────────────┐
                      │  User X25519 KeyPair    │ 秘密鍵は wrap(SK, MK) でサーバー保管
                      └───────────┬────────────┘
                                  │ sealed box (org共有時)
                                  ▼
              ┌───────────────────────────────────┐
              │            List DEK (32byte)        │
              │  個人リスト: wrap(DEK, MK)           │
              │  共有リスト: 各メンバー公開鍵でsealed  │
              └───────────────┬─────────────────────┘
                              │ XChaCha20-Poly1305
                              ▼
                 タスク/リスト/タグ等のコンテンツblob
```

List DEKと並列に、テナントごとの **Tenant Root DEK** が存在する。個人テナントは `wrap(TenantDEK, MK)` としてサーバー保管し、Orgテナントはメンバー全員の公開鍵でsealされる。tags / templates / schedules / timer_sessions 等のテナントスコープのエンティティ（3章参照）はTenant Root DEKで暗号化され、List配下のエンティティ（tasks/comments/reminders）とは異なる鍵系譜に属する（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-007参照）。

### 4.3 各鍵の定義

| 鍵 | サイズ/形式 | 生成タイミング | 保存場所 |
|---|---|---|---|
| Device Key (DK) | 32byte 乱数 | 初回起動時 | iOS: Keychain(Secure Enclave裏付け) / Android: Keystore / Desktop: OSキーチェーン |
| Master Key (MK) | 32byte 乱数 | アカウント登録時（既存ローカルデータがあればMK配下に移行） | 平文はメモリ上のみ。`wrap(MK, KEK_pw)`をサーバー、`wrap(MK, DK)`をローカル、`wrap(MK, RecoveryKey)`をサーバーに保存 |
| KEK_pw | 32byte（HKDF出力） | ログイン成功（OPAQUE成功）ごとに導出 | 保存しない（都度導出） |
| Recovery Key | 24単語程度のBIP39風ニーモニック | アカウント登録時に1回生成 | ユーザーのみが保持（サーバー・端末には保存しない） |
| User X25519 KeyPair | 秘密鍵32byte / 公開鍵32byte | アカウント登録時 | 公開鍵はサーバーに平文保存。秘密鍵は`wrap(SK, MK)`でサーバー保存（Org共有のため他デバイスからも取得できる必要がある） |
| List DEK | 32byte 乱数 | リスト作成時 | 個人: `wrap(DEK, MK)`。共有: メンバーごとに`seal(DEK, member_pubkey)` |
| Tenant Root DEK | 32byte 乱数 | テナント作成時（個人テナントは初回アカウント登録時、Orgは組織作成時） | 個人: `wrap(TenantDEK, MK)`。Org: メンバーごとに`seal(TenantDEK, member_pubkey)` |

### 4.4 未登録利用時の扱い

アカウント未登録の状態では、ローカルDBはDevice Keyのみで保護される。アカウント登録時、既存のローカルデータ（個人リスト・タスク等）はMaster Key配下に移行（再暗号化）され、以後DKはMKのラップ用途に限定される。

### 4.5 パスワード変更

パスワード変更時に必要な処理は以下のみである。

1. OPAQUEレジストレーションの再実行（新パスワードでの再登録）。
2. 新しいexportKeyからKEK_pw'を導出し、`wrap(MK, KEK_pw')`を再計算してサーバー側の値を更新。

Master Key自体は不変であるため、**タスク等のデータ本体の再暗号化は一切不要**である。これが、exportKeyを直接Master Keyとして採用せず、間接的にKEK_pwを介してMKをラップする理由である（exportKeyを直接MKとして使うと、パスワード変更のたびに全データの再暗号化が必要になる）。

### 4.6 Organization共有

- 各ユーザーはアカウントごとにX25519鍵ペアを1組保持する（複数デバイス間ではMKで秘密鍵を共有）。
- 共有リストのDEKは、メンバー各自の公開鍵に対してsealed box（NaCl方式、送信者匿名の公開鍵暗号）でラップし、`org_members.sealed_dek_map` 相当の形でサーバーに保存する。
- **メンバー除名時**は当該リストのDEKをローテーション（新DEK生成）し、既存コンテンツを新DEKで再暗号化した上で、残存メンバーの公開鍵で再ラップする。除名されたメンバーは新DEKを取得できないため、以後の更新内容を復号できない（ただし除名以前にローカルへ同期済みのデータは端末側に残存し得る点に留意する）。

### 4.7 暗号アルゴリズム・ライブラリ

| 用途 | 方式 | ライブラリ |
|---|---|---|
| 認証 | OPAQUE (PAKE) | `opaque-ke` |
| パスワードストレッチング（OPAQUE内KSF） | Argon2 | `argon2`（opaque-ke の argon2 feature 経由） |
| 鍵導出 | HKDF-SHA256 | RustCrypto `hkdf` |
| 対称暗号（AEAD） | XChaCha20-Poly1305 | RustCrypto `chacha20poly1305` |
| 鍵交換/sealed box | X25519 | RustCrypto系 `x25519-dalek` |

### 4.8 レコード暗号化方式

各エンティティ（task, list, tag, comment等）は、暗号化対象フィールドをまとめたJSONオブジェクトとしてシリアライズし、対応するDEKでXChaCha20-Poly1305暗号化した上でblobとして同期する。

- **AAD（Associated Data）**にはレコードID・テーブル種別（collection名）を含め、暗号文の入れ替え（別レコードの暗号文を別IDのレコードとして差し替える攻撃）を検知・防止する。
- ステータス・優先度・並び順などUI表示上不可欠だが機微性が低いフィールドも、原則としてすべて暗号化対象に含める（サーバー側解析面の最小化を優先する）。同期メタデータとして必要な最小限の情報（後述）のみ平文でサーバーに保持される。

**エンティティとDEKの対応**（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-007参照）

- tasks / comments / reminders → 所属リストの List DEK。
- lists自体のメタデータ（name / color / icon） → 当該 List DEK。
- tags / templates / schedules / timer_sessions → 所属テナントの Tenant Root DEK。

タスクの同一テナント内でのリスト間移動は、クライアントで復号した上で移動先List DEKによる再暗号化を要する。テナントをまたぐタスクの移動は「移動」ではなく**明示的なコピー**操作とし、新規レコードIDを発行する（鍵もHLC系譜も混ざらない）。

### 4.9 サーバーが知り得る情報 / 知り得ない情報

**サーバーが知り得る情報**

| 項目 | 内容 |
|---|---|
| アカウントID | ユーザー識別子、メールアドレス |
| OPAQUEレジストレーションレコード | パスワード検証用データ（パスワード自体は含まない） |
| ラップ済み鍵 | `wrap(MK, KEK_pw)`、`wrap(MK, RecoveryKey)`、`wrap(SK, MK)` |
| デバイスリスト | デバイス名、公開鍵、登録/失効日時 |
| 課金状態 | プラン種別、サブスクリプション有効期限 |
| Organizationメンバーシップ | 所属関係、ロール、sealed DEK |
| 暗号blob + 同期メタデータ | レコードID、collection種別、HLC（ノードID成分を含む）、seq、blobサイズ、サーバー書込み時刻 |

**サーバーが知り得ない情報**

- ユーザーのパスワード（平文はサーバーに一切送信されない）
- タスクのタイトル・メモ・タグ名・コメント本文などのコンテンツ平文
- リマインダー日時、期限日時などの時刻情報（暗号化対象のためサーバーからは不可視）

### 4.10 帰結: プッシュ通知の制約

リマインダー日時はコンテンツとして暗号化されているため、サーバーはリマインド対象・タイミングを把握できず、**サーバー起点のプッシュ通知でリマインドを送信することはできない**。したがって、リマインド通知はすべて**クライアント側のローカルスケジューリング**（OS標準のローカル通知API）によって実現する。この制約により、アプリが完全に終了・OSにより強制停止されている状況での通知信頼性は、プッシュ通知方式に比べて劣る可能性がある点を設計上の既知の制約として明記する。

## 5. ローカルストレージ

### 5.1 採用技術

- ローカルDBは素のSQLite + **SQLCipher** を採用する（Rustからは `rusqlite` の bundled-sqlcipher 系featureを想定）。
- libSQL/Tursoは採用しない（[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-002）。同期はサーバー経由のpush/pull API（6章）であり、クライアントはlibsqlクライアントを持たず、`rusqlite` + SQLCipherと通常のHTTPSクライアントのみを持つ。これにより`rusqlite`+SQLCipherとlibsqlクライアントのSQLite実装二重リンク問題（特にiOS/Androidでのシンボル衝突リスク）も発生しない。
- サーバー（Postgres）に保存されるのはDEKで暗号化済みのblobのみであるため、E2EEの保証はストレージ層の保存時暗号化には依存しない（Postgres側の保存時暗号化を多層防御として有効化してよいが、必須ではない）。
- ローカルDBは**テナントごとにSQLiteファイルを分離**する（個人テナント用ファイル1つ + 所属Orgテナントごとに1ファイル）。SQLCipherの鍵はいずれも5.3節のとおりDevice Key (DK) 由来で共通であり、ファイル分離自体が鍵構造に影響することはない。Org脱退・除名時は当該テナントのファイルを削除するだけで端末上のデータ消去が完結する。FTS5索引もテナントごとのファイル内に閉じる（5.4節参照）。詳細な採用理由は[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-007を参照。

### 5.2 暗号化の2レイヤー構造

「二重暗号化」ではなく、適用場所・脅威モデルの異なる2レイヤーであることを明確にする。

| レイヤー | 方式 | 適用場所 | 脅威モデル |
|---|---|---|---|
| コンテンツ暗号化（E2EE本体） | DEKによるXChaCha20-Poly1305 | 同期境界（push時にblob生成、pull時に復号） | サーバー・ネットワーク上の第三者 |
| 保存時暗号化 | SQLCipher（ページレベル） | ローカルDBファイル全体 | 端末紛失・盗難、他プロセスからのファイル読み取り |

- ローカルDB内のカラムは**平文**で保持する。SQLクエリ・インデックス・ソート・FTS5を機能させるためであり、DEK暗号化されたblobをローカルの正本として保持する設計は採らない。
- DEKによる暗号化は同期境界でのみ適用される。outboxに一時保持される送信待ちblobがSQLCipher内に置かれることはあるが、これは設計上意図した多層であり冗長な二重暗号化ではない。

### 5.3 ローカルDB鍵

- SQLCipherの鍵は**常にDevice Key (DK) からHKDFで導出**する（登録状態によらず一貫）。
- MK由来の鍵に切り替えない理由: `wrap(MK, DK)` が同一端末に保存される以上、MK由来にしてもDKが破られれば等しく破られるため、防御力が向上しない。鍵を一貫させることでアカウント登録時のDBリキー（PRAGMA rekey）も不要になる。
- 将来オプション（strict mode）: `wrap(MK, DK)` をローカルに保存せず、起動ごとにパスワード入力（OPAQUE）でMKを取得する運用。アプリ起動にパスワードが必須になる代わりに、端末単独ではデータを復号できなくなる。初期リリースのスコープ外とし、12章の未決事項に含める。

### 5.4 全文検索

- FTS5はSQLCipher上でそのまま動作する（SQLCipherはページレベル暗号化のためSQLite機能に透過的である）。
- FTS5の転置インデックスは平文を含むが、暗号化DBファイル内に格納されるため、脅威モデルはレイヤー2（DK保護）に帰着する。
- `tasks_fts` は `tasks` tableの `INSERT` / `UPDATE` / `DELETE` トリガーで同期する。`title` / `note` 更新、移行互換の `deleted_at` 付き行、物理削除はFTS5索引へ即時反映する（2026-07-08追記: task-62）。
- tokenizerは追加依存なしのFTS5標準 `unicode61` を用いる。検索APIは空白区切り語を前方一致queryへ変換するため、英語単語や日本語連続文字列の先頭一致は扱えるが、任意部分一致や日本語の形態素単位分割は提供しない。検索UI設計時はこの制約を前提に、必要ならtokenizer/インデックス設計を再評価する（2026-07-08追記: task-62）。

## 6. 同期プロトコル

本章の設計判断の経緯は [`05_設計判断記録.md`](./05_設計判断記録.md)（ADR-002〜ADR-005）を参照。

### 6.1 構成

同期はクライアント⇄サーバー直結ではなく、サーバー（Lambda上のaxum）が提供する**push/pull API**を経由する（ADR-002）。テナント（ユーザーごとの個人テナント、Organizationごとの Orgテナント）は物理DB分離ではなく、単一Postgres内で `tenant_id` により論理分離する（ADR-003）。

論理分離の担保は以下の多層防御による。

- 全クエリに `tenant_id` を必須とするリポジトリ層設計（クエリ条件漏れを型・レビューで検出する）。
- Postgres RLS（Row Level Security）による多層防御。
- 認可の網羅テスト。

認可・課金の有効性検証はAPIリクエストごとにサーバーが実施する。これにより、デバイス失効・課金失効は（旧設計のトークンTTLに伴う遅延なく）即時に同期アクセスへ反映される。

### 6.2 サーバースキーマ

サーバー側Postgresはコンテンツを解釈しない汎用的な同期テーブルのみを保持する。

```sql
CREATE TABLE sync_records (
    tenant_id      UUID    NOT NULL,
    record_id      UUID    NOT NULL,  -- UUIDv7
    collection     TEXT    NOT NULL,
    seq            BIGINT  NOT NULL,  -- pullカーソル用（サーバー到着順）
    hlc            TEXT    NOT NULL,  -- 競合解決用（固定幅ソート可能エンコード、ノードID内包）
    encrypted_blob BYTEA   NOT NULL,
    deleted        BOOLEAN NOT NULL DEFAULT false,
    updated_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (tenant_id, record_id),
    UNIQUE (tenant_id, seq)
);
CREATE TABLE tenant_seq (tenant_id UUID PRIMARY KEY, last_seq BIGINT NOT NULL);
CREATE TABLE sync_records_history (
    tenant_id UUID NOT NULL, record_id UUID NOT NULL, seq BIGINT NOT NULL,
    hlc TEXT NOT NULL, encrypted_blob BYTEA NOT NULL, deleted BOOLEAN NOT NULL,
    author_user_id UUID NOT NULL, overwritten_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- `sync_records` はレコード（`tenant_id`, `record_id`）ごとに**最新の暗号blobのみ**を保持する（完全append-onlyのopログ方式は不採用。理由はADR-005参照）。
- `seq` はpullカーソル専用であり、「サーバーへの到着順（＝可視になった順）」を表す。`tenant_seq` 行の `UPDATE ... RETURNING` で採番する。この採番は行ロックがコミットまで保持されるため、テナント単位でpushが直列化され「seqの順＝可視になる順」が厳密に保証される（直列化の競合範囲はテナント内、すなわち1ユーザーの数端末〜小規模Orgに限られ実害はない）。
- `hlc` は競合解決（フィールドレベルLWW）専用であり、サーバー到着順とは無関係な「編集の論理順序」である。
- `sync_records` は `device_id` カラムを持たない。HLCのノードID成分で更新元デバイスの識別は足りており、あえて別カラムを設けないことでメタデータを最小化する（ADR-005）。書き込み主体（ユーザー）の帰属は `sync_records_history.author_user_id` にのみ持たせる。
- `sync_records_history` は上書き時の旧行を退避するテーブルであり、Orgメンバーの破壊的上書きからの復旧手段として30日保持する（詳細は6.6節）。

### 6.3 変更追跡

- 各デバイスは **Hybrid Logical Clock (HLC)** を保持し、書込のたびにインクリメントする。HLCは「物理時刻（ミリ秒）+ 論理カウンタ + ノードID（デバイス識別子由来）」の3要素からなり、固定幅のソート可能な文字列としてエンコードする。この構成により全順序性と一意性が保証され、push冪等性もここから自然に得られる（同一HLCでのリトライはno-opとなり、別途の冪等キー機構を要しない）。
- マージはフィールドレベルの **Last-Write-Wins (LWW)** を基本とする。HLCの比較により、後勝ちのフィールドのみを採用する。
- 暗号blobの中にはコンテンツ本体だけでなく、フィールドごとのHLCを保持するマップ（`{fields, field_hlcs}`）を内包し、これを丸ごと暗号化する。フィールド単位の競合解決に必要な情報はすべて暗号境界の内側にあり、サーバーからは不可視である。
- `sort_order` のように頻繁かつ同時多発的に競合しうるフィールドは、LWWではなく **fractional indexing**（2レコード間に常に新しい順序値を挿入できる文字列/分数キー）を用いることで、同時並べ替えの競合を実質的に解消する。
- 削除の同期表現は同期導入時に再設計する（ADR-009はローカル削除の意味論を定めるもの）。従来案では `deleted_at`（クライアント側）/ `deleted`（サーバー側）による **tombstone** とし、同期後の一定期間経過後にサーバー側でガベージコレクションする設計だったが、ローカル削除はADR-009により物理削除へ改訂済みである。（2026-07-07改訂: docs/05 ADR-009参照）
- `seq` と `hlc` は役割が明確に分離している。`seq` は「クライアントがどこまで受信済みか」を表すpullカーソルであり、`hlc` は「フィールドのどちらの変更が勝つか」を決める競合解決キーである。HLCをカーソルとして流用すると、時計が遅れた端末による後出しのpush（古いHLC値を持つ）を他端末が永久に取り逃す問題が生じるため、両者を分離している（詳細はADR-005）。

### 6.4 同期フロー

1. ローカルでの書込は即座にローカルDBへ反映すると同時に、`outbox` テーブルへ変更操作（op）を記録する。
2. **push**: `POST /v1/tenants/{tenant_id}/push`（body: ops配列 `{record_id, collection, hlc, deleted, blob}`。`deleted` の意味論は同期導入時に再設計する）。（2026-07-07改訂: docs/05 ADR-009参照）サーバーはレコードごとに、`incoming.hlc > stored.hlc` であればupsertして新しい `seq` を採番する（`tenant_seq` 行の `UPDATE ... RETURNING` による採番。行ロックがコミットまで保持されるため、テナント内で可視順＝seq順が保証される）。そうでなければ当該レコードはsuperseded（採用されなかった）応答とする。push冪等性はHLCの一意性から自然に成立する（リトライ＝同一HLC＝no-op）。クライアントはサーバーからのACK受領まで `outbox` の当該エントリを保持する。
3. **pull**: `GET /v1/tenants/{tenant_id}/pull?since={seq}&limit={n}` → `{records, next_since, has_more}`。クライアントはテナントごとの受信済みカーソル（`seq`）をローカルDBに保持し、マージのローカルコミット後にカーソルを前進させる。自デバイスが送信したレコードのエコーは除外しない（除外すると他端末による上書きを見逃すおそれがあるため。マージは冪等であるため実害はない）。
4. 受信したレコードをDEKで復号し、フィールド単位でマージする。マージの結果ローカル側の値が勝ったフィールドがあれば、マージ済みblobを**再push**する（再push規約）。これにより全端末が最終的に同一状態へ収束する。
5. マージ完了後、ローカルUIへ変更を反映する（リアクティブなクエリ購読を想定）。

初回フル同期は `since=0` からのページング + zstd圧縮で行う。転送エンコードは初期はJSON + base64とし、将来のCBOR化に備えプロトコルバージョンをHTTPヘッダに持たせる。準リアルタイム反映は初期は「フォアグラウンド復帰時 + 数分間隔のpull」とし、SSE/long-pollは将来拡張とする。

### 6.5 競合ケースと解決方針

| ケース | 内容 | 解決方針 |
|---|---|---|
| サブタスクの親子同時編集 | デバイスAが親タスクのstatusをdoneに、デバイスBが同じ親の子タスクを新規追加、というように異なるフィールド/異なる階層を同時編集 | 親タスクと子タスクは別レコードのため、レコード単位でのマージにより両方の変更が独立に反映される。親の完了と子の追加は互いを妨げず、UI側で「完了済み親タスクに未完了の子タスクが存在する」状態を許容し警告表示するのみとする |
| 同一タスクのstatus同時更新 | デバイスAがdoneに、デバイスBがin_progressに、ほぼ同時に更新 | フィールドレベルLWWにより、HLCがより新しい方の値を採用する。ユーザー体験上の取りこぼしを避けるため、直近で上書きされた側の変更を「変更履歴」としてローカルに一定期間保持し、必要であれば手動で復元できるようにする（将来拡張。初期リリースではLWWの結果をそのまま採用） |
| 削除と同時編集の競合 | デバイスAがタスクを削除し、デバイスBが同時に当該タスクの内容を編集 | 同期導入時に再設計する。従来案ではtombstoneのHLCと編集のHLCを比較し、削除が後勝ちであればレコードは削除状態を維持し、編集が後勝ちの場合はtombstoneを取り消して復活させる設計だった。ADR-009はローカル削除の意味論を物理削除・削除Undoなしへ改訂しているため、Phase 2同期設計ではこの競合方針を再検討する。（2026-07-07改訂: docs/05 ADR-009参照） |
| 同一レコードの異なるフィールドを同時編集 | デバイスAがtitleを、デバイスBがnoteを、ほぼ同時に編集 | サーバーは `sync_records` にレコード単位で最新blobのみを保持するため、一方のpushがもう一方の変更を含まないblobとして一時的に「最新」となりうる。ただし各blob内のフィールドHLCマップにより、pull側はフィールド単位で自分のローカル値と比較し、ローカルが勝つフィールドがあれば6.4節の再push規約に従いマージ済みblobを再pushするため、両端末は最終的に両フィールドの変更を含む状態へ収束する |

### 6.6 サーバーが強制する不変条件とエッジケース

サーバーはE2EEデータの中身を解釈しない一方で、以下の不変条件をリクエスト単位で強制する（ADR-005）。

- クライアントによる物理DELETE不可と `deleted` フラグ（tombstone）の扱いは、同期導入時に再設計する（ADR-009はローカル削除の意味論を定めるもの）。従来案ではサーバー側で物理DELETEを不可とし、削除を `deleted` フラグのみで表現していた。（2026-07-07改訂: docs/05 ADR-009参照）
- 暗号blobのサイズには上限を設ける（初期値64KB）。
- push/pull 1リクエストあたりのレコード件数（バッチ）にも上限を設ける。
- pushされたレコードのHLC物理時刻がサーバー時刻より5分を超えて未来である場合、当該pushは拒否する（端末の時計異常や「未来HLCによる永久勝ち」攻撃の抑止のため）。
- 課金の有効性・Orgメンバーシップの検証はAPIリクエストごとにサーバー側で実施する。

エッジケースへの対策は以下のとおりである。

- **Orgメンバーによる破壊的上書き**: 共有リストにおいて悪意ある（または誤操作の）メンバーが他者の変更を意図せず/意図的に上書きした場合に備え、上書き時の旧行を `sync_records_history` へ退避し30日保持する。これにより復旧手段を確保する。
- **tombstone GCとオフライン端末**: tombstoneの保持・GC・オフライン端末復帰時の扱いは、同期導入時に再設計する（ADR-009はローカル削除の意味論を定めるもの）。従来案ではtombstoneを180日でガベージコレクションし、この期間（GC窓）を超えてオフラインだった端末が復帰した場合、サーバーはdeviceの最終pull時刻からこれを検知し、410 Goneを返してフル再同期させる設計だった。（2026-07-07改訂: docs/05 ADR-009参照）

## 7. アカウントフロー

### 7.1 未登録利用開始

1. アプリ初回起動時、Device Key (DK) を生成しOSキーチェーンへ保存。
2. DK由来の鍵でローカルDB（SQLite + SQLCipher）を初期化・暗号化。
3. アカウント登録なしでリスト・タスクの作成/編集がフル機能で可能（同期・共有・課金プランに紐づく上位機能を除く）。

### 7.2 新規登録（既存ローカルデータの移行を含む）

1. メールアドレス・パスワードを入力し、クライアントでOPAQUEレジストレーションを実行（パスワードはサーバーへ非送信）。
2. exportKeyからHKDFでKEK_pwを導出。
3. Master Key (MK) を新規生成。
4. 既存ローカルデータ（DK配下で暗号化済み）をMK配下へ再暗号化して移行。
5. `wrap(MK, KEK_pw)` と `wrap(MK, DK)` を生成し、それぞれサーバー・ローカルに保存。
6. Recovery Keyを生成し、`wrap(MK, RecoveryKey)` をサーバーに保存。Recovery Key自体はユーザーに提示し、安全な場所への保管を促す（サーバー・端末には平文を残さない）。
7. User X25519鍵ペアを生成し、公開鍵をサーバーへ、秘密鍵を`wrap(SK, MK)`としてサーバーへ保存。
8. サーバー側でPostgres内にテナントレコードを作成する。サインアップフローにはリージョン選択のステップが含まれる（初期はデータレジデンシー未対応のため選択肢は1つのみ。[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-006参照）。

### 7.3 ログイン・新デバイス追加

1. 新デバイスでメールアドレス・パスワードを入力し、OPAQUEログインを実行。
2. exportKeyからKEK_pwを導出し、サーバーから取得した `wrap(MK, KEK_pw)` を復号してMKを得る。
3. MKで `wrap(SK, MK)` を復号し、User X25519秘密鍵を取得（Org共有リストの復号に利用）。
4. 新デバイスのDevice Keyを生成し、`wrap(MK, DK)` をローカルに保存（4.3節の定義どおり、DKによるMKラップはローカル保存とする）。サーバーのデバイスリストへは新規デバイスのメタデータ（デバイス名・登録日時等）のみを登録。
5. セッション認証（初期はDB照合のopaqueトークン）を確立した上で同期API（6章）を利用し、`since=0` からの初回フル同期（pull）を実行してローカルDBを構築。

### 7.4 パスワード変更

1. 現在のパスワードでOPAQUE認証し、MKを取得済みの状態にする。
2. 新パスワードでOPAQUEレジストレーションを再実行。
3. 新exportKeyからKEK_pw'を導出し、`wrap(MK, KEK_pw')` を再計算してサーバー側の値を更新。
4. データ本体の再暗号化は不要（4.5節参照）。

### 7.5 リカバリーキーでの復旧

1. パスワードを忘れた場合、ユーザーが保管しているRecovery Keyを入力。
2. サーバーから `wrap(MK, RecoveryKey)` を取得し、Recovery KeyでMKを復号。
3. MKが得られたら、新しいパスワードを設定しOPAQUE再登録を実行、`wrap(MK, KEK_pw_new)` を生成してサーバーへ保存。
4. 新しいRecovery Keyを再発行し、旧Recovery Keyを無効化（サーバー側の `wrap(MK, RecoveryKey)` を新しい値で上書き）。

### 7.6 デバイス失効

1. ユーザーが既存デバイス一覧から対象デバイスを選択し失効操作を実行。
2. サーバー側で該当デバイスの `devices.revoked_at` を設定する。同期API（push/pull）はリクエストごとに `devices.revoked_at` を検証するため、**失効は即時に実効化される**（旧設計のような発行済みトークンのTTL経過待ちは発生せず、失効反映の遅延はない）。
3. Organizationの共有リストに参加していた場合、当該デバイスに紐づくユーザーの鍵自体は失効しない（ユーザー単位の鍵のため）。デバイス単位の失効はあくまで同期アクセスの遮断であり、より強固に締め出す必要がある場合（デバイス盗難等）は、別途アカウントパスワード変更・Org DEKローテーションを併用することを推奨する運用とする。

## 8. MCP / CLI

### 8.1 共通方針

CLIおよびMCPサーバーは、いずれも `core` crateを通じてローカルの暗号化DBへ直接アクセスし、**サーバーAPIを経由しない**。これにより、オフライン環境でもCLI/MCP双方がフル機能で動作し、同期は裏側で `core::sync` が非同期に処理する。

### 8.2 MCPサーバー

- トランスポートは **stdio** を採用する（ローカルプロセス間通信であり、ネットワーク経由でのE2EEデータ露出を避けるため）。
- 提供するtoolsの例:

| tool名 | 概要 |
|---|---|
| `list_tasks` | 指定リスト・条件（status、期限等）でタスク一覧を取得 |
| `create_task` | 新規タスクを作成（親タスク指定によるサブタスク作成も可） |
| `complete_task` | タスクを完了状態に更新 |
| `search_tasks` | 全文検索（ローカルFTS5インデックス経由） |

- 初回のtool呼び出し時、ローカルDBへのアクセスに対するユーザーの明示的な許可プロンプトを表示し、以後のセッションでの利用可否を確認する。

### 8.3 CLI

Rustバイナリとして `clap` を用いてサブコマンド体系を構築する。

```
todori add "牛乳を買う" --due tomorrow
todori add "資料作成" --list work --priority 2 --parent <task-id>
todori list --list work --status todo
todori done <task-id>
todori search "牛乳"
```

CLI・MCPサーバーともに、初回起動時に対象アカウントのローカルDB（未登録時はDK保護のローカルのみのDB、登録済みならMK保護のDB）へ接続するための鍵取得フロー（OSキーチェーンからのDK取得、必要であればパスワード入力によるMK取得）を実行する。

## 9. ローカルAI

### 9.1 方針

推論は**オンデバイスのみ**で実行し、タスク内容がサーバーはもとより外部のAI APIへ送信されることも一切ない。これによりE2EEの前提（サーバーがコンテンツを知り得ない）をAI機能導入後も維持する。

### 9.2 実装

- 推論エンジンには llama.cpp系のRustバインディング、または `candle` を用いる。
- モデルはGGUF量子化形式、2〜4Bパラメータクラスの軽量モデルを対象とし、ユーザーが任意のタイミングでダウンロード・削除できるモデル管理UIを提供する（初期状態ではモデル未同梱とし、アプリサイズの肥大化を避ける）。

### 9.3 対応機能

| 機能 | 概要 |
|---|---|
| 自然言語パース | 「明日の15時に資料作成」のような自然文からタイトル・期限・リストを抽出しタスク化 |
| タスク分解 | 大きなタスクをサブタスクの候補に分解して提案 |
| 週次要約 | 完了/未完了タスクの傾向をローカルで要約しレポート表示 |

### 9.4 低スペック端末フォールバック

推論に必要な最低メモリ/CPU要件を満たさない端末では、AI機能そのものをUI上非表示にし、モデルダウンロードや推論呼び出しを行わないようにする。判定は端末のRAM容量・CPUコア数等の静的情報に基づき、アプリ起動時に一度だけ実施する。

## 10. 課金・プラン制御

本章は概要である。公開向けの課金方針は `billing_overview.md` を参照する。具体価格、収益試算、provider別の運用判断はprivate repo側で管理する。

### 10.1 プラン体系

Free / Pro / Org の3プランを設け、機能をプラン単位でゲートする。

- **Free**: ローカルで完結する全機能（リスト・タスク・タグ・リマインダー・テンプレート・スケジュール自動起票・タイマー・統計・ローカルAI・CLI/MCP を含む）。単一端末利用。同期・共有は不可。
- **Pro**: E2EE同期（マルチデバイス）、暗号化クラウドバックアップ。
- **Org**: Organization共有、担当者アサイン・コメント、メンバー管理、監査ログ。

機能ゲートはサーバー依存機能（同期・共有）に限定され、ローカル機能はプランによらず制限しない。

### 10.2 実装方針

- レシート検証（App Store / Play / 外部課金プロバイダ）はすべて**サーバー側**で実施し、検証結果に基づき機能フラグをアカウント（`users.plan`等）に紐づける。
- クライアントはサーバーから取得した機能フラグに従いUI上の機能出し分けを行う。フラグの正当性はサーバー起点であり、クライアント側での改ざんに対しては、機能ゲートが必要な操作（例: Org作成、同期API（push/pull）へのアクセス）についてはサーバー側APIでリクエストごとに再検証する（6.1節参照）。トークン発行時点の一括判定ではなく、リクエスト単位の検証であるため課金失効は同期アクセスへ即時に反映される。
- サブスクリプション管理サービスは実装時点の要件に応じて選定する。エンタイトルメント判定の唯一の真実はあくまで自社サーバー側とし、外部課金サービスはその更新ソースとして抽象化する。

### 10.3 課金失効時の挙動

課金が失効（サブスクリプション解約・支払い失敗等）した場合でも、**ローカルデータは全機能で編集を継続できる**ものとする。失効により制限されるのはサーバーとの同期（およびOrg共有等のサーバー依存機能）のみであり、ユーザーのデータそのものを人質に取らない方針を明確にする。同期停止中に行われたローカル変更は、再度課金が有効化された時点で通常のpush/pullフローにより同期される。

## 11. テスト・品質

### 11.1 Rustコア

- **暗号処理**: OPAQUE、HKDF、XChaCha20-Poly1305、X25519 sealed boxについて、既知のテストベクタ（RFC/参照実装由来）を用いた単体テストを整備する。
- **同期エンジン**: HLCマージ、fractional indexing、削除同期処理について **property-based testing**（`proptest` 等）を用い、任意の操作列に対する結果整合性（最終的にすべてのデバイスが同一状態に収束すること）を検証する。削除同期処理は同期導入時に再設計する（ADR-009はローカル削除の意味論を定めるもの）。（2026-07-07改訂: docs/05 ADR-009参照）

### 11.2 Flutter

- 主要画面のwidget testと、実DBを用いたシナリオベースのintegration test（タスク作成〜同期〜他デバイス反映等）を整備する。

### 11.3 脆弱性対応

- 依存クレートの脆弱性監査に `cargo audit` をCIパイプラインへ組み込み、定期的（週次等）に実行する。
- E2EE設計・実装については、正式リリース前に**外部の第三者機関によるセキュリティ監査**を実施することを推奨する。監査範囲には鍵階層、OPAQUE実装、Org共有のsealed box実装、サーバー保存データの解析可能性を含める。

以下は本仕様書執筆時点で未確定であり、今後の技術検証・意思決定を要する事項である。

- SQLCipher統合方式の検証（rusqliteのbundled-sqlcipher系featureの各プラットフォーム（特にiOS/Android）でのビルド・動作確認）。
- strict mode（wrap(MK, DK)をローカル保存せず起動時パスワード必須とするオプション、5.3節参照）の提供要否。
- ローカルAIのモデル最終選定（候補モデル、量子化レベル、対応言語、推論速度と精度のトレードオフ）。
- デバイス失効時のOrg DEKローテーションを自動化するか、手動運用に留めるかの方針。
- リマインダーのローカルスケジューリングにおける、OSごとのバックグラウンド実行制限（特にモバイルOS）への対応方針（アプリ非起動時の通知信頼性の担保策）。
- 変更履歴・誤削除以外の競合の可視化/手動マージUIの要否と実装範囲（6.5節の「将来拡張」の具体化）。
- 翻訳管理プラットフォーム（Crowdin / Weblate 等）の選定、および追加言語の優先順位（市場規模・ストア地域統計に基づく）。
- Lambda公開エンドポイントの前段構成（CloudFront + WAF / API Gateway スロットリング）の選定。
- 同期APIのクォータ・blobサイズ上限（6.6節、初期64KB）・バッチ上限の最終値。
- `sync_records_history` 保持期間（初期30日）と削除同期データの保持/GC期間（従来案ではtombstone GC初期180日、6.6節）の最終値。削除同期設計は同期導入時に再設計する（ADR-009はローカル削除の意味論を定めるもの）。（2026-07-07改訂: docs/05 ADR-009参照）
- 転送エンコードのCBOR化（6.4節）の実施時期。
- リージョン拡張（データレジデンシー対応、[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-006参照）の実施条件。
- テナント間コピー操作の詳細仕様（サブタスク・コメント・リマインダーを含めるか、コピー時の帰属の扱い。[`05_設計判断記録.md`](./05_設計判断記録.md) ADR-007参照）。
- Neonの最新料金プラン・接続プーラーのprepared statement対応（sqlxが使用）・リージョン提供状況の確認（実装開始時に実施。ADR-008の前提検証）。

なお、CRDTライブラリの選定（自前実装 vs 既存CRDT拡張の利用）はADR-004にて自前実装（フィールドレベルLWW + HLC + fractional indexing）で決定済みである。同期トークンのTTLの最終値は、同期プロトコルがトークンTTLに依存しない設計（ADR-002・ADR-005）となったため概念自体が消滅した。コントロールプレーンの永続メタデータ保存先はADR-003にてPostgres一本化で決定済みである。サーバーレスPostgresプロバイダと接続方式はADR-008にてNeon（eu-central-1、プーラー経由TCP+sqlx）で決定済みである。
