# Todori 開発プレイブック

task文書を全作業の入口にはしない。候補を小さく保持し、実装着手が決まった標準・重要変更だけをtaskへ昇格する。

## 原則

- 状態の正本を分ける: 現在は `STATUS.md`、未着手候補は `BACKLOG.md`、実装契約と結果はtask、長期計画はPhase計画書、設計判断はADR、詳細履歴はgit / CI。
- 完了項目をBACKLOGへ残さない。task READMEへ手動一覧を作らない。
- WIPは実装中1件、Next最大3件を基本とする。
- 実装者の完了報告を合否判定として扱わない。標準・重要変更は別セッション、別エージェント、または人間が検証する。
- 観測手段のない品質目標へ着手しない。UIはスクリーンショット、性能は数値、挙動はテストまたはログを用意する。
- public/private境界を守り、private詳細をpublic taskや完了報告へ貼らない。
- ビジュアル作業は [`DESIGN_PLAYBOOK.md`](./DESIGN_PLAYBOOK.md) を併用する。ただし相談中の微調整を逐一task化しない。

## レーン選択

### 軽量レーン

task文書なしで、調査・修正・検証・コミットまで行う。

適用例:

- 誤字、リンク、文言、設定値の局所修正
- 明確な再現条件がある小さなバグ
- 既存仕様内の1〜3ファイル程度の変更
- 読み取り専用の調査
- 人間と相談しながら行う局所的なデザイン調整

次のいずれかに該当したら標準レーンへ昇格する。

- 変更が複数層へ広がった
- 受け入れ基準を書かないと完了条件が曖昧
- 同じ箇所で2回手戻りした
- schema / FRB / 暗号 / 鍵 / 新規依存 / public-private境界へ触れる

### 標準レーン

新機能、複数層変更、半日級以上、明示的な受け入れ基準が必要な変更に使う。

1. `STATUS.md` のNextから着手候補を1件選ぶ。
2. task番号を採番し、1〜8章の指示書を作る。
3. 指示書をレビューしてから実装する。
4. 実装フェーズで実装結果、品質ゲート、未解決事項を `## 9. 完了報告` へ記録する。合否は判定しない。
5. レビュー・テストフェーズで差分と必要な品質ゲートを独立に確認し、結果と合否を追記する。
6. 合格後、候補をBACKLOGから削除し、task冒頭を完了状態にする。
7. `STATUS.md` の現在・Nextを更新する。

### 重要変更レーン

標準レーンに加え、人間承認とADRまたは仕様更新を必要とする。

対象:

- 暗号、鍵、認証、同期プロトコル
- DB schema、migration、データ削除・復旧
- 新規依存、配布署名、権限・entitlement
- public/private境界、課金、法務、セキュリティ方針
- 複数frontendへ影響するアーキテクチャ

実装taskと設計判断を分ける。人間が設計を採用した時点でADRをAcceptedにできるが、実装完了とは扱わない。

## フェーズ

1. **調査・計画**: 必要な調査を並列化し、スコープ、依存関係、受け入れ基準を1つの計画へ統合する。
2. **実装・統合**: 分割可能な範囲を並列実装し、統合後に実装結果と検証事実を記録する。合否は判定しない。
3. **レビュー・テスト**: 統合後の差分を、実装を担当していないエージェントが独立に検証し、合否を判定する。
4. **修正・再検証**: 指摘を修正し、統合後の差分を再度独立に検証する。

各フェーズのエージェント数は固定しない。同一ファイルや同じ不変条件を並列編集せず、担当範囲と依存順序を先に分ける。サブエージェントを使わない場合も、実装と合否判定は論理的に分ける。

## task指示書の最小要件

- 背景と出典
- ゴール
- 変更可能範囲とやらないこと
- 実装手順
- task固有の受け入れ基準（目安10項目以内）
- 観測可能な証拠
- 未解決事項の扱い

共通規約と品質ゲートをtaskへ複製せず、[`README.md`](./README.md) を参照する。

## 検証

- 完了報告の「成功」を信用せず、リスクに応じた品質ゲートを再実行する。
- コード変更がない文書作業では `git diff --check`、リンク、正本間の矛盾を確認する。
- UI変更ではwidget testだけでなくvisual QAまたは実機確認を行う。
- 環境制約で実行不能な場合、コード失敗と区別し、コマンド・停止地点・代替証拠を記録する。
- レビュー・テストフェーズは品質評価語ではなく、合格 / 不合格と再現可能な根拠を返す。

## BACKLOG補充

毎週ではなく、Nextが1件以下になった時またはマイルストーン境界で行う。

1. Phase計画書の未達項目を確認する。
2. 完了taskの未解決事項を検索する。
3. ADRの帰結とドッグフーディング結果を確認する。
4. 出典付き候補だけをBACKLOGへ追加する。
5. Laterが20件を超える場合は、優先順位を付けずIceboxへ移すか削除する。

候補追加時にtask文書を作らない。仕様変更を伴う候補は、人間判断まではIceboxまたは人間作業欄へ置く。

## ドッグフーディング

マイルストーン境界または2週間ごとに5〜10分実機を操作する。

- 感想はまず短い観察メモとして残す。
- 再現可能な問題だけをBACKLOG候補へ変換する。
- 複数の小さな視覚調整は1つのテーマへまとめる。
- 相談中の方向探索をtask化しない。

## 終了チェック

- 意図しない未コミット差分がない。
- taskレーンでは冒頭ステータスと完了報告が更新されている。
- 完了候補がBACKLOGへ残っていない。
- `STATUS.md` の現在とNextが実態に合っている。
- 実行不能な検証がコード失敗と区別されている。
- public repoへprivate情報が混入していない。

## アンチパターン

- 候補を思いついた時点でtask番号と長い指示書を作る。
- 完了履歴をREADME、BACKLOG、STATUS、taskへ重複転記する。
- BACKLOGへ完了項目を残して進捗表として使う。
- すべての軽微な修正へ4フェーズを強制する。
- 重要変更を軽量レーンとして処理する。
- 完了報告へCIログ全文や品質評価語を貼る。
