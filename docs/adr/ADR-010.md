# ADR-010: 削除同期は空blob tombstoneとして表現する

**日付**: 2026-07-08
**状態**: Accepted（2026-07-08 プロダクトオーナー承認）

**補正**: 2026-07-10のADR-012により、既知recordのcascade tombstone、pull前preflight、stable-key fuzzy scan + delta full resyncを必須化した。ADR-014によりwire/storage表現は`deleted=true`+空byte列から、`delete_hlc`を持ちblobを持たないtagged tombstoneへ置換した。2026-07-11のADR-016により、aggregate scope / epochは導入せず、180日のtombstone保持とserver-enforced device continuity、expired-device rebase、復号後のclient-side descendant cascadeで削除へ収束する方針に確定した。

### 背景

ADR-009はローカル削除の意味論を「恒久削除・削除Undoなし」へ改訂した。一方、Phase 2の同期プロトコルでは、オフライン端末へ削除を伝播するために一定期間のサーバー側tombstoneが必要である。ここで削除済みレコードの暗号blobを保持し続けると、ユーザーに対する「削除=本当に消える」という説明と、サーバーに削除済みコンテンツ暗号文が残る実態の間にズレが残る。

E2EEではサーバーは暗号blobを復号できないが、プライバシープロダクトとしては「サーバーに内容を読めない形で残す」ことと「内容を即時消滅させ、同期に必要な最小メタデータだけ残す」ことを区別して設計する。

### 決定

1. 削除は同期ストア上で `deleted=true` のtombstoneとして表現する。
2. tombstone行の `encrypted_blob` は空byte列とする。クライアントが削除pushで非空blobを送っても、サーバーは保存時に空byte列へ正規化する。
3. サーバーが削除後に保持する同期情報は、`tenant_id`、`record_id`、`collection`、`seq`、`hlc`、`deleted=true`、`updated_at` に限定する。削除済みコンテンツ本文やフィールドHLC入り暗号blobは保持しない。
4. tombstoneは180日保持し、定期ジョブでGCする。180日という窓はADR-005の従来案を踏襲し、通常の長期オフライン端末復帰を吸収するための保守的な初期値とする。
5. tombstone GC実行時、テナントごとに `gc_horizon_seq` を記録する。これは「この値以下のseqのtombstoneは削除済みであり、サーバーは削除伝播の完全性を保証しない」境界を表す。
6. continuity preflight時、serverが保持するdevice別continuity cursor / generationが `gc_horizon_seq` 未満ならexpiredとし、フル再同期を要求する。client申告の `since` だけでpush可否を決めない。新規profileの `since=0` はexpired対象外だが、全件取得はADR-012のfuzzy scan + delta resyncを用いる（2026-07-11改訂: ADR-016）。
7. GC窓超過の検知は `devices.last_pull_at` ではなく、serverが自身の発行したpull / full-resync closureのACKから単調更新するdevice continuityと `gc_horizon_seq` の比較で行う。`devices.last_pull_at` は長期未使用デバイスの失効候補検出などのテレメトリ用途に限定する（2026-07-11改訂: ADR-016）。
8. expired-deviceフル再同期では未ACK outboxを一律保護しない。server currentにあるlocal mutationはtyped merge / rebaseし、server ACK済みなのにcurrentに不在のrecordはGC済み削除として破棄する。never-synced creation originを持つrecordは、pending key bundleとdependency closureを検証できた場合だけ新規seedとして保護する（2026-07-11改訂: ADR-016）。
9. liveからtombstoneへのCAS過程ではADR-012のsemantic clockを用いる。serverがtombstoneを受理した後はterminal deletionとし、同一record IDの`tombstone → live`をHLCの大小にかかわらず拒否する（2026-07-11改訂: ADR-016）。
10. 削除Undo導線は作らない。ユーザーが同等の内容を再作成する場合は新しいrecord IDを発行する（2026-07-11改訂: ADR-016）。

用語補足: 「outboxに未ACKエントリを持つ」とは、具体的にはクライアントの `sync_outbox` テーブルに当該record_idの行が残存していることを指す（ACKはフィールドではなくpush受理時の行削除操作である）。

補足（2026-07-11）: 2026-07-08の「GC済み削除recordの未push編集を復活させる」トレードオフはADR-016で撤回した。現行規約ではserver-seen but absent recordをrebaseで破棄し、never-synced local-new recordだけを別条件で保護する。

補足（2026-07-08 オーナーレビュー第2弾由来）: `devices.last_pull_at` はリクエスト受信時に更新され得る一方、クライアントのpullカーソルは「取得データをローカルへコミットした後」に前進する。そのため、pull直後かつローカル反映前にクラッシュした端末では、`last_pull_at` が新しく見えても実際の継続性は古いカーソルに留まる。`last_pull_at` 基準の検知はこのケースを偽陰性にし、GC済みtombstoneの取り逃がしを見落とし得る。カーソル自体を `gc_horizon_seq` と比較すれば、端末簿記や時計に依存せず、サーバーが保証できるseq範囲との関係だけで正確に判定できる。

この決定は、`docs/03_技術仕様書.md` §6.6 に残っていた「deviceの最終pull時刻から検知」という従来案との差分である。ADR-010承認に伴い`gc_horizon_seq`基準へ改訂し、ADR-016でclient申告cursorではなくserver-trusted device continuityとの比較へ強化した。

### 実装範囲

task-73時点で実装するのは、削除pushのtombstone blob空化と、180日cutoffを受け取るGC関数のみとする。`deleted=true` の転送と受信側ローカル恒久削除はtask-72で実装済みである。`gc_horizon_seq`の永続化とfuzzy full resyncはtask-95で実装し、server-trusted continuity、terminal tombstone、expired-device rebase、late descendant cascadeはADR-016の後続実装taskへ送る（2026-07-11改訂）。

将来のフル再同期実装タスクでは、受け入れ基準として少なくとも以下の自動テストを必須とする。

- server-trusted device continuityが`gc_horizon_seq`未満の端末は、full resync closureとlocal rebaseを完了するまでkey bundle / entity pushを拒否されること。
- server-seen but absent recordの未push編集は破棄され、tombstone GC後も復活しないこと。
- never-synced origin、pending List DEK bundle、dependency closureを持つ長期offline中の新規list / taskは、full resync後も保護されること。
- server不在の既存list / parentへ依存するrecordはmaterialize / pushされないこと。
- pull直後・local反映前クラッシュではserver側continuityが進まず、次回runが削除reconcileを再実行すること。

### 帰結

- 削除後、サーバーからコンテンツ暗号blobは即時に消える。E2EEの「サーバーは読めない」に加えて、削除操作への説明も「同期に必要な最小メタデータだけが最大180日残る」と誠実に表現できる。
- 180日以内に復帰した端末はtombstoneをpullしてローカル恒久削除できる。
- 180日を超えて復帰した端末、または保持期限内でもserver-trusted device continuityが `gc_horizon_seq` 未満の端末は、tombstoneを取り逃がす可能性があるためフル再同期とlocal rebaseが必須になる（2026-07-11改訂: ADR-016）。
- GC窓超過判定は端末の最終pull時刻やclient申告cursorではなく、serverがclosure ACKから更新したdevice continuityで決める。これにより、pull後クラッシュや端末時計のずれによる誤判定を避けられる。
- tombstoneにはfield HLC mapが残らないため、liveからtombstoneへの競合はfield単位ではなくtop-level `delete_hlc` と `mutation_hlc` で扱う。tombstone受理後はterminalとし、後着編集で同一record IDを復活させない（2026-07-11改訂: ADR-016）。

### 却下した代替案

- **削除tombstoneにも暗号blobを残す案**: オフライン端末とのマージ材料は増えるが、削除済み内容の暗号文がサーバーに残り続けるため、ADR-009の「削除=本当に消える」という設計意図と相性が悪い。
- **削除時にサーバー行を即時物理DELETEする案**: ほかの端末が削除を知る手段を失い、オフライン端末復帰時に削除済みレコードが再pushされて復活しやすい。
- **tombstoneを無期限保持する案**: フル再同期の必要性は減るが、削除メタデータが永続化し、同期ストアが単調増加する。
- **tombstone後も新しいHLCのliveを受理する案**: 別端末の編集を保全できる一方、ユーザーが不可逆と確認した永久削除を復活させる。archive-firstで整理し、永久削除は例外的なterminal操作とするADR-016と相反するため不採用（2026-07-11改訂）。
